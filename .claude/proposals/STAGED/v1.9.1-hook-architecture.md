## Scope

Implements the hook system whose types were stubbed in v1.8.9
(`entropic_hook_point_t`, `entropic_hook_callback_t`, `entropic_register_hook()`).
After v1.9.1, consumers can register pre/post callbacks at every engine stage
transition. Hooks are the primary extensibility mechanism for the engine --
consumers intercept, modify, inspect, and transform engine behavior without
forking or subclassing.

v1.8.9 defined:
- `entropic_hook_point_t` enum (16 values)
- `entropic_hook_callback_t` function pointer type
- `entropic_register_hook()` C API (returned `ENTROPIC_ERROR_NOT_IMPLEMENTED`)

v1.9.1 replaces the stub with a full implementation:
- `HookRegistry` -- thread-safe registration and dispatch
- Priority-ordered execution of multiple hooks per point
- Pre-hooks can modify context or cancel the operation
- Post-hooks can inspect and transform results
- Failing hooks are logged and skipped -- never crash the engine
- Hook points expanded from 16 to ~22 to cover all roadmap categories

**What this version does NOT include:**
- Compaction hooks with custom compactor registration (v1.9.9)
- Fan-out hooks (`on_fan_out`, `on_fan_in`) (3.0.0)
- Voice/STT/TTS hooks (3.0.0 -- hook points exist, consumers provide models)

v1.8.9 defined 16 hook points. v1.9.1 expands to 22 to cover all categories

    ENTROPIC_HOOK_ON_MODEL_UNLOAD,       /* 14 */

**Categories and semantics:**

| Category | Hook Points | Pre can modify/cancel? | Post can transform? |
|---|---|---|---|
| Model lifecycle | `ON_MODEL_LOAD`, `ON_MODEL_UNLOAD`, `ON_ADAPTER_SWAP`, `ON_VRAM_PRESSURE` | Yes (cancel load/swap) | No (informational) |
| Generation | `PRE_GENERATE`, `ON_STREAM_TOKEN`, `POST_GENERATE` | Yes (modify prompt, cancel) | Yes (transform result) |
| Context | `ON_CONTEXT_ASSEMBLE`, `ON_PRE_COMPACT`, `ON_POST_COMPACT` | Yes (modify assembled context) | Yes (transform compacted result) |
| Tool | `PRE_TOOL_CALL`, `POST_TOOL_CALL`, `ON_PERMISSION_CHECK` | Yes (modify args, deny) | Yes (transform result) |
| Directives | `ON_DIRECTIVE`, `ON_CUSTOM_DIRECTIVE` | Yes (modify, suppress) | No |
| Agentic loop | `ON_LOOP_START`, `ON_LOOP_ITERATION`, `ON_LOOP_END`, `ON_STATE_CHANGE` | No (informational) | No |
| Delegation | `ON_DELEGATE`, `ON_DELEGATE_COMPLETE` | Yes (cancel delegation) | Yes (transform result) |
| Error | `ON_ERROR` | No (informational) | No |

## C API Changes

### Revised Registration Signature

v1.8.9 stub had 4 parameters. v1.9.1 adds `priority`:

```c
/**
 * @brief Register a callback for an engine hook point.
 *
 * Multiple callbacks can be registered for the same hook point.
 * They execute in ascending priority order (0 = highest priority).
 * A pre-hook returning non-zero cancels the operation.
 * A pre-hook returning modified JSON (via context_json out-param)
 * modifies the operation's input.
 *
 * @param handle     Engine handle.
 * @param hook_point The hook point to register for.
 * @param callback   Function pointer invoked when the hook fires.
 * @param user_data  Opaque pointer passed to callback.
 * @param priority   Execution priority (0 = first, higher = later).
 * @return ENTROPIC_OK on success.
 *         - ENTROPIC_ERROR_INVALID_HANDLE
 *         - ENTROPIC_ERROR_INVALID_CONFIG (invalid hook_point value)
 *
 * @threadsafety Serialized per-handle.
 * @version 1.9.1
 */
ENTROPIC_EXPORT entropic_error_t entropic_register_hook(
    entropic_handle_t handle,
    entropic_hook_point_t hook_point,
    entropic_hook_callback_t callback,
    void* user_data,
    int priority);
```

### Revised Callback Signature

v1.8.9 defined the callback as returning `int` (0 = continue, non-zero = cancel).
v1.9.1 extends this: the callback can optionally write a modified JSON string
via an output parameter. The return value semantics remain the same.

```c
/**
 * @brief Hook callback function type.
 *
 * Pre-hook semantics:
 *   - Return 0 + *modified_json == NULL:  proceed unchanged
 *   - Return 0 + *modified_json != NULL:  proceed with modified context
 *   - Return non-zero:                    cancel the operation
 *
 * Post-hook semantics:
 *   - Return value ignored (post-hooks cannot cancel)
 *   - *modified_json != NULL:  transform the result
 *   - *modified_json == NULL:  no transformation
 *
 * The engine frees *modified_json after consuming it. The callback
 * MUST allocate it with entropic_alloc() (not malloc, not new, not stack).
 *
 * @param hook_point     Which hook point fired.
 * @param context_json   JSON string with hook-specific context data.
 *                       Owned by the engine, valid only for this call.
 * @param modified_json  Output: replacement JSON, or NULL for no change.
 *                       Allocated by callback with entropic_alloc(), freed by engine.
 * @param user_data      Opaque pointer from registration.
 * @return 0 to continue, non-zero to cancel (pre-hooks only).
 *
 * @version 1.9.1
 */
typedef int (*entropic_hook_callback_t)(
    entropic_hook_point_t hook_point,
    const char* context_json,
    char** modified_json,
    void* user_data);
```

**ABI note:** This changes the callback signature from v1.8.9. Acceptable because
v1.8.9 explicitly documented the hook system as `NOT_IMPLEMENTED` and the callback
type was defined for forward compatibility only. No consumer could have a working
hook callback registered against v1.8.9.

### Deregistration

```c
/**
 * @brief Deregister a previously registered hook callback.
 *
 * Matches on (hook_point, callback, user_data) triple. If no match
 * is found, returns ENTROPIC_OK (idempotent).
 *
 * @param handle     Engine handle.
 * @param hook_point The hook point to deregister from.
 * @param callback   The callback function pointer to remove.
 * @param user_data  The user_data that was passed during registration.
 * @return ENTROPIC_OK on success.
 *         - ENTROPIC_ERROR_INVALID_HANDLE
 *
 * @threadsafety Serialized per-handle.
 * @version 1.9.1
 */
ENTROPIC_EXPORT entropic_error_t entropic_deregister_hook(
    entropic_handle_t handle,
    entropic_hook_point_t hook_point,
    entropic_hook_callback_t callback,
    void* user_data);
```

## Hook Context JSON Schemas

Each hook point passes a specific JSON schema as `context_json`. Documented
here as the contract consumers code against.

### Model Lifecycle

**ON_MODEL_LOAD** (pre: can cancel)
```json
{"model_path": "/path/to/model.gguf", "config": {...}}
```

**ON_MODEL_UNLOAD** (pre: can cancel)
```json
{"model_path": "/path/to/model.gguf", "state": "WARM"}
```

**ON_ADAPTER_SWAP** (pre: can cancel)
```json
{"current_adapter": "base", "target_adapter": "lora-v2", "adapter_path": "/path/to/lora"}
```

**ON_VRAM_PRESSURE** (informational)
```json
{"vram_used_mb": 14200, "vram_total_mb": 16384, "pressure_level": "high"}
```

### Generation

**PRE_GENERATE** (pre: can modify messages, cancel)
```json
{"messages": [...], "params": {...}, "tier": "eng", "iteration": 3}
```
Modified JSON replaces `messages` and/or `params`.

**ON_STREAM_TOKEN** (informational, cannot cancel mid-stream)
```json
{"token": "Hello", "token_index": 42, "partial_content": "Hello"}
```
**Performance note:** This hook fires per token (~36 tok/s currently). JSON
serialization overhead is negligible at this rate but matters at higher
throughput. Stream token hooks MUST be lightweight — no file I/O, no network
calls, no allocations beyond the context JSON. If a consumer needs heavy
processing per token, buffer and batch-process in a post-generate hook instead.

**POST_GENERATE** (post: can transform result)
```json
{"content": "...", "tool_calls": [...], "finish_reason": "stop", "tokens_used": 150}
```
Modified JSON replaces `content` and/or `tool_calls`.

### Context

**ON_CONTEXT_ASSEMBLE** (pre: can modify assembled context)
```json
{"messages": [...], "system_prompt": "...", "anchors": [...], "token_count": 4200}
```

**ON_PRE_COMPACT** (pre: can modify, cancel)
```json
{"messages": [...], "token_count": 12000, "threshold": 10000, "force": false}
```

**ON_POST_COMPACT** (post: can transform)
```json
{"original_count": 45, "compacted_count": 12, "tokens_before": 12000, "tokens_after": 3500}
```

### Tool

**PRE_TOOL_CALL** (pre: can modify args, cancel)
```json
{"tool_name": "filesystem.read_file", "args": {"path": "/src/main.cpp"}, "tier": "eng"}
```
Modified JSON replaces `args`. Return non-zero to deny the call.

**POST_TOOL_CALL** (post: can transform result)
```json
{"tool_name": "filesystem.read_file", "args": {...}, "result": "...", "elapsed_ms": 12.5}
```

**ON_PERMISSION_CHECK** (pre: can override)
```json
{"tool_name": "bash.execute", "tier": "eng", "permission": "denied", "reason": "not in allow list"}
```
Return 0 + modified JSON with `"permission": "allowed"` to override denial.
Return non-zero to deny even if normally allowed.

### Directives

**ON_DIRECTIVE** (pre: can modify, suppress)
```json
{"type": "delegate", "params": {"target": "eng", "task": "implement feature"}}
```
Return non-zero to suppress the directive.

**ON_CUSTOM_DIRECTIVE** (pre: can modify, suppress)
```json
{"type": "custom_xyz", "params": {...}}
```
Fires for unrecognized directive types. Consumer can handle custom directives
via this hook.

### Agentic Loop

**ON_LOOP_START** (informational)
```json
{"config": {...}, "message_count": 3, "delegation_depth": 0}
```

**ON_LOOP_ITERATION** (informational)
```json
{"iteration": 5, "state": "EXECUTING", "consecutive_errors": 0, "tier": "eng"}
```

**ON_LOOP_END** (informational)
```json
{"final_state": "COMPLETE", "iterations": 7, "duration_ms": 4500, "tokens_used": 2300}
```

**ON_STATE_CHANGE** (informational)
```json
{"previous": "EXECUTING", "current": "WAITING_TOOL"}
```

### Delegation

**ON_DELEGATE** (pre: can cancel)
```json
{"target_tier": "eng", "task": "implement feature", "max_turns": -1, "depth": 1}
```

**ON_DELEGATE_COMPLETE** (post: can transform)
```json
{"target_tier": "eng", "task": "...", "summary": "...", "success": true, "duration_ms": 3200}
```

### Error

**ON_ERROR** (informational)
```json
{"error_code": "GENERATE_FAILED", "message": "...", "iteration": 3, "consecutive": 2}
```

## HookRegistry (C++ Internal)

Lives in `librentropic-core.so`. Not exposed across `.so` boundaries --
the C API in `entropic.h` is the only external interface. The facade
(`librentropic.so`) delegates `entropic_register_hook()` to the
HookRegistry owned by the engine handle.

### Class Design

```
C API (entropic.h)              Pure C boundary
  └── HookRegistry              80% logic: registration, ordering, dispatch, error handling
       (no subclasses needed -- this is a terminal class)
```

HookRegistry is not a three-layer hierarchy because there are no variants.
It is a single concrete class with all logic. This is consistent with the
architecture rule: three-layer is for subsystems with multiple implementations
(inference backends, MCP servers, storage backends). Hook dispatch has one
implementation.

### Data Structures

```cpp
/// @brief A single registered hook entry.
/// @version 1.9.1
struct HookEntry {
    entropic_hook_callback_t callback;  ///< Function pointer
    void* user_data;                    ///< Opaque consumer data
    int priority;                       ///< Execution order (ascending)
};

/// @brief Thread-safe hook registration and dispatch.
/// @version 1.9.1
///
/// Owns a vector of HookEntry per hook point. Registration and
/// deregistration take a write lock. Dispatch takes a read lock
/// and a snapshot of the entry list (so hooks can register/deregister
/// other hooks from within a callback without deadlock).
class HookRegistry {
public:
    entropic_error_t register_hook(
        entropic_hook_point_t point,
        entropic_hook_callback_t callback,
        void* user_data,
        int priority);

    entropic_error_t deregister_hook(
        entropic_hook_point_t point,
        entropic_hook_callback_t callback,
        void* user_data);

    /// @brief Fire pre-hooks. Returns modified context or cancellation.
    /// @param point The hook point.
    /// @param context_json Input context JSON.
    /// @param out_json Output: modified JSON (caller frees), or NULL.
    /// @return 0 = proceed, non-zero = cancelled.
    int fire_pre(
        entropic_hook_point_t point,
        const char* context_json,
        char** out_json);

    /// @brief Fire post-hooks. Returns transformed result.
    /// @param point The hook point.
    /// @param context_json Input context/result JSON.
    /// @param out_json Output: transformed JSON (caller frees), or NULL.
    void fire_post(
        entropic_hook_point_t point,
        const char* context_json,
        char** out_json);

    /// @brief Fire informational hooks (no modify, no cancel).
    /// @param point The hook point.
    /// @param context_json Context JSON.
    void fire_info(
        entropic_hook_point_t point,
        const char* context_json);

private:
    /// Per-hook-point entry list, sorted by priority.
    std::array<std::vector<HookEntry>, ENTROPIC_HOOK_COUNT_> entries_;

    /// Read-write lock per hook point for concurrent dispatch + safe registration.
    std::array<std::shared_mutex, ENTROPIC_HOOK_COUNT_> mutexes_;
};
```

### Dispatch Algorithm

```
fire_pre(point, context_json, out_json):
    1. Read-lock mutexes_[point]
    2. Copy entries_[point] to local snapshot
    3. Unlock
    4. accumulated_json = NULL
    5. For each entry in snapshot (sorted by priority ascending):
       a. current_input = accumulated_json ? accumulated_json : context_json
       b. modified = NULL
       c. TRY:
            rc = entry.callback(point, current_input, &modified, entry.user_data)
       d. CATCH (any):
            Log WARNING: "Hook callback failed for {point}, skipping"
            continue
       e. IF rc != 0:
            free(accumulated_json)
            free(modified)
            return rc  // cancelled
       f. IF modified != NULL:
            free(accumulated_json)
            accumulated_json = modified
    6. *out_json = accumulated_json  // NULL if no modifications
    7. return 0  // proceed

fire_post(point, context_json, out_json):
    Same as fire_pre but:
    - Return value from callback is ignored (post cannot cancel)
    - Only modified_json output is chained through

fire_info(point, context_json):
    Same dispatch loop but:
    - modified_json parameter is always NULL (ignored)
    - Return value is ignored
    - Pure notification, no accumulation
```

### Failure Handling

Failing hooks are logged at WARNING level and skipped. The engine proceeds
with the next hook in priority order. Specifically:

1. If a callback throws a C++ exception: caught at the dispatch boundary
   (HookRegistry is inside `librentropic-core.so`), logged, skipped.
2. If a callback segfaults: not recoverable. The engine documents that
   hook callbacks must not crash. This is the consumer's responsibility.
3. If a callback hangs: no timeout mechanism in v1.9.1. Consumers are
   responsible for ensuring callbacks return promptly. A hook timeout
   feature may be added in a future version if needed.

## HookRegistry Ownership

The HookRegistry is owned by the engine handle (`entropic_handle_t`).
Each engine instance has its own independent registry. Hook registrations
are not shared between engine instances.

The facade passes a `HookRegistry*` into core subsystems at construction time
via a new field on the interface contracts:

```cpp
/// @brief Hook dispatch interface injected into subsystems.
/// @version 1.9.1
///
/// Subsystems call through this pointer to fire hooks. The pointer
/// is owned by the engine handle and outlives all subsystems.
struct HookInterface {
    int (*fire_pre)(void* registry, entropic_hook_point_t point,
                    const char* context_json, char** out_json);
    void (*fire_post)(void* registry, entropic_hook_point_t point,
                      const char* context_json, char** out_json);
    void (*fire_info)(void* registry, entropic_hook_point_t point,
                      const char* context_json);
    void* registry;  ///< Opaque pointer to HookRegistry
};
```

This follows the same pattern as `InferenceInterface` from v1.8.4 --
function pointers + opaque data, no compile-time dependency between
`.so` libraries.

## Validation Criteria

### V1: Registration and dispatch work end-to-end

Register a hook for `PRE_GENERATE`, fire it, verify the callback is invoked
with correct `hook_point` and `context_json`. Verify the return from
`entropic_register_hook()` is `ENTROPIC_OK`.

### V2: Priority ordering

Register 3 hooks at priorities 10, 0, 5 for the same point. Verify they
fire in order 0, 5, 10. Each hook appends its priority to a shared log
vector; final order matches ascending priority.

### V3: Pre-hook cancellation

Register a pre-hook for `PRE_TOOL_CALL` that returns non-zero. Verify the
tool call does NOT execute. Verify subsequent hooks for the same point
do NOT fire after cancellation.

### V4: Pre-hook modification

## Implementation Notes (from adversarial review)

1. **ON_PERMISSION_CHECK is informational only (`fire_info`).** No override
   capability. Permission decisions flow through PermissionManager only. Hooks
   observe for logging/auditing but cannot modify or cancel permission checks.
   This prevents a single buggy hook from defeating all three permission layers.

2. **PRE_TOOL_CALL argument modification ordering.** The execution path is:
   permission check → hook fires (can modify args) → if args modified, re-run
   permission check on modified args → execute. Skip re-check when args
   unchanged (common case, zero overhead). This ensures modified arguments
   are still authorized.
