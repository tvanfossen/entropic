## Scope

v1.9.0 is the crossing point: the C engine becomes PRIMARY and the Python engine
becomes legacy code. Three things ship:

1. **Auto-generated Python wrapper** — a ctypes binding generated from `entropic.h`
   by a CMake build step. No manual wrapper maintenance. The wrapper exposes the
   same public API surface as the Python engine (`LibraryConfig`, `AgentEngine`,
   `GenerationResult`, etc.) so existing consumers (CLI, benchmarks, model tests,
   entropic-tui) can switch backends by changing an import path.

2. **CLI + benchmark ported** — all six CLI commands (`ask`, `status`, `init`,
   `download`, `benchmark`, `mcp-bridge`) call through the C engine via the
   generated wrapper. The benchmark runner measures tok/s from the C engine.

3. **Python engine marked deprecated** — `src/entropic/` gets deprecation warnings
   on import. Code is NOT deleted (that's v1.9.15). Both engines coexist for one
   migration window.

**What this version does NOT include:**
- Deleting `src/entropic/` (v1.9.15)
- Hook implementations (v1.9.1)
- LoRA adapter support (v1.9.2)
- Any new C engine features — this is a wrapper + migration version

**Why ctypes over cffi:**
- ctypes is in the Python standard library — zero build-time dependency
- No compilation step needed for the wrapper itself
- The C API surface is stable after v1.8.9 — ctypes maps cleanly to it
- cffi's advantage (automatic struct parsing) is unnecessary because the
  generation script already parses `entropic.h`

## Architecture

### Wrapper Generation Pipeline

```
entropic.h (C API)
    │
    ▼
scripts/generate_wrapper.py        Parses entropic.h, emits ctypes bindings
    │
    ▼
python/entropic/__init__.py        Generated ctypes wrapper module
    │
    ├── class EntropicEngine       Maps to entropic_handle_t lifecycle
    ├── class InferenceBackend     Maps to entropic_inference_backend_t
    ├── class MCPServer            Maps to entropic_mcp_server_t
    ├── error handling             Maps entropic_error_t → Python exceptions
    └── type wrappers              LibraryConfig, AgentEngine, GenerationResult, etc.
```

The generator runs as a CMake custom command:

```cmake
add_custom_command(
    OUTPUT ${CMAKE_BINARY_DIR}/python/entropic/__init__.py
    COMMAND ${Python3_EXECUTABLE} ${CMAKE_SOURCE_DIR}/scripts/generate_wrapper.py
        --header ${CMAKE_SOURCE_DIR}/include/entropic/entropic.h
        --types-dir ${CMAKE_SOURCE_DIR}/include/entropic/types/
        --output ${CMAKE_BINARY_DIR}/python/entropic/__init__.py
    DEPENDS
        ${CMAKE_SOURCE_DIR}/include/entropic/entropic.h
        ${CMAKE_SOURCE_DIR}/include/entropic/types/error.h
        ${CMAKE_SOURCE_DIR}/include/entropic/types/hooks.h
        ${CMAKE_SOURCE_DIR}/scripts/generate_wrapper.py
    COMMENT "Generating Python ctypes wrapper from entropic.h"
)
```

### Wrapper ↔ Python Engine API Mapping

The wrapper must expose a compatible API so consumers can switch with minimal
code changes. The mapping from C handles to Python classes:

| Python Engine Class | C API Handle / Functions | Wrapper Class |
|---|---|---|
| `AgentEngine` | `entropic_handle_t`, `entropic_run()`, `entropic_run_streaming()` | `AgentEngine` |
| `LibraryConfig` | `entropic_configure()`, `entropic_configure_from_file()` | `LibraryConfig` |
| `ModelOrchestrator` | `entropic_inference_*` family | `ModelOrchestrator` |
| `GenerationResult` | Parsed from `result_json` return | `GenerationResult` |
| `Message` | JSON serialization at boundary | `Message` |
| `ToolCall` | Parsed from result JSON | `ToolCall` |
| `ServerManager` | `entropic_load_mcp_plugin()`, `entropic_mcp_server_*` | `ServerManager` |
| `EngineCallbacks` | `entropic_set_error_callback()`, streaming `on_token` | `EngineCallbacks` |
| `AgentState` | Inferred from engine handle state | `AgentState` enum |
| `RoutingResult` | Parsed from run result JSON | `RoutingResult` |

Classes not mapped (internal to Python engine, not part of public C API):
- `ChatAdapter`, `get_adapter`, `register_adapter` — adapter selection is internal to the C engine
- `BaseMCPServer`, `BaseTool`, `ToolRegistry` — MCP internals are C-side
- `PromptManager`, prompt frontmatter types — prompt assembly is C-side
- `ConfigLoader` internals — config loading is C-side
- `InProcessProvider` — replaced by plugin .so loading

### Library Discovery

The wrapper must find `librentropic.so` at runtime. Discovery order:

1. `ENTROPIC_LIB_PATH` environment variable (explicit override)
2. Adjacent to the wrapper module (`python/entropic/lib/librentropic.so`) — PyPI wheel layout
3. System library paths (`ldconfig`, `/usr/local/lib`, `/usr/lib`)
4. Build directory (`${CMAKE_BINARY_DIR}/lib/`) — development mode

```python
def _find_library() -> ctypes.CDLL:
    """Locate and load librentropic.so."""
    search_paths = [
        os.environ.get("ENTROPIC_LIB_PATH"),
        os.path.join(os.path.dirname(__file__), "lib", "librentropic.so"),
        ctypes.util.find_library("rentropic"),
    ]
    for path in search_paths:
        if path and os.path.isfile(path):
            return ctypes.CDLL(path)
    raise ImportError(
        "librentropic.so not found. Set ENTROPIC_LIB_PATH or install entropic-engine."
    )
```

### CLI Migration Path

Each CLI command currently imports from `entropic` (the Python engine) and calls
async methods. After migration, they import from the wrapper and call synchronous
C engine methods (the C engine's agentic loop is synchronous from the caller's
perspective — streaming is callback-based).

```
Before (v1.7.x):
  cli.py → import entropic → AgentEngine (Python) → llama-cpp-python

After (v1.9.0):
  cli.py → import entropic_c (generated wrapper) → librentropic.so → llama.cpp
```

The CLI module (`src/entropic/cli.py`) is modified to detect which backend is
available and use the C engine by default, falling back to the Python engine
with a deprecation warning.

### Deprecation Strategy

```python
# src/entropic/__init__.py — added at top
import warnings
warnings.warn(
    "The Python entropic engine is deprecated since v1.9.0. "
    "Use the C engine wrapper (python/entropic/) instead. "
    "The Python engine will be removed in v1.9.15.",
    DeprecationWarning,
    stacklevel=2,
)
```

The deprecation warning fires on `import entropic` (the old Python package).
The new wrapper lives at `python/entropic/` and will eventually replace it on
PyPI. During the transition window (v1.9.0 → v1.9.15), both are installable.

## Implementation Plan

### Phase 1: Header Parser + Generator Script

Build `scripts/generate_wrapper.py` — the core of the auto-generation pipeline.

**Parser responsibilities:**
- Parse `entropic.h` for all `ENTROPIC_EXPORT` function declarations
- Extract function name, return type, parameter types and names
- Parse `types/error.h` for `entropic_error_t` enum values
- Parse `types/hooks.h` for `entropic_hook_point_t` enum values
- Parse opaque handle typedefs (`entropic_handle_t`, `entropic_inference_backend_t`, etc.)
- Parse callback typedefs (`entropic_error_callback_t`, `entropic_hook_callback_t`,
  streaming `on_token` signature)

**Generator responsibilities:**
- Emit ctypes function declarations with correct `argtypes` and `restype`
- Emit Python enum classes mirroring C enums
- Emit wrapper classes that manage handle lifecycle (create/destroy in `__init__`/`__del__`)
- Emit error-checking decorator that converts `entropic_error_t` returns to exceptions
- Emit `entropic_free()` calls for all `char*` returns (via context manager or explicit free)
- Emit docstrings extracted from Doxygen `@brief` comments

**Parser approach:** Regex-based, not a full C parser. The header follows strict
conventions (from v1.8.9 audit) making regex reliable:
- All exported functions match `ENTROPIC_EXPORT <return_type> <name>(`
- All enums are `typedef enum { ... } entropic_*_t;`
- All handles are `typedef struct entropic_* entropic_*_t;`
- All callbacks are `typedef <return> (*entropic_*_t)(...);`

A full C parser (pycparser, libclang) is overkill for this constrained grammar
and would add a build dependency. If the header conventions drift, the regex
parser will fail loudly rather than silently produce wrong bindings.

### Phase 2: Wrapper Class Layer

The generated wrapper needs a class layer that maps C handle lifecycle to
Pythonic patterns. This layer is part of the generator output, not hand-written.

**EntropicEngine (primary API class):**

```python
class EntropicEngine:
    """Entropic inference engine instance.

    Wraps entropic_handle_t with Python lifecycle management.
    """

    def __init__(self, config_path: str | None = None, config_json: str | None = None):
        self._handle = _lib.entropic_create()
        if not self._handle:
            raise RuntimeError("Failed to create entropic engine instance")
        if config_path:
            _check_error(self._handle, _lib.entropic_configure_from_file(
                self._handle, config_path.encode()))
        elif config_json:
            _check_error(self._handle, _lib.entropic_configure(
                self._handle, config_json.encode()))

    def run(self, prompt: str) -> GenerationResult:
        result_ptr = ctypes.c_char_p()
        _check_error(self._handle, _lib.entropic_run(
            self._handle, prompt.encode(), ctypes.byref(result_ptr)))
        try:
            return GenerationResult.from_json(result_ptr.value.decode())
        finally:
            _lib.entropic_free(result_ptr)

    def run_streaming(self, prompt: str, on_token: Callable[[str], None],
                      cancel: threading.Event | None = None) -> GenerationResult:
        ...  # Wraps entropic_run_streaming with ctypes callback

    def interrupt(self) -> None:
        _check_error(self._handle, _lib.entropic_interrupt(self._handle))

    def load_identity(self, name: str) -> None:
        _check_error(self._handle, _lib.entropic_load_identity(
            self._handle, name.encode()))

    def __del__(self):
        if hasattr(self, '_handle') and self._handle:
            _lib.entropic_destroy(self._handle)
            self._handle = None

    def __enter__(self):
        return self

    def __exit__(self, *args):
        self.__del__()
```

**Error handling pattern:**

```python
class EntropicError(Exception):
    """Base exception for entropic C engine errors."""
    def __init__(self, code: int, message: str):
        self.code = code
        self.message = message
        super().__init__(f"entropic error {code}: {message}")

def _check_error(handle, error_code: int) -> None:
    if error_code != 0:
        msg = _lib.entropic_last_error(handle)
        raise EntropicError(error_code, msg.decode() if msg else "unknown error")
```

**Memory management pattern for `char*` returns:**

```python
class _OwnedString:
    """RAII wrapper for strings allocated by the C engine."""
    def __init__(self, ptr: ctypes.c_char_p):
        self._ptr = ptr
    def __str__(self) -> str:
        return self._ptr.value.decode() if self._ptr else ""
    def __del__(self):
        if self._ptr:
            _lib.entropic_free(self._ptr)
```

### Phase 3: CMake Integration

Add the generation step to the build system.

```cmake
# python/CMakeLists.txt

find_package(Python3 REQUIRED COMPONENTS Interpreter)

set(WRAPPER_OUTPUT "${CMAKE_BINARY_DIR}/python/entropic/__init__.py")
set(WRAPPER_GENERATOR "${CMAKE_SOURCE_DIR}/scripts/generate_wrapper.py")
set(ENTROPIC_HEADER "${CMAKE_SOURCE_DIR}/include/entropic/entropic.h")

add_custom_command(
    OUTPUT ${WRAPPER_OUTPUT}
    COMMAND ${Python3_EXECUTABLE} ${WRAPPER_GENERATOR}
        --header ${ENTROPIC_HEADER}
        --types-dir ${CMAKE_SOURCE_DIR}/include/entropic/types/
        --output ${WRAPPER_OUTPUT}
        --lib-name librentropic.so
    DEPENDS
        ${ENTROPIC_HEADER}
        ${CMAKE_SOURCE_DIR}/include/entropic/types/error.h
        ${CMAKE_SOURCE_DIR}/include/entropic/types/hooks.h
        ${WRAPPER_GENERATOR}
    COMMENT "Generating Python ctypes wrapper from entropic.h"
)

add_custom_target(python_wrapper ALL DEPENDS ${WRAPPER_OUTPUT})
add_dependencies(python_wrapper rentropic)

# Install wrapper alongside .so
install(
    DIRECTORY ${CMAKE_BINARY_DIR}/python/entropic
    DESTINATION ${Python3_SITEARCH}
)
install(
    TARGETS rentropic
    LIBRARY DESTINATION ${Python3_SITEARCH}/entropic/lib
)
```

### Phase 4: CLI Port

Modify each CLI command to use the C engine wrapper. The CLI module remains
in `src/entropic/cli.py` but its imports change.

**Commands to port:**

| Command | Current Implementation | C Engine Equivalent |
|---|---|---|
| `ask` | `AgentEngine.run()` async | `EntropicEngine.run()` / `run_streaming()` sync |
| `status` | `ConfigLoader` + model state queries | `EntropicEngine` + `entropic_inference_state()` |
| `init` | `ConfigLoader.initialize_project()` | Config file generation (stays Python — no C API for this) |
| `download` | `cli_download.py` — HTTP download | Stays Python (no C API for downloads) |
| `benchmark` | `benchmark/runner.py` → Python inference | `EntropicEngine` C inference for tok/s |
| `mcp-bridge` | `ServerManager` async | `entropic_mcp_server_*` via wrapper |
| `setup-cuda` | `cli_setup_cuda.py` — system config | Stays Python (no C API for system setup) |

Commands that stay pure Python (`init`, `download`, `setup-cuda`) don't touch
the inference engine — they manage files and downloads. No port needed.

**Backend selection pattern in CLI:**

```python
def _get_engine(config_path: str) -> "AgentEngine | EntropicEngine":
    """Get the best available engine backend."""
    try:
        from entropic_c import EntropicEngine  # C engine wrapper
        return EntropicEngine(config_path=config_path)
    except ImportError:
        import warnings
        warnings.warn(
            "C engine not available, falling back to Python engine. "
            "The Python engine is deprecated and will be removed in v1.9.15.",
            DeprecationWarning, stacklevel=2)
        from entropic import AgentEngine
        return AgentEngine(config_path=config_path)
```

### Phase 5: Benchmark Runner Port

The benchmark runner (`src/entropic/benchmark/`) measures tok/s, quality scores,
and identity-specific behavior. Port it to use the C engine.

**What changes:**
- `runner.py` — replace `ModelOrchestrator` calls with `EntropicEngine.run()`
- `performance.py` — tok/s measurement uses C engine streaming callback for timing
- `quality.py` — quality checks run against C engine output
- `judge.py` — judge model runs through C engine

**What stays the same:**
- `report.py` — report formatting is pure Python
- `types.py` — benchmark result types are pure Python
- `checks.py` — validation checks are pure Python
- `cli.py` — Click command definition, delegates to runner

**Key metric:** tok/s from C engine must match or exceed Python engine (v1.7.1
baseline: 36.1 tok/s for Qwen3.5-35B-A3B-UD-IQ3_XXS). The C engine links
llama.cpp directly (no Python binding overhead), so parity is expected.

### Phase 6: Model Test Validation

Run the existing 42 model tests against the C engine via the wrapper.

The model tests (`tests/model/`) use `AgentEngine` to exercise:
- Classification accuracy (router assigns correct tier)
- Generation quality (response coherence, no special tokens)
- Tool calling (tool calls parse correctly, directives fire)
- Identity behavior (each identity stays in character)
- Performance (classification under 500ms)

**Port approach:**
- Model test fixtures that create `AgentEngine` are updated to create
  `EntropicEngine` (C wrapper) instead
- Test assertions remain identical — the C engine must produce the same
  observable behavior as the Python engine
- Any test failure indicates a behavioral regression in the C port

**Fixture change:**

The generated `EntropicEngine` class exposes `run()`, `run_streaming()`,

`entropic ask "Hello"` → response generated by C engine (verified by log output
showing `librentropic` load). `entropic status` → reports C engine model state.
`entropic benchmark` → produces tok/s measurement from C engine.

### V4: 42 model tests pass against C engine

All existing model tests in `tests/model/` pass when the engine fixture uses
the C wrapper. No test modifications (the C engine must match Python engine
behavior). Failures are C engine bugs, not test issues.

### V5: Benchmark tok/s >= Python engine baseline

C engine benchmark reports tok/s >= 36.1 (Python engine baseline for
Qwen3.5-35B-A3B-UD-IQ3_XXS on 16GB Blackwell). Expected to exceed due to
eliminated Python binding overhead.

### V6: Deprecation warnings fire correctly

`import entropic` (Python engine) emits `DeprecationWarning` mentioning v1.9.15.
`from entropic_c import EntropicEngine` (C wrapper) does NOT emit warnings.

### V7: Wrapper regenerates on header change

Modifying `entropic.h` and rebuilding triggers wrapper regeneration. The CMake
dependency tracking ensures the wrapper stays in sync.

### V8: Generator handles all C API patterns

The generator correctly handles:
- Functions returning `entropic_error_t` (error-checked)
- Functions returning `char*` (caller-freed via `entropic_free()`)
- Functions returning `const char*` (borrowed, not freed)
- Functions with callback parameters (ctypes `CFUNCTYPE`)
- Functions with output parameters (`char**`)
- Opaque handle creation/destruction
- Enum definitions with explicit values

### V9: No Python engine code changes (except deprecation markers)

The Python engine in `src/entropic/` is not modified beyond adding deprecation
warnings. No bug fixes, no feature additions, no refactoring. It is frozen
except for the deprecation markers.

## Implementation Notes (from review)

1. **Wrapper package naming collision.** `python/entropic/` and `src/entropic/`
   both resolve to `import entropic` on sys.path. During the transition window
   (v1.9.0 → v1.9.15), both exist. The implementing session must ensure the
   wrapper uses a distinct install path or the old package is renamed with a
   deprecation shim (e.g., `entropic_legacy`). Alternatively, the wrapper can
   install as `entropic` and the old package installs as `entropic-python-engine`
   with a compatibility import that emits DeprecationWarning.

2. **CLI port: async→sync.** The Python CLI uses asyncio (`async for msg in
   engine.run()`). The C engine is synchronous from the caller's perspective
   (blocking `entropic_run()` or callback-based `entropic_run_streaming()`).
   Strip all asyncio patterns when porting CLI commands — use synchronous
   wrapper calls with streaming callbacks for output.
