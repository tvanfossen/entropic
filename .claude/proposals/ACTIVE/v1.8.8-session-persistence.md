## Scope

Session persistence: SQLite conversation storage, file-based session logs with
rotation, delegation record persistence, and permission persistence. After v1.8.8,
the engine can persist conversations across restarts, maintain delegation
parent/child linkage in the database, write rotating session and model log files,
and save permission allow/deny patterns to the project config file.

This builds `librentropic-storage.so` as defined in `docs/architecture-cpp.md`.

This is a faithful port of:
- `src/entropic/storage/database.py` -- `Database`: SQLite connection, migration runner, execute/fetchone/fetchall, FTS5 virtual table
- `src/entropic/storage/backend.py` -- `SQLiteStorage`: conversation CRUD, message persistence, delegation records, search, statistics
- `src/entropic/storage/models.py` -- `ConversationRecord`, `MessageRecord`, `DelegationRecord`: typed records with to_row/from_row serialization
- `src/entropic/core/logging.py` -- `setup_logging` (session.log), `setup_model_logger` (session_model.log): file handlers with per-session overwrite
- `src/entropic/config/loader.py` -- `ConfigLoader.save_permission()`: YAML config read-modify-write for permission patterns
- `src/entropic/core/compaction.py` -- `CompactionManager._save_snapshot()`: pre-compaction snapshot persistence

v1.8.0 established spdlog as the structured logging framework. v1.8.6 established
the delegation system that creates delegation records. v1.8.8 provides the
persistence layer that both systems write to.

**What this version does NOT include:**
- spdlog setup or structured logging framework (v1.8.0) -- already done
- Delegation logic itself (v1.8.6) -- already done, this adds storage
- Structured audit log with JSONL replay (v1.9.5) -- separate concern
- Compaction hooks (v1.9.9) -- programmatic compaction invocation
- Hook architecture (v1.9.1) -- `on_pre_compact`, `on_post_compact` hooks

**What this version adds beyond the Python port:**
- spdlog file sinks with size-based rotation (Python uses mode='w' overwrite per session)
- Configurable rotation limits via `StorageConfig` (max file size, max rotated files)
- Compaction snapshot table in SQLite (Python has a TODO placeholder)
- Thread-safe SQLite access via connection-per-thread with `std::mutex` serialization (Python uses asyncio lock)
- Storage is fully optional -- engine works without it (in-memory only mode)

### Session Logger Design

The Python engine uses `logging.FileHandler(mode='w')` which overwrites each
session. The C++ port uses spdlog rotating file sinks, which is strictly better:
previous sessions are preserved as `.1`, `.2`, etc. until the rotation limit.

```
.entropic/
├── session.log          # Current session (human-readable timeline)
├── session.log.1        # Previous session (rotated)
├── session_model.log    # Current session raw model I/O
├── session_model.log.1  # Previous model log (rotated)
└── entropic.db          # SQLite database
```

`SessionLogger` wraps spdlog sink creation with engine-appropriate defaults.
It does NOT replace the global spdlog logger setup from v1.8.0 -- it adds
file sinks for session-specific output alongside the structured logging
already in place.

```cpp
/// @file include/entropic/storage/logger.h
/// @brief Session log file management with spdlog rotating sinks.
/// @version 1.8.8

namespace entropic {

/// @brief Configuration for session log files.
/// @version 1.8.8
struct SessionLogConfig {
    std::filesystem::path log_dir;          ///< Directory for log files
    size_t max_file_size = 10 * 1024 * 1024; ///< Max size before rotation (10MB)
    size_t max_files = 3;                    ///< Max rotated files to keep
    spdlog::level::level_enum session_level = spdlog::level::info;  ///< Session log level
    spdlog::level::level_enum model_level = spdlog::level::info;    ///< Model log level
};

/// @brief Manages session.log and session_model.log with rotation.
///
/// Creates spdlog rotating file sinks for:
/// - session.log: human-readable timeline of engine operations
/// - session_model.log: raw model input/output (prompt + completion)
///
/// Thread-safe. Sinks are created once and shared across the session.
///
/// @par Lifecycle:
/// @code
///   SessionLogger logger(config);
///   logger.initialize();        // Creates sinks, opens files
///   logger.session_info(...);   // Write to session.log
///   logger.model_log(...);      // Write to session_model.log
///   // Destructor flushes and closes
/// @endcode
///
/// @version 1.8.8
class SessionLogger {
public:
    /// @brief Construct with configuration.
    /// @param config Log file configuration.
    /// @version 1.8.8
    explicit SessionLogger(const SessionLogConfig& config);

    /// @brief Initialize log sinks and open files.
    /// @return true on success.
    /// @version 1.8.8
    bool initialize();

    /// @brief Log a session event (human-readable timeline).
    /// @param level Log level.
    /// @param message Event description.
    /// @version 1.8.8
    void session_log(spdlog::level::level_enum level,
                     std::string_view message);

    /// @brief Log raw model I/O.
    /// @param prompt The full prompt sent to the model.
    /// @param completion The raw model output.
    /// @param tokens_generated Token count.
    /// @param duration_ms Generation time.
    /// @version 1.8.8
    void model_log(std::string_view prompt,
                   std::string_view completion,
                   size_t tokens_generated,
                   double duration_ms);

private:
    SessionLogConfig config_;
    std::shared_ptr<spdlog::logger> session_logger_;
    std::shared_ptr<spdlog::logger> model_logger_;
};

} // namespace entropic
```

### Permission Persistence

The Python engine writes permission patterns to `.entropic/config.local.yaml`
via `ConfigLoader.save_permission()`. The C++ port needs the same capability
but uses ryml (already available from v1.8.1) for YAML read-modify-write.

```cpp
/// @file include/entropic/storage/permission_persister.h
/// @brief Persist permission allow/deny patterns to project config.
/// @version 1.8.8

namespace entropic {

/// @brief Read-modify-write permission patterns in YAML config.
///
/// Loads the project's config.local.yaml, adds patterns to the
/// permissions.allow or permissions.deny list, and writes back.
/// Thread-safe via internal mutex.
///
/// @version 1.8.8
class PermissionPersister {
public:
    /// @brief Construct with config directory path.
    /// @param config_dir Path to app directory (e.g., ".entropic/").
    /// @version 1.8.8
    explicit PermissionPersister(const std::filesystem::path& config_dir);

    /// @brief Save a permission pattern.
    /// @param pattern Permission pattern (e.g., "bash.execute:pytest *").
    /// @param allow true for allow list, false for deny list.
    /// @return true on success, false on I/O error.
    /// @version 1.8.8
    bool save_permission(std::string_view pattern, bool allow);

private:
    std::filesystem::path config_path_;
    std::mutex write_mutex_;
};

} // namespace entropic
```

### SQLite Schema (Migrations)

Direct port of the Python `MIGRATIONS` dict. The C++ `SqliteDatabase` runs
these sequentially on first `initialize()` call, tracking applied migrations
in a `migrations` table. Identical schema to Python for data compatibility.

**Migration 001 -- initial:**
```sql
CREATE TABLE IF NOT EXISTS conversations (
    id TEXT PRIMARY KEY,
    title TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    project_path TEXT,
    model_id TEXT,
    metadata TEXT DEFAULT '{}'
);

CREATE TABLE IF NOT EXISTS messages (
    id TEXT PRIMARY KEY,
    conversation_id TEXT NOT NULL,
    role TEXT NOT NULL CHECK(role IN ('user', 'assistant', 'system', 'tool')),
    content TEXT NOT NULL,
    tool_calls TEXT DEFAULT '[]',
    tool_results TEXT DEFAULT '[]',
    token_count INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_compacted BOOLEAN DEFAULT FALSE,
    identity_tier TEXT,
    FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE
);

CREATE TABLE IF NOT EXISTS tool_executions (
    id TEXT PRIMARY KEY,
    message_id TEXT,
    server_name TEXT NOT NULL,
    tool_name TEXT NOT NULL,
    arguments TEXT DEFAULT '{}',
    result TEXT,
    duration_ms INTEGER DEFAULT 0,
    status TEXT CHECK(status IN ('success', 'error', 'timeout')),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (message_id) REFERENCES messages(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_messages_conversation
    ON messages(conversation_id);
CREATE INDEX IF NOT EXISTS idx_messages_created
    ON messages(created_at);
CREATE INDEX IF NOT EXISTS idx_conversations_updated
    ON conversations(updated_at);
```

```sql
CREATE TABLE IF NOT EXISTS compaction_snapshots (
```

The Python engine has a placeholder `_save_snapshot()` that logs "Would save"
but never writes. The C++ port implements it fully -- compaction snapshots
are stored in SQLite before context is compacted, enabling future replay
and analysis (v1.9.5 audit log, v2.1.0 fine-tuning from session data).

### Thread Safety

Python uses `asyncio.Lock` for single-writer access to SQLite. The C++ port
uses `std::mutex` to serialize all database operations. SQLite is configured
with `SQLITE_OPEN_FULLMUTEX` as a safety net, but the application-level mutex
prevents any concurrent access patterns.

```cpp
/// @brief Thread-safe SQLite database wrapper.
/// @version 1.8.8
class SqliteDatabase {
public:
    /// @brief Open database at path.
    /// @param db_path Path to SQLite file. Created if absent.
    /// @version 1.8.8
    explicit SqliteDatabase(const std::filesystem::path& db_path);

    /// @brief Initialize database and run pending migrations.
    /// @return true on success.
    /// @version 1.8.8
    bool initialize();

    /// @brief Close database connection.
    /// @version 1.8.8
    void close();

    /// @brief Execute a write statement (INSERT, UPDATE, DELETE).
    /// @param sql SQL statement with ? placeholders.
    /// @param binder Function to bind parameters to the prepared statement.
    /// @return true on success.
    /// @version 1.8.8
    bool execute(std::string_view sql,
                 std::function<void(sqlite3_stmt*)> binder = nullptr);

    /// @brief Fetch a single row.
    /// @param sql SQL SELECT statement.
    /// @param binder Function to bind parameters.
    /// @param extractor Function to extract columns from result row.
    /// @return true if row found.
    /// @version 1.8.8
    bool fetch_one(std::string_view sql,
                   std::function<void(sqlite3_stmt*)> binder,
                   std::function<void(sqlite3_stmt*)> extractor);

    /// @brief Fetch all matching rows.
    /// @param sql SQL SELECT statement.
    /// @param binder Function to bind parameters.
    /// @param row_handler Called for each result row.
    /// @return Number of rows fetched.
    /// @version 1.8.8
    size_t fetch_all(std::string_view sql,
                     std::function<void(sqlite3_stmt*)> binder,
                     std::function<void(sqlite3_stmt*)> row_handler);

private:
    bool run_migrations();

    std::filesystem::path db_path_;
    sqlite3* db_ = nullptr;
    std::mutex mutex_;
};
```

### Optional Storage Pattern

Storage is nullable throughout the engine. The same pattern used in the Python
engine (`if self._storage:`) translates to nullptr checks in C++:

```cpp
// In DelegationManager (v1.8.6, receives IStorageBackend* from engine)
void DelegationManager::create_storage_records(...) {
    if (!storage_) return;  // No-op without storage
    // ...
}

// In CompactionManager (v1.8.4, receives IStorageBackend* from engine)
void CompactionManager::save_snapshot(...) {
    if (!storage_) return;  // No-op without storage
    // ...
}
```

The C API exposes `entropic_set_storage_path()` in the facade (v1.8.9
stabilizes this). If never called, the engine runs in memory-only mode.

1. Implement `SqliteDatabase` with `sqlite3_open_v2()`, `SQLITE_OPEN_FULLMUTEX`
2. Implement migration runner with `migrations` tracking table
3. Port all 4 migrations (initial, fts, delegations, compaction_snapshots)

1. Implement `ConversationRecord`, `MessageRecord`, `DelegationRecord` structs

4. ISO 8601 timestamp formatting
5. Unit tests: round-trip serialization for all record types

1. Implement `SqliteStorageBackend` wrapping `SqliteDatabase`
2. Conversation CRUD: create, save, load, list, delete, update_title
3. Message persistence with tool_calls/tool_results as JSON blobs
4. Delegation storage: create_delegation, complete_delegation, get_delegations
5. Full-text search via FTS5 MATCH queries
6. Compaction snapshot save (new -- implements Python's placeholder)
7. Statistics queries (conversation count, message count, token total)
8. Unit tests: full CRUD cycle, delegation lifecycle, search results

### Phase 4: C Interface

1. Implement `i_storage_backend.h` pure C functions

3. Exception-to-error-code translation at .so boundary
4. JSON serialization of results for cross-boundary return

### Phase 5: SessionLogger

3. Model log sink: `rotating_file_sink_mt("session_model.log", max_size, max_files)`

5. Model log format: timestamp + prompt hash + raw I/O

### Phase 6: PermissionPersister

## Validation Criteria

### Behavioral Parity with Python Engine

1. **Conversation CRUD**: create, save messages, load, list, delete -- identical schema and behavior to `SQLiteStorage`
2. **Message persistence**: all fields round-trip correctly (role, content, tool_calls JSON, tool_results JSON, token_count, is_compacted, identity_tier)
3. **Delegation records**: create_delegation returns (delegation_id, child_conversation_id), complete_delegation updates status + timestamp, get_delegations returns ordered list
4. **FTS5 search**: MATCH queries return conversation results with snippets, triggers maintain index automatically
5. **Migration runner**: runs migrations sequentially, skips already-applied, tracks in migrations table
6. **Permission persistence**: patterns written to YAML, duplicates prevented, allow/deny lists separate

### Beyond Python Parity

7. **Compaction snapshots**: full message history saved to `compaction_snapshots` table before compaction (Python has TODO placeholder)
8. **Log rotation**: session.log and model.log rotate at size limit, keep N previous files
9. **Thread safety**: concurrent access from multiple engine threads does not corrupt database or log files

### Quality Gates

10. All functions SLOC <= 50, cognitive complexity <= 15 (knots)
11. Every public symbol has Doxygen with `@brief` and `@version` tags
12. No sqlite3.h, spdlog.h, or ryml.h in interface headers
13. All exceptions caught at .so boundary, converted to `entropic_error_t`
14. Storage is fully optional -- all tests pass with nullptr storage backend
