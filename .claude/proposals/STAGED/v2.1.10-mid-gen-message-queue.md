# v2.1.10 — Mid-generation user message queue

## Session prompt

A UX feature for long-thinking interactive sessions. When a researcher-tier turn expands into 30+ tool calls running for minutes, the user today has two bad options: interrupt (destroys in-flight investigation) or wait (kills momentum). This patch adds a third option: enqueue a follow-up message that the engine consumes at the next clean turn boundary, without interrupting the current turn.

The queued message lands as a fresh user message the moment the current top-level turn reaches `AgentState::COMPLETE`. From the user's perspective it looks like they typed it the moment the previous response ended — no stream multiplexing, no mid-turn context corruption.

After this patch:
- Consumers can call `entropic_queue_user_message(handle, text)` during in-flight generation.
- Queue depth is observable via `entropic_user_message_queue_depth(handle, *count)`.
- Queued messages survive a turn completing and immediately seed the next turn.
- The streaming protocol is unchanged — observers see clean per-turn token streams.

## Design decisions already made (do not re-litigate)

- **Boundary: top-level `AgentState::COMPLETE` only.** Queued messages do NOT inject at parent-resume-after-child-delegation boundaries. Rationale: parent expects to continue from child output; injecting a new user message there breaks the parent's reasoning context. Safer to wait one more boundary at the cost of slightly longer queue latency.
- **No interruption.** The queue does NOT cancel the current turn. Consumers wanting that already have `entropic_interrupt`.
- **Queue is bounded.** Default cap 4–8 messages (consumer-configurable). Past the cap, `entropic_queue_user_message` returns an error (`ENTROPIC_ERROR_QUEUE_FULL` or similar). Pick policy: FIFO append (deferred decision; record at implementation time with rationale).
- **Streaming protocol unchanged.** Token callbacks fire only for the active turn. No mid-stream switch. The queued message becomes a fresh next turn with its own token stream.
- **Thread-safe.** Consumer threads can enqueue from any thread; engine consumes from the agent loop.

## Scope

### C ABI additions (strictly additive)

```c
/**
 * @brief Queue a user message to be injected at the next top-level
 *        turn boundary.
 *
 * Does not interrupt the in-flight generation. The queued message
 * becomes the next user input automatically when the current turn
 * reaches AgentState::COMPLETE.
 *
 * @threadsafety Thread-safe.
 * @version 2.1.10
 */
ENTROPIC_EXPORT entropic_error_t entropic_queue_user_message(
    entropic_handle_t handle,
    const char* message);

/**
 * @brief Query queue depth (for consumer UI feedback).
 *
 * @threadsafety Thread-safe.
 * @version 2.1.10
 */
ENTROPIC_EXPORT entropic_error_t entropic_user_message_queue_depth(
    entropic_handle_t handle,
    size_t* count);

/**
 * @brief Clear all queued user messages (consumer-initiated retract).
 *
 * @threadsafety Thread-safe.
 * @version 2.1.10
 */
ENTROPIC_EXPORT entropic_error_t entropic_clear_user_message_queue(
    entropic_handle_t handle);

// New error code
ENTROPIC_ERROR_QUEUE_FULL
```

### Engine plumbing

- Thread-safe bounded FIFO queue inside `AgentEngine`. Implementation: `std::mutex` + `std::deque<std::string>` or `boost::lockfree::queue` equivalent. Pick simplicity over throughput.
- Hook in agent loop state machine: on transition into `AgentState::COMPLETE` for a top-level turn, check queue; if non-empty, pop front and immediately initiate a new turn with it before yielding to caller.
- Queue depth surfaced as engine state (visible to `entropic_inspect` if relevant).
- Configurable cap via `entropic_configure` parameter or runtime field (`message_queue_capacity`, default 8).

### Streaming + observation

- Token callback fires only for the active turn (unchanged).
- A new state observer event (or extension of an existing one) when a queued message is consumed, so the TUI / consumer UI can clear the "queued" badge from scrollback.

### Tests

- Unit: queue semantics (enqueue / dequeue / clear / cap enforcement / thread safety stress test).
- Unit: state-machine boundary detection — queued message is consumed exactly at top-level `AgentState::COMPLETE` and not at parent-resume-after-child.
- Integration: in-flight turn + enqueue + wait → queued message becomes second turn cleanly; both turns' token streams are independent.

## Out of scope

- Boundary policy beyond top-level COMPLETE (e.g., parent-resume-after-child). Deferred until a real consumer demand emerges and the context-integrity risk is understood.
- Per-call boundary policy (`entropic_queue_user_message_at(handle, msg, BOUNDARY)`). Deferred — adds ABI surface without proven need.
- Queue prioritization (newest-wins, coalescing, dedup). FIFO is the v1.
- Cross-turn message editing (consumer-initiated retract of a specific queued message by id). `entropic_clear_user_message_queue` is the v1 retract; per-message dequeue can come later if a consumer needs it.
- Queue persistence across engine restarts.

## Validation criteria

1. `entropic_queue_user_message` returns `ENTROPIC_OK` during in-flight generation; returns `ENTROPIC_ERROR_INVALID_STATE` when no turn is running (consumer should just call `entropic_run_streaming` as normal).

## Implementation Log

### 2026-05-14 — Session 3 start

**Design decisions (recorded before code lands):**

- **Boundary detection by call-site, not state observer.** The queue
  drain lives in `AgentEngine::run_turn` after the call to
  `AgentEngine::run` returns. This is the structural definition of a
  top-level `AgentState::COMPLETE`: only the engine's public top-level
  entry point reaches that point. Child loops run through
  `AgentEngine::run_loop` invoked by `DelegationManager` *inside*
  `run()` and never see the `run_turn` post-loop hook. This gives the
  load-bearing property in the proposal (queued message consumed only
  at top-level COMPLETE, NOT at parent-resume-after-child) for free —
  the hook can't fire at a parent-resume boundary because that
  boundary is internal to `run()`.
- **Queuing policy = FIFO append.** `std::deque<std::string>` with
  `push_back` on enqueue, `pop_front` on consume. Rationale: simplest
  predictable semantics; consumers reason about queued order = arrival
  order. Newest-wins / coalescing / dedup deferred (out-of-scope per
  proposal).
- **Locking discipline.** Queue holds a dedicated `std::mutex
  queue_mutex_`, independent of the facade's per-handle `api_mutex`.
  Critical: `entropic_run_streaming` does NOT hold `api_mutex` for the
  duration of a turn, so a second thread can call
  `entropic_queue_user_message` without blocking on the run. The
  facade thunks for the three new ABI calls bypass `api_mutex`
  entirely and go straight to the engine's thread-safe queue methods.
- **Cap field on LoopConfig.** New `int message_queue_capacity = 8;`
  on `LoopConfig`. Not (yet) surfaced through the JSON config schema —
  the default is reasonable, and other LoopConfig fields
  (`max_iterations`, anti-spiral knobs) are similarly not in the JSON
  surface today. Engine exposes `set_message_queue_capacity(int)` so
  consumers can tune at runtime if needed; out-of-band config schema
  exposure deferred until a consumer asks.
- **Observer surface.** Added `on_queued_message_consumed(const char*
  consumed, size_t remaining, void* ud)` to `EngineCallbacks` plus
  facade `entropic_set_queue_observer`. Fires immediately before the
  engine seeds the next turn with the popped message. Single dedicated
  callback (rather than overloading `on_state_change`) so consumers
  don't have to discriminate normal PLANNING transitions from
  queue-consumption transitions.
- **Streaming protocol unchanged.** Because the drain lives in
  `run_turn` and both `run_streaming` overloads delegate to
  `run_turn`, the queued message naturally becomes a second
  agent-loop turn under the same `on_token` wiring. The synthetic
  `(token="", len=0)` end-of-run sentinel is emitted by the facade
  once per `run_streaming` call — i.e. after all queued turns drain.
- **`ENTROPIC_ERROR_INVALID_STATE` on enqueue-when-idle.** Per
  validation criterion #1: the facade checks
  `handle->engine->is_running()` (atomic flag set in `run_turn` entry,
  cleared on exit) and rejects with INVALID_STATE if false.
