## Part 1: Examples Porting

Two existing Python examples target the v1.7.1 Python engine:

| Example | Complexity | Demonstrates |
|---------|-----------|--------------|
| `hello-world/` | Minimal | Single tier, streaming, app context |
| `pychess/` | Full | Multi-tier pipeline, grammar constraints, custom MCP tool, per-tier identities, handoff |

Each gets **two new implementations** that must produce equivalent
behavior:

### hello-world

**C/C++ version** (`examples/hello-world/main.c` or `main.cpp`):
- Link against `librentropic.so` (facade)
- `entropic_create()` → `entropic_configure()` → `entropic_run_streaming()` → `entropic_destroy()`
- Streaming callback prints tokens
- Interactive prompt loop
- Config loaded from same `config.yaml`

**Python wrapper version** (`examples/hello-world/main_wrapper.py`):
- `import entropic` (auto-generated wrapper from `entropic.h`)
- Same API surface as C but Pythonic
- Streaming via callback or iterator
- Proves the wrapper round-trips correctly

### pychess

**C/C++ version** (`examples/pychess/main.cpp`):
- Custom MCP server plugin (`chess_server.so`) exposing `chess.make_move`
- Multi-tier config (thinker + executor)
- Grammar constraints loaded from `.gbnf` files
- Handoff via `entropic.handoff` directive
- Board state managed in C, moves validated via chess logic
- Depends on a C chess library or embedded move validation

**Python wrapper version** (`examples/pychess/main_wrapper.py`):
- `import entropic` wrapper + `python-chess` for board management
- Register `chess.make_move` tool via wrapper API
- Same multi-tier pipeline behavior as the Python original
- Grammar constraints passed through wrapper

### Blockers

| Blocker | Status | Required for |
|---------|--------|-------------|
| C API facade wired (not stubs) | Stubs only | Both C and Python versions |
| Python wrapper auto-generated | Not started | Python wrapper versions |
| MCP plugin loading (`dlopen`) | Implemented | pychess C version |

**The C API facade wiring is the critical path.** Examples can't be
written until `entropic_create/configure/run/destroy` actually work.

### Validation

- C/C++ hello-world: `cmake --build` + run → interactive streaming chat
- C/C++ pychess: `cmake --build` + run → play chess, AI responds with
  legal moves via grammar-constrained pipeline
- Python hello-world: `python main_wrapper.py` → identical behavior
- Python pychess: `python main_wrapper.py` → identical behavior
- All four must run against the same model on the same hardware
- Output logged for comparison (token-for-token match not required,
  behavioral equivalence sufficient)

## Part 2: Human Learning Plan

Structured walkthrough of the C++ engine, ordered by dependency graph
(bottom-up). Each item is a reading + modification exercise.

### Layer 1: Types & Build System (1-2 hours)

- Read `CMakeLists.txt` top-level + all presets
- Read `entropic_config.h.in` → trace to generated header
- Read `entropic_export.h` → understand symbol visibility
- Read `types/error.h` → trace through `entropic_error_name()`
- Read `types/config.h` → all config structs, defaults
- **Exercise:** Add a new error code, verify it round-trips through tests

### Layer 2: Config & Prompts (2-3 hours)

- Read `config/yaml_util.h` → extraction pattern
- Read `config/loader.cpp` → layered merge logic
- Read `config/validate.cpp` → cross-field validation
- Read `prompts/manager.cpp` → frontmatter parsing
- Read `prompts/classification.cpp` → router prompt builder
- **Exercise:** Add a new config field end-to-end (struct → parse → validate → test)

### Layer 3: Inference (3-4 hours)

- Read `i_inference_backend.h` → C boundary contract
- Read `inference/backend.h` → concrete base (80% logic)
- Read `inference/llama_cpp_backend.*` → 20% override
- Read `inference/adapters/adapter_base.h` → chat formatting
- Read `inference/adapters/qwen35.*` → XML tool calls
- Read `inference/orchestrator.h` → tier routing, VRAM lifecycle
- **Exercise:** Add a metric to the inference backend, expose via C API

### Layer 4: Engine Core (3-4 hours)

- Read `core/engine.h` → agentic loop, state machine
- Read `core/response_generator.*` → system prompt assembly
- Read `core/context.*` → message management, compaction trigger
- Read `core/directives.*` → tool result → engine action
- **Exercise:** Add a new directive type end-to-end

### Layer 5: MCP & Tools (2-3 hours)

- Read `i_mcp_server.h` → plugin C boundary
- Read `mcp/server_base.h` → 80% dispatch logic
- Read one server implementation (filesystem or bash)
- Read permission manager → default-deny enforcement
- **Exercise:** Write a minimal custom MCP server plugin .so

### Layer 6: Integration (2-3 hours)

- Read `entropic.h` → full C API surface
- Read facade implementation → how C API maps to internals
- Read Python wrapper → auto-generated bindings
- **Exercise:** Write a C program that uses the full lifecycle
