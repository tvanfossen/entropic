# Entropic-Powered IoT Integration Test Agent for CI/CD

## Problem Statement

BISSELL's b11_e2e validation framework currently requires Claude Code as the execution agent — a human-in-the-loop AI that interprets a 497-line CLAUDE.md procedure, makes MCP tool calls to pyCommander, collects evidence, and writes test results. This works but:

1. **Not automatable.** Claude Code requires human presence. Can't run unattended in CI/CD.
2. **Non-deterministic.** An LLM interpreting free-text procedures may execute differently each run. Test reproducibility depends on prompt engineering, not guaranteed execution.
3. **Expensive.** Cloud LLM inference per test step, per requirement, across 626 requirements. API costs scale with coverage.
4. **Not self-hosted.** Depends on Anthropic's API availability. Manufacturing validation shouldn't have external service dependencies.

### What This Proposes

Replace the Claude-driven workflow with a local, self-hosted Entropic agent where the test procedure decomposes into role-scoped identities (Executor, Analyst, Reporter) with grammar constraints and MCP tool scoping. The agent executes **one REQ at a time**, invoked by an **operator** — which may be a human, Claude on a subscription seat, a CI/CD script, or any combination. The operator decides what to run and validates results; Entropic does the work.

**Key design principle: Entropic is NOT the campaign orchestrator.** Running 293 REQs sequentially requires maintaining campaign state across hours of execution. Any LLM (local or cloud) suffers context degradation over that span. Campaign orchestration belongs to a stateless operator — a script that loops REQs, calls Entropic per-REQ, and records results. The operator needs zero memory of previous REQs to run the next one.

## Concept

### Model

All tiers run **Qwen 3.5 35A3B** (MoE, 3B active params). The tester must be genuinely intelligent — it needs to explore source repos, trace DCI call chains through firmware code, cross-reference log output against actual source implementations, and produce explicit, evidence-backed verdicts for every requirement. This is not surface-level "logs look right" checking. Every verdict must cite specific file:line evidence from the source repo.

### Operator Modes

The operator is **external to Entropic** and **stateless per-REQ**. No campaign-wide context needed.

| Operator | How | Validation | Use Case |
|---|---|---|---|
| **CI/CD script** | Loop REQs, call `entropic run-req`, collect results | None — trust grammar-enforced output | Automated PR gate, nightly regression |
| **Claude (subscription)** | Loop REQs via bash tool, review each result | Validates evidence quality, source citations, verdict reasoning | Higher-confidence results, catches reasoning gaps |
| **Human** | Invoke per-REQ manually, review output | Full manual review | Debugging, investigation, initial validation |
| **Hybrid** | Script runs all, flags failures for Claude/human review | Selective — only failures get deep review | Best of both — coverage + depth where it matters |

### Identities (Shared 35A3B Backend)

Three identities share the same Qwen 3.5 35A3B model via Entropic's backend deduplication — one GGUF loaded, different identity prompts/grammars/MCP scopes per identity. Internal delegation between identities follows the execution flow: Executor → Analyst → Reporter.

Campaign orchestration (scoping, sequencing, skip/stall decisions) is the operator's responsibility, NOT an LLM identity. This avoids context degradation over long campaigns.

#### Executor Identity

- **Identity:** Device command specialist. Sends commands to DUT via pyCommander, collects immediate responses.
- **MCP scope:** READ/WRITE on pyCommander device tools (`device.shadow_update`, `device_control.command`, `dci.in`, `dci.out`, `connection.config`, `log.manage`, `log.prepare`, `log.download`). No filesystem access — cannot read source code or write results.
- **Grammar:** Structured command + response format
- **Behavior:** Receives the REQ's acceptance criteria. Determines the correct MCP commands, clears logs, executes, waits for response, returns structured result (HTTP status, response body, timing). Does not interpret results — that's the Analyst's job.

#### Analyst Identity

- **Identity:** Evidence analyst. Deep source code explorer. Parses device logs, traces DCI event chains through IoT library source code, cross-references firmware implementations, produces explicit evidence-backed verdicts.
- **MCP scope:** READ on pyCommander log tools (`log.manage`, `log.get_status`). READ on filesystem (comprehensive — IoT library source at d_linux_bissell_iot_midea, firmware source, any referenced source repo from the PR).
- **Grammar:** Structured evidence format:
  ```json
  {
    "req_id": "REQ-XXXX",
    "log_excerpt": "full relevant log section",
    "dci_chain": [
      {"event_id": 3, "direction": "in", "name": "set_drive", "params": {"driveMode": 2}}
    ],
    "l2_source_verification": {
      "file": "source/shadow_delta_handler.c",
      "line": 1846,
      "function": "apply_valid_command",
      "doxygen": "relevant doc block",
      "code_excerpt": "actual source lines showing DCI dispatch"
    },
    "layer_verification": {
      "l1": {"result": "pass", "evidence": "HTTP 200, command accepted"},
      "l2": {"result": "pass", "evidence": "DCI set_drive emitted at shadow_delta_handler.c:1846"},
      "l3": {"result": "pass", "evidence": "MCU ACK event 3, product_status confirms driveMode=2"}
    },
    "verdict": "pass",
    "verdict_reason": "All three layers verified with source-level evidence"
  }
  ```
- **Behavior:** This is the tier that does the real work. Given command execution results from the Executor, the Analyst:
  1. Exports and parses device logs for DCI chain evidence
  2. Searches IoT library source for the handler that emitted the DCI event (file:line)
  3. Reads the actual source code to verify the implementation matches the requirement
  4. Traces the full L1 → L2 → L3 chain with specific evidence at each layer
  5. Renders a verdict with citations — not "logs look right" but "shadow_delta_handler.c:1846 calls send_dci(SET_DRIVE, {driveMode=2}) which matches the log output DCI event 3"

#### Reporter Tier

- **Identity:** Test report writer. Produces final test result Markdown, PlantUML sequence diagrams, campaign summary updates.
- **MCP scope:** READ on filesystem (Analyst output, template.md). WRITE on filesystem (test result files in campaigns/b11/<campaign>/{verdict}/). WRITE on CI server MCP (report pass/fail upstream).
- **Grammar:** Markdown template output matching existing b11_e2e template.md structure
- **Behavior:** Takes the Analyst's structured evidence and formats it into the standard test result document with all required sections: log excerpt, PlantUML diagram, L2 source verification, layer verification table.

### Per-Requirement Execution Flow

```
Lead: select next REQ from campaign scope
  │
  ├── Lead: read acceptance criteria (filesystem → csv_query)
  │   Decide: test / skip / stall based on scope_decisions + tool availability
  │
  ├── Lead: delegate to Executor
  │   ├── Executor: clear device logs (pycommander.log.manage clear)
  │   ├── Executor: execute command (pycommander.device.shadow_update / device_control)
  │   ├── Executor: wait for response, collect status
  │   └── Executor: return structured result to Lead
  │
  ├── Lead: delegate to Analyst
  │   ├── Analyst: export device logs (pycommander.log.manage export)
  │   ├── Analyst: parse DCI chain from logs
  │   ├── Analyst: search IoT library source for DCI handler (filesystem.read)
  │   ├── Analyst: read source code, trace implementation (filesystem.read)
  │   ├── Analyst: validate L1→L2→L3 chain with file:line evidence
  │   └── Analyst: return structured evidence + verdict to Lead
  │
  ├── Lead: delegate to Reporter
  │   ├── Reporter: generate PlantUML diagram
  │   ├── Reporter: write result Markdown to campaign folder
  │   └── Reporter: update campaign.yaml progress
  │
  └── Lead: advance to next REQ
```

### What This Is NOT

This is not a "procedural state machine pretending to be an AI agent." The 35A3B model at every tier means:

- The **Analyst** genuinely explores source code — it greps, reads files, traces call chains, understands C code. It doesn't follow a fixed pattern; it investigates until it finds evidence or determines evidence doesn't exist.
- The **Lead** makes real judgment calls — skip this REQ because the DCI injection path doesn't exist yet, stall that one because the log evidence is ambiguous, reorder the batch because a device reboot is needed.
- The **Executor** can handle unexpected device responses — a timeout, an error code, a malformed response — and report them structurally rather than failing silently.

The grammar constraints and MCP scoping make the output *structured and safe*, not *dumb*. The model is smart; the constraints ensure its intelligence produces parseable, evidence-backed output rather than free-form prose.

| **Per-generation grammar override** | #3 | Analyst grammar (evidence JSON) vs Reporter grammar (markdown template) vs Executor grammar (command/response) — all on shared 35A3B, switched per delegation. |
| **MCP tool call structured audit log** | #4 | The entire evidence trail IS the audit log. Every pyCommander call, every filesystem read, with timing and params. This is the test output backbone. Elevates from P2 to P1 for the tester — the game can treat it as P2 but the tester cannot function without it. |
| **BISSELL license exception** | License transition | Must be formalized before BISSELL deployment. |

**P2 — Valuable:**

| Requirement | 2.0.0 Feature # | Notes |
|---|---|---|
| **MCP server health/reconnection** | #11 | pyCommander may crash mid-campaign, device may disconnect. Tester must recover gracefully, not abort 293 REQs because the device hiccupped on REQ-47. |
| **Hook architecture** | Systematic hooks | `on_post_tool_call` for audit logging integration. `on_error` for device recovery (retry, reboot, skip). |

| 2.0.0 Feature | Why Not |
|---|---|

### VRAM Budget

```
Qwen 3.5 35A3B (Q4):   ~2-3 GB (3B active params)
Router (0.6B Q4):        ~0.5 GB
KV cache + overhead:     ~1.0 GB
─────────────────────────────────
Total:                   ~3.5-4.5 GB

Backend deduplication: all tiers share one loaded 35A3B instance.
No tier swapping needed — identity/grammar/MCP scope changes only.
```

Comfortably fits on a 12 GB card. An 8 GB card would also work at higher quantization.

## Risks & Considerations

- **35A3B source analysis quality.** Can a 3B-active MoE model reliably trace DCI call chains through C source code? Phase 0 validates this. If insufficient, consider Qwen 3.5 32B dense or similar as the Analyst tier (at higher VRAM cost).
- **Device flakiness.** Hardware-in-the-loop is inherently flaky. Lead tier must handle timeouts, retries, and device recovery gracefully. Application-level concern, not engine.
- **68 STALLED requirements.** Won't become automatable just by changing the agent. Underlying device capabilities (error injection, etc.) need to exist first.
- **pyCommander lifecycle.** CI pipeline must manage pyCommander startup/shutdown. Application-level concern.
- **BISSELL licensing exception.** Entropic LGPL + commercial dual license with BISSELL Homecare Inc. perpetual permissive exception must be formalized.
- **Source repo access in CI.** The Analyst needs read access to IoT library and firmware source from the PR branch. CI pipeline must check out correct branches and make them accessible via filesystem MCP.
