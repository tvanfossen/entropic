## Scope

Reorganize the entire test infrastructure into a coherent, maintainable
architecture. After v1.10.0, the project has:

- Conventional test mapping: test file paths mirror source paths
- Requirement traceability via doxygen-guard `@req` tags linked to
  `docs/requirements.yaml`
- doxygen-guard impact analysis for change-driven test selection
- Git diff-based test selection (`scripts/select_tests.py`)
- Model tests and benchmarks merged into a single GPU-required suite
- CI pipeline with test selection for fast PR validation
- Manual integration tests (test1-test6 style) documented but unchanged

This version produces NO new engine code. It reorganizes and enhances the
test infrastructure built across v1.8.0 through v1.9.14.

**What does NOT change:**
- No new public API surface
- No interface header modifications
- No changes to engine behavior
- No new library `.so` files
- Pre-commit still runs unit tests (fast, CPU, no GPU)
- Model tests remain developer-run with checked-in reports
- Manual integration tests remain manual

**Key shift from earlier design:** Custom `@test` annotation tooling is
replaced by doxygen-guard's `@req` tag system. The traceability unit is
`@req REQ-xxx` (linking code to requirements), not `@test path/to/test.cpp`
(linking code to test files). Test selection uses doxygen-guard's impact
analysis to determine which requirements are affected by a change, then
maps requirements to test suites via `docs/requirements.yaml`.

### Phase 2: Requirement Traceability via doxygen-guard

Replace custom `@test` annotation tooling with doxygen-guard's `@req` tag
system. Requirements are defined in `docs/requirements.yaml` (created in
this session). Every public function links to one or more requirements.

#### Annotation Format

```cpp
/**
 * @brief Generate a response from the loaded model.
 * @version 1.8.2
 * @req REQ-INFER-003
 * @req REQ-INFER-005
 * @req REQ-LOOP-006
 */
ENTROPIC_EXPORT entropic_error_t entropic_inference_generate(
    entropic_inference_backend_t backend,
    const char* messages_json,
    const char* params_json,
    char** result_json);
```

**Traceability chain:**
```
Code (@req REQ-INFER-003) в†’ docs/requirements.yaml (REQ-INFER-003)
  в†’ acceptance_criteria в†’ test suite mapping
```

#### doxygen-guard Configuration

`.doxygen-guard.yaml`:
```yaml
validate:
  presence:
    require_doxygen: true
    skip_forward_declarations: true
  version:
    require_present: true
    require_increment_on_change: true
  tags:
    req:
      pattern: "^REQ-[A-Z]+-\\d{3}$"
  exclude:
    - "^extern/"

impact:
  requirements:
    file: docs/requirements.yaml
    format: yaml
  test_mapping:
    - match: "REQ-LOOP-.*"
      suite: "Engine Loop"
      command: "ctest --test-dir build --tests-regex 'agent_engine|context_manager|directive'"
    - match: "REQ-DELEG-.*"
      suite: "Delegation"
      command: "ctest --test-dir build --tests-regex 'delegation'"
    - match: "REQ-TOOL-.*"
      suite: "Tool System"
      command: "ctest --test-dir build --tests-regex 'tool|mcp|permission'"
    - match: "REQ-IDEN-.*"
      suite: "Identity"
      command: "ctest --test-dir build --tests-regex 'identity|prompt'"
    - match: "REQ-INFER-.*"
      suite: "Inference"
      command: "ctest --test-dir build --tests-regex 'inference|backend|adapter'"
    - match: "REQ-CFG-.*"
      suite: "Config"
      command: "ctest --test-dir build --tests-regex 'config|loader|validator'"
    - match: "REQ-STOR-.*"
      suite: "Storage"
      command: "ctest --test-dir build --tests-regex 'storage|session'"
    - match: ".*"
      suite: "Full Regression"
      command: "ctest --test-dir build"
  output:
    format: markdown
```

#### Impact-Driven Test Selection

When a function with `@req REQ-INFER-003` changes, doxygen-guard's impact
analysis reports that REQ-INFER-003 is affected. The test_mapping in the
config maps that to the "Inference" suite. `scripts/select_tests.py`
consumes this to select the right ctest regex.

This replaces the custom `@test` в†’ `test_map.json` pipeline with
doxygen-guard's built-in requirement traceability.

### Phase 3: Git Diff Test Selection

`scripts/select_tests.py` вЂ” given changed files, output minimal test set.

#### Selection Algorithm

Two input sources, merged:

1. **Mirror path mapping** (structural): `src/{lib}/{file}.cpp` в†’
   `tests/unit/{lib}/{file}_test.cpp`
2. **doxygen-guard impact** (semantic): changed function has `@req REQ-xxx`
   в†’ requirements.yaml в†’ test_mapping в†’ ctest regex

The script runs `doxygen-guard impact --staged` (or `--range`) to get
affected requirements, then maps to test suites. Mirror paths provide
fallback when `@req` tags are incomplete.

#### Selection Rules

| Changed file pattern | Selected tests |
|---|---|
| `src/{lib}/{file}.cpp` | Mirror + doxygen-guard impact |
| `include/entropic/{header}.h` | All tests including that header + impact |
| `include/entropic/interfaces/i_*.h` | ALL tests (interface change = full) |
| `tests/unit/**` | The changed test file itself |
| `CMakeLists.txt` | ALL tests |
| `entropic_config.h.in` | ALL tests |
| `docs/**`, `*.md` | No tests |

#### Fallback Behavior

If doxygen-guard is not installed or `@req` tags are sparse, the script
falls back to mirror-path-only selection. Log a warning. Never fail due
to missing doxygen-guard вЂ” degrade gracefully.

### Phase 4: Model Test + Benchmark Merge

Merge model quality tests and performance benchmarks into `tests/model/`.
Both require GPU, both run at minor version bumps, both produce checked-in
reports. One suite, one gate.

`ENTROPIC_BUILD_MODEL_TESTS=OFF` by default. Developer enables for
minor version validation.

Report format: unified JSON in `test-reports/model/` with quality +
benchmark results per version.

---

### Phase 5: CI Pipeline Integration

PR builds run selected tests only. Full suite on merge/nightly.

Safety triggers forcing full runs:
- Interface header changes
- CMake/build system changes
- Test selector script changes
- Merge to main or develop

## Implementation Notes (from review)

1. **@req annotation effort is incremental.** Retroactively annotating 50+
   functions is a one-time cost. doxygen-guard tracks unmapped functions.
   The CI check starts as WARNING (v1.10.0), promoted to ERROR in v1.10.1.
   Annotate as you touch files, close gaps systematically.

2. **~25 missing test scenarios from v1.8.x (from 2026-03-20 audit).**
   Test files planned in proposals but never written during implementation:
   - v1.8.1: test_yaml_util.cpp (8 scenarios), test_env_overrides.cpp (4 scenarios)
   - v1.8.3: 3 config integration tests (parse_prompt_cache_config, defaults, disabled)
   - v1.8.4: test_response_generator.cpp (unwritten)
   These should be written as part of the test reorganization, not as
   separate backfill work. The reorganized test structure will place them
   in the correct locations.
