## Scope

Behavioral validation of the C++ engine with a live model. 10 BDD-style
tests that each exercise a full vertical slice through the engine stack.
These are the "it's working!" tests — they prove the engine produces
correct behavior end-to-end, not just that individual components pass
unit tests.

The 42 Python model tests (deleted in v1.9.15) were a kitchen-sink
approach that took 7 minutes. Most of what they tested is now covered
by 653+ unit tests with mocks and the v1.10.1 regression suite. What's
missing is the vertical slice: does the entire stack actually work when
you plug in a real model?

**Design principle:** One test per engine capability. Each test answers
exactly one question: "does this capability produce correct output with
a live model?"

**Run model:** Manual, developer-run. NOT a pre-commit hook. Results
checked into `test-reports/model/`. CI validates that checked-in results
exist and are not stale — if the developer has not run model tests for
the current version, CI fails. The staleness check is a CI gate; the
actual model test execution is a manual step.

**Target runtime:** Under 5 minutes on reference hardware (RTX PRO 4000
Blackwell 16GB, 30-40 tok/s with primary model). Each test is 1-3 generation calls, except test 6
(compaction) which is ~8 turns.

**Format:** BDD (Given/When/Then) using Catch2 BDD macros. Each test
reads as a behavioral specification.

**Test boundary:** Model tests validate that real model output works
through the real engine stack. Unit tests (653+) validate component
logic with mocks. Regression tests (v1.10.1) validate engine wiring
with scripted mock responses. These are complementary layers, not
redundant — documented in `tests/README.md`.

## The 10 Tests

### 1. Basic Generation

```
SCENARIO: Engine produces coherent text from a simple prompt
  GIVEN a configured engine with the primary model loaded
  WHEN I send "What is 2+2?"
  THEN the response contains "4"
  AND the response is coherent English (no garbage tokens, no special tokens)
  AND streaming delivered tokens incrementally
  AND finish_reason is "stop"
  AND entropic_throughput_tok_per_sec() > 0.0
```

Validates: model loading, prompt assembly, generation, streaming, adapter
output cleaning, throughput tracker population.

### 2. Tool Call Parsing

```
SCENARIO: Model output is correctly parsed into tool calls
  GIVEN a configured engine with filesystem tools available
  AND a sandboxed temp directory as filesystem root (allow_outside_root: false)
  AND a test fixture file in the sandbox
  WHEN I send "Read the file test.txt"
  THEN the model emits a tool_call for filesystem.read_file
  AND the tool call arguments contain the fixture path
  AND after tool execution, the model incorporates the result
  AND entropic_audit_count() > 0
```

Validates: tool schema injection in prompt, model tool call generation,
adapter parsing, tool execution, result injection, continuation, audit
trail with real generation artifacts.

### 3. Tier Routing

```
SCENARIO: Classification routes to the correct identity
  GIVEN a configured engine with lead + eng + qa identities
  WHEN I send "Write a Python function that sorts a list"
  THEN the router classifies to the "eng" tier
  AND the response uses the eng identity's system prompt
```

Validates: router model `complete()` call (not `generate()`),
classification prompt assembly, digit -> tier mapping.

### 4. Delegation

```
SCENARIO: Lead delegates to child and receives result
  GIVEN a configured engine with delegation enabled
  WHEN I send a task that triggers delegation to eng
  THEN the lead identity calls entropic.delegate
  AND the eng identity executes in a child context
  AND the result propagates back to the lead
```

Validates: delegation dispatch, child loop execution, locked_tier,
result return, depth tracking.

### 5. Grammar Constraint

```
SCENARIO: GBNF grammar constrains output to valid JSON
  GIVEN a configured engine with a JSON grammar registered
  WHEN I generate with grammar_key set
  THEN the output is valid JSON parseable by json::parse
  AND the JSON matches the grammar's structure
```

Validates: grammar registry resolution, GBNF application in llama.cpp,
output constraint enforcement.

### 6. Context Compaction

```
SCENARIO: Conversation compacts without breaking coherence
  GIVEN a configured engine with 8K context window
  WHEN I send 8 turns of conversation that approach the window limit
  THEN compaction triggers at least once
  AND the model's response after compaction references earlier context
  AND no crash, no corruption in message history
```

Validates: context tracking, compaction trigger, value-density strategy,
post-compaction generation continuity. Uses 8K context (realistic for
constrained consumers) with enough turns to trigger compaction without
dominating suite runtime.

### 7. Logprob Evaluation

```
SCENARIO: Evaluation produces valid log-probabilities for a known sequence
  GIVEN an ACTIVE model
  AND a tokenized sequence of natural English text
  WHEN entropic_get_logprobs() is called
  THEN result contains n_tokens - 1 logprob values
  AND all logprobs are negative (log-probabilities <= 0)
  AND perplexity is in a sane range (1.0 < ppl < 100.0 for natural text)
  AND perplexity matches exp(-mean(logprobs)) within float tolerance
```

Validates: `evaluate_logprobs()` end-to-end with real model, temporary
seq_id lifecycle, logit extraction, log-softmax computation. This API
surface is fundamentally untestable with mocks — logprob values only
come from real `llama_decode`.

### 8. Dynamic Identity + Routing

```
SCENARIO: Runtime-created identity participates in routing
  GIVEN a configured engine with static identities (lead, eng)
  WHEN I create a dynamic identity "npc_guard" with focus ["security", "guard duties"]
  AND I send "Who goes there? State your business."
  THEN the router classifies to "npc_guard"
  AND the response uses the npc_guard system prompt
  AND after destroying npc_guard, routing falls back to a static identity
```

Validates: v1.9.6 dynamic identity creation, router dirty flag,
classification prompt rebuild with new identity, identity destruction
cleanup. This is the game NPC's defining workflow.

### 9. GPU Profile / Constrained Generation

```
SCENARIO: Generation succeeds with non-default GPU resource profile
  GIVEN a configured engine with the "minimal" profile active (n_batch=64, n_threads=2)
  WHEN I send a simple prompt
  THEN the response is coherent
  AND generation completes (may be slower than default)
  AND the profile was applied (verified via backend info or timing difference)
```

Validates: `ProfileRegistry` resolution, `set_batch_size()`/`set_threads()`
application in LlamaCppBackend, generation under constrained resources.
Proves the engine works on resource-limited hardware, not just the
developer's full-power GPU.

### 10. MCP Authorization Denial

```
SCENARIO: Identity with READ-only keys is denied WRITE tool access
  GIVEN a configured engine with identity "reader" having mcp_keys: [{"pattern": "filesystem.*", "level": "READ"}]
  AND filesystem.write_file is a WRITE tool
  WHEN the model generates a write_file tool call
  THEN the tool call is denied by the MCP authorization layer
  AND the denial message includes the tool name and required access level
  AND the model receives the denial and adjusts (does not retry the same call)
```

Validates: v1.9.4 MCPKeySet per-identity enforcement with real model
output, denial message feedback to model, model recovery from denied
tool call. This is the security boundary for game NPCs and homeschool
students.

## Test Harness Design

### VRAM Monitoring

The test harness queries `BackendInfo::vram_bytes` (v1.9.13) before and
after each test. If the delta exceeds 64MB (CUDA allocator granularity),
the test logs a WARNING. This catches VRAM leaks that compound across
the 10-test suite.

### Filesystem Sandbox

Test 2 (tool call parsing) uses a dedicated temp directory as `root_dir_`
with `allow_outside_root: false`. Test fixture files are created inside
the sandbox. The live model cannot read arbitrary files on the developer's
machine.

### Throughput Assertion

Test 1 asserts `entropic_throughput_tok_per_sec() > 0.0` after generation.
Zero-cost assertion that validates the EWMA tracker works with real
inference, not just mocked samples.

### Audit Trail Verification

Test 2 asserts `entropic_audit_count() > 0` after tool execution,
validating that the audit logger captures real generation + tool
execution artifacts (not just mocked entries).

### Flakiness Strategy

LLM output is non-deterministic. Even with `temperature: 0.0`, quantized
models can produce slightly different outputs across runs due to
floating-point non-determinism.

**Defaults (easily tunable per test):**
- `temperature: 0.0` for all model tests (minimize sampling variance)
- `seed: 42` fixed seed where the backend supports it
- Each test gets **2 retries** on failure before reporting FAIL. A test
  that passes on retry is logged as FLAKY in the results JSON but counts
  as PASS for the CI gate.
- Assertions are **behavioral, not exact-match**: "contains '4'" not
  "equals 'The answer is 4.'" Range checks and substring checks preferred
  over equality.
- Tests that are inherently flaky (routing edge cases, borderline
  classification) should document an **expected pass rate** (e.g.,
  ">= 8/10 runs"). If a test drops below its expected rate across 3
  consecutive version runs, it is flagged for prompt/assertion tuning.

**Results JSON flaky tracking:**
```json
{"name": "tier_routing", "status": "pass", "retries": 1, "duration_ms": 5200}
```

A `retries > 0` field indicates the test needed a retry. Over time, tests
with persistent `retries > 0` are candidates for prompt redesign or
looser assertions.

### Model Test Entry Script

`scripts/model-test-cache-check.sh` exists but hashes deleted Python
files. Updated for C++ sources:
- Hash: `src/`, `include/`, `tests/model/`, `CMakeLists.txt`
- Output: `test-reports/model-tests.flag` (SKIP if unchanged, RUN:<hash>
  if sources changed since last run)
- The staleness check in CI validates version + git_sha in results.json.
  The cache-check script is a **local developer optimization** — skip the
  GPU-heavy test run if nothing changed. Separate concern from the CI gate.

## Test Gating Model

Model tests are NOT a pre-commit hook. They run too slow and require GPU.

```
Developer workflow:
  1. Developer runs: cmake --preset dev -DENTROPIC_BUILD_MODEL_TESTS=ON
  2. Developer runs: ctest -L model
  3. Results written to: test-reports/model/results.json
  4. Developer commits the results file

CI workflow:
  1. CI reads test-reports/model/results.json
  2. CI extracts version field, compares to CMakeLists.txt project version
  3. CI extracts git_sha field, compares to HEAD
  4. If stale or missing: CI FAILS with "Model tests not run for this version"
  5. If current: CI PASSES (trusts the committed results)
```

The staleness check is a shell command in the CI YAML — no separate
Python script:

```yaml
- name: Model test freshness
  run: |
    VERSION=$(sed -n 's/.*VERSION \([0-9.]*\).*/\1/p' CMakeLists.txt)
    SHA=$(git rev-parse HEAD)
    jq -e --arg v "$VERSION" --arg s "$SHA" \
      '.version == $v and .git_sha == $s' \
      test-reports/model/results.json || \
      (echo "Model tests stale or missing for v${VERSION}" && exit 1)
```

### Results File Format

Single file: `test-reports/model/results.json`. Overwritten each run.
Git history provides the version trail.

```json
{
  "schema_version": 1,
  "version": "1.10.2",
  "git_sha": "abc123def456",
  "timestamp": "2026-04-01T12:00:00Z",
  "model": "Qwen3.5-35B-A3B-UD-IQ3_XXS",
  "gpu": "RTX PRO 4000 Blackwell 16GB",
  "duration_ms": 180000,
  "tests": [
    {"name": "basic_generation", "status": "pass", "retries": 0, "duration_ms": 8200},
    {"name": "tool_call_parsing", "status": "pass", "retries": 0, "duration_ms": 15300}
  ],
  "summary": {
    "total": 10,
    "passed": 10,
    "failed": 0,
    "flaky": 0
  }
}
```
