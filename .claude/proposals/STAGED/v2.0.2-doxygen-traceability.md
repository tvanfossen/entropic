## Scope

Three workstreams that share a dependency on doxygen-guard stability:

1. **doxygen-guard adoption:** Review the latest doxygen-guard release,
   understand all current features and config schema, update the
   project's hook version and `.doxygen-guard.yaml` to use the full
   feature set. This is NOT an incremental config tweak — it is a
   complete review and fresh implementation against whatever the tool
   looks like when this proposal activates.

2. **Full documentation review:** Audit doxygen comments on EVERY
   function in the codebase — public headers, internal headers, and
   source files. Fix stale descriptions, missing parameters, incorrect
   return docs, missing `@version` tags. This is a comprehensive
   documentation pass, not limited to public API surface.

3. **Requirement traceability:** Add `@req REQ-xxx` tags to public
   functions, link to `docs/requirements.yaml`, configure doxygen-guard
   impact analysis, and integrate into `scripts/select_tests.py`.

4. **Test selection:** Create `scripts/select_tests.py` with both
   mirror-path mapping and doxygen-guard impact analysis. Update CI
   pipeline to use selective test runs for PRs. Until this proposal
   ships, all builds run the full test suite.

After v1.10.2, the project has:
- doxygen-guard running at its latest stable revision with full config
- Accurate, verified doxygen comments on every function in the codebase
- `@req` tags on every public function linking code to requirements
- Semantic test selection via doxygen-guard impact analysis
- Git diff-based `scripts/select_tests.py` for CI PR builds
- `@req` enforcement at WARNING (promoted to ERROR in v1.10.1)

**Dependency:** doxygen-guard must reach a stable release before this
work begins. The implementation plan below describes the SHAPE of the
work — specific config keys, CLI flags, and validation rules will be
determined by reading the actual doxygen-guard documentation at session
start.

**What does NOT change:**
- No engine code changes (documentation and config only)
- No new public API surface
- No test logic changes (select_tests.py adds a second input source)

**Relationship to v1.10.1:** Prerequisite for v1.10.1 Phase 4 (`@req`
enforcement promotion to ERROR). v1.10.1 cannot promote to ERROR until
the annotations and doxygen-guard config exist.

## Implementation Plan

### Phase 1: doxygen-guard Review & Implementation

**This phase cannot be pre-planned in detail.** doxygen-guard is
actively changing. The session must start by reading the tool's current
documentation, changelog, and config schema.

Steps:
1. Read doxygen-guard README, changelog, and config reference
2. Inventory current features: validation rules, tag types, impact
   analysis, CLI commands, pre-commit hook options
3. Compare against current `.doxygen-guard.yaml` — identify gaps and
   deprecated config
4. Update `.pre-commit-config.yaml` to latest doxygen-guard revision
5. Rewrite `.doxygen-guard.yaml` from scratch against current schema
   (do not patch — fresh config based on current capabilities)
6. Verify pre-commit hook passes on clean repo
7. Document any doxygen-guard features NOT adopted and why (in this
   proposal's implementation log)

### Phase 2: Full Documentation Review

Audit doxygen comments on every function in the codebase. This is a
file-by-file pass, not a spot check.

**Scope (all of these, not just public):**
- `include/entropic/entropic.h`
- `include/entropic/interfaces/*.h`
- `include/entropic/types/*.h`
- `include/entropic/core/*.h`
- `include/entropic/inference/*.h` and `inference/adapters/*.h`
- `include/entropic/mcp/*.h` and `mcp/servers/*.h`
- `include/entropic/config/*.h`
- `include/entropic/prompts/*.h`
- `include/entropic/storage/*.h`
- `src/**/*.cpp` — implementation files
- `src/**/*.h` — internal headers

**Per-function checklist:**
1. `@brief` present and accurate (matches current behavior, not stale
   from a prior version's design)
2. `@param` for every parameter (name matches current signature)
3. `@return` describes actual return semantics
4. `@version` present and reflects the last version that modified the
   function body
5. Role tag (`@internal`, `@utility`, `@callback`, or whatever
   doxygen-guard requires at that point) present
6. No copy-paste artifacts from templated doxygen blocks
7. Cross-references (`@see`) accurate if present

**How to handle volume:** This is ~200+ functions across ~80 files.
Work library-by-library (types → config → core → inference → mcp →
storage → facade). Commit after each library. Do not attempt all
libraries in one pass.

### Phase 3: @req Annotation

Add `@req REQ-xxx` tags to every public function, linking code to
`docs/requirements.yaml`.

**Priority order:**
1. `entropic.h` — public C API (consumer-facing)
2. `interfaces/i_inference_backend.h`
3. `interfaces/i_mcp_server.h`
4. `interfaces/i_config_loader.h`
5. `interfaces/i_storage_backend.h`
6. `interfaces/i_hook_handler.h`
7. `interfaces/i_inference_callbacks.h`

**Gap analysis:** If any public function cannot map to an existing REQ,
either:
- The function is undocumented in requirements → add a new REQ to
  `docs/requirements.yaml`
- The function should not be public → flag for architect review

### Phase 4: Impact Analysis & Test Selection

Configure doxygen-guard's impact analysis (if the feature exists in the
stable release) and integrate into `scripts/select_tests.py`.

**If impact analysis is available:**
1. Configure `impact.test_mapping` in `.doxygen-guard.yaml` linking
   REQ patterns to ctest suites
2. Update `select_tests.py` to run doxygen-guard impact as a second
   input source alongside mirror-path mapping
3. Merge results (union of both sources)
4. Graceful fallback if doxygen-guard not installed

**If impact analysis is NOT available in the stable release:**
- Skip this phase
- Document in implementation log
- Mirror-path-only selection remains sufficient
- Impact integration becomes a future v1.10.x task

## Implementation Log

### 2026-04-13 — Phase 1: doxygen-guard review & implementation

- **Hook pin:** `.pre-commit-config.yaml` bumped from v0.2.0 → v1.2.4
  (via v1.2.1/1.2.2/1.2.3 iteration — upstream fixes for namespace
  parsing, `@par`/`@throws` recognition, and constructor exemption were
  surfaced and shipped before this proposal landed).
- **Config rewrite:** `.doxygen-guard.yaml` replaced against v1.2.4
  schema — **validation + impact report only; no trace/sequence
  diagrams** (entropic is a library engine, not event-driven embedded).
  Excludes cover `extern/`, `tests/`, `build/`, `.build/`, `.old/`,
  `.venv/`. `files` regex expanded to include `.py`.
- **Upstream false-positive resolution (four categories fixed by user's
  doxygen-guard releases):**
  - `namespace A::B {` parsed as function (v1.2.2)
  - `@par` and `@throws` flagged as unknown doxygen tags (v1.2.3/4)
  - Explicit struct/class constructors incorrectly required `@return`
    and `@req` (v1.2.4)
- **One remaining edge case (not blocking):** `class ENTROPIC_EXPORT
  BundledModels` still parsed as an implicit constructor — worked
  around by adding `@return` / `@req REQ-CFG-003` to the class-level
  doxygen block. Single site. Candidate for upstream v1.2.5.
