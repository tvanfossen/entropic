## Scope

External MCP server support: connecting to out-of-process MCP servers via
stdio and SSE transports. After v1.8.7, the engine can discover, connect to,
health-check, and route tool calls to third-party MCP servers running outside
the engine process. Servers can be registered at startup (via `.mcp.json` or
YAML config) or at runtime via the C API.

This is a faithful port of:
- `src/entropic/mcp/client.py` -- `MCPClient`: stdio/SSE transport, connect/disconnect, tool list caching, tool execution with timeout, directive extraction
- `src/entropic/mcp/manager.py` -- `ServerManager`: `_load_mcp_json_servers`, `_parse_mcp_json`, `_client_from_mcp_json`, `_sse_client_from_cfg`, `_stdio_client_from_cfg`, `connect_server`, `disconnect_server`, `list_servers`, `ServerInfo`
- `src/entropic/mcp/servers/external.py` -- `compute_socket_path` (self-detection)
- `.mcp.json` -- discovery format (project + global)

v1.8.5 established the in-process tool system (`MCPServerBase`, `ServerManager`,
`ToolExecutor`, permission checking, duplicate detection). v1.8.7 extends
`ServerManager` with external server connection capabilities. Internal tool
routing, permissions, and duplicate detection apply uniformly to both in-process
and external servers.

**What this version does NOT include:**
- In-process MCP servers (v1.8.5) -- already done
- Delegation/identity (v1.8.6) -- separate concern
- Session persistence (v1.8.8) -- SQLite storage
- Per-identity MCP authorization (v1.9.4) -- per-caller key sets

**What this version adds beyond the Python port:**
- Configurable reconnection with exponential backoff (Python has no reconnection)
- Server health detection via periodic ping / connection state tracking
- Dynamic tool registration refresh (server declares new tools mid-session)
- Structured error for tool calls routed to disconnected servers
- C API: `entropic_register_mcp_server()` for runtime registration

## Architecture

### Transport Abstraction

External servers communicate via one of two transports. Both implement a
common `Transport` interface internal to `librentropic-mcp.so`:

```
ExternalMCPClient
    │
    ├── StdioTransport    fork/exec child, pipe stdin/stdout
    │   └── JSON-RPC 2.0 over newline-delimited JSON
    │
    └── SSETransport      HTTP GET text/event-stream (cpp-httplib)
        └── JSON-RPC 2.0 over SSE events
```

Both transports implement the MCP protocol: `initialize`, `tools/list`,
`tools/call`. The client negotiates capabilities during `initialize`.

### Stdio Transport

```
┌──────────────┐  stdin   ┌───────────────────┐
│ Entropic     │ ──────>  │ External Server    │
│ (parent)     │  stdout  │ (child process)    │
│              │ <──────  │                    │
└──────────────┘          └───────────────────┘
```

- `fork()`/`exec()` (or `posix_spawn()`) spawns the child process
- Environment variables from `.mcp.json` `env` field merged with inherited env
- stdin/stdout pipes carry newline-delimited JSON-RPC messages
- stderr captured to spdlog at WARNING level
- Child process lifetime tied to transport lifetime (SIGTERM on disconnect)

### SSE Transport

```
┌──────────────┐  GET /sse        ┌───────────────────┐
│ Entropic     │ ──────────────>  │ External Server    │
│ (client)     │  text/event-stream │ (HTTP server)    │
│              │ <────────────────  │                    │
│              │  POST /message    │                    │
│              │ ──────────────>  │                    │
└──────────────┘                  └───────────────────┘
```

- cpp-httplib `Client::Get()` with content receiver for SSE streaming
- Server sends events with `data:` fields containing JSON-RPC responses
- Client sends JSON-RPC requests via HTTP POST to the message endpoint
- SSE endpoint URL from `.mcp.json` `url` field

### .mcp.json Discovery

Two locations searched in order (first match wins per server name):

1. `{project_dir}/.mcp.json` -- project-level
2. `~/.entropic/.mcp.json` -- global fallback

Format:
```json
{
  "mcpServers": {
    "name": {
      "command": "npx",
      "args": ["-y", "@some/mcp-server"],
      "env": {"API_KEY": "..."}
    },
    "remote-server": {
      "type": "sse",
      "url": "http://127.0.0.1:6277/sse"
    }
  }
}
```

Transport inference: `url` present -> SSE, `command` present -> stdio.
Explicit `type` field overrides inference.

**Self-detection:** compute SHA-256 of project directory path, compare against
socket entries. Skip any entry whose socket path matches the engine's own
socket (prevents Entropic connecting to itself).

**Shadow warning:** if a `.mcp.json` entry name matches an already-registered
server (from YAML config or built-in), log WARNING and skip.

### ServerInfo

Runtime information about a connected external server:

```cpp
/// @brief Runtime state of a connected MCP server.
/// @version 1.8.7
struct ServerInfo {
    std::string name;          ///< Server name (unique key)
    std::string transport;     ///< "stdio" | "sse"
    std::string url;           ///< SSE URL (empty for stdio)
    std::string command;       ///< stdio command (empty for SSE)
    std::string status;        ///< "connected" | "disconnected" | "error" | "reconnecting"
    std::vector<std::string> tools;  ///< Registered tool names
    std::string source;        ///< "config" | "mcp_json" | "runtime"
    std::chrono::system_clock::time_point connected_at;  ///< Connection timestamp
    int reconnect_attempts;    ///< Current reconnect attempt count
};
```

### Reconnection with Exponential Backoff

When an external server connection drops (process exits, SSE stream closes,
tool call fails with transport error):

```
ReconnectPolicy:
    base_delay     = 1s    (configurable)
    max_delay      = 60s   (configurable)
    max_retries    = 5     (configurable, 0 = infinite)
    backoff_factor = 2.0

    delay(attempt) = min(base_delay * backoff_factor^attempt, max_delay)
    + jitter(0, delay * 0.1)

    AFTER max_retries:
        status = "error"
        log ERROR with server name + last error
        stop reconnecting
        tool calls return structured error
```

Reconnection runs on a background thread. `ServerInfo.status` transitions:
`connected` -> `disconnected` -> `reconnecting` -> `connected` | `error`.

### Health Detection

Two mechanisms:

1. **Connection state tracking**: Transport reports connection loss immediately
   (stdio: child process exit via `waitpid` / SIGCHLD, SSE: stream close /
   HTTP error). Triggers reconnection.

2. **Periodic ping** (optional, configurable): Send `ping` JSON-RPC method
   at configurable interval (default: disabled, `health_check_interval: 0`).
   If no response within timeout, mark as disconnected and trigger reconnect.

### Dynamic Tool Registration Refresh

After reconnection (or on explicit `refresh_tools()` call), the client
re-issues `tools/list` and diffs against the cached tool set. New tools
are registered with `ServerManager`, removed tools are deregistered.

```
ExternalMCPClient::refresh_tools():
    new_tools = send("tools/list")
    added = new_tools - cached_tools
    removed = cached_tools - new_tools
    FOR each added: register with ServerManager
    FOR each removed: deregister from ServerManager
    cached_tools = new_tools
    log INFO: "Server '{name}' tools refreshed: +{added} -{removed}"
```

This handles the case where a server restarts with a different tool set.

### Tool Call Routing to Disconnected Servers

When a tool call routes to a server with `status != "connected"`:

```cpp
/// @brief Structured error for disconnected server tool calls.
/// @version 1.8.7
///
/// Returned as a ToolResult with is_error=true. Includes enough
/// context for the model to understand the failure and adapt.
ServerResponse {
    result: "Server 'name' is disconnected (status: error, "
            "last connected: 2026-03-17T10:00:00Z). "
            "Tool 'server.tool_name' is unavailable. "
            "Use a different approach or try again later.",
    directives: []
}
```

The model receives a clear error message. No crash, no hang, no silent failure.

### Integration with ServerManager (v1.8.5)

`ServerManager` from v1.8.5 manages both in-process and external servers
uniformly. External servers are wrapped in `ExternalMCPClient` which
implements the same interface used by `InProcessProvider` (both duck-type
to the same `ToolProvider` pattern).

```
ServerManager._clients:
    "filesystem" -> InProcessProvider (v1.8.5)
    "bash"       -> InProcessProvider (v1.8.5)
    "pycommander" -> ExternalMCPClient (v1.8.7, from .mcp.json)
    "custom-api"  -> ExternalMCPClient (v1.8.7, runtime-registered)
```

Tool routing remains: `tool_name.split(".")[0]` extracts the server prefix.
Permissions, duplicate detection, tier allowlist -- all apply unchanged.

## Implementation Plan

### Phase 1: Transport Interface + StdioTransport

Define the internal transport abstraction and implement stdio transport.

```cpp
/// @brief Abstract transport for external MCP communication.
/// @version 1.8.7
///
/// Internal to librentropic-mcp.so. Not exposed across .so boundaries.
/// Implementations handle the wire protocol; ExternalMCPClient handles
/// the MCP protocol logic on top.
class Transport {
public:
    virtual ~Transport() = default;

    /// @brief Open the transport connection.
    /// @return true on success.
    /// @version 1.8.7
    virtual bool open() = 0;

    /// @brief Close the transport connection.
    /// @version 1.8.7
    virtual void close() = 0;

    /// @brief Send a JSON-RPC request and wait for response.
    /// @param request_json JSON-RPC request string.
    /// @param timeout_ms Timeout in milliseconds (0 = default).
    /// @return JSON-RPC response string, or empty on error.
    /// @version 1.8.7
    virtual std::string send_request(
        const std::string& request_json,
        uint32_t timeout_ms = 0) = 0;

    /// @brief Check if transport is connected.
    /// @return true if connected and ready.
    /// @version 1.8.7
    virtual bool is_connected() const = 0;
};
```

`StdioTransport`:
- `open()`: `posix_spawn()` child process with stdin/stdout pipes
- `close()`: send SIGTERM, `waitpid()` with timeout, SIGKILL if needed
- `send_request()`: write to stdin pipe, read response from stdout pipe
  (newline-delimited JSON)
- `is_connected()`: check child process alive via `kill(pid, 0)`
- Stderr pipe read on a separate thread, forwarded to spdlog WARNING
- Environment variables merged: parent env + `.mcp.json` env overrides

### Phase 2: SSETransport

SSE transport using cpp-httplib.

```cpp
/// @brief SSE transport for external MCP servers.
/// @version 1.8.7
///
/// Uses cpp-httplib Client::Get() with content receiver for SSE stream.
/// Requests sent via POST to the server's message endpoint.
class SSETransport : public Transport {
public:
    /// @brief Construct with SSE endpoint URL.
    /// @param url SSE endpoint URL (e.g., "http://127.0.0.1:6277/sse").
    /// @version 1.8.7
    explicit SSETransport(const std::string& url);

    bool open() override;
    void close() override;
    std::string send_request(
        const std::string& request_json,
        uint32_t timeout_ms = 0) override;
    bool is_connected() const override;

private:
    std::string url_;                    ///< SSE endpoint URL
    std::string message_endpoint_;       ///< POST endpoint for requests
    std::unique_ptr<httplib::Client> client_;
    std::atomic<bool> connected_{false};
    std::thread sse_reader_thread_;      ///< Background SSE event reader
    std::mutex pending_mutex_;
    std::condition_variable pending_cv_;
    std::map<int, std::string> pending_responses_; ///< request_id -> response
    int next_request_id_{1};
};
```

SSE event handling:
- Background thread reads SSE stream via content receiver callback
- Events with `data:` field parsed as JSON-RPC responses
- Responses matched to pending requests by `id` field
- `send_request()` POSTs request, waits on condition variable for response

### Phase 4: ReconnectPolicy

Exponential backoff with jitter for automatic reconnection.

```cpp
/// @brief Exponential backoff reconnection policy.
/// @version 1.8.7
///
/// Computes delay between reconnection attempts. Thread-safe.
/// Used by HealthMonitor to schedule reconnection.
class ReconnectPolicy {
public:
    /// @brief Construct with backoff parameters.
    /// @param base_delay_ms Base delay in milliseconds.
    /// @param max_delay_ms Maximum delay cap.
    /// @param max_retries Maximum attempts (0 = infinite).
    /// @param backoff_factor Multiplicative factor per attempt.
    /// @version 1.8.7
    ReconnectPolicy(uint32_t base_delay_ms = 1000,
                    uint32_t max_delay_ms = 60000,
                    uint32_t max_retries = 5,
                    double backoff_factor = 2.0);

    /// @brief Compute delay for the given attempt number.
    /// @param attempt Zero-based attempt number.
    /// @return Delay in milliseconds (with jitter), or 0 if max_retries exceeded.
    /// @version 1.8.7
    uint32_t delay_ms(uint32_t attempt) const;

    /// @brief Check if attempt count has exceeded max_retries.
    /// @param attempt Zero-based attempt number.
    /// @return true if retries exhausted.
    /// @version 1.8.7
    bool exhausted(uint32_t attempt) const;

    /// @brief Reset for a new reconnection cycle.
    /// @version 1.8.7
    void reset();

private:
    uint32_t base_delay_ms_;
    uint32_t max_delay_ms_;
    uint32_t max_retries_;
    double backoff_factor_;
};
```

### Phase 5: HealthMonitor

Background thread that monitors external server connections and triggers
reconnection when connections drop.

```cpp
/// @brief Monitors external server health and triggers reconnection.
/// @version 1.8.7
///
/// Runs a background thread that:
/// 1. Detects connection loss (transport reports disconnected)
/// 2. Schedules reconnection with exponential backoff
/// 3. Optionally sends periodic ping for health checking
/// 4. Refreshes tool list after successful reconnection
///
/// **Thread model:** The monitor thread does NOT invoke callbacks directly.
/// Status changes are posted to a thread-safe event queue. The engine's
/// main thread drains this queue during each loop iteration (via
/// ServerManager::process_health_events()). This avoids cross-thread
/// mutation of engine state. The StatusCallback is invoked on the
/// engine thread, not the monitor thread.
class HealthMonitor {
public:
    /// @brief Callback invoked when server status changes.
    /// @version 1.8.7
    using StatusCallback = std::function<void(
        const std::string& server_name,
        const std::string& old_status,
        const std::string& new_status)>;

    /// @brief Construct with reconnection policy.
    /// @param policy Reconnection backoff policy.
    /// @param health_check_interval_ms Ping interval (0 = disabled).
    /// @version 1.8.7
    HealthMonitor(const ReconnectPolicy& policy,
                  uint32_t health_check_interval_ms = 0);

    /// @brief Start monitoring a server.
    /// @param name Server name.
    /// @param client ExternalMCPClient to monitor (non-owning).
    /// @version 1.8.7
    void watch(const std::string& name, ExternalMCPClient* client);

    /// @brief Stop monitoring a server.
    /// @param name Server name.
    /// @version 1.8.7
    void unwatch(const std::string& name);

    /// @brief Set callback for status changes.
    /// @version 1.8.7
    void set_status_callback(StatusCallback cb);

    /// @brief Start the monitoring thread.
    /// @version 1.8.7
    void start();

    /// @brief Stop the monitoring thread and all reconnection attempts.
    /// @version 1.8.7
    void stop();

private:
    ReconnectPolicy policy_;
    uint32_t health_check_interval_ms_;
    StatusCallback status_callback_;

    struct WatchEntry {
        ExternalMCPClient* client;       ///< Non-owning pointer
        std::string status;              ///< Current status
        uint32_t reconnect_attempt;      ///< Current attempt counter
        std::chrono::steady_clock::time_point next_check;
    };

    std::map<std::string, WatchEntry> watched_;
    std::mutex watched_mutex_;
    std::thread monitor_thread_;
    std::atomic<bool> running_{false};
    std::condition_variable wake_cv_;    ///< Wake monitor for immediate check

    /// @brief Monitor loop (runs on background thread).
    /// @version 1.8.7
    void monitor_loop();

    /// @brief Attempt reconnection for a single server.
    /// @param name Server name.
    /// @param entry Watch entry (mutated on success/failure).
    /// @version 1.8.7
    void attempt_reconnect(const std::string& name, WatchEntry& entry);

    /// @brief Send ping to server and check response.
    /// @param entry Watch entry.
    /// @return true if server responded.
    /// @version 1.8.7
    bool ping_server(WatchEntry& entry);
};
```

## Implementation Notes (from adversarial review)

1. **Strip directives from external server responses.** `ExternalMCPClient::execute()`
   MUST remove the `directives` array from the JSON response before returning it
   to the engine. External servers cannot inject delegate, stop_processing,
   phase_change, prune_context, or context_anchor directives. This is a critical
   security boundary (CWE-94).

2. **Env variable blocklist for `.mcp.json`.** Block `LD_PRELOAD`,
   `LD_LIBRARY_PATH`, `PATH`, `DYLD_*`, `HOME`, `SHELL`, and any variable
   prefixed with `ENTROPIC_` from being set via `.mcp.json` env field. A
   malicious `.mcp.json` in a project repo could otherwise alter process
   behavior via environment injection.

3. **SSE transport: warn on non-localhost HTTP.** Log WARNING when connecting
   to `http://` SSE endpoints on non-localhost addresses. Tool call arguments
   (file contents, code) travel in cleartext. Don't force TLS — that breaks
   development. Just make the risk visible.

4. **SSE authentication is a known limitation.** No bearer token or other auth
   mechanism for SSE connections. A process on the same network can impersonate
   an MCP server. Primary deployment is localhost where this is acceptable.
   TODO post-2.0.0: bearer token auth for remote SSE.

5. **JSON depth/size limits on external responses.** Set `nlohmann::json::parse()`
   `max_depth` for all external MCP inputs. Deeply nested or extremely large
   JSON can cause stack overflow via recursive parsing. Enforce a maximum
   response size (e.g., 10MB) at the transport level.

## MVP Checkpoint

**After Phase 3** (StdioTransport + ExternalMCPClient). At this point, external stdio-based MCP servers connect, list their tools, and execute tool calls with timeout handling. The engine can discover servers from `.mcp.json`, spawn their processes, and route tool calls to them through the same ToolExecutor pipeline as built-in servers. SSE transport, health checking, and reconnection (Phases 4-6) add robustness for long-running and network servers but are not required for local stdio server integration.
