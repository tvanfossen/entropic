## Scope

A third reference example (`examples/explorer/`) that serves as:
1. The first thing a new user runs after downloading a model
2. An interactive guide to the entropic engine's architecture
3. Proof that the engine works on their hardware
4. A real multi-tier application demonstrating MCP tools, delegation, grammars, and identity switching
5. Potentially distributable as a standalone compiled binary

Unlike hello-world (minimal, no tools) and pychess (game, grammar focus),
the explorer is an educational tool that teaches users about the engine
they're running on — by using the engine itself.

---

## Architecture

### Knowledge Backend: Doxygen SQLite

The explorer's primary knowledge source is a SQLite database generated
from the engine's own doxygen comments. Since doxygen-guard enforces
`@brief` + `@version` on every function, the database is comprehensive.

```
Source code (with doxygen comments)
    ↓ doxygen --generate-sqlite3 (build step)
entropic_docs.db (SQLite)
    ↓ bundled with explorer
docs MCP server (query tools)
    ↓ registered with engine
Explorer identity queries during conversation
```

When the doxygen database lacks detail, the explorer falls back to
reading source files via a read-only filesystem MCP server scoped
to `include/` and `src/`.

### Identity Pipeline

Eight identities in a delegation hierarchy with four pipeline flows:

```
Lead (front_office, explicit_completion)
 │
 ├── delegate(researcher)           — fact-finding, symbol lookup
 │
 ├── delegate(architect)            — design evaluation, pattern analysis
 │
 ├── pipeline([reviewer, critic])   — adversarial change review
 │     reviewer analyzes → critic challenges findings
 │
 ├── pipeline([teacher, quizzer])   — teach concept → grammar-constrained quiz
 │     teacher explains → auto_chain → quizzer generates question
 │
 └── delegate(tracer)               — execution path tracing across .so boundaries
```

| Identity | Role | Key capability |
|----------|------|---------------|
| lead | front_office | Routes, reviews results, quality gate |
| researcher | back_office | Deep lookup — "what exists" |
| architect | back_office | Design analysis — "is this the right pattern?" |
| reviewer | back_office | Adversarial — "what's wrong" (has git access) |
| critic | back_office | Meta-adversarial — challenges reviewer findings |
| teacher | back_office | Pedagogical — explains concepts, auto_chains to quizzer |
| quizzer | back_office | Grammar-constrained quiz output |
| tracer | back_office | Sequential call-path following across .so boundaries |

### MCP Servers

| Server | Purpose | Tools |
|--------|---------|-------|
| `docs` | Doxygen SQLite queries (custom, external) | `docs.lookup_function`, `docs.lookup_class`, `docs.search`, `docs.list_files`, `docs.get_hierarchy` |
| `filesystem` | Source reading (built-in, scoped) | `filesystem.read_file` (scoped to `include/`, `src/`) |
| `git` | Repository state (built-in) | `git.status`, `git.diff`, `git.log` |
| `entropic` | Delegation + completion (built-in) | `entropic.delegate`, `entropic.pipeline`, `entropic.complete` |

### Constitutional Validation

The constitution defines architectural rules (C at boundaries, three-layer
pattern, complexity gates, doxygen coverage). The `ConstitutionalValidator`
runs as a POST_GENERATE hook on all identity output, enforcing these rules
via grammar-constrained critique and optional revision loops.

### Grammar Constraints

Quizzer output uses GBNF grammar:

```gbnf
root ::= question options answer-key
question ::= "Question: " sentence "\n"
options ::= option-a option-b option-c option-d
option-a ::= "  A) " sentence "\n"
...
answer-key ::= "Correct: " [A-D] "\n"
sentence ::= [A-Z] [^.\n]+ "."
```

## Knowledge Database

### Doxygen SQLite Schema

Doxygen `GENERATE_SQLITE3 = YES` produces a database with tables:

- `memberdef` — functions, variables, typedefs with brief, detailed, file, line
- `compounddef` — classes, structs, namespaces, files
- `contains` — compound → member relationships
- `basecompoundref` — inheritance hierarchy
- `param` — function parameters
- `reimplements` — virtual method overrides

### Custom Enrichment

Beyond raw doxygen output, the explorer needs higher-level architectural
knowledge. This is added as a supplementary table in the same database:

```sql
CREATE TABLE architecture_topics (
    id INTEGER PRIMARY KEY,
    topic TEXT NOT NULL,           -- "inference_backend", "mcp_servers", etc.
    title TEXT NOT NULL,           -- Human-readable title
    summary TEXT NOT NULL,         -- 2-3 sentence overview
    key_files TEXT NOT NULL,       -- JSON array of file paths
    key_concepts TEXT NOT NULL,    -- JSON array of concept names
    prerequisite_topics TEXT,      -- JSON array of topic IDs
    difficulty INTEGER DEFAULT 1   -- 1=beginner, 2=intermediate, 3=advanced
);
```

Populated from a YAML file (`data/architecture_topics.yaml`) during
the doxygen build step. This gives the guide structured curriculum
without depending on the model to know the architecture.

### Build Integration

```bash
# New invoke task
inv docs                    # Run doxygen, generate entropic_docs.db
inv docs --enrich           # Also populate architecture_topics from YAML

# Automatically run during full build
inv build --clean           # Includes doxygen generation
```

## MCP Docs Server

`servers/docs_server.py` — JSON-RPC 2.0 over stdio (same pattern as
`pychess/chess_server.py`):

### Tools

| Tool | Input | Output |
|------|-------|--------|
| `docs.lookup_function` | `{"name": "entropic_create"}` | Brief, params, return, file:line, version |
| `docs.lookup_class` | `{"name": "AgentEngine"}` | Brief, members, inheritance, file |
| `docs.search` | `{"query": "streaming generation"}` | Top 10 matches with context |
| `docs.list_files` | `{"pattern": "src/core/*"}` | Files with brief descriptions |
| `docs.get_hierarchy` | `{"class": "MCPServerBase"}` | Inheritance tree |

All queries run against `entropic_docs.db` via Python `sqlite3` module.
The server is stateless — each query is independent.

## Validation Criteria

1. `inv docs` generates `entropic_docs.db` with >500 documented symbols
2. `inv example -n explorer` starts interactive guide
3. User can ask "explain the inference backend" → analyst researches → structured answer
4. User can say "quiz me" → grammar-constrained question generated → answer validated
5. Filesystem fallback works: analyst reads actual source when docs DB lacks detail
6. Session logs capture the full exploration flow
7. No hallucinated architecture — all claims traceable to doxygen or source
