## Scope

Extract the Textual TUI, voice interface, and all presentation-layer code into a
standalone consumer repo (`entropic-tui`). After this version, the `entropic-engine`
package contains:

- The inference engine (core, inference, mcp, prompts, config, storage)
- The `Presenter` ABC and `HeadlessPresenter` (engine-side API boundary)
- CLI commands: `entropic status`, `entropic ask`, `entropic init`, `entropic download`,
  `entropic setup-cuda`, `entropic benchmark`, `entropic mcp-bridge`
- `LibraryConfig` as the primary config class for external consumers

The `entropic-engine` package does NOT contain:

- `entropic.ui.tui` (Textual app)
- `entropic.ui.tui_presenter` (TUI presenter)
- `entropic.ui.widgets` (Textual widgets)
- `entropic.ui.components` (Rich-based display components)
- `entropic.ui.themes` (theme definitions)
- `entropic.ui.voice_screen` / `entropic.ui.voice_widgets`
- `entropic.ui.__init__` (barrel export of TUI widgets)
- `entropic.ui.entropic.tcss` (Textual stylesheet)
- `entropic.voice.*` (entire voice subsystem)
- `EntropyConfig` (TUI-specific config subclass)
- `data/prompts/app_context_tui.md`

## Implementation Plan

### Step 1: Define the API boundary

The boundary already exists: `Presenter` ABC in `src/entropic/ui/presenter.py`.
This is the contract between `Application` and any UI. `HeadlessPresenter` proves
the engine works without Textual. These two files stay in the engine.

Relocate them out of `entropic.ui` to avoid confusion:

```
src/entropic/ui/presenter.py   -> src/entropic/core/presenter.py
src/entropic/ui/headless.py    -> src/entropic/core/headless_presenter.py
```

Update all imports accordingly. The `entropic.ui` package becomes TUI-only and
moves to `entropic-tui` entirely.

### Step 2: Decouple `Application` from `EntropyConfig`

`Application.__init__` currently requires `EntropyConfig`. Change it to accept
`LibraryConfig`:

```python
# app.py — BEFORE
from entropic.config.schema import EntropyConfig

class Application:
    def __init__(self, config: EntropyConfig, ...):

# app.py — AFTER
from entropic.config.schema import LibraryConfig

class Application:
    def __init__(self, config: LibraryConfig, ...):
```

`Application` only accesses `LibraryConfig` fields directly (models, routing,
generation, permissions, mcp, compaction, log_level, constitution, app_context,
inject_model_context, vram_reserve_mb, config_dir). The TUI-specific fields
(quality, ui, storage, lsp, voice, log_file, commands_dir) are only accessed
by `TUIPresenter` and the `cli.py` `status` command.

**Exceptions requiring refactoring:**

1. `app.py` line 15: `from entropic.config.schema import EntropyConfig` — change to
   `LibraryConfig`.

2. `app.py` line 159: `config.quality.enabled` in the `status` command — this
   reference is in `cli.py`, not `app.py`. The `status` CLI command accesses
   `config.quality.enabled` which is a TUI field. Two options:
   - Move `status` to check `hasattr(config, 'quality')` with a fallback
   - Accept that CLI `status` runs with `EntropyConfig` (loaded by `cli.py`)
     and the engine library users don't use this command

   **Decision:** CLI commands continue to load `EntropyConfig` via `reload_config()`.
   Only `Application` and engine internals use `LibraryConfig`. This is the minimum
   change for separation.

3. `app.py` line 348: lazy import of `TUIPresenter` when no presenter is provided.
   After separation, `entropic-engine` cannot import `TUIPresenter`. Change to
   raise an error if no presenter is provided:

   ```python
   # app.py — AFTER
   if self._presenter is None:
       raise RuntimeError(
           "No presenter provided. Install entropic-tui for the TUI, "
           "or pass a HeadlessPresenter/custom Presenter instance."
       )
   ```

4. `app.py` line 364: `set_voice_callbacks` — voice callbacks are optional on the
   `Presenter` ABC (default no-op). No change needed; the call is safe regardless
   of presenter implementation.

### Step 3: Decouple `ConfigLoader` from `EntropyConfig`

`ConfigLoader.load()` currently returns `EntropyConfig`. Add a `config_class`
parameter defaulting to `EntropyConfig` for backward compatibility, but allowing
library consumers to request `LibraryConfig`:

```python
class ConfigLoader:
    def __init__(self, ..., config_class: type[LibraryConfig] = EntropyConfig):
        self._config_class = config_class

    def load(self, ...) -> LibraryConfig:
        ...
        return self._config_class(**config)
```

The `cli.py` entry point continues to use the default (`EntropyConfig`). Library
consumers pass `config_class=LibraryConfig`.

### Step 4: Remove `rich` imports from `app.py`

`app.py` uses `rich.console.Console` directly for:
- Loading spinner (`rich.status.Status`) in `initialize()`
- `single_turn()` output formatting

Replace with logger-based feedback or move these to a thin console helper in
`entropic.core`. The `Console` in `single_turn` can use Python's built-in `print()`
or a lightweight callback approach consistent with the `Presenter` pattern.

**Approach:** Create `src/entropic/core/console.py` with a minimal `EngineConsole`
that wraps `print()` for headless use and can be swapped for `rich.Console` when
the `[tui]` extra is installed. `app.py` uses `EngineConsole` only.

### Step 6: Update `entropic-engine` `pyproject.toml`

Remove TUI-related optional dependencies and update extras:

```toml
[project.optional-dependencies]
# tui extra removed — install entropic-tui instead
dev = [
    # ... same as before minus TUI test deps
]

[project.scripts]
entropic = "entropic.cli:main"
# entropic-voice-server removed — lives in entropic-tui
```

Remove from `[tool.setuptools.package-data]`:
- `"ui/*.tcss"` — no longer shipped

Remove `[project.optional-dependencies]`:
- `tui` extra entirely
- `voice` extra entirely
- `all` extra (was app + voice)

**Keep `[app]` as a metapackage alias** that pulls in `entropic-tui`:

```toml
[project.optional-dependencies]
app = ["entropic-tui>=1.7.2"]  # Legacy alias — existing install guides use this
```

This preserves backward compatibility: `pip install entropic-engine[app]` still
works and gives users the TUI. All existing README instructions, setup guides,
and colleague onboarding docs continue to function. The `entropic-tui` package
doesn't exist independently yet — it ships alongside v1.7.2.

Add `aiosqlite` and `httpx` to core `dependencies` if the engine's
`ExternalMCPServer` or `SQLiteStorage` requires them, or gate those behind
optional imports.

### Step 7: Clean up `entropic.ui` in engine repo

After extraction, the `src/entropic/ui/` directory is deleted entirely.
`presenter.py` and `headless.py` already moved to `entropic.core` in Step 1.

Update `src/entropic/__init__.py` to remove `EntropyConfig` from public exports
(keep `LibraryConfig`). `EntropyConfig` moves to `entropic-tui`.

### Step 8: Update `cli.py` in engine repo

- Remove the `--headless` flag (engine is always headless now)
- Remove the lazy `TUIPresenter` import path
- The bare `entropic` command without subcommand should print help or start
  headless mode, NOT attempt to launch the TUI
- Remove `app_context_tui.md` injection logic (line 123-124)
- Keep `click` as a dependency (used by CLI commands that stay)

```python
# cli.py — interactive mode after separation
if ctx.invoked_subcommand is None:
    from entropic.app import Application
    from entropic.core.headless_presenter import HeadlessPresenter

    app = Application(
        config=app_config,
        project_dir=ctx.obj["project"],
        presenter=HeadlessPresenter(),
    )
    asyncio.run(app.run())
```

### Step 9: Move `EntropyConfig` to `entropic-tui`

`EntropyConfig` is a subclass of `LibraryConfig`. Move the class definition to
`entropic_tui/config.py`:

```python
# entropic_tui/config.py
from entropic.config.schema import LibraryConfig, ...

class EntropyConfig(LibraryConfig):
    """Full TUI configuration."""
    quality: QualityConfig = ...
    ui: UIConfig = ...
    storage: StorageConfig = ...
    lsp: LSPConfig = ...
    voice: VoiceConfig = ...
    log_file: Path | None = None
    commands_dir: ExpandedPath = ...
```

The TUI-specific sub-config classes (`QualityConfig`, `UIConfig`, `StorageConfig`,
`LSPConfig`, `VoiceConfig` and their children) move with it.

`LibraryConfig` stays in `entropic.config.schema` — it is the engine's public
config API.

### Step 10: Lock TUI to engine v1.7.1

The `entropic-tui` dependency pin is `entropic-engine>=1.7.1,<2.0`. Initial release
tracks the v1.7.1 engine API surface. As the engine evolves through 1.8.x, the TUI
updates its pin to track compatible releases.

## Validation Criteria

1. **Engine installs clean without TUI deps:**
   ```bash
   pip install entropic-engine
   python -c "from entropic.config.schema import LibraryConfig; print('OK')"
   python -c "from entropic.core.presenter import Presenter; print('OK')"
   python -c "from entropic.core.headless_presenter import HeadlessPresenter; print('OK')"
   ```
   Must succeed. `import textual` must NOT be triggered.

2. **Engine unit tests pass:**
   ```bash
   pre-commit run pytest --all-files
   ```
   All 878 unit tests pass. No imports of `textual`, `entropic.ui.tui`,
   `entropic.ui.widgets`, or `entropic.voice` remain in engine code.

3. **Headless mode works:**
   ```bash
   entropic ask "Hello"
   entropic status
   entropic benchmark run --dry-run
   ```
   All CLI commands function without TUI installed.

4. **TUI installs and works:**
   ```bash
   pip install entropic-tui
   entropic-tui
   ```
   TUI launches, connects to engine, full interactive session works.

5. **No `EntropyConfig` in engine:**
   ```bash
   grep -r "EntropyConfig" src/entropic/
   ```
   Returns zero matches (except maybe deprecation shim if added).

6. **No textual/rich imports in engine core:**
   ```bash
   grep -rE "from (textual|rich)" src/entropic/
   ```
   Returns zero matches. `rich` usage in `cli.py` status command and
   `benchmark/cli.py` is acceptable (CLI-only, lazy imports, `rich` is
   a lightweight dep). If strict separation is desired, replace with
   plain `print()` formatting.

   **Decision:** Keep `rich` as an optional engine dependency for CLI
   pretty-printing. It is NOT in core dependencies — CLI commands that
   use it do lazy imports with graceful fallback.

7. **Import graph verification:**
   No engine module imports from `entropic.ui` (package deleted).
   No engine module imports from `entropic.voice` (package deleted).
