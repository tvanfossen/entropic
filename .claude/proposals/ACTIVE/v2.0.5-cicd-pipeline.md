## Scope

Turn entropic from a source-tree-only project into a first-class
distributable library. Two channels (GitHub Releases + PyPI), one
consumer experience: `find_package(entropic)` or `pip install
entropic-engine` and you're done.

Three workstreams:
1. **Packaging infrastructure** — install rules, CMake package config,
   single-binary distribution
2. **Release automation** — tag-driven GitHub Actions workflows
3. **PR + nightly gating** — PR checks, scheduled model tests

Primary goal: **low friction with reliable outcomes**. A consumer should
never touch LD_LIBRARY_PATH, manage sublib dependencies, or set
compile-time paths. If they know the version, installation is trivial.

## Consumer Experience (the target)

### C++ consumer

```cmake
# any-consumer/CMakeLists.txt
cmake_minimum_required(VERSION 3.21)
project(my-app CXX)

find_package(entropic 2.0 REQUIRED)

add_executable(my-app src/main.cpp)
target_link_libraries(my-app PRIVATE entropic::entropic)
```

```bash
# Consumer installs entropic from GitHub release once:
tar -xf entropic-2.0.5-linux-x86_64-cuda.tar.gz -C /usr/local
# Build their app:
cd my-app && cmake -B build && cmake --build build
./build/my-app
```

That's it. No `ENTROPIC_LIB_DIR`, no manual RPATH, no LD_LIBRARY_PATH.

### Python consumer

```bash
pip install entropic-engine
```

```python
from entropic import EntropicEngine

engine = EntropicEngine()
engine.configure_dir(".my-app")
```

The wheel bundles everything. `entropic` CLI is on PATH via
`[project.scripts]`.

## Packaging Infrastructure

### Single shared library

**Distribution ships `librentropic.so` only** (plus its external deps:
llama, sqlite3, system libs). The source tree keeps separate
`entropic-core`, `entropic-mcp`, `entropic-config`, etc. for
build-time modularity and incremental compilation, but at link time
they are absorbed into the facade as OBJECT or STATIC targets.

Why: The "Pure C at .so boundaries" design rule governs the **ABI
contract** (what types cross library boundaries), not the binary
layout. At runtime, only `entropic.h`'s C API escapes `librentropic.so`
— the contract holds regardless of whether the implementation is
internally split into multiple object files or one.

Consumer cost of multiple .so files (8+ RPATH entries, coordinated
SONAME bumps, `LD_LIBRARY_PATH` juggling) buys nothing we actually
use — the hypothetical `dlopen`-based backend swap isn't implemented
and third-party MCP plugins are already stdio subprocesses. When
runtime plugin loading is actually built, it becomes an additive
layer; it doesn't require splitting the core library today.

**Implementation**: change sublib targets from SHARED to STATIC (or
convert to OBJECT libraries). Only `entropic-facade` is SHARED. The
facade's link command absorbs all sublibs' objects into the single
output library.

### CMake package config

`cmake/entropic-config.cmake.in` template + install-export rules:

```
<install-prefix>/lib/cmake/entropic/
├── entropic-config.cmake           # find_package entry point
├── entropic-config-version.cmake   # version compatibility check
└── entropic-targets.cmake          # exported entropic::entropic target
```

The exported target carries:
- Include dirs (`<prefix>/include/`)
- Link library (`<prefix>/lib/librentropic.so`)
- C++ standard requirement (C++20)
- RPATH hint so `entropic::entropic` resolves its runtime location
- SOVERSION for ABI-compatible upgrades

Consumer call `find_package(entropic 2.0 REQUIRED)` resolves all of
this automatically.

### Install layout

```
<prefix>/
├── bin/
│   └── entropic                    # Single CLI binary (mcp-bridge + version)
├── include/
│   └── entropic/
│       ├── entropic.h              # Public C API
│       ├── entropic_config.h       # Generated build config
│       └── entropic_export.h       # Export macros
├── lib/
│   ├── librentropic.so.2.0.5       # Versioned library
│   ├── librentropic.so.2           # SOVERSION symlink → .so.2.0.5
│   ├── librentropic.so             # Development symlink → .so.2
│   └── cmake/
│       └── entropic/
│           ├── entropic-config.cmake
│           ├── entropic-config-version.cmake
│           └── entropic-targets.cmake
└── share/
    └── entropic/
        ├── bundled_models.yaml
        ├── prompts/               # Default identities, constitution
        ├── tools/                 # Built-in MCP tool JSON definitions
        └── grammars/              # Built-in GBNF grammars
```

The data files under `share/entropic/` are what the engine needs at
runtime. Discovery order:
1. `ENTROPIC_DATA_DIR` env var (explicit override)
2. Project config's `config_dir` field (consumer-specific)
3. `<prefix>/share/entropic/` (from installed package, via
   compile-time `ENTROPIC_INSTALL_PREFIX` baked into the facade)
4. Source-tree fallback (dev-only, when building from source)

### Model file resolution

`bundled_models.yaml` currently has hardcoded `~/models/gguf/...`
paths. That won't work on CI runners or fresh installs. Add
`ENTROPIC_MODEL_DIR` env var with the same discovery layering:

1. `ENTROPIC_MODEL_DIR` env var
2. Per-model explicit absolute path in config
3. `~/.entropic/models/` (user convention)
4. `/opt/entropic/models/` (system convention, advisory)

The registry records the model filename; path resolution happens at
model-load time using the discovery order. Documented in the release
README: "Download your GGUF, export `ENTROPIC_MODEL_DIR=/path/to/dir`,
you're done."

### SONAME + RPATH

- `librentropic.so` gets `VERSION 2.0.5` and `SOVERSION 2`. Minor/patch
  versions bump without breaking the ABI link; major version bumps
  break cleanly.
- Build-tree RPATH points at the build dir for dev builds.
- Install RPATH is `$ORIGIN/../lib` so the CLI binary (in `bin/`)
  finds the library (in `lib/`) regardless of install prefix.
- Consumers that link via `find_package` inherit an RPATH hint via
  the imported target.

## Release Automation

### Trigger: tag push

```bash
# Developer cuts a release:
git tag v2.0.5
git push origin v2.0.5
```

GitHub Actions matches `v*` and runs `release.yml`.

### release.yml: build matrix

| OS | Arch | Backend | Compiler | Runner |
|----|------|---------|----------|--------|
| Ubuntu 22.04 | x86_64 | CPU | GCC 13 | GitHub hosted |
| Ubuntu 22.04 | x86_64 | CUDA 12.x | GCC 13 | Self-hosted GPU |

Each matrix job:
1. Checkout + submodules (llama.cpp)
2. Install system deps (cmake, ninja, nlohmann-json, ryml via
   FetchContent, spdlog via FetchContent)
3. Verify tag matches `CMakeLists.txt` project VERSION
4. Configure with the matrix's CMake preset
5. Build
6. Run unit tests (headless example as smoke test)
7. `cmake --install build --prefix stage/`
8. Package:
   - **tar.gz**: `entropic-{VERSION}-linux-x86_64-{backend}.tar.gz`
   - **wheel**: via `cibuildwheel` (CPU build → manylinux_2_28;
     CUDA build → non-manylinux tagged wheel)

### release.yml: publish

After all matrix jobs succeed:
- **GitHub Release**: Created at the tag with all `.tar.gz` artifacts
  attached + autogenerated changelog from commits since last tag.
- **PyPI**: Wheels published via OIDC trusted publishing. CI already
  has the necessary project configuration on pypi.org.

### release.yml: validation

After publish, a smoke-test job:
1. Download the published `.tar.gz` from the release
2. Extract to `/tmp/entropic-smoke/`
3. Build a throwaway consumer project using `find_package`
4. Run it against a tiny CPU model (or mock backend) to verify the
   link and launch works
5. `pip install entropic-engine==<VERSION>` in a fresh venv; verify
   import + CLI entrypoint works

Failure at this stage marks the release as draft so it can be
yanked without users seeing broken artifacts.

## PR + Nightly Gating

### ci.yml (PR checks)

Triggers on PR to `develop` and `main`.

| Step | What | Time |
|------|------|------|
| Pre-commit | All hooks (knots, flake8, doxygen-guard, ruff, build, tests) | ~3min |
| C99 compat | `entropic-c99-test` compiles | included |
| Coverage | lcov → artifact upload | ~30s |
| Wrapper drift | Verify auto-generated Python wrapper matches committed | ~5s |

All steps must pass for merge. No GPU needed — this is CPU-only.

### nightly.yml (scheduled)

Triggers 02:00 UTC against `develop`. Runs on self-hosted GPU runner.

| Step | What |
|------|------|
| Build (CUDA) | Full preset |
| Unit tests (GPU) | Entire test suite on CUDA build |
| Model tests | Full model + benchmark suite (x.y.0 gate requirement) |
| Headless example | End-to-end smoke test |
| Benchmark | Regression detection (>10% tok/s, >15% TTFT alerts) |
| Coverage | Full report + badge update |

On failure: opens a GitHub Issue tagged `nightly-failure` with the
failed step, logs, and commit hash.

On pass for a `x.y.0` target: marks develop commit as eligible for
minor release tagging.

## Python Wheel Specifics

### Build path

`cibuildwheel` with `pyproject.toml` configuration:
- Build backend: `scikit-build-core` (cmake-native)
- Python versions: 3.11, 3.12, 3.13
- Manylinux: `manylinux_2_28_x86_64` for CPU build
- CUDA wheel: `linux_x86_64` tag (non-manylinux, requires host CUDA
  runtime)

### Wheel contents

```
entropic_engine-2.0.5-cp311-*.whl
├── entropic/
│   ├── __init__.py               # Auto-generated ctypes wrapper
│   ├── _native.py                # Library loader + path resolution
│   ├── librentropic.so           # Bundled shared library
│   ├── entropic                  # CLI binary (goes on PATH via entry point)
│   └── data/
│       ├── bundled_models.yaml
│       ├── prompts/
│       ├── tools/
│       └── grammars/
└── entropic_engine-2.0.5.dist-info/
    └── entry_points.txt          # exposes `entropic` command
```

The CLI binary is shipped inside the wheel; `pyproject.toml`
`[project.scripts]` entry exposes it as `entropic` on the user's
PATH. Wheel consumers and tarball consumers get the same CLI.

### Native library loader

`_native.py` handles the wheel vs tarball discovery:
```python
def _find_lib():
    # 1. Env override (power users, testing)
    if p := os.environ.get("ENTROPIC_LIB_PATH"): return p
    # 2. Bundled in wheel (pip install path)
    bundled = Path(__file__).parent / "librentropic.so"
    if bundled.exists(): return bundled
    # 3. System install (tarball path)
    return ctypes.util.find_library("rentropic")
```

## Implementation Plan

### Phase 1: Packaging infrastructure (no CI yet)

1. **Convert sublibs to STATIC/OBJECT** targets; facade is the only
   SHARED target. Verify build still produces a single
   `librentropic.so` that contains everything.
2. **Install rules** in every `src/*/CMakeLists.txt` and top-level.
   Library install goes to `lib/`, CLI to `bin/`, data to
   `share/entropic/`, headers to `include/entropic/`.
3. **CMake package config**: create `cmake/entropic-config.cmake.in`,
   use `install(EXPORT entropic-targets ...)` to generate the
   targets file, write version check file.
4. **SONAME + install RPATH** on the facade target and CLI binary.
5. **Data dir discovery**: verify and extend
   `src/config/data_dir.cpp` for the 4-level fallback.
6. **Model dir discovery**: add `ENTROPIC_MODEL_DIR` env var to
   `src/config/bundled_models.cpp`.
7. Smoke-test locally: `cmake --install build --prefix /tmp/test`,
   then build a consumer project using `find_package(entropic)`.
   Must compile, link, launch against /tmp/test without any manual
   path setup.

### Phase 2: PR CI (ci.yml)

8. `.github/actions/setup-entropic/action.yml` — composite action
   for deps + venv + preset config.
9. `.github/workflows/ci.yml` — PR check workflow. Pre-commit + build
   + tests + coverage upload.
10. `.github/CODEOWNERS`.
11. Verify on a test PR against develop.

### Phase 3: Release workflow (release.yml)

12. `pyproject.toml` migration to `scikit-build-core` + cibuildwheel
    configuration.
13. Verify local wheel build: `pipx run cibuildwheel --platform linux`.
14. `.github/workflows/release.yml`:
    - Tag validation job
    - Matrix build (CPU + CUDA)
    - Package jobs (tar.gz + wheel)
    - Publish jobs (GitHub Release + PyPI via OIDC)
    - Post-publish smoke test job
15. Register self-hosted GPU runner on user's hardware.
16. Configure PyPI trusted publishing.

### Phase 4: Nightly (nightly.yml)

17. `.github/workflows/nightly.yml` — scheduled run.
18. Benchmark baseline + regression detection logic.
19. Auto-filed GitHub Issues on failure.

### Phase 5: Test release (v2.0.5-rc1)

20. Cut `v2.0.5-rc1` on a prerelease branch.
21. Verify GitHub Release creates correctly.
22. Verify `pip install entropic-engine==2.0.5rc1 --pre` works from PyPI.
23. Verify a throwaway consumer can `find_package(entropic)` from the
    extracted tarball.
24. If all smoke tests pass, cut `v2.0.5` proper.

## Validation Criteria

1. `cmake --install` produces a self-contained prefix with library,
   CLI, headers, data, and CMake package config.
2. A consumer project with just `find_package(entropic 2.0 REQUIRED)`
   and `target_link_libraries(... entropic::entropic)` builds and
   runs without any manual LD_LIBRARY_PATH or include path setup.
3. PR to develop triggers CI; all hooks pass on green.
4. `git tag v2.0.5 && git push --tags` triggers release workflow.
5. GitHub Release at v2.0.5 has both CPU and CUDA tarballs.
6. `pip install entropic-engine==2.0.5` installs cleanly in a fresh
   venv; `entropic version` works from the installed wheel.
7. Nightly workflow runs unattended and posts results.
