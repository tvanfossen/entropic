# v2.1.7 — `mcp-bridge` becomes a pure relay

## Scope

The proposal content is the GitHub issue and its architect-decision
comment. This file is a thin pointer + implementation log only.

- **Issue:** [gh#34](https://github.com/tvanfossen/entropic/issues/34)
- **Architect decision comment:**
  https://github.com/tvanfossen/entropic/issues/34#issuecomment-4442473983
- **Split out:** [gh#35](https://github.com/tvanfossen/entropic/issues/35)
  — engine idle-exit policy (separate version, separate proposal).

## Summary of the change

`entropic mcp-bridge` becomes a pure stdio↔unix-socket relay. It does
not call `entropic_create`, owns no model, and exits fast when no
engine is reachable. `mcp-connect` is absorbed into `mcp-bridge` and
removed as a separate subcommand. The current standalone-engine-on-
stdio behavior is dropped, not flagged behind `--standalone`.

Engine host is a separate, orthogonal concept. The bridge sits in
front of any engine host (TUI, consumer app, future headless
`entropic serve`) identically.

## Unix-socket hardening (in scope)

- **SO_PEERCRED UID check** on `accept()` — reject if peer UID does
  not match engine UID.
- **Containing directory `~/.entropic/socks/` mode 0700**, socket file
  mode 0600 — explicit `chmod`/`fchmod`, not relying on umask.
- **Symlink-safe bind** — refuse to bind if the path is a symlink or a
  pre-existing non-socket file.

Session-token handshake was considered and dropped — same-uid
attackers can read the token, cross-uid attackers can't reach the
socket. Adds protocol surface without changing the trust boundary.

## Out of scope (filed / documented elsewhere)

- **Idle-exit policy for engine hosts** — gh#35.
- **TCP transport** — when it ships, that is the trigger for
  proper token/OAuth auth. Note added to `docs/architecture-cpp.md`.
- **External bridge concurrency** — already supported via per-client
  threads since v2.1.2 (issue #4). Not a limitation; do not document
  as one.

## Validation criteria

1. `entropic mcp-bridge` exits 1 with diagnostic message when no
   socket exists for the project_dir hash. Message names the cwd,
   canonical path, hash, and computed socket path.
2. With a running engine on the computed socket, `mcp-bridge` relays
   JSON-RPC requests and responses byte-for-byte.
3. External bridge rejects connections from a peer with a different
   UID (SO_PEERCRED test).
4. External bridge refuses to bind if the socket path is a symlink or
   a pre-existing non-socket file (rejected with a clear log line).
5. Socket file mode is 0600, containing dir 0700, after bridge start.
6. Repo's own `.mcp.json` example updated; pre-commit passes;
   unit tests pass.

### Decisions
- **`--socket PATH` retained on `mcp-bridge`** (not just a `mcp-connect`
  legacy). Justified by (a) deterministic relay round-trip testing,
  (b) users with non-standard engine deployments whose socket path
  doesn't follow the `compute_socket_path` hash. Default discovery
  remains the project_dir hash.
- **No session-token handshake.** Considered. Rejected: same-uid attackers
  can read the token; cross-uid attackers cannot reach the socket
  (file perms + SO_PEERCRED). Adds protocol surface without changing
  the trust boundary. Recorded in architecture decision #37.
- **Concurrency limitation removed from the v2.1.7 scope.** The original
  architect comment noted `listen(fd, 1)` as a single-client limitation;
  on re-reading the v2.1.2 fix for issue #4, per-client threads already
  exist. Backlog of 1 is unchanged and correct. No action needed.
