## Scope

Per-identity MCP access control with runtime-grantable keys. After v1.9.4,
each identity/caller has a discrete set of authorized MCP tool keys with
READ/WRITE access levels. Keys are grantable and revocable at runtime (one
caller grants keys to another). Key sets are serializable for persistence
and restoration. The engine enforces boundaries -- a caller cannot invoke
tools outside its key set.

This is the enforcement point promised in architecture design rule #8:
"Permissions ship with tools. Default-deny from day one. Per-identity
granularity added later." v1.8.5 established `PermissionManager` with
global allow/deny glob patterns. v1.8.6 added per-tier `allowed_tools`
filtering from identity frontmatter. v1.9.4 adds a FINER layer on top:
per-identity MCP key sets with READ/WRITE access levels, runtime grant/revoke,
and serialization.

**Relationship to existing permission layers:**

```
Layer 3 (v1.9.4): MCPKeySet per identity — READ/WRITE per tool key
    ↓ checked first (finest grain)
Layer 2 (v1.8.6): allowed_tools per tier — identity frontmatter filter
    ↓ checked second
Layer 1 (v1.8.5): PermissionManager — global allow/deny glob patterns
    ↓ checked third (coarsest grain)
```

All three layers must pass for a tool call to execute. A tool allowed by
the global permission manager can still be blocked by the identity's key set.
This is defense-in-depth -- each layer narrows the set of permissible actions.

**What this version does NOT include:**
- Key set persistence to SQLite (v1.8.8 storage layer handles serialized blobs)
- Hook integration (v1.9.1 `on_permission_check` fires but doesn't gate)
- Audit logging of grant/revoke events (v1.9.5)
- Dynamic identity creation with MCP keys (v1.9.6)

## Access Level Model

Two access levels per tool key:

```cpp
/// @brief MCP tool access level.
/// @version 1.9.4
enum class MCPAccessLevel : uint8_t {
    NONE  = 0,   ///< No access (default for ungranted keys)
    READ  = 1,   ///< Read-only operations (e.g., read_file, list_directory, git.status)
    WRITE = 2,   ///< Read + write operations (e.g., write_file, execute, git.commit)
};
```

Access levels are ordered: `WRITE` implies `READ`. A caller with `WRITE`
access to `filesystem.*` can also read files. A caller with `READ` access
cannot write.

**Tool classification** (which tools are READ vs WRITE):

Each MCP server plugin declares the access level required per tool:

```cpp
/// @brief Declare the minimum access level required for this tool.
/// @version 1.9.4
/// Default returns WRITE (safe default -- tools must opt into READ).
virtual MCPAccessLevel required_access_level() const {
    return MCPAccessLevel::WRITE;
}
```

Override examples:
- `filesystem.read_file` -> `READ`
- `filesystem.list_directory` -> `READ`
- `filesystem.glob` -> `READ`
- `filesystem.grep` -> `READ`
- `filesystem.write_file` -> `WRITE`
- `filesystem.edit_file` -> `WRITE`
- `bash.execute` -> `WRITE`
- `git.status` -> `READ`
- `git.diff` -> `READ`
- `git.log` -> `READ`
- `git.commit` -> `WRITE`
- `git.add` -> `WRITE`
- `git.checkout` -> `WRITE`
- `git.reset` -> `WRITE`
- `git.branch` -> `WRITE`
- `web.web_fetch` -> `READ`
- `web.web_search` -> `READ`
- `diagnostics.diagnostics` -> `READ`
- `diagnostics.check_errors` -> `READ`
- `entropic.todo` -> `WRITE`
- `entropic.delegate` -> `WRITE`
- `entropic.pipeline` -> `WRITE`
- `entropic.complete` -> `WRITE`
- `entropic.phase_change` -> `WRITE`
- `entropic.prune_context` -> `WRITE`

The default (`WRITE`) is the safe choice. New tools must explicitly opt into
`READ` if they have no side effects. This follows the default-deny principle.

## MCPKeySet

The key set is the per-identity authorization state. Each identity has one.

```cpp
/// @brief A single authorized MCP key with access level.
/// @version 1.9.4
struct MCPKey {
    std::string tool_pattern;   ///< Tool pattern (e.g., "filesystem.*", "git.status")
    MCPAccessLevel level;       ///< Granted access level (READ or WRITE)
};

/// @brief Per-identity set of authorized MCP tool keys.
/// @version 1.9.4
///
/// Default state is EMPTY -- no tools authorized. Keys must be explicitly
/// granted. This enforces the default-deny principle at the identity level.
///
/// Tool patterns support:
/// - Exact match: "filesystem.read_file"
/// - Server wildcard: "filesystem.*"
/// - Full wildcard: "*" (all tools -- use sparingly)
///
/// Grant resolution: most specific pattern wins. "filesystem.read_file:WRITE"
/// overrides "filesystem.*:READ" for that specific tool.
///
/// @par Thread safety:
/// grant/revoke acquire key_mutex_. has_access() is lock-free (reads from
/// snapshot). Serialization acquires key_mutex_ for consistency.
class MCPKeySet {
public:
    /// @brief Grant a tool key with an access level.
    /// @param pattern Tool pattern string.
    /// @param level Access level to grant.
    /// @version 1.9.4
    ///
    /// If the pattern already exists, the level is updated (not additive).
    /// Granting READ to a pattern that had WRITE downgrades it.
    void grant(const std::string& pattern, MCPAccessLevel level);

    /// @brief Revoke a tool key entirely.
    /// @param pattern Tool pattern string to revoke.
    /// @return true if pattern was found and removed.
    /// @version 1.9.4
    bool revoke(const std::string& pattern);

    /// @brief Check if a specific tool is authorized at the required level.
    /// @param tool_name Fully-qualified tool name (e.g., "filesystem.read_file").
    /// @param required Minimum access level needed.
    /// @return true if authorized.
    /// @version 1.9.4
    ///
    /// Resolution order (most specific wins):
    /// 1. Exact match on tool_name
    /// 2. Server wildcard (e.g., "filesystem.*" matches "filesystem.read_file")
    /// 3. Full wildcard ("*")
    /// 4. No match -> denied
    ///
    /// Access level comparison: granted >= required means authorized.
    /// WRITE (2) >= READ (1) is true. READ (1) >= WRITE (2) is false.
    bool has_access(const std::string& tool_name,
                    MCPAccessLevel required) const;

    /// @brief List all granted keys.
    /// @return Vector of MCPKey entries.
    /// @version 1.9.4
    std::vector<MCPKey> list() const;

    /// @brief Number of granted keys.
    /// @version 1.9.4
    size_t size() const;

    /// @brief Remove all granted keys.
    /// @version 1.9.4
    void clear();

    /// @brief Serialize key set to JSON string.
    /// @return JSON array of {pattern, level} objects.
    /// @version 1.9.4
    ///
    /// Format: [{"pattern": "filesystem.*", "level": "WRITE"}, ...]
    /// Level serialized as string: "NONE", "READ", "WRITE".
    std::string serialize() const;

    /// @brief Deserialize key set from JSON string.
    /// @param json JSON array string (same format as serialize output).
    /// @return true if parsed successfully.
    /// @version 1.9.4
    ///
    /// Replaces all current keys with the deserialized set.
    /// Invalid entries are logged as WARNING and skipped.
    bool deserialize(const std::string& json);

private:
    /// @brief Granted keys, keyed by pattern for O(1) exact lookup.
    std::unordered_map<std::string, MCPAccessLevel> keys_;

    /// @brief Guards mutations to keys_ map.
    mutable std::mutex key_mutex_;

    /// @brief Find the best matching pattern for a tool name.
    /// @return Iterator to best match, or keys_.end() if no match.
    /// @version 1.9.4
    std::unordered_map<std::string, MCPAccessLevel>::const_iterator
    find_best_match(const std::string& tool_name) const;
};
```

## MCPAuthorizationManager

Central manager that maps identities to key sets and handles cross-identity
grant/revoke operations.

```cpp
/// @brief Per-identity MCP authorization with runtime grant/revoke.
/// @version 1.9.4
///
/// Manages MCPKeySet instances per identity. Provides the enforcement
/// check called by ToolExecutor before tool execution. Handles
/// cross-identity key granting (identity A grants keys to identity B).
///
/// @par Ownership:
/// Owned by AgentEngine. One MCPAuthorizationManager per engine instance.
///
/// @par Default behavior:
/// If an identity has no MCPKeySet registered, the authorization check
/// PASSES (backward compatible -- v1.8.5/v1.8.6 behavior). Once a key
/// set is registered for an identity (even an empty one), authorization
/// is enforced. This allows incremental adoption.
///
/// @par Thread safety:
/// All public methods acquire auth_mutex_. The authorization check
/// (check_access) is on the hot path but expected to be fast --
/// single map lookup + pattern matching.
class MCPAuthorizationManager {
public:
    /// @brief Register an empty key set for an identity.
    /// @param identity_name Identity/tier name.
    /// @version 1.9.4
    ///
    /// After registration, the identity has ZERO authorized tools.
    /// Keys must be explicitly granted. This enables per-identity
    /// authorization enforcement.
    void register_identity(const std::string& identity_name);

    /// @brief Check if an identity has authorization enforcement enabled.
    /// @param identity_name Identity/tier name.
    /// @return true if the identity has a registered MCPKeySet.
    /// @version 1.9.4
    bool is_enforced(const std::string& identity_name) const;

    /// @brief Grant a tool key to an identity.
    /// @param identity_name Target identity.
    /// @param pattern Tool pattern string.
    /// @param level Access level to grant.
    /// @return ENTROPIC_OK on success.
    ///         ENTROPIC_ERROR_NOT_FOUND if identity not registered.
    /// @version 1.9.4
    entropic_error_t grant(const std::string& identity_name,
                           const std::string& pattern,
                           MCPAccessLevel level);

    /// @brief Revoke a tool key from an identity.
    /// @param identity_name Target identity.
    /// @param pattern Tool pattern string.
    /// @return ENTROPIC_OK on success.
    ///         ENTROPIC_ERROR_NOT_FOUND if identity or pattern not found.
    /// @version 1.9.4
    entropic_error_t revoke(const std::string& identity_name,
                            const std::string& pattern);

    /// @brief Check if a tool call is authorized for an identity.
    /// @param identity_name Caller identity.
    /// @param tool_name Fully-qualified tool name.
    /// @param required_level Minimum access level needed.
    /// @return true if authorized (or if identity has no key set registered).
    /// @version 1.9.4
    ///
    /// If the identity has no registered key set, returns true
    /// (backward compatible -- no enforcement). If the identity has
    /// a registered key set, delegates to MCPKeySet::has_access().
    bool check_access(const std::string& identity_name,
                      const std::string& tool_name,
                      MCPAccessLevel required_level) const;

    /// @brief One identity grants a key to another identity.
    /// @param granter_name Identity performing the grant.
    /// @param grantee_name Identity receiving the key.
    /// @param pattern Tool pattern to grant.
    /// @param level Access level to grant.
    /// @return ENTROPIC_OK on success.
    ///         ENTROPIC_ERROR_NOT_FOUND if either identity not registered.
    ///         ENTROPIC_ERROR_PERMISSION_DENIED if granter lacks the key.
    /// @version 1.9.4
    ///
    /// The granter must possess the key at >= the granted level.
    /// You cannot grant WRITE access to a tool you only have READ for.
    /// You CAN grant READ access to a tool you have WRITE for.
    entropic_error_t grant_from(const std::string& granter_name,
                                const std::string& grantee_name,
                                const std::string& pattern,
                                MCPAccessLevel level);

    /// @brief List all keys for an identity.
    /// @param identity_name Identity to query.
    /// @return Vector of MCPKey entries, or empty if identity not registered.
    /// @version 1.9.4
    std::vector<MCPKey> list_keys(const std::string& identity_name) const;

    /// @brief Serialize all identity key sets to JSON.
    /// @return JSON object: {"identity_name": [key_array], ...}
    /// @version 1.9.4
    std::string serialize_all() const;

    /// @brief Deserialize all identity key sets from JSON.
    /// @param json JSON object string (same format as serialize_all output).
    /// @return true if parsed successfully.
    /// @version 1.9.4
    bool deserialize_all(const std::string& json);

    /// @brief Remove an identity's key set (disables enforcement).
    /// @param identity_name Identity to unregister.
    /// @version 1.9.4
    void unregister_identity(const std::string& identity_name);

private:
    /// @brief Key sets per identity.
    std::unordered_map<std::string, MCPKeySet> key_sets_;

    /// @brief Guards mutations to key_sets_ map.
    mutable std::mutex auth_mutex_;
};
```

## ToolExecutor Integration

`ToolExecutor` (v1.8.5) gains an authorization check in its execution path.
The check sits between the existing tier allowlist check and the actual
tool execution.

```
ToolExecutor::process_tool_calls()
    FOR each tool_call:
        1. check_tier_allowed()      ← v1.8.6 (identity frontmatter allowed_tools)
        2. check_mcp_authorization() ← v1.9.4 (MCPKeySet per identity)
        3. check_duplicate()         ← v1.8.5 (duplicate detection)
        4. check_approval()          ← v1.8.5 (global PermissionManager)
        5. execute_tool()            ← v1.8.5 (actual execution)
```

New method:

```cpp
/// @brief Check MCP authorization for a tool call.
/// @param ctx Loop context (provides current identity name).
/// @param call Tool call to check.
/// @return Error message if denied, nullopt if authorized.
/// @version 1.9.4
std::optional<Message> check_mcp_authorization(
    const LoopContext& ctx,
    const ToolCall& call) const;
```

The method:
1. Gets the current identity name from `ctx.locked_tier` (or default tier)
2. Gets the tool's `required_access_level()` from the server
3. Calls `authorization_manager_.check_access(identity, tool_name, required_level)`
4. If denied: returns an error message to the model explaining the denial
5. If authorized: returns `nullopt` (proceed)

The error message includes:
- Which tool was denied
- What access level was required
- What access level the identity has (if any)
- Suggestion: "Use entropic.delegate to hand off to an identity with access"

This feedback is critical -- the model needs to know WHY a tool was denied
so it can delegate appropriately rather than retrying the same denied tool.

## ToolBase Access Level Declaration

`ToolBase` (v1.8.5) gains a virtual method for access level declaration:

```cpp
/// @brief Minimum access level required to execute this tool.
/// @return MCPAccessLevel::WRITE by default (safe default).
/// @version 1.9.4
///
/// Override in read-only tools (e.g., read_file, list_directory, status).
/// The ToolExecutor checks this against the caller's MCPKeySet before
/// executing.
virtual MCPAccessLevel required_access_level() const {
    return MCPAccessLevel::WRITE;
}
```

Each built-in server's tool classes override this appropriately. Third-party
plugin authors must also override it -- the safe default (`WRITE`) means
unclassified tools require the highest access level.

## Identity Frontmatter Integration

Identity frontmatter gains an optional `mcp_keys` field for static key
configuration:

```yaml
# identity_eng.md frontmatter
allowed_tools:
  - "filesystem.*"
  - "bash.*"
  - "git.*"
  - "entropic.*"
mcp_keys:
  - pattern: "filesystem.*"
    level: WRITE
  - pattern: "bash.*"
    level: WRITE
  - pattern: "git.*"
    level: WRITE
  - pattern: "entropic.*"
    level: WRITE
  - pattern: "diagnostics.*"
    level: READ
```

On identity load, if `mcp_keys` is present, the `MCPAuthorizationManager`
registers the identity and populates its key set from the frontmatter.
If `mcp_keys` is absent, no key set is registered (backward compatible --
no enforcement for that identity).

This means adoption is incremental:
1. Identities WITHOUT `mcp_keys` continue to work as before (layers 1+2 only)
2. Identities WITH `mcp_keys` get layer 3 enforcement
3. Runtime grant/revoke modifies the key set beyond what frontmatter declared

/// @brief Access level enum for MCP authorization.

typedef enum {

## Key Grant Flow (Runtime)

The primary novel capability: identity A grants tool access to identity B
at runtime.

```
Lead (depth=0)                          MCPAuthorizationManager
      │                                        │
      │ "eng needs filesystem access"          │
      │──────────────────────────────────────►│
      │ entropic_grant_mcp_key_from(           │
      │   "lead", "eng",                       │
      │   "filesystem.*", WRITE)               │
      │                                        │
      │                         validate:      │
      │                         lead has       │
      │                   "filesystem.*:WRITE"?│
      │                         yes ─────────►│
      │                                        │
      │                         eng.key_set.   │
      │                         grant(         │
      │                   "filesystem.*",WRITE)│
      │◄──────────────────────────────────────│
      │   ENTROPIC_OK                          │
      │                                        │
      │ entropic.delegate(target="eng",        │
      │   task="Write hello.py")               │
      │                                        │
             Eng (depth=1)
                  │
                  │ filesystem.write_file("hello.py", ...)
                  │
                  │ ToolExecutor.check_mcp_authorization()
                  │   identity="eng"
                  │   tool="filesystem.write_file"
                  │   required=WRITE
                  │   eng.has_access("filesystem.write_file", WRITE)?
                  │   matches "filesystem.*:WRITE" → AUTHORIZED
                  │
                  │ → tool executes
```

Consumer scenarios:

1. **TUI developer**: All identities get `mcp_keys` via frontmatter. Lead has
   all keys at WRITE. Child roles get scoped keys. Runtime grant not needed
   unless ad-hoc access is required.

2. **Game NPC**: NPCs get minimal key sets (read-only filesystem for dialog
   files, no bash/git). The game orchestrator grants additional keys to
   specific NPCs as needed (e.g., a "builder" NPC gets filesystem.write_file).

3. **Homeschool**: Student identity gets READ-only access. Teacher identity
   has WRITE. Teacher can grant temporary WRITE to student for exercises.
   Grant is revoked after the exercise completes.

## Serialization Format

JSON wire format for MCPKeySet:

```json
[
    {"pattern": "filesystem.*", "level": "WRITE"},
    {"pattern": "git.status", "level": "READ"},
    {"pattern": "git.log", "level": "READ"},
    {"pattern": "entropic.complete", "level": "WRITE"}
]
```

JSON wire format for full authorization state (serialize_all):

```json
{
    "lead": [
        {"pattern": "*", "level": "WRITE"}
    ],
    "eng": [
        {"pattern": "filesystem.*", "level": "WRITE"},
        {"pattern": "bash.*", "level": "WRITE"},
        {"pattern": "git.*", "level": "WRITE"},
        {"pattern": "entropic.*", "level": "WRITE"},
        {"pattern": "diagnostics.*", "level": "READ"}
    ],
    "qa": [
        {"pattern": "filesystem.*", "level": "READ"},
        {"pattern": "bash.execute", "level": "WRITE"},
        {"pattern": "git.*", "level": "READ"},
        {"pattern": "entropic.*", "level": "WRITE"},
        {"pattern": "diagnostics.*", "level": "READ"}
    ]
}
```

This format is what `entropic_serialize_mcp_keys()` returns and
`entropic_deserialize_mcp_keys()` accepts. It is also the format stored
by the storage layer (v1.8.8) for session persistence.

## Implementation Plan

### Phase 1: MCPAccessLevel Enum + MCPKey Struct

Add the access level enum and key struct to the types library.

**Files:**
- `include/entropic/types/config.h` (modified -- add MCPAccessLevel, MCPKey)

### Phase 2: MCPKeySet Class

Core key set implementation: grant, revoke, has_access, list, serialize,
deserialize, pattern matching. Thread-safe mutation guards.

Pattern matching logic:
```
find_best_match("filesystem.read_file"):
    1. Exact: keys_.find("filesystem.read_file") → hit? return it
    2. Server wildcard: extract "filesystem", keys_.find("filesystem.*") → hit? return it
    3. Full wildcard: keys_.find("*") → hit? return it
    4. No match → keys_.end()
```

This is O(1) for exact matches, O(1) for wildcards (three map lookups max).
No iteration over the key set. Pattern matching is deliberately simple --
only three forms are supported (exact, server.*, *). This avoids the
performance cost of arbitrary glob matching on the hot path while covering
all practical authorization patterns.

## Validation Criteria

1. **Empty key set blocks all tools**: Register identity with empty key set. Verify all tool calls denied.
2. **Exact pattern match grants access**: Grant "filesystem.read_file:READ". Verify read_file succeeds, write_file denied.
3. **Server wildcard grants access**: Grant "filesystem.*:WRITE". Verify all filesystem tools succeed.
4. **Full wildcard grants all access**: Grant "*:WRITE". Verify all tools succeed.
5. **READ does not authorize WRITE tools**: Grant "filesystem.*:READ". Verify read_file succeeds, write_file denied.
6. **WRITE implies READ**: Grant "filesystem.*:WRITE". Verify both read_file and write_file succeed.
7. **Most specific pattern wins**: Grant "filesystem.*:READ" + "filesystem.write_file:WRITE". Verify write_file succeeds.
8. **Revoke removes access**: Grant then revoke "filesystem.*". Verify tools denied after revoke.
9. **Runtime grant takes effect immediately**: Grant key during active session. Next tool call uses new key set.
10. **Cross-identity grant validates granter**: Granter without the key cannot grant it. Returns PERMISSION_DENIED.
11. **Cross-identity grant succeeds when valid**: Granter with WRITE grants READ to grantee. Grantee can read.
12. **Serialize/deserialize round-trip**: Serialize key sets, clear, deserialize. Verify identical state.
13. **Identity without key set passes authorization**: Identity with no mcp_keys in frontmatter has no enforcement (backward compatible).
14. **Identity with mcp_keys enforced from startup**: Identity with mcp_keys in frontmatter enforced immediately.
15. **Denial feedback message is actionable**: Denied tool call produces message with tool name, required level, and delegation suggestion.
16. **Three permission layers compose correctly**: Tool allowed by global PermissionManager + tier allowlist but denied by key set is blocked. Tool allowed by key set but denied by global PermissionManager is blocked.
17. **C API functions work across .so boundary**: grant, revoke, list, grant_from, serialize, deserialize all callable from C consumer.
18. **Thread safety**: Concurrent grant/revoke/check_access from multiple threads produces no data races (ThreadSanitizer clean).
19. **SLOC <= 50 per function**: All functions pass knots complexity gates.
20. **Doxygen complete**: Every public symbol has @brief and @version.
