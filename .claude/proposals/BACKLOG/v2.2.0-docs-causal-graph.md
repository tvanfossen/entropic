## Summary

Replace the current FTS5 + LIKE-based documentation search with a
causal-graph AST lookup over the C++ source tree. Agents (especially
the `tracer` tier) should be able to ask *"what calls function X,
reading which structs, producing which outputs, and where does the
result flow?"* — queries the current search surface can't answer
without multiple round-trips and heuristic synthesis by the model.

Current state: `inv docs` produces a SQLite database with doxygen
XML output and a text-indexed symbol table. `docs.search(query)`,
`docs.lookup_function(name)`, `docs.lookup_class(name)`, and
`docs.get_hierarchy(name)` provide read access. These answer
*locality* questions ("what is function X?") but not *causality*
questions ("how does data flow from X to Y?").

## Motivation

Tracer-tier and researcher-tier agents routinely ask causality
questions by firing multiple `docs.search` probes and stitching the
results together in the model's own reasoning. Example from recent
explorer sessions:

- `docs.search "run_turn"`
- `docs.search "execute_iteration"`
- `docs.search "process_tool_results"`
- `docs.search "run_loop"`
- `docs.search "engine loop state"`

All to answer a single "trace the engine loop" question. Each search
returns a bag of symbol matches with snippets; the model re-assembles
the call graph by pattern-matching identifier names. This is the slow
path:

- High tool-call budget (5+ per question).
- Error-prone: misses cross-library calls, mis-attributes indirect
  dispatches, hallucinates caller/callee relationships not present in
  the source.
- No awareness of `.so`-boundary crossings (critical for our
  architecture where the C API is the contract surface).

A graph lookup turns this into one query:

```
docs.causal_graph(symbol="AgentEngine::run_turn",
                  direction="forward",
                  depth=3,
                  filter_layer="core")
```

Returns: sequence of calls `run_turn → execute_iteration → …`, each
node annotated with file:line, library membership (`librentropic-core`),
and edge types (direct call, virtual dispatch, callback).

## Scope

### Extraction

- Replace doxygen-XML-only extraction with clang `libTooling` or
  `tree-sitter-cpp` AST walk over the source tree.
- Per translation unit, emit:
  - Symbol definitions (functions, classes, methods, fields, typedefs).
  - Call edges (caller → callee, annotated with call-kind: direct /
    virtual / function-pointer / macro-expanded).
  - Read/write edges (function → struct field).
  - Include edges (TU → header).
  - Boundary annotations (crosses `.so` boundary? Uses `ENTROPIC_EXPORT`?
    Inside `extern "C"` block?).
- Preserve source coordinates (file:line:column) for every edge.

### Storage

New SQLite schema alongside the existing FTS5 surface (don't replace —
existing `docs.search` stays for noisy exploration):

```sql
CREATE TABLE ast_node (
    id INTEGER PRIMARY KEY,
    kind TEXT,            -- 'function', 'class', 'field', ...
    qualified_name TEXT,  -- 'entropic::AgentEngine::run_turn'
    file TEXT,
    line INTEGER,
    library TEXT,         -- 'librentropic-core', ...
    is_boundary BOOLEAN   -- true if crosses .so boundary
);

CREATE TABLE ast_edge (
    id INTEGER PRIMARY KEY,
    src_id INTEGER REFERENCES ast_node(id),
    dst_id INTEGER REFERENCES ast_node(id),
    kind TEXT,            -- 'calls', 'reads', 'writes', 'includes', 'inherits'
    call_kind TEXT,       -- 'direct', 'virtual', 'fptr', 'macro' (NULL for non-calls)
    file TEXT,
    line INTEGER
);

CREATE INDEX idx_edge_src ON ast_edge(src_id, kind);
CREATE INDEX idx_edge_dst ON ast_edge(dst_id, kind);
```

Graph traversal runs over edges via recursive CTE; up to depth 5
answers most practical causality questions in <50 ms on the entropic
codebase.

### Query surface (new MCP tools)

- `docs.causal_graph(symbol, direction, depth, filter_layer)` —
  returns the call/data-flow graph starting at `symbol`, walking
  `forward` (callees) or `backward` (callers) up to `depth`, optionally
  restricted to a library layer.
- `docs.boundary_calls(src_library, dst_library)` — returns all edges
  crossing between two library layers. Useful for enforcing architecture
  rules in the tracer's output.
- `docs.data_flow(field)` — for a struct field, returns writers → field
  → readers.

Existing `docs.search`, `docs.lookup_function`, etc. stay as-is.

### Index maintenance

- `inv docs --ast` re-runs the AST walk + rebuilds the ast_node /
  ast_edge tables.
- `inv docs` (default) rebuilds both FTS5 + AST in one pass.
- CI check: a pre-commit hook or CI step that fails if source changed
  without docs rebuild — prevents stale graph in shipped docs database.

## Implementation plan

### Phase 1 — extractor prototype

Write standalone `scripts/extract_ast.py` using `tree-sitter-cpp`
(simpler dep than clang libTooling). Target: emit JSON with nodes +
edges for a single compilation unit. Validate on
`src/core/engine.cpp` + verify graph matches hand-traced expectations.

### Phase 2 — schema + loader

Add SQLite schema to the existing docs database init. Write loader
that consumes extractor JSON and populates ast_node / ast_edge with
upserts keyed by (qualified_name, file, line).

### Phase 3 — MCP tool surface

Add `docs.causal_graph`, `docs.boundary_calls`, `docs.data_flow` to
the entropic server. Definition JSON goes in
`data/tools/entropic/` (or a new `data/tools/docs/` subdir — decide
during implementation). Return format: JSON graph with explicit node
+ edge arrays.

### Phase 4 — migrate tracer identity

Update `prompts/identity_tracer.md` to teach the tier to prefer
`docs.causal_graph` over multi-probe searches for execution-flow
questions. Keep `docs.search` available for fuzzy exploration.

### Phase 5 — integration test

Run a tracer-tier end-to-end on "explain run_turn's execution flow";
compare tool-call count + answer quality before/after. Target: 1
tool call replaces the current 5+.

## Trade-offs

- **Complexity.** ~800-1500 LOC of new extraction + tool code. Index
  rebuild goes from seconds (doxygen) to ~30s (full AST walk).
- **Binding fragility.** clang libTooling ties extraction to a specific
  clang version. `tree-sitter-cpp` is more portable but produces
  lower-fidelity AST (no full type resolution). Start with tree-sitter,
  upgrade to clang if the precision gap matters.
- **Schema churn.** The initial ast_edge schema will evolve as we
  discover edge types we need. Budget for 2-3 schema revisions before
  stabilizing.
- **Storage cost.** entropic is ~80 kLOC C++; estimated ~300 MB docs
  database (vs. ~50 MB today). Not a problem at current scale.

## Out of scope

- Cross-repo graphs (entropic consumers — e.g., PyChess example — linking
  to entropic symbols). Only in-repo AST for now.
- Runtime dynamic-dispatch resolution beyond virtual method tables.
  Function-pointer edges are annotated as unresolved; tracer tier
  handles via manual follow-up.
- LLM-friendly natural-language summaries of graph paths. The raw graph
  is structured enough for the model to narrate itself.

## Validation

Given the same "explain run_turn" question before and after:

- **Before:** ≥5 `docs.search` calls, final answer missing at least
  one cross-`.so` boundary hop.
- **After:** 1 `docs.causal_graph(symbol="AgentEngine::run_turn",
  direction=forward, depth=5, filter_layer=core)` call, final answer
  contains every call edge and correctly attributes library layers.

Regression test:
`tests/integration/test_docs_causal_graph.cpp` — for 5 known
call chains, assert the graph tool returns exactly the expected
nodes + edges.
