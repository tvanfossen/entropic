### 2026-05-14 — #49 landed (`f0fef29`)

The streaming-interrupt cancel chain on develop is correctly wired:
engine.interrupt() -> interrupt_flag_ -> events_.interrupt ->
stream_token_callback per-token poll -> *acc->cancel_flag = 1 ->
c_api wrapper polls -> std::atomic cancel set -> llama_cpp's decode
loop exits with finish_reason="cancelled". The v2.1.10
EngineCallbacks-shuffle fix (gh#40 fallout) is the most likely
incidental fix that closed the v2.1.7 race.

Added a single-line diagnostic log at response_generator.cpp:215
("Stream interrupt observed at token N; raising backend cancel_flag")
so any future "interrupt didn't propagate" report has a
consumer-side confirmation channel: presence of the log = chain
fires; absence = chain still broken.

Added regression test exercising the cancel chain end-to-end with
the existing MockInference (stream_token_by_token=true, on_stream_chunk
fires engine.interrupt() after the 2nd token, asserts the stream
stops within 1 more token).

### 2026-05-14 — #50 landed (`a0fcd30`)

New `entropic_set_critique_callbacks(handle, start_cb, end_cb, ud)`
C ABI fires from `ConstitutionalValidator::run_critique` immediately
before and after the synchronous `inference_->generate()` call.
Both callbacks fire regardless of verdict (compliant or violating).

Persistent-slot pattern: handle owns the authoritative slot
(`critique_start_cb` / `critique_end_cb` / `critique_cb_data` on
`engine_handle.h`), `rewire_critique_callbacks` helper re-applies
to each newly-built validator after every configure() reconstruction.
Validator-side state is a `CritiqueCallbacks` struct guarded by
`critique_cbs_mutex_`, mirroring the existing `AttemptBoundaryCb`
layout. Snapshot-then-invoke-outside-lock so a consumer handler
that re-registers cannot self-deadlock.

Why Option B over A/C is documented inline in the commit message and
the public header docstring. Option A (extend hook ctx with grammar
markers) couples consumer visibility to the hook system without a
clear schema; Option C (reuse VERIFYING) would conflate the 3ms
post-iteration verification with the 20-30s critique generation.

Test: "Critique start/end callbacks fire around the generate call
(gh#50)" — covers fire-order (S before E), counts (exactly once
each per validate()), and the cleared-slot edge.
