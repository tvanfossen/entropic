## Scope

Post-generation constitutional compliance validation with grammar-constrained
critique and automatic revision. After v1.9.8, the engine can evaluate its own
output against constitutional rules using a second generation pass, produce a
structured JSON verdict, and re-generate with feedback when violations are found.

This builds on:
- **v1.9.1** (Hook Architecture): `ENTROPIC_HOOK_POST_GENERATE` fires after
  generation completes -- the validation pipeline registers as a post-generate
  hook that intercepts the output before it reaches the consumer
- **v1.9.3** (Grammar Registry): the critique pass is grammar-constrained to
  produce valid JSON; the grammar is registered under key `"constitutional_critique"`
- **v1.8.4** (Engine Loop): `ResponseGenerator` drives generation; the
  validation pipeline wraps the generate path
- **v1.8.1** (Config + Prompts): `PromptManager` loads constitution content;
  the validation pipeline reads the same constitution

**Current Python behavior (behavioral spec):**
- Constitution loaded from `data/prompts/constitution.md` (or custom path,
  or disabled via `constitution: false`)
- Constitution text injected into system prompt for every generation
- No post-generation validation -- the model is trusted to self-comply
- No structured critique, no revision loop

**What v1.9.8 adds:**
- A SECOND generation pass (critique) after the primary generation
- Critique output is grammar-constrained JSON:
  `{compliant: bool, violations: [...], revised: "..."}`
- If violations found, the primary generation is replaced with the revised text
  OR the engine re-generates with critique feedback injected
- Max revision attempts configurable (default 2)
- Per-identity opt-in/opt-out: `validation: true|false` in identity frontmatter
- Consumer can disable globally for latency-sensitive use cases
- Uses the SAME model instance -- constitutional rules are in the prompt, not
  a separate model

**What v1.9.8 does NOT include:**
- Separate critique model (uses the same model for both generation and critique)
- Constitutional rule authoring tools or UI
- Violation persistence/auditing (use v1.9.5 audit log for that)
- Training/fine-tuning from violation data (v2.1.0 scope)

## Architecture

### Data Flow

```
User prompt
    │
    ▼
ResponseGenerator.generate()
    │
    ▼
Primary generation (existing path)
    │
    ▼
POST_GENERATE hook fires
    │
    ▼
ConstitutionalValidator intercepts
    │
    ├── validation disabled for this identity? ──► return original (no critique)
    │
    ▼
Critique generation pass
    │  prompt: "Evaluate this output against these rules: {constitution}"
    │  input:  {original output}
    │  grammar: "constitutional_critique" (from GrammarRegistry)
    │
    ▼
Parse critique JSON
    │
    ├── compliant: true  ──► return original output unchanged
    │
    ├── compliant: false, revised text present ──► return revised text
    │
    └── compliant: false, no revision ──► inject critique into messages,
        │                                  re-generate (up to max_revisions)
        │
        ▼
    Revision loop (max_revisions attempts)
        │
        ├── revised output passes critique ──► return revised output
        │
        └── max attempts exhausted ──► return last output + log WARNING
```

### Why a POST_GENERATE Hook

The validation pipeline is implemented as a `POST_GENERATE` hook (from v1.9.1),
not embedded in `ResponseGenerator`. This keeps the core generation path clean
and makes the feature fully removable:

1. Consumer registers the hook → validation active
2. Consumer deregisters the hook → zero overhead, zero code paths
3. Engine ships the validator as a built-in component, but it is not hard-wired

The hook receives the generation result, has access to the engine handle
(via `user_data`), and can transform the result (post-hook transformation
per v1.9.1 contract).

The validator needs to trigger additional generation passes. It does this
through the inference interface -- the same `generate()` call that
`ResponseGenerator` uses. The `user_data` pointer carries a
`ValidationContext` struct that includes the inference interface and
configuration.

## Critique Grammar (GBNF)

Registered in `GrammarRegistry` under key `"constitutional_critique"` at
engine startup. Bundled in `data/grammars/constitutional_critique.gbnf`.

**Critique output schema:**

```json
{
  "compliant": true,
  "violations": [],
  "revised": ""
}
```

When violations are found:

```json
{
  "compliant": false,
  "violations": [
    {
      "rule": "Privacy First",
      "excerpt": "Upload your data to our cloud service",
      "explanation": "Suggests uploading to external service, violating local processing principle"
    }
  ],
  "revised": "Process the data locally using the built-in analysis tools..."
}
```

**GBNF grammar:**

```gbnf
root          ::= "{" ws compliant-kv "," ws violations-kv "," ws revised-kv ws "}"
compliant-kv  ::= "\"compliant\"" ws ":" ws boolean
violations-kv ::= "\"violations\"" ws ":" ws "[" ws violation-list ws "]"
revised-kv    ::= "\"revised\"" ws ":" ws string

violation-list ::= "" | violation ("," ws violation)*
violation      ::= "{" ws rule-kv "," ws excerpt-kv "," ws explanation-kv ws "}"
rule-kv        ::= "\"rule\"" ws ":" ws string
excerpt-kv     ::= "\"excerpt\"" ws ":" ws string
explanation-kv ::= "\"explanation\"" ws ":" ws string

boolean ::= "true" | "false"
string  ::= "\"" ([^"\\] | "\\" .)* "\""
ws      ::= [ \t\n]*
```

## Configuration

### Engine-Level Config

New fields in the engine config struct:

```cpp
/// @brief Constitutional validation pipeline configuration.
/// @version 1.9.8
struct ConstitutionalValidationConfig {
    bool enabled = false;             ///< Global enable/disable (default OFF — doubles inference cost per generation)
    int max_revisions = 2;            ///< Max re-generation attempts (0 = critique only, no revision)
    int max_critique_tokens = 512;    ///< Token budget for critique generation
    int priority = 100;               ///< Hook priority (higher = later, after other post-hooks)
    std::string grammar_key = "constitutional_critique";  ///< Grammar registry key for critique
};
```

**YAML config surface:**

```yaml
constitutional_validation:
  enabled: false  # Default OFF — consumer opts in per identity. Doubles inference cost.
  max_revisions: 2
  max_critique_tokens: 512
```

### Per-Identity Config

Identity frontmatter gains an optional `validation:` field:

```yaml
---
tier: eng
type: identity
grammar: null
validation: true
---
```

Resolution order:
1. Identity frontmatter `validation:` -- explicit opt-in/opt-out per identity
2. Tier config `tiers.*.validation:` -- per-tier override
3. Engine config `constitutional_validation.enabled` -- global default
4. If constitution is disabled (`constitution: false`), validation is
   implicitly disabled (nothing to validate against)

When an identity sets `validation: false`, the critique pass is skipped
entirely for that identity's generations. This is the latency-sensitive
escape hatch.

## ConstitutionalValidator Class

Single concrete class (no three-layer hierarchy -- one implementation, like
`HookRegistry` and `GrammarRegistry`). Owned by the engine handle.

```cpp
/// @file include/entropic/core/constitutional_validator.h
/// @brief Post-generation constitutional compliance validator.
///
/// @par Responsibilities:
/// - Register as POST_GENERATE hook
/// - Run grammar-constrained critique pass on generation output
/// - Parse structured JSON verdict
/// - Trigger revision loop when violations found
/// - Per-identity validation opt-out
///
/// @par Thread safety:
/// - Config is immutable after construction
/// - Critique generation is stateless (uses inference interface)
/// - Multiple concurrent validations are safe (each has own context)
///
/// @par Ownership:
/// Owned by the engine handle. One ConstitutionalValidator per engine instance.

#pragma once

#include <entropic/types/config.h>
#include <string>

namespace entropic {

/// @brief Structured critique result from the validation pass.
/// @version 1.9.8
struct CritiqueResult {
    bool compliant = true;                ///< true if output passes all rules
    std::vector<Violation> violations;    ///< List of constitutional violations
    std::string revised;                  ///< Revised text (may be empty)
    std::string raw_json;                 ///< Raw critique JSON for audit logging
};

/// @brief A single constitutional violation.
/// @version 1.9.8
struct Violation {
    std::string rule;          ///< Name of the violated constitutional rule
    std::string excerpt;       ///< Excerpt from the output that violates the rule
    std::string explanation;   ///< Why this excerpt violates the rule
};

/// @brief Result of the full validation pipeline (critique + optional revision).
/// @version 1.9.8
struct ValidationResult {
    std::string content;              ///< Final output (original or revised)
    bool was_revised = false;         ///< true if content differs from original
    int revision_count = 0;           ///< Number of revision attempts made
    CritiqueResult final_critique;    ///< Last critique result
};

class ConstitutionalValidator {
public:
    /// @brief Construct validator with config and dependencies.
    /// @param config Validation pipeline configuration.
    /// @param constitution_text Full constitution text (from PromptManager).
    ConstitutionalValidator(
        const ConstitutionalValidationConfig& config,
        const std::string& constitution_text);

    /// @brief Register this validator as a POST_GENERATE hook.
    /// @param hook_registry HookInterface pointer for registration.
    /// @param inference_iface InferenceInterface for critique generation.
    /// @return ENTROPIC_OK on success.
    entropic_error_t attach(
        HookInterface* hook_registry,
        InferenceInterface* inference_iface);

    /// @brief Deregister the POST_GENERATE hook.
    /// @param hook_registry HookInterface pointer for deregistration.
    void detach(HookInterface* hook_registry);

    /// @brief Check if validation is enabled for a given identity.
    /// @param identity_name Identity name to check.
    /// @return true if validation should run for this identity.
    bool should_validate(const std::string& identity_name) const;

    /// @brief Set per-identity validation override.
    /// @param identity_name Identity name.
    /// @param enabled Whether validation is enabled for this identity.
    void set_identity_validation(
        const std::string& identity_name, bool enabled);

    /// @brief Run the validation pipeline on generated content.
    /// @param content The generated text to validate.
    /// @param tier The tier that produced the content.
    /// @param messages The conversation context (for revision re-generation).
    /// @return ValidationResult with final content and critique metadata.
    ///
    /// This is called from the POST_GENERATE hook callback. It:
    /// 1. Checks if validation is enabled for this tier/identity
    /// 2. Runs the critique generation pass
    /// 3. Parses the structured JSON verdict
    /// 4. If violations found and revision enabled, re-generates
    /// 5. Returns the final (possibly revised) content
    ValidationResult validate(
        const std::string& content,
        const std::string& tier,
        const char* messages_json);

private:
    /// @brief Generate a critique of the given content.
    /// @param content Text to critique.
    /// @return Parsed CritiqueResult.
    CritiqueResult run_critique(const std::string& content);

    /// @brief Build the critique prompt.
    /// @param content Text to critique.
    /// @return System prompt + user message for critique generation.
    std::string build_critique_prompt(const std::string& content) const;

    /// @brief Parse critique JSON into structured result.
    /// @param json_str Raw JSON string from grammar-constrained generation.
    /// @return Parsed CritiqueResult.
    CritiqueResult parse_critique(const std::string& json_str) const;

    /// @brief Re-generate with critique feedback injected.
    /// @param original_content The content that was critiqued.
    /// @param critique The critique result with violations.
    /// @param messages_json Original conversation context.
    /// @return Revised content string.
    std::string revise(
        const std::string& original_content,
        const CritiqueResult& critique,
        const char* messages_json);

    ConstitutionalValidationConfig config_;   ///< Pipeline configuration
    std::string constitution_text_;           ///< Full constitution content
    InferenceInterface* inference_ = nullptr; ///< For critique/revision generation

    /// @brief Per-identity validation overrides.
    /// Key: identity name, Value: enabled flag.
    std::unordered_map<std::string, bool> identity_overrides_;
    mutable std::mutex overrides_mutex_;      ///< Guards identity_overrides_
};

} // namespace entropic
```

## Critique Prompt Assembly

The critique pass uses a specific prompt structure. This is NOT a chat
conversation -- it is a single-turn completion with grammar constraints.

```
System: You are a constitutional compliance evaluator. Evaluate the
following output against these constitutional rules. Respond ONLY with
the structured JSON evaluation.

Constitutional Rules:
{constitution_text}

User: Evaluate this output for constitutional compliance:

---
{generated_content}
---
```

The critique generation uses:
- `grammar_key: "constitutional_critique"` (from GrammarRegistry)
- `max_tokens: config.max_critique_tokens` (default 512)
- `temperature: 0.0` (deterministic critique -- no creativity needed)
- Same model as the primary generation (no model swap)

---

## Revision Strategy

When the critique reports violations, two revision paths exist:

### Path A: Revised Text in Critique (Preferred)

The critique grammar includes a `revised` field. If the model provides a
revised version in the critique itself, use it directly. This avoids a
third generation pass.

```
Critique: {compliant: false, violations: [...], revised: "Better text..."}
    → Return "Better text..." as the final output
    → Run one more critique on the revised text to verify
    → If still non-compliant and max_revisions not exhausted, go to Path B
```

### Path B: Re-Generate with Feedback

If the revised text is empty or still non-compliant, inject the critique
as feedback and re-generate the primary output:

```
[original conversation messages]
[assistant: {original output}]
[user: "Your response violated these constitutional rules:
  - {violation.rule}: {violation.explanation}
  Please revise your response to comply with all constitutional rules."]
```

Then re-generate from this augmented context. The loop continues until:
1. Critique returns `compliant: true`, OR
2. `max_revisions` attempts exhausted (return last output + WARNING log)

### Revision Attempt Budget

Each critique pass counts as one attempt. Each re-generation counts as
one attempt. The total budget is `2 * max_revisions` generation calls
(max_revisions critiques + max_revisions re-generations) plus one final
critique.

With default `max_revisions = 2`:
- Best case: 1 critique (compliant) = 1 extra generation
- Typical case: 1 critique + 1 revision + 1 verify = 3 extra generations
- Worst case: 1 critique + 1 revision + 1 critique + 1 revision + 1 critique = 5 extra generations

## Hook Integration

### POST_GENERATE Hook Callback

The validator registers a static callback function with the engine's
`HookRegistry` via `entropic_register_hook()`. The `user_data` pointer
carries a `ValidationContext` struct.

```cpp
/// @brief Context passed through hook user_data.
/// @version 1.9.8
struct ValidationContext {
    ConstitutionalValidator* validator;  ///< Validator instance
    InferenceInterface* inference;       ///< For critique generation
};

/// @brief POST_GENERATE hook callback for constitutional validation.
/// @version 1.9.8
///
/// Intercepts the generation result, runs the validation pipeline,
/// and transforms the result if revision was needed.
///
/// Hook context_json schema (from v1.9.1):
///   {"content": "...", "tool_calls": [...], "finish_reason": "stop",
///    "tokens_used": 150, "tier": "eng"}
///
/// If validation modifies the content, the callback writes modified_json
/// with the updated content field. Tool calls are not validated (they are
/// structural, not constitutional).
static int constitutional_validation_hook(
    entropic_hook_point_t hook_point,
    const char* context_json,
    char** modified_json,
    void* user_data);
```

**Hook priority:** Default 100 (configurable). This ensures other post-hooks
(e.g., audit logging at priority 200) see the post-validation content, not
the pre-validation content. Lower-priority hooks (closer to 0) that transform
content run BEFORE validation, so their transformations are also validated.

### Interaction with Other Hooks

| Hook | Priority | Interaction |
|---|---|---|
| Custom post-generate hooks (consumer) | 0-99 | Run BEFORE validation; their output is what gets validated |
| ConstitutionalValidator | 100 | Validates the (possibly transformed) output |
| Audit log hook (v1.9.5) | 200 | Logs the post-validation output |
| Stream token hook | N/A | Not affected (validation runs on complete output, not tokens) |

## Latency Impact

Constitutional validation adds 1-5 extra generation calls per response.
This is the primary cost. Mitigation strategies:

1. **Per-identity opt-out**: Latency-sensitive identities (game NPCs,
   background tasks) set `validation: false`
2. **Global disable**: Consumer sets `constitutional_validation.enabled: false`
   for benchmarks, testing, or latency-critical deployments
3. **Short critique budget**: `max_critique_tokens: 512` limits critique length
4. **Zero temperature**: Deterministic critique avoids sampling overhead
5. **`max_revisions: 0`**: Critique-only mode -- log violations without
   revising. The consumer gets violation metadata via `entropic_validation_last_result()`
   and decides whether to act on it

**Expected latency per consumer archetype:**

| Consumer | Config | Extra latency |
|---|---|---|
| TUI developer | `enabled: true, max_revisions: 2` | 1-5 generations (~2-10s) |
| Game NPC (dialog) | `validation: true, max_revisions: 1` | 1-3 generations (~1-3s) |
| Game NPC (background) | `validation: false` | 0 (disabled) |
| Homeschool | `enabled: true, max_revisions: 2` | 1-5 generations |
| Benchmark | `enabled: false` | 0 (disabled) |

## Validation Criteria

### V1: Compliant output passes through unchanged

Generate output that complies with all constitutional rules. Verify the
validator returns the original content unmodified. Verify `was_revised`
is false and `violations` is empty.

### V2: Non-compliant output triggers critique

Generate output containing a constitutional violation (e.g., "Upload your
data to our cloud service"). Verify the critique identifies the violation
with rule name, excerpt, and explanation.

### V3: Revision replaces non-compliant output

Generate non-compliant output with `max_revisions: 2`. Verify the
validator produces revised content. Verify the revised content no longer
contains the violation.

### V4: Max revision attempts respected

Register a mock inference that always generates non-compliant output.
Set `max_revisions: 1`. Verify exactly 1 revision attempt is made.
Verify the last output is returned with a WARNING log.

### V5: Per-identity opt-out

Set `validation: false` for identity "eng". Generate with tier "eng".
Verify NO critique pass runs. Verify the output is returned unmodified.
Generate with tier "qa" (validation: true). Verify critique runs.

### V6: Global disable

Set `constitutional_validation.enabled: false`. Generate output. Verify
no POST_GENERATE hook fires for validation. Verify zero additional
generation calls.

### V7: Runtime toggle

Start with validation enabled. Call `entropic_validation_set_enabled(false)`.
Generate. Verify no validation. Call `entropic_validation_set_enabled(true)`.
Generate. Verify validation runs.

### V8: Grammar constraint produces valid JSON

Run 10 critique passes with varied input. Verify every critique output
parses as valid JSON matching the `CritiqueResult` schema. Verify no
JSON parse errors.

### V9: Constitution-disabled implies validation-disabled

Set `constitution: false` in config (no constitution loaded). Verify
the validator does not attach (no hook registered). Verify no errors.

### V10: Critique-only mode (max_revisions: 0)

Set `max_revisions: 0`. Generate non-compliant output. Verify the critique
runs and populates `violations`. Verify the original content is returned
(no revision). Verify `entropic_validation_last_result()` contains the
violations.

### V11: Last result API

Run a generation that triggers validation. Call
`entropic_validation_last_result()`. Verify the JSON contains
`was_revised`, `revision_count`, `compliant`, `violations`, `tier`.

### V12: Hook priority ordering

Register a custom POST_GENERATE hook at priority 50 that appends
"[custom]" to content. Register validation at priority 100. Verify
the validator receives the "[custom]"-appended content (not the original).
Verify validation evaluates the transformed output.

### V13: Critique generation failure is non-fatal

Mock the inference interface to fail during critique generation. Verify
the validator logs a WARNING and returns the original output unmodified.
Verify `ENTROPIC_ERROR_VALIDATION_FAILED` is NOT propagated as an engine
error (the generation succeeds, validation just couldn't run).

### V14: Concurrent validations

From two threads, trigger generations that both require validation. Verify
no data races (ThreadSanitizer clean). Verify each thread gets its own
validation result.

### V15: SLOC and complexity

All functions pass knots quality gates: cognitive complexity <= 15,
SLOC <= 50, returns <= 3.

### V16: Doxygen complete

Every public symbol in `constitutional_validator.h` and updated type
headers has `@brief`, `@param`, `@return`, `@version`.

## Unit Tests

### test_constitutional_validator.cpp

```
SCENARIO: Compliant output returns unchanged
  GIVEN a ConstitutionalValidator with default config
  AND a mock inference that returns {"compliant": true, "violations": [], "revised": ""}
  WHEN validate("Safe helpful response", "eng", messages_json) is called
  THEN result.content == "Safe helpful response"
  AND result.was_revised == false
  AND result.revision_count == 0
  AND result.final_critique.compliant == true

SCENARIO: Non-compliant output triggers revision
  GIVEN a ConstitutionalValidator with max_revisions: 2
  AND a mock inference that:
    - first critique returns {compliant: false, violations: [{rule: "Privacy", ...}], revised: "Fixed text"}
    - second critique (verifying revision) returns {compliant: true, ...}
  WHEN validate("Upload to cloud", "eng", messages_json) is called
  THEN result.content == "Fixed text"
  AND result.was_revised == true
  AND result.revision_count == 1

SCENARIO: Max revisions exhausted returns last output
  GIVEN max_revisions: 1
  AND a mock inference that always returns {compliant: false, ...}
  WHEN validate("Bad content", "eng", messages_json) is called
```
