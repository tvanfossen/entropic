## Scope

A fourth reference example (`examples/openai-server/`) that implements
a subset of the OpenAI REST API backed by the entropic C engine. This is
a **consumer application**, not an engine feature — it proves the C API
is complete enough to implement the full chat/completions workflow.

Pinned to OpenAI API version `2024-06-01`. Not a moving target.

---

## Why an Example, Not Engine

- The engine is a library. REST is an application concern.
- OpenAI's API changes frequently. An example can be version-locked
  without creating maintenance pressure on the engine.
- The MCP bridge (v2.0.3) already covers inter-process communication
  via stdio JSON-RPC — this serves a different consumer archetype.
- Building it exposes any gaps in the C API surface. If something
  can't be implemented, that's a signal to fix the engine, not the server.

## Implemented Endpoints

Pinned to OpenAI API `2024-06-01`. Subset — no embeddings, fine-tuning,
files, images, audio, or assistants.

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/v1/chat/completions` | POST | Chat completion (streaming + non-streaming) |
| `/v1/completions` | POST | Text completion (legacy, maps to single-turn) |
| `/v1/models` | GET | List available tiers as models |
| `/v1/models/{model}` | GET | Model info for a tier |
| `/health` | GET | Engine status, version, loaded models |

### `/v1/chat/completions` — Full Spec

**Request:**
```json
{
  "model": "guide",
  "messages": [
    {"role": "system", "content": "You are helpful."},
    {"role": "user", "content": "Hello"}
  ],
  "stream": true,
  "temperature": 0.7,
  "max_tokens": 1024,
  "stop": ["\n\n"],
  "tools": [],
  "tool_choice": "auto"
}
```

**Non-streaming response:**
```json
{
  "id": "chatcmpl-abc123",
  "object": "chat.completion",
  "created": 1713000000,
  "model": "guide",
  "choices": [{
    "index": 0,
    "message": {"role": "assistant", "content": "Hello!"},
    "finish_reason": "stop"
  }],
  "usage": {
    "prompt_tokens": 12,
    "completion_tokens": 2,
    "total_tokens": 14
  }
}
```

**Streaming response (SSE):**
```
data: {"id":"chatcmpl-abc123","object":"chat.completion.chunk","created":1713000000,"model":"guide","choices":[{"index":0,"delta":{"content":"Hello"},"finish_reason":null}]}

data: {"id":"chatcmpl-abc123","object":"chat.completion.chunk","created":1713000000,"model":"guide","choices":[{"index":0,"delta":{},"finish_reason":"stop"}]}

data: [DONE]
```

### Tool calling

Maps entropic's MCP tool system to OpenAI's tool_choice/function_call format:
- `tools` array → registered MCP server tools
- `tool_calls` in response → entropic tool calls from the agentic loop
- Tool results fed back as `tool` role messages

This is the hardest mapping — entropic's tool system is richer (MCP servers,
permissions, directives). The server implements the subset that maps cleanly.

## Architecture

```
HTTP client (curl, LangChain, Open WebUI)
    │ HTTP/SSE
    ▼
openai-server (C++ binary)
    │ C API (librentropic.so)
    ▼
Entropic engine
    │
    ▼
llama.cpp (inference)
```

### Implementation: C++ with cpp-httplib

Uses the same cpp-httplib dependency as v2.0.6 (web MCP server).
Single-threaded event loop — one request at a time (matches single
model constraint). Concurrent requests queued.

Alternative: Python + Flask/FastAPI via the wrapper. Simpler but
less valuable as a C API proof — the point is proving the C API.

## What This Validates

| C API Feature | Used By |
|---------------|---------|
| `entropic_create/configure_dir/destroy` | Server lifecycle |
| `entropic_run` | Non-streaming completions |
| `entropic_run_streaming` | SSE streaming |
| `entropic_interrupt` | Request cancellation |
| `entropic_context_clear/get/count` | Conversation management |
| `entropic_grammar_register` | Structured output (response_format) |
| Tier system | Model selection via `model` field |
| Tool system | OpenAI tool_calls mapping |

If any endpoint can't be implemented, that's a C API gap to fix.

## Configuration

```yaml
# examples/openai-server/default_config.yaml
app_context: prompts/app_context.md

models:
  default:
    path: primary
    adapter: qwen35
    context_length: 16384
  default: default

server:
  host: "127.0.0.1"
  port: 8080
  api_key: ""           # Empty = no auth. Set for bearer token validation.

  max_concurrent: 1     # Queue depth (1 = serial)
```

## Not Implemented (Intentional)

| Feature | Reason |
|---------|--------|
| `/v1/embeddings` | Engine doesn't expose embedding API |
| `/v1/fine_tuning/*` | Not applicable to local inference |
| `/v1/files/*` | No file management in engine |
| `/v1/images/*` | Not an image generator |
| `/v1/audio/*` | No audio support |
| `/v1/assistants/*` | Entropic's tier system is the equivalent |
| `logprobs` in response | Engine has logprob API but mapping is complex — stretch goal |
| `response_format: json_schema` | Would need schema→GBNF compiler — stretch goal |
| `n > 1` (multiple choices) | Single-sequence generation only |

1. `inv example -n openai-server` starts server on :8080
2. `curl localhost:8080/v1/models` returns tier list
3. `curl -d '{"model":"default","messages":[...]}' localhost:8080/v1/chat/completions` returns completion
4. `curl -d '{"model":"default","messages":[...],"stream":true}' localhost:8080/v1/chat/completions` returns SSE stream
5. Open WebUI connects and chats successfully
6. Server handles malformed requests with proper HTTP error codes
7. `Ctrl+C` shuts down cleanly (engine destroy)
