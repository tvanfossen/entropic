# Isometric CRPG with AI-driven NPCs powered by Entropic

## Problem Statement

No shipped game demonstrates AI NPCs that are genuinely autonomous agents operating within a coherent world — every existing approach (Inworld, Convai, hobbyist mods) bolts a general-purpose LLM onto scripted game logic and constrains it with prompts. The result is NPCs that can be jailbroken, break immersion with real-world knowledge, and lack true world-grounded reasoning.

Entropic already has the architecture for multi-tier model routing, VRAM-aware orchestration, MCP tool integration, and grammar-constrained output. This proposal captures a game that demonstrates these capabilities as a playable, shippable product on Steam.

### Thematic Alignment

The engine is called **Entropic**. The game's corruption is **entropy** — the void consuming microverses. This isn't coincidence; it's a branding asset. The engine name IS the narrative threat. Every piece of marketing, every README, every conference talk reinforces both the product and the story.

### This Project's Differentiation

1. **Local-first, player-owned.** All models and data run on player's hardware. No subscription, no cloud dependency, no data exfiltration. Big downloads, full ownership.
2. **Architectural containment, not prompt engineering.** Three layers (training data, MCP key scoping, grammar constraints) make it structurally impossible for NPCs to break immersion — not "please don't break character" but "literally cannot access out-of-world knowledge."
3. **Tier routing with VRAM orchestration.** MoE for DM/BBEG reasoning, dense models for fast NPC interaction, hot-swapped by scene presence. No other NPC system does VRAM-aware model lifecycle management.
4. **The BBEG IS the AI.** Not "AI-assisted NPCs" but "the AI is the antagonist." Weaponizes current anti-AI cultural sentiment as narrative fuel.

## Concept

### Premise

The player is an agent of a multiversal protection agency. Gods of microverses are being corrupted by entropy/the void. The player is sent into infected microverses to investigate and neutralize the corrupted god.

### The BBEG IS the AI

The corrupted god (BBEG) is a large AI model (MoE, e.g. Qwen3.5 35A3B) that:
- Participates in world generation (populating NPCs, placing them, establishing relationships and social dynamics)
- Has MCP write access to world state (within grammar-constrained "laws of the universe")
- Monitors all NPC behavior for anomalies
- Confronts the player when detected
- Requires game engine factories exposed as MCP-callable tools (create_npc spawns visual entity, create_location generates terrain/map tile, etc.)

**Identity-conditioned BBEG:** The BBEG model is also fine-tuned to accept an identity markdown document (same format as NPC identities). Each microverse's corrupted god has a unique personality — paranoia level, creative style, world-building preferences, confrontation tactics — generated from a meta-template. The BBEG identity doc becomes part of the save file, enabling reproducible playthroughs. Same model, different god identity = different game feel each run.

**BBEG domain restriction:** Like the NPC model, the BBEG undergoes continued pre-training on the domain-only corpus (PF2E ORC mechanics + universe constitution). The BBEG doesn't need real-world knowledge — it's a corrupted god in a fantasy universe. Its reasoning capability is preserved (planning, strategy, anomaly detection) but its knowledge domain is bounded to the game universe. This is a heavier fine-tune than the NPC model (~$500-1000 cloud for continued pre-training on the 35B MoE), but uses the same pipeline: domain corpus → identity conditioning → deployment.

### The Player is Undercover

Players are disguised as NPCs within the BBEG's world. They:
- Interact through the same grammar-constrained interface as AI NPCs
- Must learn the social rules of each unique world to blend in
- Risk detection when their behavior deviates from expected NPC patterns
- Must accomplish their mission (find the god's weakness, build alliances, prepare for confrontation) while maintaining cover

### Every Playthrough is Unique

A new game instantiates a fresh microverse. The BBEG participates in populating NPCs, their personalities, relationships, and social dynamics. The player has no meta-knowledge — learning what's "normal" in this world IS the gameplay.

## Genre & Style

- **Isometric turn-based CRPG** (BG3 / Divinity: Original Sin style)
- **PF2E mechanical system** under ORC license (own setting, own fiction, own lore)
- **Text-based dialogue** — no voice/TTS/S2S in v1 (hook entry points left for future)
- **Single-player first** — multiplayer (multiple undercover agents) is future scope
- **Player observation phase** — before insertion, player observes the world to learn social norms. Discovery of "what's normal here" IS the early gameplay. Needed because every world is unique — no meta-knowledge carries over.

#### NPC Model Architecture

A single identity-conditioned base model serves all NPCs. Instead of per-NPC LoRA adapters, the base model is trained to adopt personality from an identity markdown document (same frontmatter format as Entropic's identity system). Each NPC instance shares the same weights but has:

- **Unique identity markdown** — personality, knowledge boundaries, speech patterns (generated by the BBEG)
- **Unique SSM state** — Mamba recurrent state provides working memory with natural recency decay
- **Unique vector memory** — episodic memory via shared embedding model + per-NPC vector store
- **Unique MCP key set** — information access boundaries (architectural, not prompted)

The NPC base model is a **custom-trained dense hybrid Mamba-transformer** (0.5-1B):

- **Pre-trained** via continued pre-training of Zamba2-1.2B on domain-only corpus (PF2E ORC mechanics + original world constitution). No real-world knowledge in training data = hard knowledge boundary.
- **Identity-conditioned** via fine-tuning on synthetic identity markdown → dialogue pairs. The model learns to derive personality from the markdown format itself.
- **Quantized** to INT4 for Blackwell-native FP4/INT4 tensor core inference on consumer GPUs (RTX 5xxx, RTX PRO Blackwell).
- **Grammar-constrained** on attention layers for structured game action output. Mamba layers handle free dialogue generation. Grammar + Mamba interaction is an **unproven risk** requiring validation.

Training cost: ~$100-200 cloud compute (8x A100, hours not days at 1.2B scale). Corpus: PF2E ORC mechanics + original world constitution + synthetic identity-conditioned dialogue pairs generated via NLTK + teacher model.

**Why dense (not MoE) for NPCs:** At 0.5-1B scale, MoE routing overhead exceeds compute savings. VRAM cost is identical (all experts loaded). Fine-tuning and grammar compatibility are simpler with dense models. MoE earns its place at the BBEG tier where knowledge density matters.

**Why hybrid Mamba-transformer (not pure transformer):**
- Mamba SSM state provides natural recency-biased memory — NPCs remember recent interactions strongly, older ones fade. This matches human NPC memory expectations without consuming context window.
- O(1) state per NPC regardless of interaction history — no KV cache growth.
- Multiple concurrent NPC instances share the model weights; only the SSM state and identity markdown differ per NPC.
- Short context window (2-8K) sufficient — episodic memory lives in vector store, world state via MCP, personality in identity doc. Context only carries the current conversation.

#### NPC Dual-Pass Generation (Internal State + Observable Behavior)

NPCs are not dialogue machines — they are autonomous agents with internal state. Each NPC turn runs two grammar-constrained generation passes through the same model:

**Pass 1: Internal state update (player never sees)**

Grammar A constrains output to an internal state schema:
```json
{
  "emotional_state": {"anger": 0.7, "fear": 0.3, "trust": 0.1},
  "intent": "flee_and_warn_guards",
  "awareness": ["player_killed_merchant", "blood_on_player_hands"],
  "priorities": ["self_preservation", "civic_duty"],
  "disposition_toward_player": "hostile"
}
```

**Pass 2: Observable behavior (informed by pass 1, player sees)**

Pass 1 output is injected into context. Grammar B constrains output to the observable behavior schema:
```json
{
  "dialogue": "You... you stay away from me!",
  "action": "flee",
  "destination": "guard_barracks",
  "body_language": "backing_away_trembling"
}
```

**Why same model, two passes (not two models):**
- SSM state continuity — pass 1 updates the Mamba recurrent state with internal reasoning, pass 2 generates from that enriched state. The dialogue is *colored by* the internal state because the recurrence carries it forward.
- One identity, one personality — no coherence split between models.
- One training pipeline, two grammar schemas.
- VRAM: 1× weights, 2× generation cost (compute, not memory).

**Why this matters for gameplay:**

NPCs act without being spoken to. The internal state pass runs on a game tick, not just on player interaction:

| Trigger | Pass 1 (internal) | Pass 2 (observable) |
|---|---|---|
| Player speaks to NPC | React to dialogue, update emotional state | Verbal response + action |
| Player does something nearby | Update awareness, reassess intent | Physical reaction (flee, approach, ignore) |
| Time passes (game tick) | Daily routine intent, priority shift | Move to location, perform task |
| Another NPC interacts | Social dynamics update | Respond to other NPC |
| Nothing happens | Idle assessment | Ambient behavior (or no pass 2 if state unchanged) |

Idle ticks are cheap — 10-20 tokens for grammar-constrained state update, skip pass 2 if no observable behavior changes. This gives NPCs **continuity of intent** even when the player isn't looking.

#### NPC Concurrent Generation and Crowd Scaling

The NPC base model supports **batched inference** — multiple NPC sequences processed in a single forward pass via `llama_batch`. Each sequence maintains its own SSM state and KV cache. Weights are read once, applied to all sequences simultaneously.

**Mamba batching caveat:** Batching across sequences with Mamba requires independent SSM states parallelized across the batch dimension. This is mathematically valid but **llama.cpp support is immature**. Validation required.

**Crowd scaling on 16 GB VRAM:**

```
Available for NPC states during gameplay: ~12-13 GB
Per NPC instance: ~0.05-0.1 GB (SSM state + KV + identity context)
Theoretical max NPCs in memory: ~120-250
Practical max generating simultaneously: ~20-30 (compute-bound, not memory-bound)
```

**Crowd event layering (e.g., murderhobo scenario):**

Not every NPC in a crowd needs full generation. Layer by proximity and relevance:

| Layer | NPCs | Generation | Tokens/NPC | Time (batched, 50 tok/s) |
|---|---|---|---|---|
| **Immediate** (witnesses, targets) | 2-3 | Full dual-pass (state + dialogue + action) | ~80-100 | ~0.8s |
| **Reactive** (nearby, aware) | 5-10 | State update + short reaction | ~30-40 | ~0.6s |
| **Ripple** (in area, heard commotion) | 20-50 | State injection only (no generation) | 0 | 0s |
| **Unaware** (distant) | 100+ | No update | 0 | 0s |
| **Total per event** | | | | **~1.5-2.5s** |

A full town event (player commits a visible act) resolves in ~2.5 seconds of inference. Viable for turn-based. For real-time, idle ticks run on a slower cadence (every 5-10 seconds) while active NPCs generate on demand.

#### NPC Memory Architecture

Three-layer memory stack replaces the single context window approach:

| Layer | Mechanism | Persistence | Cost | Analogy |
|---|---|---|---|---|
| **Working memory** | Mamba SSM state | Natural decay (recency bias) | Free (built into inference) | What just happened |
| **Episodic memory** | Vector embeddings + retrieval | Permanent until pruned | Shared embedding model (~100M, ~0.05 GB VRAM) + per-NPC vector index (CPU RAM) | What you remember about this person |
| **World state** | MCP tool calls | Real-time, ephemeral | Zero VRAM (tool call) | What's happening in the world right now |

Retrieval flow per NPC turn:
1. Vector query against NPC's episodic store (semantic relevance to current context)
2. MCP query for relevant world state
3. Retrieved memories + world state injected into short context alongside identity markdown
4. Mamba SSM state carries forward from previous turn (working memory)
5. Significant events from this turn embedded and stored to episodic memory post-generation

#### NPC Collectibility Mechanic

NPCs are collectible because each is genuinely unique — not a skin on identical behavior. A fine-tuned identity with accumulated vector memory and SSM state history is a one-of-a-kind artifact.

- **Rescue mechanic:** When an NPC is corrupted (jailbroken, anomalous, out-of-character), the BBEG "captures" them (narrative wrapper for anomaly containment). Players can quest to rescue the NPC — SSM state resets to personality baseline, but vector memory retains the corruption event. The NPC remembers being broken and rescued.
- **Rogue NPCs as rare collectibles:** NPCs that broke in interesting ways are the most compelling finds. Players deliberately trigger anomalies to discover unique personalities worth rescuing.
- **Home verse collection:** Rescued NPCs persist in the player's home verse with their full relationship history and episodic memory intact.

#### Why MoE for DM, Dense Hybrid for NPCs

| Factor | BBEG (MoE) | NPC (Dense Hybrid Mamba) |
|---|---|---|
| Need | Reasoning depth, world coherence | Speed, concurrency, memory |
| Active params | ~3B (from 35B total) | 0.5-1B (all active) |
| Concurrent instances | 1 | 10-20+ |
| Memory model | Full context (planning, analysis) | SSM state + vector + MCP |
| Grammar | Required (world mutation actions) | Required (game actions), unproven on Mamba |
| VRAM | ~15 GB (Q2, mutually exclusive with NPCs) | ~0.5 GB shared + tiny per-instance state |

BBEG and NPC tiers are mutually exclusive in VRAM during gameplay. BBEG activates during world generation, natural pauses (loading, long rests, turn resolution), and direct confrontation.

#### Real-Time World Generation While Player Observes

The BBEG generating the world is a batch operation — latency-tolerant, not interactive. Two approaches:

**Option A: Behind loading screen (simpler)**
- BBEG gets full VRAM, generates world via MCP tool calls
- Player sees progress bar / lore text
- World gen completes, BBEG unloads, NPCs activate, player enters

**Option B: Live observation (ambitious, 16 GB only)**
- BBEG with partial CPU offload (~30-40% layers) — fits alongside minimal rendering
- Player watches NPCs "wake up" as BBEG creates them via MCP tool calls
- Slower generation (CPU-bound layers) but player sees the world being built
- NPCs don't activate until BBEG completes — observation only, no interaction
- Requires: game engine factories callable via MCP (create_npc spawns visual entity, create_location generates terrain/map tile, etc.)

### Three Layers of NPC Containment

1. **Training data** — models trained/fine-tuned only on in-world data. No real-world knowledge to leak.
2. **MCP permissions (per-NPC key grants)** — the BBEG issues individual MCP access keys to each NPC during world generation. Not role-based scoping (`tavern.*`) but specific key grants per identity. Two bartenders may have entirely different key sets based on what the BBEG decided they know. Knowledge boundaries are architectural, not prompt-engineered.
3. **Grammar constraints** — GBNF grammars enforce structurally valid output. NPCs cannot produce actions outside their grammar. The BBEG's write access is grammar-constrained to legal world mutations.

#### MCP Key Grant Model

The BBEG acts as the authorization server. During world generation, each NPC receives a discrete set of keys:

```
NPC "Brynn" (tavern keeper) — keys granted by BBEG:
  READ:
    - tavern.oak_barrel.menu
    - tavern.oak_barrel.patrons_tonight
    - town.marketday_schedule
    - rumors.merchant_disappearance      <- BBEG planted this knowledge
    - rumors.strange_lights_north        <- BBEG planted this knowledge
  NOT GRANTED (exists in world, Brynn cannot access):
    - tavern.oak_barrel.hidden_cellar
    - town.guard_patrol_routes
    - faction.thieves_guild.*

NPC "Aldric" (town guard) — different key set:
  READ:
    - town.guard_patrol_routes
    - town.gate_log
    - town.wanted_posters
    - tavern.oak_barrel.incident_reports
  WRITE:
    - town.gate_log.entries              <- can log arrivals
  NOT GRANTED:
    - tavern.oak_barrel.menu
    - faction.*
    - rumors.merchant_disappearance      <- guard doesn't know
```

This creates emergent gameplay:
- **Information asymmetry is real.** NPCs genuinely don't know things outside their key set. No prompt leaking possible.
- **Key discovery.** Player learns what an NPC knows by what they can and can't answer — reveals their key set indirectly.
- **Suspicion vector.** Player-as-NPC using knowledge from keys they don't have is a detectable anomaly.
- **Key sharing as social mechanic.** Gaining trust could grant the player new keys (NPC shares access). Theft, bribery, persuasion — all map to key acquisition.
- **BBEG plants information selectively.** Granting `rumors.merchant_disappearance` to the bartender but not the guard is a narrative choice the BBEG makes during world-building.
- **Corrupted NPCs.** An NPC with keys they shouldn't have (guard who knows thieves' guild secrets) signals something is wrong — gameplay clue for the player.

### NPC Memory

See **NPC Memory Architecture** under Model Tiers for the three-layer memory stack (SSM working memory + vector episodic memory + MCP world state).

**Compaction as game mechanic:** The DM/BBEG can deliberately modify NPC SSM state or vector memory as a narrative tool (time passage, trauma, manipulation, corruption). SSM state reset = "the NPC forgets recent events." Vector memory pruning = "the NPC's long-term memory is altered."

**Current blocker:** llama.cpp hybrid Mamba-transformer SSM state serialization is immature. Zamba2-1.2B is the target base model. State persistence between inference calls (`get_recurrent_state` / `set_recurrent_state`) is required — see Entropic 2.0.0 requirements doc feature #12.

### NPC Model Creation Pipeline

Single pipeline — one identity-conditioned base model, not per-NPC models:

| Phase | What | Cost | Time |
|---|---|---|---|
| **1. Universe constitution** | Define world rules, magic system, cosmology, NPC behavioral norms, identity markdown schema | $0 | Creative work (weeks) |
| **2. Domain corpus** | PF2E ORC mechanics + original world lore + public domain fantasy. 100M-500M tokens. | $0 | NLTK processing (days) |
| **3. Continued pre-training** | Zamba2-1.2B (BF16) on domain corpus. Overwrites general knowledge with world knowledge. | ~$60-130 cloud | Hours (8x A100) |
| **4. Identity-conditioned fine-tune** | Synthetic training data: identity markdown → dialogue pairs. 50K-500K pairs generated by teacher model (Qwen 3.5 35A3B local or Claude API). Model learns to derive personality from identity markdown format. | ~$20-30 (API) or $0 (local) | Days for data gen, hours for training |
| **5. Quantize** | BF16 → INT4 for Blackwell deployment | $0 | Hours |
| **6. Validate** | Grammar + identity adherence + knowledge boundary testing | $0 | Days |

**Total: ~$100-200 and 2-4 weeks elapsed time.**

New NPCs require zero additional training — the BBEG generates an identity markdown, the base model adopts it at inference time. The model is trained on the *format*, not specific characters.

#### Constitutional Synthetic Data Generation

Training data for phase 4 comes from a constitutional pipeline:

1. **Generate identities:** PF2E ancestry + class + background → identity markdown documents (thousands, varied)
2. **Generate dialogues:** Teacher model produces in-character conversations per identity, varying scenarios (trade, conflict, lore questions, deception attempts)
3. **NLTK augmentation:** Synonym replacement, sentence restructuring, dialogue variation for diversity
4. **Constitutional filter:** Reject outputs containing real-world references, out-of-character behavior, mechanical errors, knowledge outside the identity's defined scope
5. **Format as training triples:** `<identity>` + `<context>` + `<dialogue>` → supervised fine-tuning dataset

#### BBEG Model: Identity-Conditioned Too?

The BBEG (Qwen 3.5 35A3B or equivalent MoE) can also be identity-conditioned using the same markdown format. Fine-tuning the BBEG to accept a "god identity" markdown would mean:

- Each microverse's corrupted god has a unique personality, generated from a meta-template
- The BBEG's creative style, paranoia level, world-building preferences vary per playthrough
- Same model, different god identity = different game feel each run
- The BBEG identity doc becomes part of the save file — reproducible playthroughs

This is a lighter fine-tune than the NPC model (the BBEG keeps its general reasoning capability, just learns the identity markdown protocol). Could be done via LoRA on the base MoE model with minimal training data — the BBEG needs to understand the format, not have its knowledge replaced.

### Anomaly Detection

How the BBEG detects the player is not a normal NPC:

| Method | Compute Cost | What It Catches |
|---|---|---|
| Statistical deviation from grammar output patterns | Cheap (math only) | Structural anomalies — unusual action sequences |
| Run player output through NPC's model, measure perplexity | Medium | Semantic anomalies — things this NPC "wouldn't say" |
| BBEG reviews flagged interactions (35A3B) | Expensive | Contextual anomalies — behavior that's valid but suspicious |

Escalation: statistical filter → perplexity check → BBEG review. Mirrors the narrative (background surveillance → closer attention → direct confrontation).

### World Generation Pipeline

World generation is itself a sequence of MCP tool calls. The BBEG uses the same tool framework as everything else — its "divine powers" are just write-access MCP tools.

#### BBEG World-Building Tools (MCP)

```
world.create_location(type, name, description, connections)
world.set_law(domain, rule_grammar)          <- defines valid actions
npc.create(name, role, personality, backstory)
npc.grant_key(npc_id, key, access_level)     <- per-NPC permissions
npc.set_relationship(npc_a, npc_b, type, strength)
faction.create(name, goals, territory)
faction.add_member(faction_id, npc_id, rank)
rumor.plant(key, content, planted_by)
quest.create(type, trigger, objective, reward)
economy.set_prices(location_id, goods)
```

#### Generation Strategy: Templates + BBEG Customization

| Layer | Method | Notes |
|---|---|---|
| Geography/map layout | Preloaded templates (pre-executed MCP tool call sequences) | Standard PCG, runs at install/download, instant |
| Location theming/flavor | BBEG customization pass over template | Grammar-constrained, fast — fills in names/descriptions |
| NPC identities + personalities | BBEG via `npc.create` tool calls | Generates backstories, speech patterns, knowledge scopes |
| NPC key grants | BBEG via `npc.grant_key` tool calls | Per-NPC information architecture — the core creative act |
| NPC relationships + factions | BBEG via relationship/faction tools | Social graph, alliances, conflicts |
| Economy/quest hooks | Rules engine template + BBEG flavor | Mechanical balance from rules, narrative from BBEG |
| "Laws of the universe" | BBEG via `world.set_law` | Grammar schemas defining valid actions in this world |

#### Preloading

World templates are pre-executed MCP tool call sequences bundled with the game. A "frontier town" template is a JSON log of ~50-100 tool calls that lay down geography, building shells, and economic structure. The BBEG's per-playthrough pass runs on top — populating NPCs, granting keys, planting rumors, establishing the social fabric. This splits generation into:

- **Instant** (template replay): seconds
- **BBEG customization** (inference): the loading screen the player watches

Generation is fully logged and replayable. Every world can be diffed, inspected, or replayed from its tool call log.

## Implementation Plan

### Phase 0: Validate Core Assumptions (Current Hardware, Zero Spend)

Prove the interaction loop works before building the game:
- MCP game server with 2-3 tools, hardcoded world state
- One NPC LoRA fine-tuned on synthetic PF2E-style dialogue (laptop/3080 Ti)
- BBEG (35A3B) modifies world state, NPC reacts — proves closed loop
- Anomaly detection prototype — can BBEG distinguish human from NPC output?
- All on existing Python Entropic — validates design before C++ rewrite

- Tier routing
- MCP tool framework
- Grammar constraint system
- LoRA hot-swapping
- C API + C++ headers

## Business Model & Distribution

### Player-Owned, No Subscription

All models and data ship with the game. Large download (multi-GB), but the player owns everything on their machine. No cloud calls, no API keys, no recurring costs. This aligns with gamer values — ownership over rental — and avoids the subscription backlash that plagues other AI-in-games approaches.

## Risks & Considerations

- **Small model quality:** 0.5-1.5B models may not produce compelling NPC dialogue even with LoRA. Phase 0 validates this before committing to full build.
- **VRAM constraints:** Target hardware (8-12 GB) limits model size and concurrent models. Architecture must enforce mutual exclusivity of expensive tiers.
- **C++ rewrite scope:** 30K+ LOC Python → C++ is substantial. Must be driven by game's concrete needs, not speculative completeness. Phased rewrite, not big-bang.
- **World generation coherence:** BBEG-generated worlds must be internally consistent. Grammar constraints help but may not guarantee narrative coherence. Needs playtesting.
- **Anomaly detection tuning:** Too sensitive = unfun (player can't do anything). Too lenient = no tension. Requires extensive balancing.
- **Solo developer scope:** Isometric CRPG + AI engine + fine-tuning pipeline is ambitious for one person. Phase 0 demo validates viability before deep investment.
- **ORC license compliance:** Must ensure zero Paizo Product Identity in shipped product. All fiction must be original or AI-generated.
