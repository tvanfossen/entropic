## Scope

Runtime generation budgeting and hardware adaptation. After v1.9.7, the engine
enforces wall-clock time caps per tier alongside token limits, tracks real-time
throughput during generation, auto-adapts `max_tokens` when hardware is slower
than expected, and exposes named GPU resource profiles that let the same model
run differently based on use case.

This absorbs two prior proposals:
- P2-20260225-020 (GPU Resource Management) -- `n_batch`, `n_threads` knobs
- P2-20260225-021 (Time-Based Generation Cap) -- per-tier wall-clock limits

**Why this matters for each consumer archetype:**

1. **Game engine**: NPCs must respond within frame budgets. A background NPC
   thinking about inventory uses the `minimal` profile (small batch, fewer
   threads). A player-facing dialog NPC uses `maximum`. If hardware is slower
   than expected (thermal throttle, shared GPU), the engine auto-reduces
   `max_tokens` rather than blowing the time budget.

2. **Homeschool (9-14B models)**: Running on consumer hardware with variable
   load. Throughput tracking detects when another process is competing for GPU
   and reduces generation length to stay within the time cap. The `balanced`
   profile gives good throughput without monopolizing the GPU.

3. **TUI developer**: Long code generations can run with `maximum` profile
   and generous time caps. Background tasks (compaction, scribe summaries)
   run with `background` profile to avoid starving the foreground generation.

**Dependencies:**
- **v1.8.2** (Inference Backend): `InferenceBackend`, `LlamaCppBackend`,
  `GenerationParams`, `GenerationResult`, `ModelOrchestrator`. The backend
  already exposes `n_batch` and `n_threads` in `ModelConfig` -- this version
  makes them dynamically switchable via profiles rather than fixed at load time.
- **v1.8.4** (Engine Loop): `ResponseGenerator`, `LoopMetrics`, per-loop timing.
  The engine loop already records `generation_time_ms` in `GenerationResult` --
  this version uses that data to feed the throughput tracker.

**What v1.9.7 does NOT include:**
- Dynamic model offloading (moving GPU layers to CPU under pressure) -- that
  is VRAM orchestration, not generation budgeting
- Per-token latency SLAs (first-token-time guarantees) -- future work
- Multi-GPU profile splitting -- `tensor_split` is a `ModelConfig` field, not
  a runtime-switchable profile knob
- Automatic profile selection based on system load -- the consumer picks profiles,
  the engine enforces them

## GPUResourceProfile Struct

Named presets that control hardware utilization during inference. Profiles
are applied at the `LlamaCppBackend` level before each generation call.

```cpp
/// @file include/entropic/types/config.h (addition)
/// @brief Named GPU resource profile for controlling inference hardware knobs.
/// @version 1.9.7
struct GPUResourceProfile {
    std::string name;           ///< Profile name ("maximum", "balanced", "background", "minimal")
    int n_batch = 512;          ///< Batch size for prompt processing (1-2048)
    int n_threads = 0;          ///< CPU threads for generation (0 = auto-detect)
    int n_threads_batch = 0;    ///< CPU threads for batch processing (0 = use n_threads)
    std::string description;    ///< Human-readable description
};
```

### Bundled Profiles

Four profiles ship with the engine. Consumers can register custom profiles.

| Profile | n_batch | n_threads | n_threads_batch | Use case |
|---|---|---|---|---|
| `maximum` | 2048 | 0 (auto) | 0 (auto) | Foreground generation, latency-sensitive |
| `balanced` | 512 | 0 (auto) | 0 (auto) | Default. Good throughput without monopolizing |
| `background` | 256 | max(1, auto/2) | max(1, auto/2) | Background tasks (compaction, scribe) |
| `minimal` | 64 | 2 | 2 | Lowest resource usage (background NPCs, idle) |

`n_threads = 0` means auto-detect (`std::thread::hardware_concurrency()`).
For `background`, halving the auto-detected count yields the thread count.

## ThroughputTracker Class

Measures and smooths tok/s during generation. Feeds the auto-adapt logic
that reduces `max_tokens` when hardware is slower than expected.

Single concrete class (no three-layer hierarchy -- one implementation, like
`GrammarRegistry` and `HookRegistry`). Owned by `ModelOrchestrator`.

```cpp
/// @file include/entropic/inference/throughput_tracker.h
/// @brief ThroughputTracker -- real-time throughput measurement and prediction.
///
/// @par Responsibilities:
/// - Record per-generation throughput samples (tokens, wall-clock time)
/// - Maintain exponentially weighted moving average (EWMA) of tok/s
/// - Predict time required for N tokens based on recent throughput
/// - Recommend max_tokens given a time budget
///
/// @par Thread safety:
/// - record() acquires mutex_ (called from generation thread)
/// - tok_per_sec() and predict_tokens() are lock-free (read from atomic EWMA)
/// - One tracker per model (not per tier) -- multiple tiers sharing a model
///   share throughput data, which is correct (same hardware, same model)
///
/// @par Ownership:
/// Owned by ModelOrchestrator. One ThroughputTracker per loaded model
/// (keyed by model path, same dedup as model pool).

#pragma once

#include <atomic>
#include <mutex>
#include <cstdint>

namespace entropic {

class ThroughputTracker {
public:
    /// @brief Record a completed generation sample.
    /// @param tokens_generated Number of tokens produced.
    /// @param elapsed_ms Wall-clock generation time in milliseconds.
    ///
    /// Updates the EWMA. Ignores samples with 0 tokens or 0 ms
    /// (degenerate generations).
    void record(int tokens_generated, int64_t elapsed_ms);

    /// @brief Current smoothed throughput estimate.
    /// @return Tokens per second (EWMA). 0.0 if no samples recorded.
    double tok_per_sec() const;

    /// @brief Predict wall-clock time for generating N tokens.
    /// @param token_count Desired token count.
    /// @return Predicted milliseconds. 0 if no throughput data.
    int64_t predict_ms(int token_count) const;

    /// @brief Recommend max_tokens to fit within a time budget.
    /// @param time_budget_ms Available wall-clock time.
    /// @param headroom Fraction of budget to target (e.g., 0.9 = 90%).
    /// @param floor Minimum token count to return (never recommend fewer).
    /// @return Recommended max_tokens. Returns floor if no throughput data.
    int recommend_tokens(int64_t time_budget_ms,
                         float headroom = 0.9f,
                         int floor = 64) const;

    /// @brief Number of recorded samples.
    int sample_count() const;

    /// @brief Reset all throughput data.
    void reset();

private:
    /// @brief EWMA smoothing factor. 0.3 = 30% weight on newest sample.
    /// Chosen to be responsive to hardware changes (thermal throttle,
    /// competing processes) while smoothing single-sample noise.
    static constexpr double kAlpha = 0.3;

    /// @brief Minimum tokens for a sample to count (filters degenerate runs).
    static constexpr int kMinTokens = 4;

    std::atomic<double> ewma_tok_s_{0.0};   ///< Smoothed tok/s estimate
    std::atomic<int> samples_{0};           ///< Number of recorded samples
    std::mutex mutex_;                      ///< Guards EWMA update (write path only)
};

} // namespace entropic
```

### EWMA Update Logic

```
on record(tokens, elapsed_ms):
    if tokens < kMinTokens or elapsed_ms <= 0:
        return  // skip degenerate samples

    sample_tok_s = tokens / (elapsed_ms / 1000.0)

    lock(mutex_)
    if samples_ == 0:
        ewma_tok_s_ = sample_tok_s  // first sample = initial value
    else:
        ewma_tok_s_ = kAlpha * sample_tok_s + (1 - kAlpha) * ewma_tok_s_
    samples_++
    unlock(mutex_)
```

The EWMA design means the tracker adapts within 3-5 generations to a
sustained throughput change (e.g., GPU thermal throttle). Single-sample
spikes (GC pause, context switch) are dampened by the 0.7 weight on history.

## ProfileRegistry Class

Manages named GPU resource profiles. Similar pattern to `GrammarRegistry`
but simpler (no validation step, no file loading -- profiles are small
structs).

```cpp
/// @file include/entropic/inference/profile_registry.h
/// @brief ProfileRegistry -- named GPU resource profile management.
///
/// @par Responsibilities:
/// - Store bundled profiles (maximum, balanced, background, minimal)
/// - Register/deregister custom profiles at runtime
/// - Resolve profile name to GPUResourceProfile struct
///
/// @par Thread safety:
/// - get() is lock-free (read from snapshot)
/// - register/deregister acquire mutex_
///
/// @par Ownership:
/// Owned by ModelOrchestrator. One registry per engine instance.

#pragma once

#include <entropic/types/config.h>

#include <mutex>
#include <string>
#include <unordered_map>
#include <vector>

namespace entropic {

class ProfileRegistry {
public:
    /// @brief Initialize with bundled profiles (maximum, balanced, background, minimal).
    void load_bundled();

    /// @brief Register a custom profile.
    /// @param profile Profile to register. profile.name is the key.
    /// @return true on success. false if name already exists.
    bool register_profile(const GPUResourceProfile& profile);

    /// @brief Remove a profile by name.
    /// @param name Profile name.
    /// @return true if removed. false if not found.
    bool deregister(const std::string& name);

    /// @brief Get a profile by name.
    /// @param name Profile name.
    /// @return Profile struct. Returns the "balanced" profile if name not found
    ///         (with WARNING log). Returns empty profile if "balanced" also
    ///         missing (should never happen after load_bundled).
    GPUResourceProfile get(const std::string& name) const;

    /// @brief Check if a profile name exists.
    bool has(const std::string& name) const;

    /// @brief List all registered profile names.
    std::vector<std::string> list() const;

    /// @brief Number of registered profiles.
    size_t size() const;

private:
    std::unordered_map<std::string, GPUResourceProfile> profiles_;
    mutable std::mutex mutex_;
};

} // namespace entropic
```

## Time Limit Enforcement

The time limit is enforced at two levels:

### Level 1: Backend (hard limit)

`LlamaCppBackend::do_generate()` and `do_generate_streaming()` check elapsed
time after each token. If `time_limit_ms` is exceeded, generation stops
immediately with `finish_reason = "time_limit"` and `result.time_limited = true`.

This uses the existing `cancel` flag mechanism: the orchestrator sets the
cancel flag when the time limit fires. No new threading infrastructure needed.

```
do_generate(messages, params):
    start = steady_clock::now()
    deadline = start + milliseconds(params.time_limit_ms)  // 0 = no deadline

    for each generated token:
        if cancel_flag:
            break
        if deadline != 0 and steady_clock::now() >= deadline:
            cancel_flag = true
            finish_reason = "time_limit"
            break
        emit token
```

### Level 2: Orchestrator (auto-adapt)

Before calling the backend, the orchestrator checks if the throughput tracker
can predict whether the requested `max_tokens` will fit within the time budget.
If not, it reduces `max_tokens`.

```
orchestrator.generate(tier, messages, params):
    resolve profile -> apply n_batch/n_threads
    resolve time_limit_ms from tier config

    if params.time_limit_ms > 0 and params.auto_adapt:
        tracker = throughput_trackers_[model_path]
        if tracker.sample_count() > 0:
            recommended = tracker.recommend_tokens(
                params.time_limit_ms, params.adapt_headroom)
            if recommended < params.max_tokens:
                params.original_max_tokens = params.max_tokens  // save for result
                params.max_tokens = recommended
                log INFO "Auto-adapted max_tokens: {original} -> {recommended} "
                         "(throughput: {tracker.tok_per_sec()} tok/s, "
                         "budget: {params.time_limit_ms}ms)"

    result = backend.generate(messages, params)

    // Record sample for future predictions
    tracker.record(result.token_count, result.generation_time_ms)
    result.throughput_tok_s = result.token_count /
        (result.generation_time_ms / 1000.0)

    return result
```

The two levels are complementary: auto-adapt tries to prevent hitting the hard
limit, but the hard limit is the safety net. Consumers see `time_limited=true`
when the hard limit fires, and `original_max_tokens > 0` when auto-adapt
reduced the token count.

## Profile Application

GPU resource profiles are applied by the backend before each generation call.
This requires `LlamaCppBackend` to support runtime modification of `n_batch`
and `n_threads` without reloading the model.

### llama.cpp Support

- `n_batch` can be changed between calls -- it controls prompt processing
  batch size, not model parameters. `llama_decode()` accepts batches up to
  `n_batch` tokens, so reducing it just means more decode calls per prompt.
- `n_threads` / `n_threads_batch` can be set per `llama_decode()` call via
  `llama_set_n_threads()`. This is a thread-local setting, not a model parameter.

No model reload needed. Profile switching is sub-millisecond.

### Application Flow

```
orchestrator.apply_profile(backend, profile_name):
    profile = profile_registry_.get(profile_name)
    backend.set_batch_size(profile.n_batch)
    backend.set_threads(profile.n_threads, profile.n_threads_batch)
    log DEBUG "Applied profile '{profile.name}': n_batch={profile.n_batch}, "
              "n_threads={profile.n_threads}"
```

### InferenceBackend Extension

```cpp
/// @brief Set batch size for prompt processing.
/// @param n_batch Batch size (1-2048). Applied on next generate() call.
/// @version 1.9.7
void set_batch_size(int n_batch);

/// @brief Set CPU thread counts for inference.
/// @param n_threads Generation threads (0 = auto).
/// @param n_threads_batch Batch processing threads (0 = use n_threads).
/// @version 1.9.7
void set_threads(int n_threads, int n_threads_batch = 0);
```

These are non-virtual methods on `InferenceBackend` (base class). They store
the values and the `do_generate` / `do_generate_streaming` overrides apply
them via `llama_set_n_threads()` before each generation.

## Orchestrator Integration

### Startup Sequence

```
orchestrator.initialize(config)
    |
    +-- Load models (existing v1.8.2)
    +-- Load adapters (existing v1.8.2)
    +-- Load grammars (existing v1.9.3)
    |
    +-- profile_registry_.load_bundled()
    |   +-- Register: maximum, balanced, background, minimal
    |   +-- Register custom profiles from config.profiles
    |   +-- Log: "Loaded N GPU resource profiles"
    |
    +-- Create ThroughputTracker per model in model_pool_
    |   +-- throughput_trackers_[model_path] = ThroughputTracker{}
    |
    +-- Ready for generation
```

### Per-Generation Flow

```
orchestrator.generate(tier, messages, params)
    |
    +-- Resolve grammar (existing v1.9.3)
    |
    +-- Resolve time config from tier
    |   +-- params.time_limit_ms == 0? -> check tier config -> check global config
    |   +-- params.profile empty? -> check tier config -> check global config -> "balanced"
    |
    +-- Apply profile to backend
    |   +-- profile = profile_registry_.get(params.profile)
    |   +-- backend.set_batch_size(profile.n_batch)
    |   +-- backend.set_threads(profile.n_threads, profile.n_threads_batch)
    |
    +-- Auto-adapt max_tokens (if time_limit_ms > 0 and auto_adapt)
    |   +-- tracker = throughput_trackers_[model_path]
    |   +-- recommended = tracker.recommend_tokens(time_limit_ms, headroom)
    |   +-- if recommended < max_tokens: reduce max_tokens, save original
    |
    +-- backend.generate(messages, params)  // time limit enforced in backend
    |
    +-- Record throughput sample
    |   +-- tracker.record(result.token_count, result.generation_time_ms)
    |
    +-- Populate result.throughput_tok_s
    +-- Return result
```

## Hook Integration (v1.9.1)

  logging, alerting, or dynamic profile switching.

extended structs.

## Validation Criteria

### V1: Bundled profiles loaded at startup

Initialize the engine. Verify `profile_registry.size() >= 4`. Verify
`has("maximum")`, `has("balanced")`, `has("background")`, `has("minimal")`
all return true.

### V2: Custom profile registration

Call `entropic_profile_register(handle, json)` with a custom profile. Verify
`entropic_profile_get(handle, "custom")` returns the profile. Verify it
appears in `entropic_profile_list()`.

### V3: Profile application changes backend knobs

Apply the `minimal` profile to a backend. Verify `n_batch` and `n_threads`
are set to the profile values. Apply `maximum` profile. Verify values changed.

### V4: Throughput tracking records samples

Generate 5 responses. Verify `tracker.sample_count() == 5`. Verify
`tracker.tok_per_sec() > 0`. Verify `result.throughput_tok_s > 0` on each
result.

### V5: EWMA responds to throughput changes

Record 10 samples at 30 tok/s, then 5 samples at 15 tok/s. Verify EWMA
has shifted toward 15 tok/s (within 20% after 5 samples at new rate).

### V6: recommend_tokens respects headroom

Set tracker to 20 tok/s. Call `recommend_tokens(10000, 0.9)`.
Expected: `20 * (10000/1000) * 0.9 = 180`. Verify result is approximately 180.

### V7: recommend_tokens respects floor

Set tracker to 1 tok/s. Call `recommend_tokens(1000, 0.9, floor=64)`.
Expected: `1 * 1 * 0.9 = 0.9`, but floor is 64. Verify result is 64.

### V8: Time limit terminates generation

Set `params.time_limit_ms = 100` with `max_tokens = 100000` (intentionally
oversized). Generate. Verify `result.time_limited == true`. Verify
`result.generation_time_ms <= 200` (100ms + reasonable overshoot). Verify
`result.finish_reason == "time_limit"`.

### V9: Auto-adapt reduces max_tokens

Set tracker to 10 tok/s. Set `params.time_limit_ms = 5000`,
`params.max_tokens = 1000`, `params.auto_adapt = true`. The orchestrator
should reduce `max_tokens` to approximately `10 * 5 * 0.9 = 45`. Verify
`result.original_max_tokens == 1000` and `result.token_count <= 50`.

### V10: Auto-adapt disabled skips adaptation

Set `params.auto_adapt = false` with low throughput and tight time limit.
Verify `result.original_max_tokens == 0` (no adaptation occurred). Generation
may hit the time limit instead.

### V11: No time limit means no enforcement

Set `params.time_limit_ms = 0` (default). Generate with large `max_tokens`.
Verify generation completes normally. Verify `result.time_limited == false`.

### V12: Identity frontmatter time config

Configure identity with `time_limit_ms: 15000` and `profile: background`.
Generate for that identity. Verify the background profile was applied (check
`n_batch`, `n_threads` on backend). Verify time limit is 15000ms.

### V13: Phase-level override

Configure identity with two phases: `planning` (time_limit_ms: 30000,
profile: maximum) and `execution` (time_limit_ms: 15000, profile: balanced).
Generate in each phase. Verify different profiles are applied.

### V14: Config YAML profile definition

Define a custom profile in config YAML `profiles:` section. Initialize
engine. Verify the custom profile is registered and usable.

### V15: Resolution order precedence

Set `time_limit_ms = 5000` in config YAML for tier "eng". Set
`time_limit_ms = 10000` in identity frontmatter. Set
`time_limit_ms = 2000` in `GenerationParams`. Verify 2000 is used
(per-call wins).

### V16: Throughput tracker per model (not per tier)

Two tiers sharing the same model. Generate on tier A, then on tier B.
Verify tier B's auto-adapt uses throughput data from tier A's generation
(same underlying model).

### V17: Profile fallback on unknown name

Set `params.profile = "nonexistent"`. Verify a WARNING is logged and the
`balanced` profile is used as fallback.

### V18: Thread safety on concurrent profile operations

From thread A, register profiles in a loop. From thread B, call `get()` in
a loop. Verify no data races (ThreadSanitizer clean).

### V19: SLOC and complexity

All functions pass knots quality gates: cognitive complexity <= 15,
SLOC <= 50, returns <= 3.

### V20: Doxygen complete

Every public symbol in new headers and updated type headers has
`@brief`, `@param`, `@return`, `@version`.

### test_throughput_tracker.cpp

```
SCENARIO: First sample sets initial EWMA
  GIVEN an empty ThroughputTracker
  WHEN record(100, 2000) is called  // 50 tok/s
  THEN tok_per_sec() returns 50.0
  AND sample_count() returns 1

  GIVEN a tracker with one sample at 50 tok/s
  WHEN record(100, 5000) is called  // 20 tok/s
  THEN tok_per_sec() is between 20 and 50
  AND tok_per_sec() is approximately 0.3*20 + 0.7*50 = 41.0

SCENARIO: Degenerate samples are ignored
  GIVEN a tracker with one sample at 50 tok/s
  WHEN record(0, 1000) is called  // 0 tokens
  THEN sample_count() remains 1
  AND tok_per_sec() remains 50.0

SCENARIO: Zero-ms samples are ignored
  GIVEN a tracker with one sample at 50 tok/s
  WHEN record(100, 0) is called  // 0 ms
  THEN sample_count() remains 1

SCENARIO: predict_ms estimates generation time
  GIVEN a tracker at 40 tok/s
  WHEN predict_ms(200) is called
  THEN returns approximately 5000  // 200/40 * 1000

SCENARIO: predict_ms returns 0 with no data
  GIVEN an empty tracker
  WHEN predict_ms(200) is called
  THEN returns 0

SCENARIO: recommend_tokens with headroom
  GIVEN a tracker at 20 tok/s
  WHEN recommend_tokens(10000, 0.9) is called
  THEN returns approximately 180  // 20 * 10 * 0.9

SCENARIO: recommend_tokens respects floor
  GIVEN a tracker at 1 tok/s
  WHEN recommend_tokens(1000, 0.9, floor=64) is called
  THEN returns 64

SCENARIO: recommend_tokens returns floor with no data
  GIVEN an empty tracker
  WHEN recommend_tokens(10000, 0.9, floor=64) is called
  THEN returns 64

SCENARIO: reset clears all data
  GIVEN a tracker with 5 samples
  WHEN reset() is called
  THEN sample_count() returns 0
  AND tok_per_sec() returns 0.0

SCENARIO: EWMA adapts to sustained throughput change
  GIVEN a tracker with 10 samples at 30 tok/s
  WHEN 5 samples at 15 tok/s are recorded
  THEN tok_per_sec() is within 20% of 15 tok/s

SCENARIO: Concurrent record and read are thread-safe
  GIVEN an empty tracker
  WHEN thread A records 100 samples

  THEN no data races (ThreadSanitizer clean)
```
