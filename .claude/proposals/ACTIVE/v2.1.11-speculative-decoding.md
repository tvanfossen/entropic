# v2.1.11 — Speculative decoding + SecondaryModelLoader

## Session prompt

This is the largest patch in the 2.2.0 plan and has the most unknowns. The goal is to add speculative decoding as a first-class inference accelerator: a small draft model proposes a window of tokens, the primary GPU model verifies them in a single batched forward pass, accepted prefix commits. Expected speedup on long generations with a well-paired draft: 1.8–4× decode throughput at the **same output distribution** (correctness contract: bit-identical to plain decode on rejection cases).

This patch ALSO lands SecondaryModelLoader (gh#27), the unified lifecycle for non-primary models (router, draft, future thinking-model). Without it, speculative-draft loading would duplicate the existing router-loading path. The two issues land together because the speculative work would otherwise rewrite the router lifecycle ad-hoc.

After this patch:
- A new `inference.speculative.*` config section enables draft-based acceleration when configured.
- Speculative is **off by default**; consumers opt-in by setting a draft model in config.
- Token-stream protocol is unchanged — callbacks fire once per *accepted* token, not per *proposed*.
- Draft model placement (CPU / GPU / hybrid) is a config axis, not a code change.
- Acceptance-rate and speedup metrics surface as engine counters.

## Design decisions already made (do not re-litigate)

- **Decoupled from router.** Speculative decoding is its own config-level setting. The fact that Qwen3-0.6B is currently bundled as the router does NOT mean the speculative draft reuses the router model handle. `routing.model` and `inference.speculative.draft_model` are independent config keys. (The router and draft may happen to point at the same GGUF if the user configures it that way — incidental, not architectural.)
- **Draft pairing is tokenizer compatibility, NOT same-family.** `common_speculative_is_compat` (in llama.cpp at the current pin) validates tokenizer + vocab + special-token compatibility at runtime. Same family is the useful heuristic but not the formal requirement.
- **Draft model selection.** Use a **Qwen3.5 small variant** as the draft for the Qwen3.5/3.6-A3B primary (NOT Qwen3-0.6B — different generation, likely tokenizer mismatch). User has the responsibility of picking the specific Qwen3.5 small variant; v2.1.9's registry expansion should provide candidates (`qwen3_5_0_8b`, `qwen3_5_2b`, etc.).
- **Placement is a config axis.** `inference.speculative.draft_n_gpu_layers` (or equivalent llama.cpp knob) is configurable:
  - **CPU-resident draft** (default): GPU stays saturated verifying; CPU headroom does the drafting. Minimal VRAM cost.
  - **GPU-resident draft** (opt-in when VRAM allows): minimizes CPU contention. Costs additional VRAM (e.g., ~600 MB for a 0.6B Q8 draft).
  - **Hybrid offload** (partial `n_gpu_layers`): standard llama.cpp pattern.
- **No llama.cpp pin bump required.** Current pin (`7f2cbd9a4`, gguf-v0.18.0-246) already exposes `common_speculative_is_compat`, `_init`, `_draft`, `_accept`, `_free` in `extern/llama.cpp/common/speculative.h`. A pin bump is optional optimization (tree-based speculation, lower per-draft overhead), not a prerequisite.
- **Token callback semantics unchanged.** Callback fires once per *accepted* token, not per *proposed*. Consumers see no protocol change.
- **Correctness contract.** Output distribution must be bit-identical to plain decode on rejection cases. Test the contract explicitly.
- **Roadmap update.** Speculative decoding was previously listed under "Deferred to 3.0.0" in `docs/roadmap.md:852` with the rationale "Blocked by single-GPU VRAM constraint." That rationale is invalidated by CPU-led draft as the default placement — bringing speculative forward to 2.1.11 is consistent with the new design. The roadmap edit landed in Session 0.

## Scope

### SecondaryModelLoader (gh#27)

- New class under `src/inference/secondary_model_loader.{h,cpp}`.
- Unified lifecycle for non-primary models: router, draft, future thinking-model. Each gets its own slot keyed by role (`"router"`, `"draft"`, `"thinking"`).
- Slot policy: lazy load on first use; explicit unload via `release_role(role)`; refcount or guard so concurrent users don't double-release.
- Refactor existing router-loading code to use the new loader (no behavioral change for router users).

### Speculative decoding (gh#36)

- New config section in `entropic_config` / `entropic_config.yaml` schema:
  ```yaml
  inference:
    speculative:
      enabled: false              # default off
      draft_model: <bundled key or path>
      n_draft: 16                 # window size
      draft_n_gpu_layers: 0       # 0 = CPU; >0 = GPU layers; -1 = full GPU
      draft_cpu_threads: 4        # subset of host threads
  ```
- New backend capability flag: `InferenceBackend::do_supports(SPECULATIVE_DECODING)` returns true for llama.cpp-backed primaries when a compatible draft is configured.
- `LlamaCppBackend::do_generate_speculative(...)` wired to `common_speculative_*` from `extern/llama.cpp/common/speculative.h`.
- Compatibility check at config-load time: `entropic_speculative_compat(handle)` exposes `common_speculative_is_compat` result. If the configured draft is incompatible with the primary's tokenizer, load fails fast with a clear diagnostic. Add a new public API:
  ```c
  ENTROPIC_EXPORT entropic_error_t entropic_speculative_compat(
      entropic_handle_t handle,
      bool* compatible,
      char** diagnostic);
  ```
- Acceptance metrics surfaced via existing engine counter mechanism:
  - `inference.speculative.accept_rate`
  - `inference.speculative.tokens_per_primary_step` (mean over a window)
  - `inference.speculative.drafts_total` / `drafts_accepted`

### Tests

- **Unit tests** (mock draft + verifier):
  - Acceptance accounting correctness
  - Output stream bit-identical to plain decode on rejection cases (correctness contract)
  - Compatibility check returns false for tokenizer-mismatched draft
- **Model test** under `tests/model/`:
  - Qwen3.5/3.6-A3B primary + Qwen3.5 small-variant draft (specific size: see registry from v2.1.9; user picks)
  - Measure tokens/sec with and without speculative on a fixed prompt suite
  - Gate: ≥1.8× speedup on long generations (>500 tokens)
  - Verify accept-rate is sane (target range 0.5–0.8 for a well-paired draft)
- **Integration test**: enable speculative via config, run a turn end-to-end through `entropic_run_streaming`, verify token callbacks fire correctly and acceptance counters update.

## Out of scope

- Tree-based speculation / multi-branch drafts. Sequential covers our throughput case; tree variants would be a follow-up if a specific family ever has a draft whose linear acceptance rate is too low.
- Self-speculation / Medusa heads. Distinct architecture; separate proposal if/when it becomes interesting.
- Speculative decoding for hybrid Mamba models (Nemotron, Falcon-H1R). llama.cpp auto-disables speculative for recurrent models; honor that.
- Thinking-model lifecycle (gh#25 territory) — SecondaryModelLoader provides the slot, but loading and routing semantics for thinking-model are out of scope here.
- Per-family draft auto-pairing for #17 adapter expansion families (Llama, Gemma, Mistral, etc.) — that ratchets in 2.3.x as adapters are added.
- llama.cpp pin bump.

## Validation criteria

1. `SecondaryModelLoader` replaces existing router-loading code with no behavioral change for router users (existing router tests still pass unchanged).
2. `inference.speculative.enabled: false` is the shipping default — existing configs see no behavior change.
3. `entropic_speculative_compat` returns `false` + diagnostic when a tokenizer-incompatible draft is configured; engine refuses to enable speculative in that case.
4. With a compatible draft configured + `enabled: true`:
   - Token callback fires once per accepted token (same protocol as plain decode).
   - Output distribution matches plain decode on rejection cases (bit-identical test).
   - Acceptance counters update sanely.
5. Model test gate: ≥1.8× speedup on long generations (>500 tokens) with Qwen3.5/3.6-A3B + Qwen3.5 small-variant draft, on the dev machine GPU.
6. CPU-resident draft works without VRAM cost beyond the primary's footprint.
7. GPU-resident draft works when configured (VRAM budget permitting).
8. Doxygen on every new public symbol includes `@brief` and `@version 2.1.11`.
9. Pre-commit gates pass (knots, flake8, doxygen-guard, unit tests).
10. `docs/architecture-cpp.md` decision log updated to record the speculative-decoding design decision (independent of router, CPU-default placement, accept-token streaming contract).

## Implementation Log

### 2026-05-14 — Session 4 kickoff: verification gates

llama.cpp pin confirmed at `253ba110bcd372207ca7b0bb56f1ea10d60d53fd` (the pin
landed by v2.1.9). All four pre-implementation gates run; results below.

#### Gate 1 — llama.cpp speculative API at the current pin: **SHIFTED**

The proposal's locked-decision section (line 38, 44, 72) references API
symbols that do **not exist** at the current pin. The surface has
restructured significantly from the older `7f2cbd9a4` snapshot the proposal
was drafted against. Detailed delta:

| Proposal-referenced symbol | Status at 253ba110b |
|---|---|
| `common_speculative_is_compat` (public) | **Removed.** Replaced by `static bool common_speculative_are_compatible(model_tgt, model_dft)` inside `extern/llama.cpp/common/speculative.cpp` (line 41). Not callable externally. |
| `common_speculative_init(ctx_tgt, ctx_dft)` | **Signature changed.** Now `common_speculative_init(common_params_speculative & params, uint32_t n_seq)` — contexts now live inside `params.draft.ctx_tgt` / `params.draft.ctx_dft`. |
| `common_speculative_draft(...)` returning tokens | **Restructured.** Now `void common_speculative_draft(spec)` — caller pre-populates `common_speculative_draft_params` (via `common_speculative_get_draft_params(spec, seq_id)`); draft tokens land in `dparams.result`. |
| `common_speculative_accept(spec, accepted_tokens)` | **Signature changed.** Now `common_speculative_accept(spec, llama_seq_id, uint16_t n_accepted)` — accumulated count, not token list. |
| `common_speculative_free(spec)` | Present, unchanged. |

New surface introduced since the older pin:

- `common_speculative_begin(spec, seq_id, prompt)` — optional per-generation init.
- `common_speculative_process(spec, batch) -> bool` — feeds verifier batch into draft state.
- Multi-implementation chain: `common_speculative_type` now includes
  `DRAFT_SIMPLE`, `DRAFT_EAGLE3`, `NGRAM_SIMPLE`, `NGRAM_MAP_K`,
  `NGRAM_MAP_K4V`, `NGRAM_MOD`, `NGRAM_CACHE`. The proposal targets
  `DRAFT_SIMPLE` semantics — that path still exists and is the closest
  analogue to the original sketch.
- Compatibility check is now **implicit and fatal**: the
  `common_speculative_impl_draft_simple` ctor (`speculative.cpp:193`) calls
  `common_speculative_are_compatible(...)` and **throws
  `std::runtime_error`** if the vocabs disagree. There is no
  query-without-commit path.

**Gate 1 conclusion:** the upstream surface is stable at the pinned commit
(not a moving target — it's a fixed snapshot), but the proposal's
pseudocode is out of date. The feature is still implementable, but the
specifics of `entropic_speculative_compat` and the `do_generate_speculative`
wiring need to be rewritten against the new shape. Concretely:

- `entropic_speculative_compat(...)` cannot map 1:1 to a public llama.cpp
  call. Implementation options:
  1. **Mirror the upstream logic in entropic.** Inline an
     `entropic::speculative::is_compatible(model_tgt, model_dft)` using
     the same `llama_vocab_*` accessors the upstream `static` helper uses
     (vocab type, BOS/EOS parity, vocab size diff, token text strcmp on
     the prefix). Robust against private-API access.
  2. **Try-init, catch, free.** Build a throwaway
     `common_speculative_impl_draft_simple` inside a try/catch; if it
     throws, return `compatible=false` + the message; otherwise free it
     and return `compatible=true`. Ergonomically clean but allocates real
     ctx state for a query.

  Option (1) is preferred — purely metadata-level inspection, no model
  state required, exercise-able from a unit test with mock vocabs.
- `do_generate_speculative` wires to `_init` / `_get_draft_params` /
  `_process` / `_draft` / `_accept` / `_free` in their current shapes.

**Action:** proceed under Option (1) for the compat path; rewrite the
proposal's per-symbol references in `Scope > Speculative decoding (gh#36)`
to match the actual API. Surfaced this divergence to the user before any
implementation per Gate 1's STOP directive.

#### Gate 2 — tokenizer compatibility (Qwen3.6-A3B vs Qwen3.5 small): **DEFERRED**

Runtime verification of `common_speculative_are_compatible(...)` between
the primary (`qwen3_6_a3b`) and candidate draft (`qwen3_5_0_8b`) requires
loading both models against a CUDA context — out of reach for a non-GPU
code-only session. Static observation: the new generation tag between
Qwen3.5 and Qwen3.6 is a real risk for tokenizer drift; the proposal
nominates "Qwen3.5 small variant for Qwen3.5/3.6-A3B" on that basis, but
the compatibility verdict is empirical.

**Action:** the model-test harness added by this patch (per Validation
Criterion 5) carries the compatibility check as its first assertion.
Developer runs that harness on the dev GPU before declaring v2.1.11
shippable. If the verdict is "incompatible," the speculative pairing
contracts to whatever IS compatible (e.g., another Qwen3.5 size, or
deferring speculative for the Qwen3.6 primary entirely until a Qwen3.6
small variant is bundled); the SecondaryModelLoader half ships regardless.

#### Gate 3 — hybrid-Mamba speculative auto-disable: **NOT automatic in current speculative.cpp**

Searched `extern/llama.cpp/common/speculative.cpp` and `speculative.h` for
`recurrent` / `mamba` / `llama_model_is_recurrent` calls — no matches.
`llama_model_is_recurrent(model)` IS available as a public llama.cpp API
(`extern/llama.cpp/include/llama.h:616`) and correctly returns true for
Mamba/RWKV/hybrid architectures, but the speculative layer does **not**
self-gate on it at the current pin.

**Implication:** Nemotron-3-Nano-4B (hybrid-Mamba, shipped v2.1.9) will
not be silently disabled by upstream. If a user configures it as the
primary with a draft, the implicit vocab check inside
`common_speculative_impl_draft_simple` may even *pass* (if some Qwen
small shares a tokenizer) — but the underlying speculative-decoding
math assumes a non-recurrent target, and behavior would be incorrect.

**Action:** entropic must add the gate itself, not rely on upstream.
Adding to scope:
- `entropic_speculative_compat(...)` returns `false` + diagnostic
  `"primary model is recurrent / hybrid; speculative decoding not
  supported for this architecture"` when
  `llama_model_is_recurrent(target_model)` is true. Test fixture under
  `tests/inference/` asserts the compat call returns false with this
  diagnostic for a recurrent-shaped target.
- `do_generate_speculative` early-exits (refusing to engage) if the
  primary is recurrent — defensive, even though `compat` should prevent
  reaching it.

#### Gate 4 — current router model load site: **`src/inference/orchestrator.cpp`**

Router lifecycle today:
- **Owner:** `ModelOrchestrator::router_` — `std::shared_ptr<LlamaCppBackend>`
  member (private).
- **Creation:** `create_tier_backends(config)` lines 80–82 — if
  `config.models.router` is set, `router_ = std::make_shared<LlamaCppBackend>()`.
- **Activation:** `activate_router(config)` lines 130–137 — calls
  `router_->load_and_activate(*config.models.router)`. Logs success/failure.
- **Use:** `classify_task(...)` in the routing path calls
  `router_->complete(prompt, params)` with `max_tokens=1`, `temperature=0`.
- **Teardown:** `shutdown()` lines 183–185 — `router_->unload()` if loaded.
- **Diagnostics:** `loaded_models()` and `configured_roles()` report
  `"router"` when applicable (lines 487, 503).
- **Cache management:** `clear_prompt_cache()` clears the router's cache
  too (line 729).

SecondaryModelLoader plan (no behavioral change for router callers):
- New class `entropic::inference::SecondaryModelLoader` owning a
  `std::unordered_map<std::string, std::shared_ptr<LlamaCppBackend>>`
  keyed by role (`"router"`, `"draft"`, future `"thinking"`).
- API: `ensure_loaded(role, ModelConfig)`, `get(role) -> LlamaCppBackend*`,
  `release_role(role)`, `is_loaded(role)`, `loaded_roles()`.
- `ModelOrchestrator` replaces `router_` with a `SecondaryModelLoader`
  member; existing call sites (`router_->complete(...)`, `router_->unload()`,
  etc.) route through the loader (`loader_.get("router")->complete(...)`).
- Router tests untouched — public observable behavior preserved.
