## Scope

Three related MCP workstreams that complete the engine's MCP story:

1. **MCP Bridge** — entropic as an MCP SERVER (Claude Code calls into it)
2. **`.mcp.json` Discovery** — entropic as an MCP CLIENT (auto-registers external servers)
3. **Headless Example** — scripted validation example, testable by CI/Claude without interactive terminal

---

## Part 1: MCP Bridge (entropic as server)

### Problem

The `.mcp.json` config references `entropic mcp-bridge` but the command
doesn't exist. Claude Code cannot connect to the engine as an MCP tool
provider.

### Solution

A Python CLI subcommand that:
1. Speaks JSON-RPC 2.0 over stdio (MCP protocol)
2. Creates an `EntropicEngine` handle via the Python wrapper
3. Configures via `configure_dir()` (layered config)
4. Exposes engine capabilities as MCP tools
5. Forwards tool calls to the C engine and returns results

### MCP Tools to Expose

| Tool | Description |
|------|-------------|
| `entropic.ask` | Single-turn inference (run_streaming, returns complete text) |
| `entropic.status` | Engine version, model info, tier list, context usage |
| `entropic.context_clear` | Reset conversation |
| `entropic.context_get` | Get conversation history as JSON |
| `entropic.context_count` | Message count |

### Architecture

```
Claude Code (MCP client)
    │ stdio (JSON-RPC 2.0)
    ▼
entropic mcp-bridge (Python CLI)
    │ ctypes wrapper (EntropicEngine)
    ▼
librentropic.so (C engine)
    │
    ▼
llama.cpp (inference)
```

- Bridge is Python (CLI layer) — thin MCP protocol handler
- Engine lifecycle: create once on bridge startup, destroy on exit
- Config: `configure_dir()` with project dir from CWD or CLI flag
- Streaming: bridge accumulates tokens and returns complete result
  (MCP doesn't support streaming responses natively)
- Session state: persistent conversation across tool calls within
  a single bridge session

### Validation

1. `.mcp.json` connects: `/mcp` in Claude Code shows entropic server
2. `entropic.ask "What is 2+2?"` returns a response
3. `entropic.status` shows version and model info
4. Multi-turn: `ask` → `ask` (follow-up) retains context
5. `context_clear` → `ask` starts fresh

## Part 2: .mcp.json Discovery (entropic as client)

### Problem

Regression from v1.7.1. The Python engine read `.mcp.json` to discover
and connect to external MCP servers (stdio/SSE). The C engine has the
transport layer and `entropic_register_mcp_server()` but lost the
automatic discovery.

### Solution

During `entropic_configure_dir()`, after config loading:
1. Check for `.mcp.json` in the project directory
2. Check for `.mcp.json` in CWD
3. Parse the JSON: `{"mcpServers": {"name": {"type": "stdio|sse", ...}}}`
4. Call `entropic_register_mcp_server()` for each entry
5. Log registered external servers to session.log

### Format

```json
{
  "mcpServers": {
    "chess": {
      "type": "stdio",
      "command": "python3",
      "args": ["chess_server.py"]
    },
    "api": {
      "type": "sse",
      "url": "http://localhost:8080/sse"
    }
  }
}
```

### Where It Lives

This is config loader responsibility — NOT facade logic. The
`load_layered()` function (or a post-configure hook) discovers and
registers external servers. The facade just calls configure; the
config layer handles discovery.

### Validation

1. Place `.mcp.json` in project dir with a test server
2. `configure_dir` auto-registers the server
3. `entropic_list_mcp_servers()` shows it
4. Model can call tools on the auto-registered server

## Part 3: Headless Example

### Problem

All current examples require interactive terminal input. There's no
example that can be validated by CI, automated testing, or by Claude
reviewing log files — without a human typing prompts.

### Solution

`examples/headless/` — a scripted conversation that:
1. Creates engine, configures from project dir
2. Runs a fixed sequence of prompts (no stdin)
3. Validates responses programmatically (contains expected text, no errors)
4. Writes session.log + session_model.log
5. Exits with 0 (success) or 1 (failure) + error report

### Script Sequence

```
1. "What is 2+2?" → verify response contains "4"
2. "What did I just ask?" → verify multi-turn context (references "2+2")
3. Register test MCP server → "Call test.echo with 'hello'" → verify tool call works
4. context_clear → "What did I just ask?" → verify no history
5. Exit with pass/fail summary
```

### Architecture

```
examples/headless/
├── main.c                    # C version
├── main_wrapper.py           # Python wrapper version
├── default_config.yaml       # Minimal config
├── test_server.py            # Minimal MCP server (echo tool)
├── .headless/
│   ├── session.log
│   └── session_model.log
└── expected_results.yaml     # Validation criteria
```

### Integration

```bash
inv example -n headless          # Run headless C version
inv example -n headless --wrapper # Run headless Python version

# CI usage:
inv example -n headless && echo "PASS" || echo "FAIL"
```

The headless example is also the regression test for the wrapper and
MCP client functionality. Claude can validate by reading session.log
and session_model.log without needing interactive access.

### Validation

1. `inv example -n headless` exits 0
2. session.log shows: configure → run × 4 → context_clear → run → destroy
3. session_model.log shows all 5 prompts + responses
4. Response validation: each response meets criteria from expected_results.yaml
5. Tool call validation: test.echo tool called and returned correctly

### Decisions
- Bridge is a compiled C++ binary (not Python) to keep "free integration"
  truly free — no Python runtime dependency on consumer machines.
  nlohmann/json is already an engine dependency, so adding the bridge
  costs only ~150 lines of C++.
- Single `entropic` CLI dispatcher rather than separate
  `entropic-mcp-bridge` binary. Future subcommands (status, models, etc.)
  can be added without growing PATH entries.
- `.mcp.json` discovery lives in the config loader, not the facade. This
  keeps the facade dumb (just calls configure) and the config layer
  responsible for all source-of-truth merging.
