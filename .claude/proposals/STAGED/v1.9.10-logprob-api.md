## Scope

Evaluation-only API for computing per-token log-probabilities and sequence
perplexity against any ACTIVE model. After v1.9.10, consumers can score
arbitrary token sequences without generating new tokens, mutating KV cache
state, or affecting in-flight generation.

This builds on:
- **v1.8.2** (Inference Backend): `InferenceBackend` concrete base,
  `LlamaCppBackend`, VRAM lifecycle (COLD/WARM/ACTIVE), `ModelOrchestrator`

**What v1.9.10 adds:**
- `get_logprobs(model_id, tokens)` — per-token log-probabilities for a
  token sequence, evaluated against a loaded model
- `compute_perplexity(model_id, tokens)` — sequence perplexity as
  `exp(-mean(log_probs))` over the input
- Evaluation uses `llama_decode()` with `logits_all=true` on a temporary
  batch — no KV cache mutation, no sampling, no generation
- Works on any model in ACTIVE state (on GPU, ready for inference)
- C API: `entropic_get_logprobs()`, `entropic_compute_perplexity()`

**What v1.9.10 does NOT include:**
- Token-level generation (sampling, beam search)
- KV cache modifications or stateful evaluation sessions
- Batch evaluation across multiple sequences (single sequence per call)
- Top-k alternative token probabilities per position
- Text-to-token conversion (caller provides token IDs)

**Consumer use cases:**
- Model quality assessment (compare perplexity across models/quants)
- Confidence scoring on generated output (evaluate own output post-hoc)
- Fine-tuning data selection (filter training data by perplexity)
- Constitutional validation scoring (how "surprising" is a response?)
- A/B testing quantization levels (IQ3_XXS vs Q4_K_M perplexity delta)

## Architecture

### Evaluation Flow

```
Consumer calls entropic_get_logprobs(handle, model_id, tokens, n_tokens, &result)
    |
    v
Facade: validate handle, resolve model_id via ModelOrchestrator
    |
    v
ModelOrchestrator: assert model state == ACTIVE
    |
    v
InferenceBackend::evaluate_logprobs(tokens, n_tokens)
    |
    +-- Acquire eval_mutex_ (separate from generation mutex)
    |
    v
LlamaCppBackend::do_evaluate_logprobs(tokens, n_tokens)
    |
    +-- Allocate temporary llama_batch (logits_all = true)
    +-- Copy tokens into batch
    +-- llama_decode(ctx, batch) with temporary KV cache slot
    +-- Extract logits for each position
    +-- softmax → log_prob for each token[i+1] given tokens[0..i]
    +-- Free temporary batch
    |
    v
Return LogprobResult { per_token_logprobs[], perplexity, n_tokens }
```

### Why a Separate Mutex

Generation uses the model's KV cache and modifies it across streaming
calls. Evaluation must NOT touch the KV cache used by generation. Two
isolation strategies exist:

1. **Separate KV cache slot** — llama.cpp supports multiple sequences
   (`seq_id`). Evaluation uses a dedicated `seq_id` that is allocated,
   used, and freed within a single `evaluate_logprobs()` call. The
   generation `seq_id` is untouched.

2. **Separate mutex** — `eval_mutex_` serializes evaluation calls but
   does NOT block generation (which holds `generate_mutex_`). Both can
   proceed concurrently on the same model because they use different
   `seq_id` slots in the KV cache.

Strategy 1 provides the isolation. Strategy 2 prevents evaluation from
blocking generation. Together: evaluation and generation can run
concurrently on the same ACTIVE model with zero interference.

```
                   ┌──────────────────────────────┐
                   │     ACTIVE model (GPU)        │
                   │                               │
 generate() ──────►  seq_id=0  (generation KV)    │
                   │                               │
 evaluate() ──────►  seq_id=N  (temporary, freed)  │
                   │                               │
                   └──────────────────────────────┘
```

If the model's `n_ctx` is fully consumed by generation, evaluation
returns `ENTROPIC_ERROR_EVAL_CONTEXT_FULL` — it never evicts generation
cache entries.

### llama.cpp Mechanics

The evaluation call chain inside `LlamaCppBackend`:

```cpp
// 1. Allocate a temporary sequence ID
llama_seq_id eval_seq = allocate_temp_seq_id();

// 2. Build batch with logits_all = true
llama_batch batch = llama_batch_init(n_tokens, 0, 1);
for (int i = 0; i < n_tokens; i++) {
    batch.token[i]    = tokens[i];
    batch.pos[i]      = i;
    batch.n_seq_id[i] = 1;
    batch.seq_id[i]   = &eval_seq;
    batch.logits[i]   = true;   // request logits at every position
}
batch.n_tokens = n_tokens;

// 3. Decode (populates logits, does NOT affect other seq_ids)
int rc = llama_decode(ctx_, batch);

// 4. Extract log-probabilities
//    For position i, logits predict token at position i+1.
//    log_prob[i] = log(softmax(logits[i])[tokens[i+1]])
//    Position 0 has no "previous" prediction — skip or use BOS context.
float* logprobs = extract_logprobs(batch, tokens, n_tokens);

// 5. Clean up: remove temporary sequence from KV cache
llama_kv_cache_seq_rm(ctx_, eval_seq, 0, -1);
llama_batch_free(batch);
release_temp_seq_id(eval_seq);
```

Key detail: `llama_decode()` with a specific `seq_id` only writes to
KV cache entries for that `seq_id`. Removing the `seq_id` afterwards
leaves the KV cache identical to before the call. This is what makes
evaluation non-mutating from the perspective of any ongoing generation.

### Logprob Extraction

For a sequence of N tokens `[t_0, t_1, ..., t_{N-1}]`:

- Position `i` produces logits predicting position `i+1`
- `logprob[i] = log_softmax(logits_at_position_i)[t_{i+1}]` for `i` in `[0, N-2]`
- Position `N-1` has no "next token" to score — its logits are discarded
- Result contains `N-1` log-probability values

Perplexity over the sequence:

```
perplexity = exp( -1/(N-1) * sum(logprob[0..N-2]) )
```

Lower perplexity = the model found the sequence less surprising.

### Batching for Long Sequences

If `n_tokens > n_batch` (the model's batch size, typically 512), the
evaluation is chunked:

```
tokens[0..511]   → llama_decode → logits[0..511]
tokens[512..1023] → llama_decode → logits[512..1023]
...
```

Each chunk extends the temporary `seq_id`'s KV cache. The chunking is
internal to `do_evaluate_logprobs()` — the caller sees a single result.
After all chunks, the entire temporary `seq_id` is freed.

If the total sequence length exceeds `n_ctx` (context window), the call
returns `ENTROPIC_ERROR_EVAL_CONTEXT_FULL` without partial results.

**Base class `evaluate_logprobs()` implementation (the 80%):**

```cpp
LogprobResult InferenceBackend::evaluate_logprobs(
    const int32_t* tokens,
    int n_tokens)
{
    // 1. State check (lock-free atomic read)
    if (state_.load() != ModelState::ACTIVE) {
        spdlog::error("evaluate_logprobs: model not ACTIVE (state={})",
                      static_cast<int>(state_.load()));
        throw std::runtime_error("Model must be ACTIVE for evaluation");
    }

    // 2. Input validation
    if (n_tokens < 2) {
        spdlog::error("evaluate_logprobs: need >= 2 tokens, got {}", n_tokens);
        throw std::runtime_error("Need at least 2 tokens for logprob evaluation");
    }

    // 3. Eval mutex (does not block generation)
    std::lock_guard<std::mutex> lock(eval_mutex_);

    spdlog::info("evaluate_logprobs: evaluating {} tokens", n_tokens);
    auto start = std::chrono::steady_clock::now();

    // 4. Delegate to backend implementation
    LogprobResult result = do_evaluate_logprobs(tokens, n_tokens);

    // 5. Compute perplexity from logprobs
    result.total_logprob = 0.0f;
    for (float lp : result.logprobs) {
        result.total_logprob += lp;
    }
    float mean_logprob = result.total_logprob / result.n_logprobs;
    result.perplexity = std::exp(-mean_logprob);

    auto elapsed = std::chrono::steady_clock::now() - start;
    auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(elapsed);
    spdlog::info("evaluate_logprobs: {} tokens, perplexity={:.2f}, {}ms",
                 n_tokens, result.perplexity, ms.count());

    return result;
}

float InferenceBackend::compute_perplexity(
    const int32_t* tokens,
    int n_tokens)
{
    return evaluate_logprobs(tokens, n_tokens).perplexity;
}
```

### do_evaluate_logprobs Implementation Sketch

```cpp
LogprobResult LlamaCppBackend::do_evaluate_logprobs(
    const int32_t* tokens,
    int n_tokens)
{
    int n_vocab = llama_n_vocab(model_);
    int n_batch = llama_n_batch(ctx_);
    int n_ctx   = llama_n_ctx(ctx_);

    if (n_tokens > n_ctx) {
        throw std::runtime_error(
            fmt::format("Sequence length {} exceeds context window {}",
                        n_tokens, n_ctx));
    }

    llama_seq_id eval_seq = allocate_temp_seq_id();
    if (eval_seq < 0) {
        throw std::runtime_error("No available sequence ID for evaluation");
    }

    LogprobResult result;
    result.tokens.assign(tokens, tokens + n_tokens);
    result.n_tokens = n_tokens;
    result.n_logprobs = n_tokens - 1;
    result.logprobs.reserve(n_tokens - 1);

    // Process in chunks of n_batch
    for (int offset = 0; offset < n_tokens; offset += n_batch) {
        int chunk_size = std::min(n_batch, n_tokens - offset);

        llama_batch batch = llama_batch_init(chunk_size, 0, 1);
        for (int i = 0; i < chunk_size; i++) {
            batch.token[i]    = tokens[offset + i];
            batch.pos[i]      = offset + i;
            batch.n_seq_id[i] = 1;
            batch.seq_id[i]   = &eval_seq;
            batch.logits[i]   = true;
        }
        batch.n_tokens = chunk_size;

        int rc = llama_decode(ctx_, batch);
        if (rc != 0) {
            llama_batch_free(batch);
            llama_kv_cache_seq_rm(ctx_, eval_seq, 0, -1);
            release_temp_seq_id(eval_seq);
            throw std::runtime_error(
                fmt::format("llama_decode failed with rc={}", rc));
        }

        // Extract logprobs for this chunk
        for (int i = 0; i < chunk_size; i++) {
            int global_pos = offset + i;
            // Skip last token overall — no next token to score
            if (global_pos >= n_tokens - 1) break;

            const float* logits = llama_get_logits_ith(ctx_, i);
            float lp = extract_token_logprob(
                logits, tokens[global_pos + 1], n_vocab);
            result.logprobs.push_back(lp);
        }

        llama_batch_free(batch);
    }

    // Clean up temporary sequence
    llama_kv_cache_seq_rm(ctx_, eval_seq, 0, -1);
    release_temp_seq_id(eval_seq);

    return result;
}
```

### extract_token_logprob

```cpp
float LlamaCppBackend::extract_token_logprob(
    const float* logits,
    int32_t next_token,
    int n_vocab)
{
    // log_softmax: log(exp(logits[t]) / sum(exp(logits)))
    // Numerically stable: logits[t] - log(sum(exp(logits)))

    float max_logit = *std::max_element(logits, logits + n_vocab);
    float sum_exp = 0.0f;
    for (int v = 0; v < n_vocab; v++) {
        sum_exp += std::exp(logits[v] - max_logit);
    }
    float log_sum_exp = max_logit + std::log(sum_exp);

    return logits[next_token] - log_sum_exp;
}
```

### Temporary Sequence ID Management

The pool starts with a configurable number of reserved eval seq_ids
(default: 4, set at model load time). These are seq_ids above the
range used by generation.

```cpp
llama_seq_id LlamaCppBackend::allocate_temp_seq_id()
{
    std::lock_guard<std::mutex> lock(seq_id_mutex_);
    if (free_seq_ids_.empty()) {
        return -1;
    }
    llama_seq_id id = free_seq_ids_.back();
    free_seq_ids_.pop_back();
    return id;
}

void LlamaCppBackend::release_temp_seq_id(llama_seq_id seq_id)
{
    std::lock_guard<std::mutex> lock(seq_id_mutex_);
    free_seq_ids_.push_back(seq_id);
}
```

## C API

### entropic.h Additions

```c
/**
 * @brief Per-token log-probability result.
 *
 * Contains per-token log-probabilities for a sequence evaluated
 * against a loaded model. The logprobs array has (n_tokens - 1)
 * entries: logprobs[i] is the log-probability of tokens[i+1]
 * given tokens[0..i].
 *
 * @version 1.9.10
 */
typedef struct entropic_logprob_result {
    float* logprobs;        /**< Per-token log-probabilities (N-1 values). */
    int32_t* tokens;        /**< Input tokens echoed back (N values). */
    float perplexity;       /**< exp(-mean(logprobs)) over the sequence. */
    float total_logprob;    /**< Sum of all logprob values. */
    int n_tokens;           /**< Number of input tokens. */
    int n_logprobs;         /**< Number of logprob values (n_tokens - 1). */
} entropic_logprob_result_t;

/**
 * @brief Evaluate per-token log-probabilities for a token sequence.
 *
 * Runs the token sequence through the model and returns the
 * log-probability of each token given its preceding context.
 * This is evaluation-only — no sampling, no generation, no KV
 * cache mutation visible to ongoing generation.
 *
 * The model identified by model_id must be in ACTIVE state
 * (loaded on GPU). Use entropic_inference_state() to check.
 *
 * @param handle     Engine handle.
 * @param model_id   Model identifier (tier name or model key).
 * @param tokens     Array of token IDs to evaluate.
 * @param n_tokens   Number of tokens (minimum 2).
 * @param result     Output: logprob result. Caller frees with
 *                   entropic_free_logprob_result().
 *
 * @par Token ordering:
 * result->logprobs[i] = log P(tokens[i+1] | tokens[0..i])
 *
 * @par Perplexity:
 * result->perplexity = exp(-mean(result->logprobs))
 *
 * @par Example:
 * @code
 * int32_t tokens[] = {1, 15043, 3186, 29889};  // "Hello world."
 * entropic_logprob_result_t result;
 * entropic_error_t err = entropic_get_logprobs(
 *     handle, "primary", tokens, 4, &result);
 * if (err == ENTROPIC_OK) {
 *     printf("perplexity: %.2f\n", result.perplexity);
 *     for (int i = 0; i < result.n_logprobs; i++) {
 *         printf("  token %d -> %d: logprob=%.4f\n",
 *                result.tokens[i], result.tokens[i+1],
 *                result.logprobs[i]);
 *     }
 *     entropic_free_logprob_result(&result);
 * }
 * @endcode
 *
 * @return ENTROPIC_OK on success.
 *         - ENTROPIC_ERROR_INVALID_HANDLE
 *         - ENTROPIC_ERROR_MODEL_NOT_FOUND (unknown model_id)
 *         - ENTROPIC_ERROR_MODEL_NOT_ACTIVE (model not in ACTIVE state)
 *         - ENTROPIC_ERROR_EVAL_CONTEXT_FULL (sequence exceeds n_ctx)
 *         - ENTROPIC_ERROR_INVALID_CONFIG (n_tokens < 2)
 *         - ENTROPIC_ERROR_EVAL_FAILED (llama_decode error)
 *
 * @threadsafety Serialized per-model via eval_mutex. Does not block
 *               generation on the same model.
 * @version 1.9.10
 */
ENTROPIC_EXPORT entropic_error_t entropic_get_logprobs(
    entropic_handle_t handle,
    const char* model_id,
    const int32_t* tokens,
    int n_tokens,
    entropic_logprob_result_t* result);

/**
 * @brief Compute perplexity for a token sequence.
 *
 * Convenience function that calls entropic_get_logprobs() internally
 * and returns only the perplexity value. Use entropic_get_logprobs()
 * when you need per-token detail.
 *
 * @param handle     Engine handle.
 * @param model_id   Model identifier (tier name or model key).
 * @param tokens     Array of token IDs to evaluate.
 * @param n_tokens   Number of tokens (minimum 2).
 * @param perplexity Output: perplexity value.
 *
 * @return ENTROPIC_OK on success. Same error codes as entropic_get_logprobs().
 *
 * @threadsafety Same as entropic_get_logprobs().
 * @version 1.9.10
 */
ENTROPIC_EXPORT entropic_error_t entropic_compute_perplexity(
    entropic_handle_t handle,
    const char* model_id,
    const int32_t* tokens,
    int n_tokens,
    float* perplexity);

/**
 * @brief Free a logprob result allocated by entropic_get_logprobs().
 *
 * Frees the internal logprobs and tokens arrays. The result struct
 * itself is caller-owned (stack or heap) and is NOT freed.
 *
 * @param result Pointer to result struct. NULL-safe (no-op on NULL).
 *
 * @version 1.9.10
 */
ENTROPIC_EXPORT void entropic_free_logprob_result(
    entropic_logprob_result_t* result);
```

## C API Result Struct Semantics

The `entropic_logprob_result_t` struct is caller-owned (declared on
stack or heap by the consumer). The `logprobs` and `tokens` arrays
inside are engine-allocated and must be freed via
`entropic_free_logprob_result()`.

This matches the pattern used elsewhere in the C API where out-params
are struct pointers with engine-allocated internal arrays.

```c
// Stack-allocated result — only internal arrays need freeing
entropic_logprob_result_t result;
entropic_get_logprobs(handle, "primary", tokens, n, &result);
// use result.perplexity, result.logprobs[i], etc.
entropic_free_logprob_result(&result);

// result struct itself goes out of scope normally — no leak
```

## ModelOrchestrator Integration

`ModelOrchestrator` (v1.8.2) manages the model pool and VRAM lifecycle.
v1.9.10 adds model resolution for evaluation calls — the orchestrator
maps `model_id` (a tier name like `"primary"` or `"eng"`) to the
corresponding `InferenceBackend` instance.

No new methods are needed on `ModelOrchestrator`. The existing
`get_backend(model_id)` or equivalent lookup is sufficient. The facade
resolves the model_id to a backend handle, asserts ACTIVE state, and
calls `evaluate_logprobs()`.

If the model is not ACTIVE, the C API returns
`ENTROPIC_ERROR_MODEL_NOT_ACTIVE`. The engine does NOT auto-activate
for evaluation — the consumer must ensure the model is ACTIVE before
calling. This prevents surprise VRAM allocation from an evaluation call.

## Consumer Use Cases

### Use Case 1: Model Quality Comparison

Compare perplexity of the same text across two quantization levels:

```c
// Tokens for a reference passage
int32_t tokens[512];
int n = tokenize(passage, tokens, 512);

float ppl_iq3, ppl_q4;
entropic_compute_perplexity(handle, "model_iq3", tokens, n, &ppl_iq3);
entropic_compute_perplexity(handle, "model_q4k", tokens, n, &ppl_q4);

printf("IQ3_XXS: %.2f, Q4_K_M: %.2f, delta: %.2f\n",
       ppl_iq3, ppl_q4, ppl_q4 - ppl_iq3);
```

### Use Case 2: Confidence Scoring on Generated Output

After generating a response, score how confident the model is in
its own output:

```c
// Generate response
entropic_inference_generate(backend, messages, params, &response);

// Tokenize the response
int32_t* resp_tokens;
int n_resp;
tokenize_response(response, &resp_tokens, &n_resp);

// Score confidence
entropic_logprob_result_t result;
entropic_get_logprobs(handle, "primary", resp_tokens, n_resp, &result);

// Flag low-confidence tokens
for (int i = 0; i < result.n_logprobs; i++) {
    if (result.logprobs[i] < -5.0f) {
        printf("Low confidence at position %d: logprob=%.2f\n",
               i + 1, result.logprobs[i]);
    }
}
entropic_free_logprob_result(&result);
```

### Use Case 3: Fine-Tuning Data Selection

Filter training data to exclude passages the model already knows well
(low perplexity) and focus on challenging examples:

```c
for (int s = 0; s < n_samples; s++) {
    float ppl;
    entropic_compute_perplexity(
        handle, "primary", samples[s].tokens, samples[s].n, &ppl);

    if (ppl > perplexity_threshold) {
        // Model found this surprising — good training candidate
        add_to_training_set(samples[s]);
    }
}
```

### Use Case 4: Constitutional Validation Scoring

Score how "natural" a response is under the model — responses the model
finds surprising may indicate hallucination or off-distribution output:

```c
float ppl;
entropic_compute_perplexity(handle, "primary", response_tokens, n, &ppl);

if (ppl > constitutional_threshold) {
    spdlog::warn("Response perplexity {:.2f} exceeds threshold {:.2f}",
                 ppl, constitutional_threshold);
    // Trigger re-generation or flag for review
}
```

## Validation Criteria

### V1: Basic logprob evaluation

Given a model in ACTIVE state and a sequence of 10 tokens, call
`entropic_get_logprobs()`. Verify result contains 9 logprob values.
Verify all logprobs are negative (log-probabilities are <= 0). Verify
`n_tokens == 10`, `n_logprobs == 9`.

### V2: Perplexity computation

Given the same token sequence, call `entropic_compute_perplexity()`.
Verify perplexity equals `exp(-mean(logprobs))` from V1. Verify
perplexity > 0.

### V3: Token echo-back

Verify `result.tokens` matches the input tokens exactly. This catches
off-by-one errors in buffer management.

### V4: Short sequence (minimum)

Call with exactly 2 tokens. Verify result contains 1 logprob value.
Verify no crash, no error.

### V5: Sequence too short

Call with 1 token. Verify `ENTROPIC_ERROR_INVALID_CONFIG` is returned.
Call with 0 tokens. Same error.

### V6: Model not ACTIVE

Call `entropic_get_logprobs()` on a WARM or COLD model. Verify
`ENTROPIC_ERROR_MODEL_NOT_ACTIVE` is returned. Verify no crash,
no state change on the model.

### V7: Unknown model_id

Call with a model_id that does not exist. Verify
`ENTROPIC_ERROR_MODEL_NOT_FOUND` is returned.

### V8: Sequence exceeds context window

Call with `n_tokens > n_ctx`. Verify `ENTROPIC_ERROR_EVAL_CONTEXT_FULL`
is returned. Verify no partial results, no KV cache pollution.

### V9: Long sequence chunking

Call with `n_tokens > n_batch` but `<= n_ctx`. Verify result contains
`n_tokens - 1` logprob values. Verify logprobs are consistent (no
discontinuity at chunk boundaries).

### V10: Evaluation does not affect generation

Start a streaming generation. While streaming, call
`entropic_get_logprobs()` on the same model. Verify generation
completes normally. Verify evaluation returns correct results.
Verify no KV cache corruption (generation output is identical
with and without concurrent evaluation).

### V11: Evaluation does not mutate KV cache

Before evaluation, snapshot the generation seq_id's KV state (or
verify generation output deterministically). After evaluation, verify
generation produces identical output. The temporary eval seq_id must
be fully cleaned up.

### V12: Multiple concurrent evaluations

Call `entropic_get_logprobs()` from two threads on the same model.
Verify both return correct results (serialized by eval_mutex_).
Verify no data races (ThreadSanitizer clean).

### V13: Free result is NULL-safe

Call `entropic_free_logprob_result(NULL)`. Verify no crash.
Call `entropic_free_logprob_result()` on an already-freed result.
Verify no double-free (pointers zeroed after free).

### V14: Deterministic results

Call `entropic_get_logprobs()` twice with the same tokens. Verify
logprobs are identical (evaluation is deterministic — no sampling).

### V15: Perplexity math correctness

Hand-compute expected perplexity from the returned logprobs array.
Verify `result.perplexity == exp(-mean(result.logprobs))` within
float tolerance (1e-5).

### V16: SLOC and complexity

All functions pass knots quality gates: cognitive complexity <= 15,
SLOC <= 50, returns <= 3.

### V17: Doxygen complete

Every public symbol in `logprob_result.h` and updated `entropic.h`
has `@brief`, `@param`, `@return`, `@version`, `@threadsafety`
where applicable.

## Implementation Notes (from review)

1. **`entropic_inference_tokenize()` needed.** The C API needs a tokenize function
   so callers can convert text → token IDs before calling `get_logprobs()`.
   Without it, the API requires callers to have their own tokenizer. Add:
   ```c
   entropic_error_t entropic_inference_tokenize(
       entropic_inference_backend_t backend,
       const char* text,
       int32_t** tokens,
       size_t* n_tokens);
   ```
   Caller frees `tokens` via `entropic_free()`. This also benefits other
   subsystems that need token-level operations.
