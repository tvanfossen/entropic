## Scope

Delegation system and identity management: the last subsystem needed for the
engine to function as a multi-identity agentic system. After v1.8.6, the engine
can spawn child inference loops with role-locked identities, execute multi-stage
pipelines through shared worktrees, enforce identity-aware tool filtering, and
auto-chain between tiers based on delegation depth.

This is a faithful port of:
- `src/entropic/core/delegation.py` -- `DelegationManager`: child loop creation, worktree lifecycle, pipeline execution, todo list save/restore
- `src/entropic/core/worktree.py` -- `WorktreeManager`: git worktree create/merge/discard, develop branch model, `scoped_worktree` directory swapping
- `src/entropic/core/engine.py` -- Delegation directive handlers: `_directive_delegate`, `_directive_pipeline`, `_directive_complete`, `_execute_pending_delegation`, `_execute_pending_pipeline`, `_try_auto_chain`, `_should_auto_chain`, `MAX_DELEGATION_DEPTH`
- `src/entropic/core/engine_types.py` -- `LoopContext` delegation fields: `delegation_depth`, `parent_conversation_id`, `child_conversation_ids`, `active_phase`
- `src/entropic/prompts/__init__.py` -- `IdentityFrontmatter`: `allowed_tools`, `phases`, `PhaseConfig`, `auto_chain`, `explicit_completion`, `role_type`
- `src/entropic/inference/orchestrator.py` -- `get_tier_param`, `get_allowed_tools`, `can_handoff`, `get_handoff_targets`

v1.8.5 provided the MCP tool system including the `EntropicServer` with
delegate/pipeline/complete/phase_change tools. Those tools emit directives
(`Delegate`, `Pipeline`, `Complete`, `PhaseChange`) that the engine's
`DirectiveProcessor` (v1.8.4) already dispatches. v1.8.6 implements the
directive *handlers* that execute delegation and identity logic.

**What this version does NOT include:**
- External MCP server support (v1.8.7) -- stdio/SSE transport
- Session persistence (v1.8.8) -- SQLite storage for delegation records
- Per-caller MCP authorization (v1.9.4) -- per-identity key sets
- Hook architecture (v1.9.1) -- `on_delegate`, `on_delegate_complete` hooks

**What this version DOES include beyond the Python port:**
- Nothing. This is a strict behavioral port. The Python delegation system is
  already well-designed and v1.8.6 replicates it faithfully in C++.

## Architecture

### Delegation Flow

```
Lead (depth=0)                    Child (depth=1)
────────────────                  ──────────────────
model generates
  → tool_call: entropic.delegate
  → DelegateTool returns
    Delegate directive + StopProcessing

engine._directive_delegate()
  → stores pending_delegation in ctx.metadata
  → result.stop_processing = true

execute_iteration() sees pending_delegation
  → _execute_pending_delegation()
    → depth check (< MAX_DELEGATION_DEPTH=2)
    → state → DELEGATING
    → DelegationManager.execute_delegation()
      → resolve tier via orchestrator
      → build_child_context()
        ├── delegation_depth = parent + 1      child_ctx created
        ├── locked_tier = target               fresh messages
        ├── system prompt rebuilt for tier      [system, user(task)]
        └── completion_instructions appended
      → create worktree (or use shared)
      → scoped_worktree swaps dirs             child loop runs
        → engine._loop(child_ctx)   ──────→   generates, uses tools
                                               auto_chain at depth>0
                                                 → COMPLETE
        ← DelegationResult         ◄──────    loop exits
      → merge/discard worktree
    → inject result into parent context
    → state → EXECUTING
lead continues with delegation result
```

### Pipeline Flow

```
Lead calls entropic.pipeline(stages=["eng","qa"], task="...")
  └── DelegationManager.execute_pipeline()
      ├── create shared worktree (one for entire pipeline)
      ├── stage 1: execute_delegation("eng", task + pipeline context)
      │   └── eng works in shared worktree, auto-chains → COMPLETE
      ├── stage 2: execute_delegation("qa", task + pipeline context)
      │   └── qa works in same worktree, sees eng's artifacts
      └── merge shared worktree into develop
```

### Auto-Chain Logic

```
model finishes generation (no tool calls)
  └── _evaluate_no_tool_decision()
      └── _try_auto_chain()
          ├── depth == 0 (root):
          │   auto_chain target from identity frontmatter
          │   → TierChange directive (switch identity in-place)
          └── depth > 0 (child):
              → set state COMPLETE (return to parent)
```

### Worktree Branch Model

```
main          pristine, only updated via accept_to_main()
  └── develop     lead operates here, created from main at first delegation
        ├── delegation/eng-abc12345     child worktree, merges back to develop
        └── delegation/qa-def67890      child worktree, merges back to develop
```

### Identity System Data Flow

```
IdentityFrontmatter (loaded by PromptManager from identity_*.md)
  ├── allowed_tools: ["entropic.*", "bash.*"]     → ToolExecutor filters
  ├── phases:
  │   ├── default: {temperature: 0.7, max_output_tokens: 4096}
  │   └── execute: {temperature: 0.3, max_output_tokens: 8192, bash_commands: ["make","cmake"]}
  ├── auto_chain: "lead"                           → auto_chain target
  ├── explicit_completion: true                    → requires entropic.complete
  ├── bash_commands: ["python","pip","git"]         → ToolExecutor prefix allowlist
  └── role_type: "back_office"                     → routing classification
```

### No New `.so` Boundary

Delegation, worktree management, and identity resolution all live in
`librentropic-core.so`. The `EntropicServer` tools (delegate, pipeline,
complete, phase_change) already live in `librentropic-mcp-entropic.so`
from v1.8.5. Directives cross the `.so` boundary as JSON in the
`ServerResponse` envelope (established in v1.8.5).

## Implementation Plan

### Phase 1: Identity System Extensions

Extend `IdentityFrontmatter` and `PhaseConfig` (scaffolded in v1.8.1) with
the fields needed for delegation and identity-aware behavior. These structs
are loaded by the PromptManager and consumed by the orchestrator and engine.

`PhaseConfig` fields:
```cpp
/// @brief Inference parameters for a single identity phase.
/// @version 1.8.6
///
/// Each role has one or more named phases. The engine resolves
/// inference params from the active phase at generation time.
/// Phase-specific bash_commands override the identity-level list
/// when set (nullopt = inherit from identity).
struct PhaseConfig {
    float temperature = 0.7f;                    ///< Sampling temperature
    int max_output_tokens = 4096;                ///< Max generation tokens
    bool enable_thinking = false;                ///< Think-block generation
    float repeat_penalty = 1.1f;                 ///< Repetition penalty
    std::optional<std::vector<std::string>>
        bash_commands;                           ///< Phase-specific bash allowlist
};
```

`IdentityFrontmatter` delegation-relevant fields (extending v1.8.1 scaffold):
```cpp
/// @brief Identity frontmatter for a tier.
/// @version 1.8.6
///
/// Extended with delegation and phase fields.
struct IdentityFrontmatter {
    // ... existing fields from v1.8.1 ...
    std::optional<std::string> auto_chain;               ///< Target tier on completion
    std::optional<std::vector<std::string>> allowed_tools; ///< Tool filter (nullopt = all)
    std::optional<std::vector<std::string>> bash_commands;  ///< Bash command prefix allowlist
    bool explicit_completion = false;                     ///< Require entropic.complete
    bool interstitial = false;                            ///< Interstitial role (no routing)
    bool routable = true;                                 ///< Visible to router
    std::string role_type = "front_office";               ///< front_office|back_office|utility
    std::optional<std::map<std::string, PhaseConfig>> phases; ///< Named phases
};
```

The orchestrator's `get_tier_param()`, `get_allowed_tools()`, `can_handoff()`,
and `get_handoff_targets()` already exist from v1.8.4 (they were needed for
tier-filtered tool lists in ResponseGenerator). Phase resolution for inference
params is new.

**Phase resolution for generation parameters:**
```
1. Check ctx.active_phase (set by PhaseChange directive)
2. Look up phase in identity frontmatter phases map
3. If found: use phase's temperature, max_output_tokens, etc.
4. If not found: use identity-level defaults
5. Config overrides always win (get_tier_param resolution order)
```

### Phase 2: WorktreeManager

Port `worktree.py`. Git worktree management for delegation isolation.

```cpp
/// @brief Tracks a created worktree's path and branch for later merge/discard.
/// @version 1.8.6
struct WorktreeInfo {
    std::filesystem::path path;       ///< Worktree directory
    std::string branch;               ///< Branch name (e.g., "delegation/eng-abc12345")
    std::string delegation_id;        ///< Delegation UUID (full)
};

/// @brief Create, merge, and discard git worktrees for delegation isolation.
/// @version 1.8.6
///
/// Ensures a `develop` branch exists before creating delegation worktrees.
/// The main working tree is checked out to `develop` so that lead operates
/// there. Delegation branches are created from and merged back to `develop`.
/// Main is only updated via `accept_to_main()`.
///
/// @code
/// WorktreeManager mgr(repo_dir);
/// mgr.ensure_develop();                    // main → develop
/// auto wt = mgr.create_worktree(id, "eng"); // develop → delegation/eng-xxx
/// // ... child works in wt.path ...
/// mgr.merge_worktree(wt);                  // delegation/eng-xxx → develop
/// mgr.accept_to_main();                    // develop → main (explicit)
/// @endcode
class WorktreeManager {
public:
    /// @brief Construct with repository root directory.
    /// @param repo_dir Path to the git repository root.
    /// @version 1.8.6
    explicit WorktreeManager(const std::filesystem::path& repo_dir);

    /// @brief Create develop branch from current HEAD and check it out.
    /// @return true if the working tree is on develop.
    /// @version 1.8.6
    bool ensure_develop();

    /// @brief Create a git worktree for a delegation, branching from develop.
    /// @param delegation_id UUID for the delegation.
    /// @param tier Target tier name (used in branch naming).
    /// @return WorktreeInfo on success, nullopt on failure.
    /// @version 1.8.6
    std::optional<WorktreeInfo> create_worktree(
        const std::string& delegation_id,
        const std::string& tier);

    /// @brief Merge worktree branch back to develop.
    /// @param info Worktree to merge.
    /// @return true if merge succeeded.
    /// @version 1.8.6
    bool merge_worktree(const WorktreeInfo& info);

    /// @brief Remove worktree and delete branch without merging.
    /// @param info Worktree to discard.
    /// @version 1.8.6
    void discard_worktree(const WorktreeInfo& info);

    /// @brief Merge develop into main. Called on explicit acceptance.
    /// @return true if merge succeeded.
    /// @version 1.8.6
    bool accept_to_main();

    /// @brief Discard develop branch, returning to main.
    /// @return true if discard succeeded.
    /// @version 1.8.6
    bool discard_develop();

private:
    /// @brief Auto-commit uncommitted changes before merge.
    /// @version 1.8.6
    void auto_commit_if_dirty(const WorktreeInfo& info);

    /// @brief Remove worktree directory and delete branch.
    /// @version 1.8.6
    void cleanup(const WorktreeInfo& info);

    /// @brief Find the main branch name (main or master).
    /// @return Branch name, or empty string if not found.
    /// @version 1.8.6
    std::string find_main_branch() const;

    std::filesystem::path repo_dir_;        ///< Repository root
    std::filesystem::path worktree_base_;   ///< .worktrees/ directory
    bool develop_ready_ = false;            ///< develop branch checked out
};
```

Git operations shell out via `popen()` (matching Python behavior). Each
`run_git()` call captures stdout/stderr and returns `{success, output}`.

context manager. Swaps filesystem, bash, and git server directories to the
worktree path on construction, restores on destruction.

### Phase 3: DelegationManager

Port `delegation.py`. Orchestrates child loop creation and execution.

```cpp
/// @brief Result returned from a child delegation loop.
/// @version 1.8.6
struct DelegationResult {
    std::string summary;                   ///< Final summary from child
    bool success = false;                  ///< Whether child reached COMPLETE
    std::string target_tier;               ///< Tier that executed
    std::string task;                      ///< Original task text
    int turns_used = 0;                    ///< Iterations consumed
    std::vector<Message> child_messages;   ///< Full child message history
};

/// @brief Orchestrates child loop creation and execution.
/// @version 1.8.6
///
/// Builds a fresh LoopContext for the target tier, runs the engine's
/// loop to completion, and extracts the final assistant message as
/// the delegation result. Manages worktree lifecycle (create, merge,
/// discard) and todo list save/restore across child contexts.
///
/// @code
/// DelegationManager mgr(engine, nullptr, repo_dir);
/// auto result = mgr.execute_delegation(parent_ctx, "eng", "Implement X", 5);
/// // result.summary contains the child's final output
/// @endcode
class DelegationManager {
public:
    /// @brief Construct with engine reference and optional storage/repo.
    /// @param engine AgentEngine instance (child loops reuse its loop machinery).
    /// @param storage Optional storage backend for delegation persistence (v1.8.8).
    /// @param repo_dir Optional repo root for worktree isolation.
    /// @version 1.8.6
    DelegationManager(AgentEngine& engine,
                      IStorageBackend* storage = nullptr,
                      const std::filesystem::path& repo_dir = {});

    /// @brief Run a child inference loop for the target tier.
    /// @param parent_ctx Parent loop context (depth, tools, system prompt).
    /// @param target_tier Tier name to delegate to.
    /// @param task Task description for the child.
    /// @param max_turns Optional iteration limit for child loop.
    /// @param parent_conversation_id Optional parent conversation ID for storage.
    /// @return DelegationResult with summary, success, and child messages.
    /// @version 1.8.6
    DelegationResult execute_delegation(
        LoopContext& parent_ctx,
        const std::string& target_tier,
        const std::string& task,
        std::optional<int> max_turns = std::nullopt,
        const std::string& parent_conversation_id = "");

    /// @brief Run a multi-stage delegation pipeline sequentially.
    /// @param parent_ctx Parent loop context.
    /// @param stages Ordered list of tier names.
    /// @param task Task description (shared across stages).
    /// @param parent_conversation_id Optional parent conversation ID.
    /// @return DelegationResult from the final stage.
    /// @version 1.8.6
    DelegationResult execute_pipeline(
        LoopContext& parent_ctx,
        const std::vector<std::string>& stages,
        const std::string& task,
        const std::string& parent_conversation_id = "");

private:
    /// @brief Build a fresh LoopContext for the child delegation.
    /// @version 1.8.6
    LoopContext build_child_context(
        const LoopContext& parent_ctx,
        const ModelTier& tier,
        const std::string& task);

    /// @brief Build completion instructions based on explicit_completion flag.
    /// @version 1.8.6
    std::string completion_instructions(const ModelTier& tier) const;

    /// @brief Extract the delegation summary from child context.
    /// @version 1.8.6
    std::string extract_final_summary(const LoopContext& child_ctx) const;

    /// @brief Run child loop with engine state save/restore.
    /// @version 1.8.6
    DelegationResult run_child_loop(
        LoopContext& child_ctx,
        const std::string& target_tier,
        const std::string& task,
        std::optional<int> max_turns);

    /// @brief Save current TodoList state from EntropicServer.
    /// @version 1.8.6
    std::string save_todo_list();

    /// @brief Install fresh empty TodoList for child delegation.
    /// @version 1.8.6
    void install_fresh_todo_list();

    /// @brief Restore TodoList state to EntropicServer.
    /// @version 1.8.6
    void restore_todo_list(const std::string& saved_json);

    /// @brief Finalize worktree based on delegation result.
    /// @version 1.8.6
    void finalize_worktree(
        const std::optional<WorktreeInfo>& wt_info,
        const DelegationResult& result);

    AgentEngine& engine_;
    IStorageBackend* storage_;                       ///< Optional (v1.8.8)
    std::optional<WorktreeManager> worktree_mgr_;
    std::optional<WorktreeInfo> shared_worktree_;    ///< Set during pipeline
};
```

Key behaviors from Python spec:
1. **Child context assembly**: fresh `LoopContext` with `delegation_depth = parent + 1`,
   `locked_tier = target`, system prompt rebuilt for target tier, task as user message
2. **Engine state save/restore**: context anchors, loop config, todo list saved before
   child loop, restored after (even on failure)
3. **Worktree ownership**: per-delegation worktrees are owned and finalized by
   `run_child_loop()`. Pipeline shared worktrees are owned by `execute_pipeline()`.
4. **Pipeline context injection**: each stage gets a `[PIPELINE CONTEXT]` prefix with
   stage position and scope enforcement ("stay within your role")
5. **Pipeline worktree lifecycle**: single worktree for entire pipeline, always merged
   (even on partial failure -- completed stages produce valuable artifacts)
6. **Todo list isolation**: child gets fresh empty TodoList, parent's restored after
7. **Summary extraction**: prefers `explicit_completion_summary` from `entropic.complete`
   tool, falls back to last assistant message

### Phase 4: Engine Directive Handlers

Wire delegation handlers into `AgentEngine`. These fill in the directive
handler registrations that v1.8.4 left as stubs and add the post-tool-call
delegation execution logic.

**Delegate handler** (`_directive_delegate` equivalent):
```
handle_delegate(ctx, directive, result):
    log target, task, depth
    store {target, task, max_turns} in ctx.metadata["pending_delegation"]
    result.stop_processing = true
```

**Pipeline handler** (`_directive_pipeline` equivalent):
```
handle_pipeline(ctx, directive, result):
    log stages, task
    store {stages, task} in ctx.metadata["pending_pipeline"]
    result.stop_processing = true
```

**Complete handler** (`_directive_complete` equivalent):
```
handle_complete(ctx, directive, result):
    log depth, summary
    store summary in ctx.metadata["explicit_completion_summary"]
    set_state(ctx, COMPLETE)
    result.stop_processing = true
```

**PhaseChange handler** (`_directive_phase_change` equivalent):
```
handle_phase_change(ctx, directive, result):
    look up phase in current tier's identity frontmatter
    if valid: ctx.active_phase = directive.phase
    else: log warning with available phases
```

**Pending delegation execution** (in `execute_iteration()` after tool processing):
```
IF "pending_delegation" in ctx.metadata:
    ctx.metrics.iterations -= 1   // delegation turns are free
    execute_pending_delegation(ctx)
    return

IF "pending_pipeline" in ctx.metadata:
    ctx.metrics.iterations -= 1
    execute_pending_pipeline(ctx)
    return
```

`execute_pending_delegation()`:
```
pop pending_delegation from ctx.metadata
depth check: if ctx.delegation_depth >= MAX_DELEGATION_DEPTH (2)
    inject rejection message, return
set state DELEGATING
fire on_delegation_start callback
create DelegationManager
result = manager.execute_delegation(ctx, target, task, max_turns)
inject result message: "[DELEGATION {COMPLETE|FAILED}: {tier}] {summary}"
fire on_delegation_complete callback
set state EXECUTING
```

`execute_pending_pipeline()`:
```
pop pending_pipeline from ctx.metadata
depth check: same as delegation
set state DELEGATING
create DelegationManager
result = manager.execute_pipeline(ctx, stages, task)
inject result: "[PIPELINE {COMPLETE|FAILED}: {stages}] {summary}"
set state EXECUTING
```

### Phase 5: Auto-Chain Logic

Port `_should_auto_chain()` and `_try_auto_chain()` into the engine's
no-tool-call evaluation path.

```
should_auto_chain(ctx, finish_reason, content):
    if !ctx.locked_tier: return false
    if !get_tier_param(ctx.locked_tier, "auto_chain"): return false
    if finish_reason == "length": return true
    if finish_reason == "stop" AND is_response_complete(content): return true
    return false

try_auto_chain(ctx, finish_reason, content):
    if !should_auto_chain(ctx, finish_reason, content): return false
    if ctx.delegation_depth > 0:
        // Child: completing returns to parent
        set_state(ctx, COMPLETE)
        return true
    // Root: tier change to auto_chain target
    target = resolve_auto_chain_target(ctx)
    if target: emit TierChange directive
    return target != nullptr
```

`resolve_auto_chain_target()`:
```
chain_target = get_tier_param(ctx.locked_tier, "auto_chain")  // string tier name
target = find_tier(chain_target)
if target: return target
// Fallback: route_among handoff targets
targets = get_handoff_targets(ctx.locked_tier)
if empty: return nullptr
return route_among(targets)  // currently returns first candidate
```

Integration point in `evaluate_no_tool_decision()`:
```
if finish_reason == "interrupted": stop
if finish_reason == "length": continue
if tools_were_attempted: continue (model retries with feedback)
if try_auto_chain(ctx, finish_reason, content): continue
if explicit_completion AND tool_calls > 0: wait for entropic.complete
else: check is_response_complete → COMPLETE
```

### Phase 6: Repo Initialization

Port `_get_repo_dir()` and `_init_project_repo()`. The engine lazily discovers
the project's git repository for worktree isolation.

```
get_repo_dir():
    if already checked: return cached result
    project_dir = filesystem server's root_dir
    if project_dir has .git: cache and return
    if repo_init callback: try it
    else: git init + git add -A + git commit --allow-empty
    cache result (Path or nullptr)
    return cached result
```

The `repo_init` callback in `EngineCallbacks` allows consumers to override
the default `git init` behavior (e.g., use a different VCS or skip entirely).

## Validation Criteria

1. **Lead → eng delegation completes**: lead delegates, eng produces output, result injected into lead's context
2. **Pipeline eng → qa completes**: sequential execution, qa sees eng's artifacts in shared worktree
3. **Depth limit enforced**: delegation at `MAX_DELEGATION_DEPTH` (2) returns rejection message
4. **Auto-chain at depth 0**: fires TierChange to configured target
5. **Auto-chain at depth > 0**: fires COMPLETE (returns to parent)
6. **Worktree isolation**: filesystem/bash/git directories swapped during child loop, restored after
7. **Worktree merge on success**: child branch merged to develop
8. **Worktree discard on failure**: child branch deleted, worktree removed
9. **Pipeline shared worktree**: all stages share one worktree, merged once at end
10. **Pipeline always merges**: even on partial failure, completed stage artifacts preserved
11. **Todo list isolation**: child gets fresh list, parent's restored after
12. **Phase change works**: PhaseChange directive updates ctx.active_phase, affects generation params
13. **Phase resolution order**: phase config → identity defaults → config override
14. **explicit_completion enforced**: roles with `explicit_completion=true` require entropic.complete
15. **Delegation turns free**: delegation iterations don't count against parent's budget
16. **Child context fresh**: child gets new messages (system + task), no parent history
17. **Completion instructions injected**: children with explicit_completion get instructions appended
18. **State transitions correct**: EXECUTING → DELEGATING → EXECUTING on delegation return
19. **Repo auto-init**: missing `.git` triggers `git init` (or callback)
20. **SLOC <= 50 per function**: all functions pass knots complexity gates
21. **Doxygen complete**: every public symbol has @brief and @version

## Implementation Notes (from adversarial review)

1. **ScopedWorktree uses `entropic_mcp_server_set_working_dir()`** (added to
   `i_mcp_server.h` in v1.8.5) per-server instead of direct C++ member access.
   This crosses the `.so` boundary cleanly via the C plugin interface.

2. **bash.execute bypasses worktree path containment by design.** A child loop
   can run `bash.execute` with absolute paths (`/home/user/.ssh/id_rsa`) — not
   constrained by `set_working_dir` (which changes cwd only). The filesystem
   server's `realpath()` check (v1.8.5) protects filesystem tools, but bash is
   inherently unconstrained. The permission system (Layer 1 allow/deny) is the
   security boundary for bash, not path containment. Document this explicitly.

## MVP Checkpoint

**After Phase 3** (DelegationManager execute_delegation). At this point, a single delegation works end-to-end: the parent engine spawns a child loop with a fresh context locked to the target tier, the child executes to completion, and the result is returned to the parent. Worktree isolation (Phase 2) provides git branch safety. Auto-chain logic, pipeline execution, and engine directive handler wiring (Phases 4-6) add multi-stage and identity-aware behavior but are not required for basic parent-child delegation testing.
