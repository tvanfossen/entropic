## Summary

The doxygen API reference at `https://tvanfossen.github.io/entropic/`
ships as static HTML on GitHub Pages. Add a chat panel that runs a
small LLM client-side via WebAssembly with GBNF-constrained tool
calling, where "tools" navigate the doxygen index. The result is a
live in-browser demo of the same capabilities the engine ships
(constrained generation, tool routing) running against the docs of
the engine itself — dogfood as marketing.

This is deferred from v2.1.0; the v2.1.0 doxygen site lands as a
plain reference. v2.2 adds the embedded-LLM layer.

## Motivation

Two reasons to do this beyond a normal docs nav:

1. **Live capability demo.** Entropic is an inference engine with
   GBNF tool calling. Running that exact pattern in the browser, on
   the docs site, is a memorable "try it without leaving the page"
   pitch. Static demos of inference engines are common; in-browser
   demos that drive site navigation are rare.

2. **Search beyond keyword/symbol match.** The doxygen built-in
   search is keyword-based. An LLM-mediated nav can answer
   "where do I register a hook for post-validation?" with the
   right symbol pointer instead of a keyword-AND match.

## Architecture (proposed)

Stack:
- **wllama** — llama.cpp compiled to WebAssembly. ~2 MB. Supports
  GBNF natively. https://github.com/ngxson/wllama
- **Model** — Qwen2.5-0.5B-Q4_K_M (~280 MB) or SmolLM-360M-Q4
  (~200 MB). Sub-1B is sufficient for tool-routing intent. The
  generation step doesn't need to produce long-form text; it
  produces a structured tool call.
- **GBNF grammar** — constrains output to:
  ```
  {
    "tool": "search_symbol" | "show_file" | "navigate" | "explain",
    "args": { ... }
  }
  ```
- **JS tool handlers**:
  - `search_symbol(query)` — query the doxygen-generated
    `search/searchdata.js` (already shipped)
  - `show_file(path)` — load and render the source HTML
  - `navigate(target)` — `window.location` to a doxygen anchor
  - `explain(symbol)` — emit a brief response based on the doxygen
    `\brief` text from the SQLite index
- **Search corpus** — the `entropic_docs.db` SQLite index
  (`GENERATE_SQLITE3 = YES` in Doxyfile) loaded in-browser via
  sql.js. Or the simpler `search/searchdata.js` if SQLite is
  overkill.

Hosting:
- WASM runtime + JS glue: GitHub Pages alongside doxygen output
  (zero extra cost).
- GGUF model: **Hugging Face Hub** (free model hosting, free CDN
  bandwidth). Fetched on first visit, cached in IndexedDB.
- Avoids the GitHub Pages 100 GB/month bandwidth cap.

UX:
- Sticky chat panel in the lower-right of every doxygen page.
- First visit: explicit "Loading 280 MB model (cached after)"
  banner with progress.
- Cached subsequent visits load in seconds.
- "Power user" toggle to disable for slow connections.

## Risks and mitigations

- **Hallucinated symbol names.** Even with GBNF constraining the
  *shape* of the tool call, sub-1B models can produce
  valid-shape-but-wrong-content args. Mitigation: constrain
  `args.symbol` to an enum drawn from the doxygen index at panel-
  load time. The grammar is generated dynamically from the index
  so the model can't request a symbol that doesn't exist.
- **Cold-start UX.** 280 MB on first visit = 15-45 s depending on
  connection. Mitigation: show explicit progress; defer chat panel
  visibility until the model is ready; fall back to plain doxygen
  search if WebAssembly support is missing or the user dismisses
  the load banner.
- **Battery drain on mobile.** Inference is CPU-heavy. Default the
  panel to closed on touch devices; offer manual open.
- **Sub-1B model quality.** May surface plausible-but-wrong
  results. Mitigation: deterministic retrieval (the tool handler)
  produces the actual answer; the model is only routing intent.
  If retrieval finds nothing, the model returns "no match" rather
  than fabricating one.

## Out of scope

- Server-side compute. Pages is static-only; this is by design.
- Web-LLM via WebGPU. Tempting (faster), but Chrome/Edge-only
  excludes Firefox users. WASM works everywhere.
- Conversational follow-up beyond a single tool call. Each turn
  is intent → tool call → result. Multi-turn context wastes
  cycles for a nav helper.
- Local development indexing. The docs index is generated at
  release time via `docs.yml`; no live re-indexing on the
  client.

## Validation

- Cold-load on a 50 Mbps connection completes within 60 seconds.
- Cached load (second visit) completes within 5 seconds.
- 10 sample queries against the docs ("how do I register a hook",
  "what's the difference between fast and primary tier",
  "which header has entropic_run_streaming", etc.) return a
  correct symbol pointer.
- Model never produces an out-of-vocabulary symbol name (verified
  by the dynamically-generated grammar).
- Pages bandwidth stays within 100 GB/month free tier (GGUF on
  HuggingFace, not on Pages).

## Open questions

- Which exact model? Qwen2.5-0.5B vs SmolLM-360M vs something
  smaller — needs a quality bake-off against the docs corpus.
- Should the chat panel persist its history across sessions, or
  be ephemeral per page-load? Ephemeral is simpler; persistence
  is more useful.
- Tool surface: just nav (search/show/navigate) or also include
  "compare two functions" / "show callers of X"? The latter
  needs richer queries against the SQLite index.
- Pre-generate embeddings for symbol descriptions (semantic
  search bonus) or stick with keyword + LLM intent only?

## Implementation plan (rough)

1. Decide on model and bake off quality on a held-out set of
   navigation queries.
2. Generate the dynamic GBNF grammar from the doxygen index at
   release time; commit alongside the doxygen output.
3. Stand up the chat panel UI as a separate static asset bundle
   in the docs.yml workflow.
4. Wire wllama + model fetch + JS glue.
5. Smoke against staging Pages before promoting.
