## Scope

Finalize the `InferenceBackend` interface to accommodate non-transformer
architectures, non-GGUF model formats, and multi-sequence inference. After
v1.9.13, the interface contract is locked — any future backend (GDN/recurrent,
NPU, batched) can implement it without modifying the interface headers.

This is NOT a new backend implementation. This is interface finalization:
reviewing the v1.8.2 `InferenceBackend` and `i_inference_backend.h` contracts,
identifying transformer-specific assumptions baked in, and replacing them with
architecture-agnostic abstractions.

**What changes:**
- Capability query system (`BackendCapability` enum + `supports()` method)
- State abstraction: "model state" replaces "KV cache" in the interface vocabulary
- Model format agnosticism: load path accepts any format, backend validates
- Sequence ID parameter on generation methods (foundation for 3.0.0 fan-out)
- Backend metadata struct for introspection

**What does NOT change:**
- `InferenceBackend` remains the concrete base class with template method pattern
- `LlamaCppBackend` remains the only implementation (reference backend)
- The COLD/WARM/ACTIVE lifecycle state machine
- The C boundary `i_inference_backend.h` function signatures (additive only)
- `ChatAdapter` hierarchy (unchanged — adapters are model-family-specific,
  not architecture-specific)

**Dependencies consumed:**
- v1.8.2: `InferenceBackend` base class, `LlamaCppBackend`, `ModelOrchestrator`,
  C interface `i_inference_backend.h`, `GenerationParams`, `GenerationResult`
- v1.9.2: `AdapterManager`, LoRA hot-swap (adapter lifecycle is orthogonal to
  backend abstraction but validates that the interface is composable)

## Why This Matters

The v1.8.2 interface was designed with transformers in mind. Several assumptions
leak through that would force non-transformer backends into awkward workarounds:

| Assumption | Where | Problem for non-transformers |
|---|---|---|
| KV cache exists | `llama_kv_cache_clear()` in orchestrator | GDN/recurrent models have hidden state, not KV cache |
| GGUF model format | `ModelConfig::path` implies `.gguf` | NPU backends use `.axmodel` (Pulsar2 compiled) |
| Single sequence per context | `generate()` takes messages, produces one result | Fan-out routing needs parallel sequences on one model |
| All backends support streaming | No capability negotiation | NPU backends may only support batch inference |
| All backends support `complete()` | Raw completion assumed universal | Some backends may only support chat-mode |
| All backends support grammar | `GenerationParams::grammar` passed blindly | NPU backends have no GBNF support |
| Context length is fixed | `context_length()` returns int | Recurrent models have theoretically unlimited context |

## BackendCapability Enum

Architecture-agnostic capability flags. Backends declare what they support;
the engine queries before using optional features.

```cpp
/// @file include/entropic/types/backend_capability.h
/// @brief Backend capability flags for architecture-agnostic queries.
/// @version 1.9.13

#pragma once

namespace entropic {

/// @brief Capabilities that an inference backend may or may not support.
///
/// The engine queries capabilities before using optional features.
/// A backend that does not support a capability returns graceful errors
/// (not crashes) if the feature is invoked anyway.
///
/// @par Design rationale:
/// Enum rather than string tags for compile-time exhaustiveness checking.
/// New capabilities are appended (never reordered) for ABI stability.
enum class BackendCapability : int {
    /// @brief KV cache state management (save/load/clear).
    /// Transformers: yes. GDN/recurrent: no (use HIDDEN_STATE instead).
    KV_CACHE = 0,

    /// @brief Recurrent hidden state management (save/load/reset).
    /// GDN/Falcon-H1, Mamba, RWKV: yes. Pure transformers: no.
    HIDDEN_STATE = 1,

    /// @brief Streaming token-by-token generation.
    /// Most GPU backends: yes. Some NPU backends: no (batch only).
    STREAMING = 2,

    /// @brief Raw text completion without chat template.
    /// llama.cpp: yes. Backends with fused chat logic: maybe not.
    RAW_COMPLETION = 3,

    /// @brief GBNF grammar-constrained generation.
    /// llama.cpp: yes. NPU backends: no.
    GRAMMAR = 4,

    /// @brief LoRA adapter hot-swapping (v1.9.2).
    /// llama.cpp: yes. NPU backends compiled from frozen graph: no.
    LORA_ADAPTERS = 5,

    /// @brief Multiple concurrent sequences on one model instance.
    /// Foundation for 3.0.0 fan-out routing. llama.cpp batch API
    /// supports this, but Entropic uses single-sequence through 2.0.0.
    MULTI_SEQUENCE = 6,

    /// @brief Token counting / tokenizer access.
    /// Backends with embedded tokenizer: yes. Pre-tokenized NPU: no.
    TOKENIZER = 7,

    /// @brief Log-probability retrieval (v1.9.10).
    /// llama.cpp: yes. NPU backends: unlikely.
    LOG_PROBS = 8,

    /// @brief Vision / multimodal input (v1.9.11).
    /// Models with mmproj: yes. Text-only: no.
    VISION = 9,

    /// @brief Speculative decoding compatibility.
    /// Pure transformers: yes. Recurrent models: no.
    /// (llama.cpp auto-disables via common_speculative_is_compat())
    SPECULATIVE_DECODING = 10,

    /// @brief Prompt cache prefix save/load (v1.8.3).
    /// Transformers with KV cache: yes. Recurrent: different mechanism
    /// (hidden state save/restore, also supported if HIDDEN_STATE is set).
    PROMPT_CACHING = 11,

    /// @brief Sentinel — must be last. Used for iteration/array sizing.
    _COUNT
};

} // namespace entropic
```

## BackendInfo Struct

Metadata struct returned by backends for introspection and engine decisions.

```cpp
/// @brief Backend metadata for introspection.
/// @version 1.9.13
///
/// Returned by InferenceBackend::info(). The engine uses this for
/// logging, diagnostics (v1.9.12), and routing decisions.
struct BackendInfo {
    std::string name;                    ///< Backend identifier (e.g. "llama.cpp", "axcl")
    std::string version;                 ///< Backend version string
    std::string compute_device;          ///< "cuda", "vulkan", "cpu", "npu"
    std::string model_format;            ///< "gguf", "axmodel", "onnx", etc.

    /// @brief Architecture family of the loaded model.
    /// "transformer", "gdn", "mamba", "rwkv", "hybrid", "unknown"
    /// Populated after load(). Empty when COLD.
    std::string architecture;

    /// @brief Maximum context length.
    /// For transformers: fixed window from model metadata.
    /// For recurrent: -1 (theoretically unlimited, practically memory-bound).
    int max_context_length = 0;

    /// @brief VRAM consumed by the loaded model (bytes). 0 if COLD.
    size_t vram_bytes = 0;

    /// @brief RAM consumed by the loaded model (bytes). 0 if COLD.
    size_t ram_bytes = 0;

    /// @brief Number of parameters (from model metadata).
    size_t parameter_count = 0;

    /// @brief Quantization type string (e.g. "IQ3_XXS", "Q8_0", "fp16").
    std::string quantization;
};
```

## InferenceBackend Interface Changes

Additive changes to the v1.8.2 `InferenceBackend` concrete base class. No
existing methods are removed or have their signatures changed. The base class
gains new public methods with default implementations that existing subclasses
inherit without modification.

### New Public Methods (base class)

```cpp
/// @file include/entropic/inference/backend.h
/// @brief Additions to InferenceBackend for v1.9.13.

class InferenceBackend {
public:
    // ... existing v1.8.2 methods unchanged ...

    // ── Capability queries (v1.9.13) ──

    /// @brief Query whether this backend supports a capability.
    /// @param cap Capability to query.
    /// @return true if supported.
    ///
    /// Base class returns false for all capabilities. Subclasses override
    /// do_supports() to declare their capabilities.
    /// Lock-free — no state transitions involved.
    bool supports(BackendCapability cap) const;

    /// @brief Get all supported capabilities as a vector.
    /// @return Vector of capabilities this backend supports.
    ///
    /// Convenience method. Iterates BackendCapability enum, calls supports()
    /// on each.
    std::vector<BackendCapability> capabilities() const;

    // ── Backend metadata (v1.9.13) ──

    /// @brief Get backend metadata.
    /// @return BackendInfo populated from model metadata after load().
    ///         Returns default (empty) info if COLD.
    ///
    /// Base class returns a default-constructed BackendInfo with name
    /// from do_backend_name(). Subclasses override do_info() to populate
    /// architecture, quantization, memory usage, etc.
    BackendInfo info() const;

    // ── Model state management (v1.9.13, architecture-agnostic) ──

    /// @brief Save model state to buffer.
    /// @param seq_id Sequence identifier (0 for single-sequence backends).
    /// @param buffer Output buffer. Caller owns the returned data.
    /// @return true on success. false if backend does not support state save
    ///         (check supports(KV_CACHE) or supports(HIDDEN_STATE) first).
    ///
    /// For transformers: saves KV cache state for the sequence.
    /// For recurrent: saves hidden state.
    /// The opaque buffer can be restored with restore_state().
    bool save_state(int seq_id, std::vector<uint8_t>& buffer) const;

    /// @brief Restore model state from buffer.
    /// @param seq_id Sequence identifier to restore into.
    /// @param buffer Previously saved state buffer.
    /// @return true on success. false if buffer is incompatible or capability
    ///         not supported.
    bool restore_state(int seq_id, const std::vector<uint8_t>& buffer);

    /// @brief Clear/reset model state for a sequence.
    /// @param seq_id Sequence identifier (-1 for all sequences).
    /// @return true on success.
    ///
    /// For transformers: clears KV cache (replaces direct llama_kv_cache_clear).
    /// For recurrent: resets hidden state to initial values.
    /// This is what the orchestrator calls on adapter swap instead of
    /// directly calling llama_kv_cache_clear().
    bool clear_state(int seq_id = -1);

    // ── Multi-sequence (v1.9.13, foundation for 3.0.0 fan-out) ──

    /// @brief Generate with explicit sequence ID.
    /// @param seq_id Sequence identifier for multi-sequence backends.
    ///        Single-sequence backends ignore this (treat as 0).
    /// @param messages Conversation history.
    /// @param params Generation parameters.
    /// @return GenerationResult.
    ///
    /// Default implementation: ignores seq_id, delegates to generate().
    /// Multi-sequence backends override do_generate_seq() to route to
    /// the correct batch slot.
    GenerationResult generate_seq(
        int seq_id,
        const std::vector<Message>& messages,
        const GenerationParams& params);

    /// @brief Streaming generation with explicit sequence ID.
    /// @param seq_id Sequence identifier.
    /// @param messages Conversation history.
    /// @param params Generation parameters.
    /// @param on_token Per-token callback.
    /// @param cancel Cancellation flag.
    /// @return GenerationResult.
    GenerationResult generate_streaming_seq(
        int seq_id,
        const std::vector<Message>& messages,
        const GenerationParams& params,
        std::function<void(std::string_view token)> on_token,
        std::atomic<bool>& cancel);

protected:
    // ... existing v1.8.2 pure virtuals unchanged ...

    // ── New overridable methods (v1.9.13) ──

    /// @brief Declare supported capabilities.
    /// @param cap Capability to check.
    /// @return true if this backend supports the capability.
    ///
    /// Default: returns false for everything. Subclasses override to
    /// declare their capabilities.
    virtual bool do_supports(BackendCapability cap) const;

    /// @brief Return backend name identifier.
    /// @return Short name (e.g. "llama.cpp", "axcl").
    virtual std::string do_backend_name() const = 0;

    /// @brief Populate backend metadata.
    /// @return BackendInfo with model-specific details.
    ///
    /// Default: returns BackendInfo with name only.
    /// Called by info() — subclass populates architecture, memory, etc.
    virtual BackendInfo do_info() const;

    /// @brief Save model state (KV cache or hidden state).
    /// Default: returns false (not supported).
    virtual bool do_save_state(int seq_id,
                               std::vector<uint8_t>& buffer) const;

    /// @brief Restore model state.
    /// Default: returns false (not supported).
    virtual bool do_restore_state(int seq_id,
                                  const std::vector<uint8_t>& buffer);

    /// @brief Clear/reset model state.
    /// Default: returns false (not supported).
    virtual bool do_clear_state(int seq_id);

    /// @brief Generate with sequence ID.
    /// Default: ignores seq_id, delegates to do_generate().
    virtual GenerationResult do_generate_seq(
        int seq_id,
        const std::vector<Message>& messages,
        const GenerationParams& params);

    /// @brief Streaming generation with sequence ID.
    /// Default: ignores seq_id, delegates to do_generate_streaming().
    virtual GenerationResult do_generate_streaming_seq(
        int seq_id,
        const std::vector<Message>& messages,
        const GenerationParams& params,
        std::function<void(std::string_view token)> on_token,
        std::atomic<bool>& cancel);
};
```

### Base Class Implementation (the 80%)

The new public methods follow the same template method pattern as v1.8.2:

1. **`supports(cap)`**: Delegates to `do_supports(cap)`. No locking. Pure query.

2. **`capabilities()`**: Iterates `BackendCapability::KV_CACHE` through
   `BackendCapability::_COUNT - 1`, returns vector of capabilities where
   `supports()` returns true.

3. **`info()`**: Calls `do_info()`. Base implementation returns `BackendInfo`
   with `name = do_backend_name()`. If state is not COLD, subclass populates
   remaining fields from loaded model metadata.

4. **`save_state()`**: Checks if ACTIVE. If not, returns false. Delegates to
   `do_save_state()`. Logs timing.

5. **`restore_state()`**: Checks if ACTIVE. Delegates to `do_restore_state()`.
   Logs timing.

6. **`clear_state()`**: Checks if WARM or ACTIVE. Delegates to
   `do_clear_state()`. Logs which seq_id was cleared.

7. **`generate_seq()`**: Same guard as `generate()` (requires ACTIVE). Delegates
   to `do_generate_seq()`. Default `do_generate_seq()` ignores `seq_id` and
   calls `do_generate()` — single-sequence backends work without overriding.

8. **`generate_streaming_seq()`**: Same pattern.

## LlamaCppBackend Changes

`LlamaCppBackend` becomes the reference implementation of the finalized
interface. Changes are additive — existing overrides are untouched.

### Capability Declaration

```cpp
/// @brief LlamaCppBackend v1.9.13 additions.

bool LlamaCppBackend::do_supports(BackendCapability cap) const {
    switch (cap) {
        case BackendCapability::KV_CACHE:
            return !is_recurrent();  // GDN models use hidden state
        case BackendCapability::HIDDEN_STATE:
            return is_recurrent();
        case BackendCapability::STREAMING:
            return true;
        case BackendCapability::RAW_COMPLETION:
            return true;
        case BackendCapability::GRAMMAR:
            return true;
        case BackendCapability::LORA_ADAPTERS:
            return true;
        case BackendCapability::MULTI_SEQUENCE:
            return false;  // Single-sequence through 2.0.0
        case BackendCapability::TOKENIZER:
            return true;
        case BackendCapability::LOG_PROBS:
            return true;
        case BackendCapability::VISION:
            return has_mmproj();  // From v1.9.11
        case BackendCapability::SPECULATIVE_DECODING:
            return !is_recurrent();  // Per llama.cpp PR #19270
        case BackendCapability::PROMPT_CACHING:
            return true;  // Both transformer and recurrent (PR #19045)
        default:
            return false;
    }
}
```

### Architecture Detection

```cpp
/// @brief Detect model architecture from GGUF metadata.
///
/// llama.cpp exposes architecture info via model metadata.
/// GDN models (Falcon-H1 family) have the "falcon-h1" arch tag.
/// Hybrid models (Mamba-Transformer) have architecture-specific tags.
///
/// This drives capability reporting — whether the model uses KV cache
/// or hidden state, whether speculative decoding is compatible, etc.
bool LlamaCppBackend::is_recurrent() const {
    // Query llama.cpp model architecture metadata
    // GDN, Mamba, RWKV → recurrent
    // Transformer, hybrid with KV → not recurrent
    // Cached after do_load() to avoid repeated metadata queries
    return is_recurrent_;
}
```

The `is_recurrent_` flag is set during `do_load()` by querying the GGUF
metadata for the model architecture. llama.cpp provides this through the
model's architecture string in metadata. The flag is cached as a member.

### State Management

```cpp
bool LlamaCppBackend::do_save_state(int seq_id,
                                     std::vector<uint8_t>& buffer) const {
    if (is_recurrent()) {
        // Save recurrent hidden state
        // llama.cpp handles this through llama_state_seq_get_data()
        // same API, different internal state (hidden vs KV)
        return save_sequence_state(seq_id, buffer);
    }
    // Save KV cache state for sequence
    return save_sequence_state(seq_id, buffer);
}

bool LlamaCppBackend::do_clear_state(int seq_id) {
    if (is_recurrent()) {
        // Reset recurrent hidden state
        // llama.cpp: llama_memory_seq_rm() not supported for recurrent
        // (per PR #19045), use llama_kv_cache_clear() which internally
        // handles both KV and recurrent state
        llama_kv_cache_clear(ctx_);
        return true;
    }
    if (seq_id < 0) {
        llama_kv_cache_clear(ctx_);
    } else {
        llama_kv_cache_seq_rm(ctx_, seq_id, -1, -1);
    }
    return true;
}
```

### Backend Info

```cpp
BackendInfo LlamaCppBackend::do_info() const {
    BackendInfo bi;
    bi.name = "llama.cpp";
    bi.version = /* llama.cpp version string */;
#if defined(ENTROPIC_BACKEND_CUDA)
    bi.compute_device = "cuda";

    bi.compute_device = "vulkan";

        // Populate from llama.cpp model metadata

        bi.quantization = /* from GGUF metadata */;
        bi.vram_bytes = llama_model_vram_bytes(model_);
        bi.ram_bytes = llama_model_ram_bytes(model_);
    }
    return bi;
}
```

## Orchestrator Changes

The `ModelOrchestrator` is updated to use the abstracted state management
instead of direct llama.cpp calls.

### Adapter Swap State Clear

Before v1.9.13, the orchestrator called `llama_kv_cache_clear(ctx)` directly
after adapter swap (v1.9.2). This is replaced with `backend->clear_state(-1)`:

```
// v1.9.2 (before):
llama_kv_cache_clear(ctx_);

// v1.9.13 (after):
backend->clear_state(-1);
```

This works identically for `LlamaCppBackend` (which delegates to
`llama_kv_cache_clear` internally) but will also work for future backends
that have different state representations.

### Prompt Cache Integration

The v1.8.3 prompt cache layer uses `llama_state_seq_get_data` /
`llama_state_seq_set_data` directly. After v1.9.13, it uses the backend's
`save_state()` / `restore_state()` methods instead:

```
// v1.8.3 (before):
llama_state_seq_get_data(ctx, buf, seq_id);

// v1.9.13 (after):
backend->save_state(seq_id, buffer);
```

This ensures prompt caching works for recurrent models too (llama.cpp PR #19045
fixed prompt caching for recurrent models — the state saved is hidden state
rather than KV cache, but the API is the same).

### Capability-Gated Features

The orchestrator checks capabilities before invoking optional features:

```cpp
// Router uses raw completion — verify backend supports it
if (!router_backend->supports(BackendCapability::RAW_COMPLETION)) {
    spdlog::error("Router backend does not support raw completion");
    return default_tier_;
}

// Grammar constraint — only if backend supports it
if (!params.grammar.empty() &&
    !backend->supports(BackendCapability::GRAMMAR)) {
    spdlog::warn("Backend does not support grammar, ignoring constraint");
    params.grammar.clear();
}

// LoRA adapter swap — only if backend supports it
if (tier_config.adapter_path.has_value() &&
    !backend->supports(BackendCapability::LORA_ADAPTERS)) {
    spdlog::warn("Backend does not support LoRA adapters, "
                  "tier '{}' adapter_path ignored", tier_name);
}
```

## Future Backend Contract

This section documents the contract that future backend implementations must
satisfy. It is NOT implemented in v1.9.13 — it is the specification that
v1.9.13 enables.

### GDN / Recurrent Backend (3.0.0)

A GDN backend for Falcon-H1 family models. Uses the same `LlamaCppBackend`
class (GDN kernels are merged into llama.cpp via PR #20340) but with different
capability reporting:

```
supports(KV_CACHE) = false
supports(HIDDEN_STATE) = true
supports(STREAMING) = true
supports(SPECULATIVE_DECODING) = false  (PR #19270)
supports(PROMPT_CACHING) = true  (PR #19045)
```

The architecture is detected from GGUF metadata at load time. No separate
backend class needed — `LlamaCppBackend` handles both architectures through
the `is_recurrent()` flag. The capability query system makes this transparent
to the engine.

### NPU Backend (3.0.0+)

An `AxeraBackend` for AX630C NPU inference:

```
supports(KV_CACHE) = false
supports(HIDDEN_STATE) = false
supports(STREAMING) = false  (batch inference only)
supports(RAW_COMPLETION) = false  (pre-compiled graph)
supports(GRAMMAR) = false
supports(LORA_ADAPTERS) = false  (compiled model)
supports(TOKENIZER) = false  (tokenizer in host-side Python/C++)
supports(PROMPT_CACHING) = false
```

The NPU backend would subclass `InferenceBackend` and override only the
core `do_load` / `do_activate` / `do_generate` methods. The Pulsar2/AXCL
SDK provides the model loading and inference API. Model format is `.axmodel`.

### Multi-Sequence Backend (3.0.0)

Fan-out routing needs multiple sequences generated in parallel from one
model. The `seq_id` parameter on `generate_seq()` provides the hook:

```cpp
// Fan-out coordinator (3.0.0):
auto eng_future = pool.submit([&] {
    return backend->generate_seq(1, messages, eng_params);
});
auto qa_future = pool.submit([&] {
    return backend->generate_seq(2, messages, qa_params);
});
// Both sequences share the model's KV cache prefix
// (system prompt + shared context), diverge at generation
```

This requires `supports(MULTI_SEQUENCE) = true`. The llama.cpp batch API
supports this already, but `LlamaCppBackend` returns false through 2.0.0.

## Validation Criteria

### V1: Capability queries reflect backend

Instantiate `LlamaCppBackend`, load a transformer model. Verify
`supports(KV_CACHE)` returns true, `supports(HIDDEN_STATE)` returns false,
`supports(STREAMING)` returns true, `supports(MULTI_SEQUENCE)` returns false.

### V2: Capability bitmask matches individual queries

Call `entropic_inference_capabilities()`. Verify the bitmask has bit N set
for exactly those capabilities where `entropic_inference_supports(N)` returns 1.

### V3: BackendInfo populated after load

Load a model. Call `info()`. Verify `name`, `compute_device`, `model_format`,
`architecture`, `max_context_length`, `parameter_count`, and `quantization`
are all non-empty. Verify `vram_bytes > 0` when ACTIVE.

### V4: BackendInfo empty when COLD

Call `info()` on a COLD backend. Verify `name` is populated (static), all
other fields are default/empty/zero.

### V5: save_state / restore_state round-trip

Generate some tokens (populate KV cache). Save state for seq 0. Clear state.
Restore state. Generate again. Verify the generation continues coherently
from the restored state (not from scratch).

### V6: clear_state replaces direct KV cache call

Trigger adapter swap via orchestrator. Verify the orchestrator calls
`backend->clear_state(-1)` (not `llama_kv_cache_clear` directly). Verify
generation after swap works correctly.

### V7: generate_seq defaults to generate

Call `generate_seq(42, messages, params)` on `LlamaCppBackend`. Verify it
produces the same result as `generate(messages, params)` — seq_id is ignored
by single-sequence backends.

### V8: Unsupported capability returns graceful error

Call `entropic_inference_save_state()` on a backend that does not support
`KV_CACHE` or `HIDDEN_STATE`. Verify it returns
`ENTROPIC_ERROR_NOT_SUPPORTED`, not a crash.

### V9: Grammar ignored when unsupported

Set `grammar` in `GenerationParams`. Call `generate()` on a backend where
`supports(GRAMMAR)` is false. Verify grammar is silently cleared and
generation succeeds without constraint.

### V10: model_format validation

Set `model_format = "axmodel"` in ModelConfig. Attempt to load with
`LlamaCppBackend`. Verify `ENTROPIC_ERROR_LOAD_FAILED` with diagnostic
message indicating format mismatch.

### V11: Architecture detection sets is_recurrent

Load a transformer model. Verify `info().architecture == "transformer"`.
If a GDN GGUF is available (Falcon-H1), load it and verify
`info().architecture == "gdn"` and `supports(KV_CACHE) == false`,
`supports(HIDDEN_STATE) == true`.

### V12: Orchestrator capability gates

Configure a tier with `adapter_path` but load on a backend that reports
`supports(LORA_ADAPTERS) == false`. Verify WARNING logged and adapter
silently skipped. Verify generation proceeds with base model.

### V13: C API functions operational

Call all new C API functions (`entropic_inference_supports`,
`entropic_inference_capabilities`, `entropic_inference_info`,
`entropic_inference_save_state`, `entropic_inference_restore_state`,
`entropic_inference_clear_state`, `entropic_inference_generate_seq`).
Verify correct return values and no memory leaks (caller frees returned
strings).

### V14: SLOC and complexity

All new and modified functions pass knots quality gates: cognitive
complexity <= 15, SLOC <= 50, returns <= 3.

### V15: Doxygen complete

Every new public symbol has `@brief`, `@param`, `@return`, `@version`.
Every enum value has `///<` inline doc.

### V16: Backward compatibility

Compile the v1.8.2 unit tests against the modified `InferenceBackend`
header without changes. All existing tests pass — no regressions from
additive interface changes.

### V17: Plugin API version bumped

`entropic_plugin_api_version()` returns a new version number for inference

## Design Decisions

### Why enum rather than string capabilities?

Compile-time exhaustiveness. A `switch` over `BackendCapability` with no
`default` case warns on missing cases. String-based capability tags
(`backend->supports("kv_cache")`) silently fail on typos and cannot be
validated at compile time. The enum values are appended only (never
reordered) for ABI stability.

### Why default implementations rather than new pure virtuals?

Adding pure virtuals would break every existing `InferenceBackend` subclass.
Default implementations (`do_supports()` returns false, `do_generate_seq()`
delegates to `do_generate()`) ensure backward compatibility. Existing code
compiles and runs identically. New capabilities are opt-in.

### Why seq_id on generate rather than a separate BatchBackend class?

A separate `BatchBackend` hierarchy would require the orchestrator to handle
two backend types at runtime, with casting and type checking. A single
`seq_id` parameter on the existing interface is simpler — single-sequence
backends ignore it (default implementation delegates to the seq-unaware
method). When multi-sequence support is implemented in 3.0.0, the same
interface accommodates it without hierarchy changes.

### Why is state management in InferenceBackend, not a separate class?

State (KV cache, hidden state) is fundamentally tied to the inference context.
A separate `StateManager` class would need access to the backend's internal
handles, creating tight coupling without encapsulation benefit. The backend
owns its state — save/restore/clear are natural methods on the backend itself.
The v1.8.3 prompt cache layer becomes a consumer of the backend's state API
rather than a peer that reaches into llama.cpp directly.

### Why not finalize the interface at v1.8.2?

v1.8.2 was a faithful port of the Python engine. Finalizing the interface
before implementing LoRA (v1.9.2), grammar registry (v1.9.3), logprobs
(v1.9.10), and vision (v1.9.11) would have required speculative interface
design. By v1.9.13, all features that exercise the interface have been
implemented, revealing which assumptions need abstraction.

## Implementation Notes (from review)

1. **seq_id on generate is a C API change.** Adding `seq_id` to existing
   `entropic_inference_generate()` would break existing consumers. The approach
   is: add NEW functions (`entropic_inference_generate_seq()`,
   `entropic_inference_generate_streaming_seq()`) alongside the originals.
   The originals delegate to the new functions with `seq_id=0`. This avoids
   an API version bump while preserving backward compatibility. The original
   functions are NOT deprecated — they remain the primary API for
   single-sequence consumers (which is all consumers through 2.0.0).
