## Scope

Port the 42 Python model tests (deleted in v1.9.15) to C++ in
`tests/model/`. These are GPU-required, developer-run tests that
validate end-to-end engine behavior with a live model. Results are
checked into `test-reports/model/`.

This was originally v1.9.0 Phase 6, deferred because the C engine
wasn't complete. The engine is now complete (v1.9.15). The
`tests/model/CMakeLists.txt` skeleton exists from v1.10.0.

**This proposal requires interactive development.** Model test
thresholds, enabled tests, and quality criteria need hands-on tuning
with the actual model. The plan below defines structure, not final
test content.

## Implementation Plan

### Phase 1: Test Infrastructure

Populate `tests/model/CMakeLists.txt` with real targets. Gate behind
`ENTROPIC_BUILD_MODEL_TESTS=ON`.

Test harness needs:
- C API engine initialization (`entropic_create`, `entropic_configure`)
- Model loading (requires GGUF on disk, path from env or config)
- Result capture (JSON output → assertions)
- Report generation (unified JSON in `test-reports/model/`)
- Sequence diagram generation (PlantUML, matching Python format)

### Phase 2: Classification Tests (highest value, fastest)

Port the 19 classification tests first — they're the fastest (~0.1s each)
and validate the router model works with the C engine.

- `model_routing_test.cpp`:
  - Example classifies to own tier (8 tiers)
  - Novel prompt classifies correctly (8 prompts)
  - Classification under 500ms
  - Multiple classifications consistent
  - Route to non-default tier

### Phase 3: Generation Quality Tests

Port the 6 generation quality tests. These take 4-65s each and validate
the primary model produces usable output.

- `model_inference_test.cpp`:
  - Hello produces response
  - Response ends cleanly
  - No special tokens in response
  - Code request produces code
  - Code response has Python syntax
  - Explanation is substantive

### Phase 4: Advanced Tests

Port the remaining tests — grammar, delegation, multi-turn. These are
the most complex and may need the most tuning.

- `model_identity_test.cpp`:
  - Grammar constrains output
  - Grammar auto-chain fires
  - Grammar without auto-chain is terminal
  - Multi-tier pipeline completes
  - Multiple messages accumulate history
  - Confidence above threshold (8 tiers)

### Phase 5: Benchmark Integration

Port the performance benchmarks into the same suite:

- `bench_inference.cpp`: tok/s measurement
- `bench_engine_overhead.cpp`: engine loop overhead vs raw inference
- `bench_memory_profile.cpp`: VRAM/RAM usage at load, generate, idle

## Interactive Decisions (to be made during implementation)

These cannot be pre-decided — they require running the tests:

1. **Which tests stay?** Some Python tests may not map cleanly to the
   C engine's behavior. Drop or adapt case by case.
2. **Threshold tuning:** Classification accuracy, response length,
   latency budgets — all need calibration against the C engine.
3. **Confidence thresholds:** The Python tests used per-tier confidence
   thresholds. These may need recalibration.
4. **Report format:** JSON vs text vs both. What goes in git, what's
   ephemeral.
5. **Benchmark baselines:** Establish C engine baselines (tok/s, latency,
   VRAM). Python baseline was 36.1 tok/s.
