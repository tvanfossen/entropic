## Summary

When an external MCP server connects or disconnects mid-session, the
model's tool list is stale — it was injected into the system prompt at
session start. The model has no way to discover new capabilities or
learn that a server went away.

## Design

Hybrid approach: keep tool schema injection as the default (small models
need it), but use `entropic.inspect` as the refresh mechanism.

When an external MCP server connects or disconnects:
1. Engine fires `ON_TOOLS_CHANGED` hook (new hook point).
2. Hook handler injects a system message into the conversation:
   `[TOOLS UPDATED] Call entropic.inspect to see current capabilities.`
3. Model calls `entropic.inspect` (no params) and receives the updated
   tier-relative tool list including the new/removed server's tools.

This avoids re-injecting the full tool schema into the system prompt
(which would require context rewriting) while giving the model a
discoverable path to the current state.

## Prerequisites

- v1.8.7 external MCP (STAGED) — runtime `connect_mcp_server` API
- v2.0.3 MCP bridge (STAGED) — stdio/SSE transport
- v2.0.6+ state provider wiring — `entropic.inspect` functional

## Out of scope

- Replacing tool schema injection with discovery-only (rejected:
  small models can't reliably self-orient without explicit tool list)
- Per-tool hot-swap (connect replaces the full server, not individual tools)

## Open questions

- Should `ON_TOOLS_CHANGED` be a pre-hook (cancellable) or info-only?
- Should the injected system message include a summary of what changed,
  or just a generic "call inspect"?
- Rate limiting: if a flaky server reconnects rapidly, suppress
  duplicate notifications within a window?
