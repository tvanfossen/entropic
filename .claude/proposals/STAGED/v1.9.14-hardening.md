## Scope

Systematic validation of the complete C++ engine under extreme conditions.
After v1.9.14, every subsystem from v1.8.0 through v1.9.13 has been stress-tested,
sanitizer-verified, and benchmarked against the Python engine baseline.

This is NOT new features. This is proving the engine is production-ready:
no leaks, no data races, no undefined behavior, graceful degradation under
resource exhaustion, and measurable performance gains over the Python engine.

**What this version produces:**
- Stress test suite (long sessions, context pressure, rapid interrupts)
- Sanitizer CI jobs (ASan, TSan, MSan, UBSan) running the full test suite
- Valgrind memcheck pass over the full test suite
- Edge case test suite (malformed inputs, crash recovery, OOM)
- Concurrency test suite (multi-threaded C API access patterns)
- Performance benchmark suite with Python engine comparison
- CI pipeline configuration for sanitizer builds

**What does NOT change:**
- No new public API surface
- No interface header modifications
- No new library `.so` files
- No changes to existing test infrastructure (additive only)

**Dependencies consumed:**
- v1.9.13 (and transitively all of v1.8.x and v1.9.x): the complete engine.
  Every subsystem must be functional before hardening can validate it.

---

## Why This Matters

The C++ engine has been built subsystem-by-subsystem across 22 versions
(v1.8.0 through v1.9.13). Each version has unit tests proving its own
correctness. What is missing:

| Gap | Risk | Example |
|---|---|---|
| Cross-subsystem leaks | Memory grows unbounded in long sessions | Context compaction frees messages but leaks adapter state |
| Data races | Corruption under concurrent access | Interrupt during delegation while hook fires on same state |
| Undefined behavior | Silent corruption, intermittent crashes | Signed integer overflow in token counting, use-after-free on cancel |
| Resource exhaustion | Engine crashes instead of degrading | CUDA OOM during generation kills the process instead of returning error |
| Performance regression | C++ engine slower than Python in some paths | JSON serialization at `.so` boundaries adds latency Python didn't have |

Unit tests prove components work. Hardening proves the system survives.

## Test Categories

### 1. Stress Tests

Long-running, high-volume tests that exercise the engine under sustained load.
These run outside pre-commit (too slow). Developer-run with checked-in reports.

#### 1.1 Long Session Endurance

Simulate extended interactive sessions. Validates no resource drift over time.

| Test | Description | Duration | Pass Criteria |
|---|---|---|---|
| `stress_long_session_1000_turns` | 1000 user/assistant turns, mixed tool calls | ~30 min | RSS delta < 5% from turn 100 baseline |
| `stress_long_session_delegation_heavy` | 500 turns, each triggers delegation chain (lead → eng → qa) | ~45 min | No leaked child contexts, RSS stable |
| `stress_session_with_compaction` | 2000 turns forcing repeated context compaction | ~60 min | Compaction count > 10, no corruption in message history |

**Measurement:** RSS sampled every 10 turns. Linear regression on RSS after
warmup (first 100 turns). Slope must be < 1 KB/turn.

#### 1.2 Context Pressure

Push the context window to its limits and verify graceful behavior.

| Test | Description | Pass Criteria |
|---|---|---|
| `stress_context_fill_to_limit` | Generate until context window is 95% full | Compaction triggers, generation continues |
| `stress_context_rapid_fill` | 100 large tool results in rapid succession | No OOM, compaction fires, oldest messages evicted |
| `stress_context_anchor_churn` | Rapidly replace context anchors (100 file reads, same file) | Only one anchor instance in context at any time |

#### 1.3 Rapid Interrupts

Cancel tokens fired at high frequency during generation.

| Test | Description | Pass Criteria |
|---|---|---|
| `stress_interrupt_during_generation` | Cancel after every 5 tokens, 500 cycles | No crash, no leaked GPU memory, clean state after each cancel |
| `stress_interrupt_during_delegation` | Cancel parent while child is mid-generation | Child terminates, parent returns to idle, no orphaned threads |
| `stress_interrupt_during_tool_exec` | Cancel during filesystem/bash tool execution | Tool execution abandoned cleanly, no partial state |
| `stress_rapid_cancel_restart` | Start generation, cancel after 1 token, restart — 1000 cycles | Engine state is IDLE before each restart, no cumulative drift |

### 2. Sanitizer Builds

Compile the full engine and test suite with sanitizers enabled. Each sanitizer
is a separate CI job because they are mutually exclusive and add significant
overhead.

#### 2.1 AddressSanitizer (ASan)

Detects: heap/stack/global buffer overflows, use-after-free, double-free,
memory leaks (leak sanitizer integrated).

```
CMake flags:
  -DCMAKE_C_FLAGS="-fsanitize=address -fno-omit-frame-pointer"
  -DCMAKE_CXX_FLAGS="-fsanitize=address -fno-omit-frame-pointer"
  -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=address"
```

| Job | What runs | Pass Criteria |
|---|---|---|
| `ci-asan-unit` | Full unit test suite | Zero ASan errors |
| `ci-asan-integration` | Integration tests (engine loop, tool calls, delegation) | Zero ASan errors |
| `ci-asan-stress-short` | Abbreviated stress suite (100-turn versions) | Zero ASan errors, no leaks reported by LeakSanitizer |

**Expected overhead:** 2-3x slowdown. Unit tests remain under 5 min.

#### 2.2 ThreadSanitizer (TSan)

Detects: data races, lock-order inversions, deadlocks.

```
CMake flags:
  -DCMAKE_C_FLAGS="-fsanitize=thread"
  -DCMAKE_CXX_FLAGS="-fsanitize=thread"
  -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=thread"
```

| Job | What runs | Pass Criteria |
|---|---|---|
| `ci-tsan-unit` | Full unit test suite | Zero TSan warnings |
| `ci-tsan-concurrency` | Dedicated concurrency test suite (Section 5) | Zero TSan warnings |
| `ci-tsan-interrupt` | Interrupt stress tests (1.3) | Zero data races on cancel path |

**Expected overhead:** 5-15x slowdown. Concurrency suite designed to be fast
despite overhead (targeted scenarios, not endurance).

**Critical paths to verify:**
- `std::atomic<ModelState>` reads concurrent with `transition_mutex_` writes
- Cancel token (`std::atomic<bool>`) set from consumer thread, read from generation thread
- Hook registry iteration concurrent with hook registration
- Context anchor replacement concurrent with context read for prompt assembly
- Audit log append concurrent with log query

#### 2.3 MemorySanitizer (MSan)

Detects: reads of uninitialized memory.

```
CMake flags:
  -DCMAKE_C_FLAGS="-fsanitize=memory -fno-omit-frame-pointer"
  -DCMAKE_CXX_FLAGS="-fsanitize=memory -fno-omit-frame-pointer"
  -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=memory"
```

| Job | What runs | Pass Criteria |
|---|---|---|
| `ci-msan-unit` | Full unit test suite | Zero MSan errors |

**Constraint:** MSan requires all linked libraries (including libc++) to be
instrumented. This means llama.cpp must also be built with MSan, or MSan
runs are limited to subsystems that don't call into llama.cpp. Evaluate
feasibility during implementation — if llama.cpp MSan build is impractical,
scope MSan to `librentropic-core`, `librentropic-config`, `librentropic-mcp`,
`librentropic-storage`, and `librentropic-types` only.

#### 2.4 UndefinedBehaviorSanitizer (UBSan)

Detects: signed integer overflow, null pointer dereference, misaligned access,
shift by negative/overwide amount, division by zero.

```
CMake flags:
  -DCMAKE_C_FLAGS="-fsanitize=undefined -fno-omit-frame-pointer"
  -DCMAKE_CXX_FLAGS="-fsanitize=undefined -fno-omit-frame-pointer"
  -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=undefined"
```

| Job | What runs | Pass Criteria |
|---|---|---|
| `ci-ubsan-unit` | Full unit test suite | Zero UBSan errors |
| `ci-ubsan-integration` | Integration tests | Zero UBSan errors |

**ASan + UBSan combo:** These two can run together. Consider a combined
`ci-asan-ubsan` job to reduce CI time while catching both classes.

### 3. Valgrind Memcheck

Complements ASan with different detection characteristics. Valgrind catches
some patterns ASan misses (conditional jumps on uninitialized values) and
does not require recompilation.

| Job | What runs | Pass Criteria |
|---|---|---|
| `valgrind-unit` | Full unit test suite under `valgrind --leak-check=full --show-leak-kinds=all` | Zero "definitely lost", zero "indirectly lost" |
| `valgrind-integration` | Integration tests | Same as above |

**Suppression file:** `tests/valgrind.supp` for known llama.cpp allocations
that are freed at process exit (GPU driver allocations, CUDA context).
Suppressions must be reviewed — every suppression needs a comment explaining
why it is acceptable.

**Expected overhead:** 10-30x slowdown. Valgrind runs are CI-only, not
developer-run routinely.

### 4. Edge Case Tests

Targeted tests for failure modes. These are fast (unit-test speed) and run
in the standard test suite.

#### 4.1 Malformed Tool Calls

| Test | Input | Expected Behavior |
|---|---|---|
| `edge_malformed_json_tool_args` | `{"tool": "bash", "args": "{not json}"}` | `ENTROPIC_ERROR_GENERATE_FAILED`, error callback with parse details |
| `edge_missing_tool_name` | Tool call JSON with no `name` field | Structured error, engine continues to next turn |
| `edge_unknown_tool_name` | `{"tool": "nonexistent.tool", "args": "{}"}` | `ENTROPIC_ERROR_TOOL_NOT_FOUND`, error logged, engine continues |
| `edge_tool_args_wrong_type` | String where object expected, array where string expected | Schema validation error, engine continues |
| `edge_tool_result_exceeds_context` | Tool returns 1MB of text | Result truncated or compaction triggered, no OOM |
| `edge_empty_tool_result` | Tool returns empty string | Engine handles gracefully, no null pointer dereference |
| `edge_tool_result_binary_data` | Tool returns bytes with null characters | No string corruption, proper handling or rejection |
| `edge_recursive_delegation` | Tool call attempts infinite delegation depth | `max_delegation_depth` enforced, returns error at limit |

#### 4.2 Model Crash Recovery

| Test | Scenario | Expected Behavior |
|---|---|---|
| `edge_model_segfault_recovery` | Inject fault during `do_generate()` (signal handler test) | Engine catches signal, returns error code, remains usable |
| `edge_model_returns_empty` | `do_generate()` returns zero tokens | Engine retries up to `max_consecutive_errors`, then returns error to consumer |
| `edge_model_returns_garbage` | `do_generate()` returns invalid UTF-8 | Adapter detects and reports, engine returns error |
| `edge_model_hangs` | `do_generate()` blocks indefinitely (simulated) | Cancel token fires after timeout, engine recovers |
| `edge_model_load_corrupt_file` | Load a truncated/corrupt GGUF file | `ENTROPIC_ERROR_LOAD_FAILED` with descriptive message, no crash |
| `edge_model_load_wrong_format` | Pass a JPEG as model path | Same as above |

**Implementation note:** Signal-based recovery (SIGSEGV from model crash) is
platform-specific and inherently unreliable. The `edge_model_segfault_recovery`
test validates that the engine's signal handler at least attempts recovery.
Document that model crashes in the inference backend are not guaranteed
recoverable — the engine makes a best-effort attempt.

#### 4.3 OOM Recovery

| Test | Scenario | Expected Behavior |
|---|---|---|
| `edge_cuda_oom_during_load` | Attempt to load model exceeding available VRAM | `ENTROPIC_ERROR_LOAD_FAILED`, VRAM freed, state returns to COLD |
| `edge_cuda_oom_during_generate` | Trigger OOM mid-generation (large context + batch) | `ENTROPIC_ERROR_GENERATE_FAILED`, partial VRAM freed, model state recoverable |
| `edge_host_oom_during_compaction` | Compaction allocator fails (simulated via allocation hook) | Error propagated, original context preserved (compaction is non-destructive) |
| `edge_host_oom_prompt_cache` | Prompt cache exceeds configured RAM limit | LRU eviction fires, no crash, cache degrades gracefully |
| `edge_oom_during_plugin_load` | `dlopen` succeeds but plugin init allocates too much | Plugin rejected, error reported, engine continues without that plugin |

**CUDA OOM detection:** llama.cpp surfaces CUDA OOM as error returns from
`llama_decode()` and `llama_model_load()`. The `LlamaCppBackend` must
translate these into `entropic_error_t` codes without calling `abort()` or
`exit()`.

#### 4.4 Config Edge Cases

| Test | Input | Expected Behavior |
|---|---|---|
| `edge_config_empty_file` | Zero-byte config YAML | Defaults applied, warning logged |
| `edge_config_invalid_yaml` | Syntactically broken YAML | `ENTROPIC_ERROR_INVALID_CONFIG` with line number |
| `edge_config_unknown_fields` | Config with extra fields not in schema | Unknown fields ignored, warning logged |
| `edge_config_negative_context_length` | `context_length: -500` | Validation rejects, clear error message |
| `edge_config_missing_model_path` | Config with no model path and no bundled model key | `ENTROPIC_ERROR_MODEL_NOT_FOUND` |
| `edge_config_circular_delegation` | Tier A delegates to B, B delegates to A | Cycle detected at config load, rejected |

### 5. Concurrency Tests

Multi-threaded access patterns exercising the C API from multiple threads
simultaneously. These are the primary TSan targets.

#### 5.1 Multi-Thread C API Access

| Test | Pattern | Pass Criteria |
|---|---|---|
| `conc_simultaneous_generate` | 4 threads call `entropic_inference_generate()` on same backend | Serialized by backend mutex, all return valid results or queuing error |
| `conc_generate_while_loading` | Thread A loads model, thread B calls generate before load completes | Thread B gets `ENTROPIC_ERROR_LOAD_FAILED` or blocks until ready (defined behavior, not UB) |
| `conc_config_reload_during_generate` | Thread A generates, thread B reloads config | Config reload waits for generation to complete, or generation uses old config atomically |
| `conc_tool_registry_concurrent_access` | 8 threads call different tools simultaneously | Tool executor handles concurrent dispatch, no registry corruption |
| `conc_hook_fire_during_registration` | Thread A registers hook, thread B triggers hook point | No crash, hook either fires or doesn't (atomic visibility) |

#### 5.2 Interrupt Concurrency

| Test | Pattern | Pass Criteria |
|---|---|---|
| `conc_cancel_from_multiple_threads` | 4 threads set cancel flag simultaneously | Single cancellation, clean shutdown, no double-free |
| `conc_interrupt_during_delegation` | Parent cancels while child is mid-tool-execution | Child cleans up, parent receives cancellation result |
| `conc_destroy_during_generate` | Consumer calls `entropic_destroy()` while generation active | Generation cancelled first, then resources freed, no use-after-free |

#### 5.3 Plugin Lifecycle Concurrency

| Test | Pattern | Pass Criteria |
|---|---|---|
| `conc_plugin_load_unload` | Load and unload MCP plugins while tools are executing | Executing tools complete, unloaded plugin tools return error on next call |
| `conc_external_mcp_disconnect` | External MCP server disconnects mid-tool-call | Structured error returned, reconnection attempted on next call |

### 6. Performance Benchmarks

Quantitative comparison of C++ engine vs Python engine. These benchmarks
establish the performance baseline for 2.0.0 release notes.

#### 6.1 Inference Performance

| Metric | Test | Python Baseline | Target | Method |
|---|---|---|---|---|
| tok/s (generation) | Single-turn, 512 token response | 36.1 tok/s (IQ3_XXS) | >= 36.1 tok/s (parity or better) | Median of 10 runs |
| tok/s (prompt processing) | 4096 token prompt, 1 token response | Measured | >= Python baseline | Median of 10 runs |
| Time to first token | Single-turn, cold model | Measured | <= Python baseline | Median of 10 runs |
| Time to first token | Single-turn, warm model (cached prefix) | Measured | <= Python baseline | Median of 10 runs |

#### 6.2 Engine Overhead

| Metric | Test | Target | Method |
|---|---|---|---|
| Prompt assembly latency | System prompt + tools + 50 messages | < 1 ms | 1000 iterations, measure p50/p95/p99 |
| Tool dispatch latency | Tool call parse + dispatch + result formatting | < 0.5 ms (excluding tool execution) | 1000 iterations |
| Config load time | Full config with 11 identities | < 10 ms | 100 iterations |
| Plugin load time | Load all 6 built-in MCP `.so` files | < 50 ms total | 100 iterations |
| JSON serialization overhead | 100-message context across `.so` boundary | < 0.1 ms per crossing | 10000 iterations |
| Context compaction time | Compact 8K token context to 4K | Measured (establish baseline) | 100 iterations |

#### 6.3 Memory Profile

| Metric | Test | Target | Method |
|---|---|---|---|
| Engine RSS (no model) | Create engine, load config, register tools | < 50 MB | `/proc/self/status` VmRSS |
| Engine RSS (model loaded, idle) | Load IQ3_XXS model, no generation | Measured (establish baseline) | Same |
| RSS per 1000 context messages | Generate 1000 turns, measure growth | < 100 KB/message average | RSS delta / message count |
| Peak VRAM | Full generation at max context | <= Python baseline | `nvidia-smi` sampling |

#### 6.4 Benchmark Reporting

Results are machine-readable JSONL files checked into `benchmark/results/cpp/`:

```json
{"version": "1.9.14", "test": "tok_s_generation", "value": 38.2, "unit": "tok/s", "runs": 10, "p50": 38.2, "p95": 37.1, "gpu": "RTX 5090", "model": "Qwen3.5-35B-A3B-UD-IQ3_XXS", "timestamp": "2026-03-18T12:00:00Z"}
```

A comparison script reads both `benchmark/results/` (Python) and
`benchmark/results/cpp/` (C++) to produce a side-by-side report.
