## Scope

Host-memory KV cache save/load layer within `librentropic-inference`. After
v1.8.3, the engine can cache processed system prompt prefixes in host RAM and
restore them on tier swap, eliminating the most expensive component of
first-token latency on role change.

This addresses roadmap task #51 (KV cache prefix injection) via llama.cpp's
native `llama_state_seq_get_data` / `llama_state_seq_set_data` API.

**What this is:** A caching layer that saves the KV cache state AFTER
processing the system prompt prefix (identity + constitution + tool defs).
On subsequent activations of the same tier, the saved state is restored
instead of re-processing the prompt tokens.

**What this is NOT:** The `-cram` server flag. This is not server-side
caching. This is a host-RAM caching layer managed by the inference backend
directly.

No new `.so` boundaries. No new C interface functions. This lives entirely
within `librentropic-inference` as an internal implementation detail of
`LlamaCppBackend`.

## Conceptual Flow

```
Tier "eng" activated (first time):
  ┌─────────────────────────────────────────────────────────┐
  │ 1. Tokenize system prompt (identity + constitution +    │
  │    tool definitions)                                    │
  │ 2. Process tokens through model (llama_decode)          │
  │ 3. KV cache now contains prompt prefix entries          │
  │ 4. Save KV state → PromptCache, keyed by content hash  │
  │ 5. Continue with user message processing                │
  └─────────────────────────────────────────────────────────┘

Tier swap to "qa":
  ┌─────────────────────────────────────────────────────────┐
  │ 1. Clear KV cache (llama_kv_cache_clear)                │
  │ 2. Cache miss for "qa" hash → process qa system prompt  │
  │ 3. Save KV state → PromptCache, keyed by qa hash       │
  │ 4. Continue with user message                           │
  └─────────────────────────────────────────────────────────┘

Tier swap back to "eng":
  ┌─────────────────────────────────────────────────────────┐
  │ 1. Clear KV cache                                       │
  │ 2. Cache HIT for "eng" hash → restore saved KV state    │
  │ 3. Skip prompt processing entirely                      │
  │ 4. Continue with user message                           │
  └─────────────────────────────────────────────────────────┘
```

**Expected latency improvement:** System prompts for identity tiers are
typically 1000-3000 tokens. At ~15 tok/s prompt processing, that is 65-200ms
of processing eliminated per cached restore. The save/restore is a memcpy
operation — sub-millisecond for typical cache sizes (10-50 MB per entry).

## Design Decisions

### 1. Cache Key: XXH3 hash of (prompt_text + model_path)

The cache key is `SHA256(system_prompt_content || "\x00" || model_path)`.

- **prompt_text**: The fully assembled system prompt string (identity +
  constitution + tool definitions). Content changes → different hash → cache
  miss → automatic invalidation.
- **model_path**: Different models produce different KV representations for
  identical prompts. A model swap MUST invalidate all cached entries.
- **NOT included**: Generation params (temperature, top_k, etc.). These do
  not affect the KV cache state produced by prompt processing — they only
  affect sampling during generation. Including them would unnecessarily
  fragment the cache.
- **NOT included**: KV cache quantization type (`cache_type_k/v`). These are
  model-load-time settings in `ModelConfig`. If they change, the model is
  reloaded, which clears the entire `PromptCache` (see invalidation rules).

The null byte separator prevents collisions between prompts that share a
common prefix with the model path suffix.

### 2. Cache Storage: `PromptCache` Class

```cpp
/// @file src/inference/prompt_cache.h
/// @brief Host-memory KV cache state storage with LRU eviction.
///
/// Internal to librentropic-inference. Does not cross .so boundaries.
/// Thread-safe: all public methods acquire the internal mutex.

class PromptCache {
public:
    /// @brief Construct with maximum RAM budget.
    /// @param max_bytes Maximum total bytes across all cached entries.
    ///        0 = caching disabled.
    explicit PromptCache(size_t max_bytes);

    /// @brief Store a KV cache snapshot.
    /// @param key SHA-256 hash of prompt content + model path.
    /// @param data Raw KV cache bytes from llama_state_seq_get_data.
    /// @param token_count Number of prompt tokens this entry covers.
    /// @return true if stored (may evict older entries), false if entry
    ///         exceeds max_bytes entirely and cannot be stored.
    bool store(const CacheKey& key,
               std::vector<uint8_t>&& data,
               int token_count);

    /// @brief Retrieve a cached KV snapshot.
    /// @param key SHA-256 hash to look up.
    /// @return Pointer to cached entry if found, nullptr on miss.
    ///         Pointer valid until next store() or clear() call.
    ///         Updates LRU ordering on hit.
    const CacheEntry* lookup(const CacheKey& key);

    /// @brief Evict all entries. Called on model reload.
    void clear();

    /// @brief Current total bytes consumed by cached entries.
    size_t bytes_used() const;

    /// @brief Number of cached entries.
    size_t entry_count() const;

    /// @brief Cache hit/miss statistics.
    CacheStats stats() const;
};
```

**Internal types:**

```cpp
/// @brief 64-bit hash used as cache lookup key.
/// @version 1
///
/// XXH3 hash of (prompt_text + '\0' + model_path). Not a security hash —
/// this is a cache key where speed matters and collision across <20 entries
/// is negligible.
struct CacheKey {
    uint64_t hash;

    bool operator==(const CacheKey& other) const { return hash == other.hash; }
};

/// @brief Hash function for CacheKey in unordered containers.
struct CacheKeyHash {
    size_t operator()(const CacheKey& key) const;
};

/// @brief Single cached KV state snapshot.
struct CacheEntry {
    std::vector<uint8_t> data;   ///< Raw KV cache bytes
    int token_count;             ///< Prompt tokens covered
    size_t data_size;            ///< data.size() for quick accounting
};

/// @brief Cumulative cache performance counters.
struct CacheStats {
    uint64_t hits = 0;          ///< Successful lookups
    uint64_t misses = 0;        ///< Failed lookups
    uint64_t evictions = 0;     ///< LRU evictions
    uint64_t stores = 0;        ///< Successful stores
    size_t peak_bytes = 0;      ///< High-water mark
};
```

**Storage internals:** `std::unordered_map<CacheKey, CacheEntry, CacheKeyHash>`
for O(1) lookup, paired with a `std::list<CacheKey>` for LRU ordering. The
list front is most-recently-used; eviction pops from the back.

### 3. RAM Limit and Eviction

- Configurable via `PromptCacheConfig::max_bytes` in the config hierarchy.
- Default: 512 MB. Sufficient for ~10-50 tier prefixes depending on model
  context length and KV quantization.
- Eviction policy: LRU. When a `store()` would exceed `max_bytes`, evict
  least-recently-used entries until space is available.
- If a single entry exceeds `max_bytes`, it is not cached (logged at WARNING).
- Setting `max_bytes = 0` disables caching entirely. The backend skips
  save/restore calls and operates as if v1.8.2.

### 4. Integration Point: LlamaCppBackend

The cache integrates at two points in `LlamaCppBackend`:

**Save point** — after system prompt processing, before user message decoding:

```
LlamaCppBackend::do_generate()
  └── apply_chat_template()          // Assemble full prompt
  └── tokenize()                     // Full token sequence
  └── identify_system_prefix()       // Find system prompt token boundary
  └── IF cache.lookup(key) HIT:
  │     └── llama_kv_cache_clear()
  │     └── llama_state_seq_set_data()  // Restore cached prefix
  │     └── llama_decode() remaining tokens only (user msg onward)
  └── ELSE (cache MISS):
        └── llama_kv_cache_clear()
        └── llama_decode() full token sequence
        └── cache.store(key, snapshot)  // Save prefix state
```

**System prompt boundary detection:** The chat adapter already knows where
the system message ends (it assembles the prompt). The token count of the
system prefix is computed by tokenizing the system portion separately. This
boundary is the save/restore point.

**Restore flow in detail:**

1. Compute `CacheKey` from the system prompt text + model path.
2. `lookup(key)` — if hit, restore the KV state for `seq_id=0` using
   `llama_state_seq_set_data`. Then decode only the remaining tokens
   (user message, conversation history) starting from the cached
   `token_count` position.
3. If miss, decode the full sequence. After decoding the system prefix
   tokens, snapshot the KV state via `llama_state_seq_get_data` and
   `store()` it.

### 5. Cache Invalidation

| Trigger | Action | Reason |
|---------|--------|--------|
| Prompt content changes | Automatic — different hash → miss | Content-addressed by design |
| Model reload (`unload` + `load`) | `cache.clear()` | KV representations are model-specific |
| Config change to `cache_type_k/v` | Triggers model reload → `clear()` | Different quantization = different KV bytes |
| Manual `cache.clear()` via config | Immediate eviction | Operator control |
| RAM pressure (LRU eviction) | Oldest entries evicted | Bounded memory |

No timer-based expiry. Cache entries are valid as long as the model is
loaded and the prompt content has not changed. Content-addressing handles
invalidation naturally.

### 6. Thread Safety

`PromptCache` uses a single `std::mutex` guarding all mutations (store, evict,
clear) and lookups. This is sufficient because:

- Cache access is infrequent — once per generate call at most, and only for
  the system prefix check. This is not a hot path.
- The expensive operations (llama_decode, llama_state_seq_get/set_data) happen
  OUTSIDE the cache mutex. The mutex only guards the hashmap lookup and the
  byte vector copy/move.
- Lock contention would only occur with concurrent generate calls to the same
  backend, which is already serialized by the inference backend's design
  (single model context per backend instance).

A lock-free design adds complexity for zero benefit here.

## Config Integration

New config struct consumed by `LlamaCppBackend`, loaded by the config
layer from v1.8.1:

```cpp
/// @brief Prompt caching configuration.
/// @par YAML key: inference.prompt_cache
struct PromptCacheConfig {
    size_t max_bytes = 536870912;  ///< 512 MB default
    bool enabled = true;           ///< Master switch (false = no caching)
    bool log_hits = true;          ///< Log cache hit/miss at INFO level
};
```

YAML example:

```yaml
inference:
  prompt_cache:
    enabled: true
    max_bytes: 536870912   # 512 MB
    log_hits: true
```

Setting `enabled: false` or `max_bytes: 0` both disable caching. The
`enabled` flag exists for clarity in config files.

## Validation Criteria

### Functional

1. **Cache hit eliminates prompt processing:** Given a tier activated twice
   with the same system prompt, the second activation restores from cache
   and does NOT call `llama_decode` for the system prefix tokens.

2. **Cache miss processes normally:** Given a tier activated for the first
   time, the full prompt is processed and the KV state is saved to cache.

3. **Content-addressed invalidation:** Given a tier whose identity prompt
   changes between activations, the cache key differs and the old entry is
   not restored.

4. **Model reload clears cache:** Given a model unload + reload cycle, all
   cached entries are evicted.

5. **LRU eviction under pressure:** Given a cache with `max_bytes = N` and
   entries totaling > N, the least-recently-used entries are evicted to make
   room.

6. **Single entry exceeding limit:** Given `max_bytes = 1MB` and a prompt
   producing 2MB of KV state, the entry is not cached and a WARNING is logged.

7. **Disabled caching:** Given `enabled: false`, no save or restore calls
   occur. Behavior matches v1.8.2 exactly.

8. **Generated output identical:** Given the same prompt and generation
   params, output produced via cache restore must be byte-identical to
   output produced via full prompt processing (deterministic sampling assumed).

### Performance

9. **Restore faster than reprocess:** Cache restore completes in < 5ms for
   a typical 2000-token system prompt. Full processing of the same prompt
   takes 65-200ms.

10. **Memory accounting accurate:** `bytes_used()` matches the sum of
    `data.size()` across all cached entries at all times.

### Thread Safety

11. **Concurrent access safe:** Multiple threads calling `lookup()` and
    `store()` simultaneously do not produce data races (verified with TSAN).

| Dependency | Version | What it provides |
|------------|---------|------------------|
| v1.8.2 | `InferenceBackend`, `LlamaCppBackend` | Base class, lifecycle, `do_generate()` integration point |
| v1.8.1 | Config loader, `PromptManager` | `PromptCacheConfig` parsing, system prompt content for hashing |
| v1.8.0 | Types, spdlog, test framework | `ModelConfig`, logging, unit test infrastructure |
| llama.cpp | Submodule | `llama_state_seq_get_size`, `llama_state_seq_get_data`, `llama_state_seq_set_data`, `llama_kv_cache_clear` |
| xxHash (XXH3) or similar | Header-only | Hash for cache key computation |

**Hash dependency note:** SHA-256 is overkill for a cache key — this is not
a security context. XXH3 (xxHash, header-only, BSD-2) is preferred: 64-bit
hash, extremely fast, collision probability negligible across <20 cached
entries. Alternatives:
- **XXH3** (header-only, 1 file, BSD-2) — preferred for speed and simplicity
- **std::hash** — no dependency but implementation-defined, not portable
- **llama.cpp's internal GGML hash** — investigate if usable, avoids new dep

The `CacheKey` type changes from `std::array<uint8_t, 32>` to `uint64_t`
with XXH3. The interface remains: `CacheKey compute_cache_key(string_view prompt, string_view model_path)`.

## Implementation Notes (from review)

1. **Token boundary precision.** The cache save/restore point must be at an
   exact token offset, not a character offset. The adapter knows the character
   boundary of the system prompt; the tokenizer maps that to tokens. The
   implementing session must verify the token count matches between save and
   restore — if the tokenizer splits a multi-byte character at the boundary
   differently than expected, the save/restore point could be off by a token.
   **Mitigation:** After tokenizing the system prompt, record the exact token
   count in the cache entry. On restore, verify the stored token count matches
   the current tokenization. If mismatched, treat as cache miss and reprocess.
   Test with non-ASCII identity prompts.
