## Summary

Design and ship the `entropic-engine` Python package for v2.x. This
replaces the v1.7.1 Python-only engine (still on PyPI) with a thin
wrapper around the native C library.

v2.0.5 ships the C/C++ distribution only (tarballs + install.sh).
This proposal covers how Python consumers install and use the v2.x
engine, including the architectural decisions that v2.0.5 deferred.

## Non-goals (locked)

- **No per-environment PyPI project split.** One project name:
  `entropic-engine`. Not `entropic-engine-cpu`, not
  `entropic-engine-cuda`. Ever.
- **Python is a PURE wrapper.** The `entropic` CLI entry point
  dispatches to the bundled native binary. No click reimplementation
  of subcommands that exist in C.
- **ctypes bindings (auto-generated from entropic.h) stay.** These
  are the programmatic API for `from entropic import EntropicEngine`.

## Architecture decision matrix

### Option A: Fat wheel (single wheel, CUDA + CPU bundled)

Wheel contains both `librentropic-cuda.so` (~400 MB) and
`librentropic-cpu.so` (~4 MB). Python wrapper detects CUDA at import
time and loads the right one.

| Pro | Con |
|---|---|
| Single `pip install` for all users | ~400 MB download per install, including CPU-only |
| No infra beyond PyPI | PyPI 100 MB per-file limit → exemption request required |
| Simple CI | 5 GB/year PyPI storage at 12 releases/year |

**Risk:** PyPI exemption request turnaround is days-to-weeks. Blocks
the first release.

### Option B: Index-URL pattern (PyTorch-style)

Default `pip install entropic-engine` → CPU wheel from PyPI (~4 MB).
CUDA variant served from a self-hosted index (GitHub Pages, S3, or
a GitHub Release-based simple index):

```
pip install entropic-engine --index-url https://tvanfossen.github.io/entropic/whl/cu128/
```

| Pro | Con |
|---|---|
| Small default wheel, no PyPI exemption | Self-hosted index to maintain |
| CPU users unaffected | CUDA users need to know the `--index-url` flag |
| Industry standard (PyTorch, JAX) | Index hosting: ~$0 (GitHub Pages), but operational surface |

### Option C: On-demand .so download (first-import fetch)

Small wheel (~4 MB) with ctypes wrapper only. First `import entropic`
detects hardware and downloads the matching `librentropic.so` from the
GitHub Release into a local cache (`~/.entropic/lib/`).

| Pro | Con |
|---|---|
| Tiny wheel, no PyPI exemption | Offline install breaks |
| Auto-adapts to hardware changes | First-import latency (download + cache) |
| No self-hosted index | Network dependency at runtime |

### Option D: Thin wheel + explicit download step

`pip install entropic-engine` ships ctypes + CLI shim only. User runs
`entropic download-native` (or `entropic setup`) to fetch the right
`.so` into `~/.entropic/lib/`. Explicit, user-driven.

| Pro | Con |
|---|---|
| Tiny wheel, clear contract | Two-step install (pip + download) |
| Works offline after download | More user friction than "just pip install" |
| No hidden network access | Needs CLI support in native binary |

## Recommendation

**Option B (index-URL)** for v2.1.1. It's the industry standard for
GPU Python libraries, keeps the default experience lightweight, and
doesn't require a PyPI exemption. The self-hosted index is a one-time
GitHub Pages setup (~30 min) that we own forever.

If users push back on the `--index-url` flag (UX friction), revisit
Option D as a simpler alternative.

## CLI surface

### Native binary (canonical)

The v2.0.5 native `entropic` binary ships with: `mcp-bridge`,
`download`, `version`. Future additions are driven by consumer
demand.

### Python entry point (thin shim)

`python/entropic/cli.py` becomes:

```python
import os, sys
from pathlib import Path

def main():
    native = Path(__file__).parent / "bin" / "entropic"
    os.execv(str(native), [str(native)] + sys.argv[1:])
```

`[project.scripts] entropic = entropic.cli:main` in pyproject.toml
makes `entropic` available on PATH via pip. Delegates 100% to the
native binary. Zero click code.

### `cli_download.py`

Deleted. `entropic download` in the native binary is canonical.
`httpx` dependency dropped.

## CUDA runtime resolution

The wheel's bundled `librentropic-cuda.so` dynamically links to
`libcudart.so.12` + `libcublas.so.12`. Options for the consumer:

1. **Host CUDA 12.8+ installed** (apt / conda / runfile). Documented
   as a prerequisite, same as the tarball path.
2. **Declare pip dependency on `nvidia-cuda-runtime-cu12`** (~200 MB
   pip package from NVIDIA that ships the runtime libs). Auto-resolves
   for pip users; adds to install size.
3. **Bundle CUDA runtime in the wheel.** Adds ~800 MB. Not
   recommended — PyTorch does this but their scale justifies the
   tradeoff; ours doesn't.

Recommend option 1 initially, with option 2 as an opt-in extra:
`pip install entropic-engine[cuda-runtime]`.

## Migration from v1.7.1

`entropic-engine` v1.7.1 is a pure-Python engine on PyPI. The v2.x
package is fundamentally different (C library wrapper). When v2.1.1
ships:

- v1.7.1 stays on PyPI (never yanked; some users may depend on it).
- v2.1.1 is published as a higher version → auto-upgrade for users
  with `>=` version pins.
- **Breaking changes** from v1.7.1 → v2.1.1:
  - Python engine code deleted; native library required.
  - CLI surface changes (click → native binary shim).
  - Config format may differ (YAML structure changes).
  - Model path convention changed (`~/models/gguf/` → `~/.entropic/models/`).
- **Mitigation**: major version bump + prominent release notes +
  `pip install entropic-engine<2` for users needing to pin to v1.x.

## Implementation plan

1. **Architecture decision** — confirm Option B (or alternative).
2. **GitHub Pages index setup** — `tvanfossen.github.io/entropic/whl/`.
3. **`cli.py` → execv shim**, delete `cli_download.py`.
4. **`pack_wheel.py` update** — bundle native binary + correct `.so`.
5. **release.yaml** — add wheel-build + index-publish jobs.
6. **External smoke (smoke-test-v3)** — `pip install` from self-hosted
   index, verify `entropic version` + `from entropic import *`.
7. **Migration notes** — changelog + README warning about v1.x → v2.x.

## Validation

- `pip install entropic-engine` on a fresh machine: CPU wheel, ctypes
  wrapper loads, `entropic version` prints 2.1.1.
- `pip install entropic-engine --index-url <cuda-url>`: CUDA wheel,
  GPU inference at ~30 tok/s on Blackwell.
- v1.7.1 users with `entropic-engine<2` pin: unaffected.
