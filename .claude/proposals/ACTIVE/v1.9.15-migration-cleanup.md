## Scope

v1.9.15 is the point of no return. The deprecated Python engine is deleted,
the PyPI package is restructured around the C library, and the examples are
rewritten against the C API. After this version, `entropic-engine` on PyPI
IS `librentropic.so` + the auto-generated ctypes wrapper + a thin CLI.

**What ships:**

1. **Python engine deleted** — the entire `src/entropic/` tree is removed.
   Every module (inference, core, config, mcp, prompts, storage, ui, voice,
   lsp, quality, benchmark, data) is gone. The v1.9.0 deprecation window is
   over.

2. **PyPI package restructured** — `entropic-engine` now ships:
   - `librentropic.so` (pre-built CUDA wheel, or source build)
   - Auto-generated ctypes wrapper (`entropic/` Python package)
   - CLI entry point (`entropic` command) using the wrapper
   - Bundled data files (prompts, grammars, tool schemas, `bundled_models.yaml`)

3. **CLI fully on C engine** — no more fallback-to-Python detection logic from
   v1.9.0. The CLI imports from the wrapper unconditionally. If `librentropic.so`
   is missing, it fails with a clear error.

4. **Examples rewritten** — `hello-world/` and `pychess/` use the C API via
   the wrapper. No imports from the old `entropic` Python package.

5. **README rewritten** — reflects 2.0.0 architecture: C library + Python
   wrapper + CLI. Installation instructions updated for the new package layout.

**What this version does NOT include:**
- New C engine features (engine is frozen post-v1.9.14 hardening)
- Test framework changes (that's v1.10.0)
- License change (that's v2.0.0)
- TUI port (post-2.0.0, separate repo)

**Why this is irreversible:**
The Python engine is the only implementation of several subsystems that
have been faithfully ported to C++ across v1.8.0–v1.8.9. Deleting it means
the C++ engine is now the sole source of truth. If a behavioral regression
surfaces post-deletion, the fix goes into the C++ engine — there is no
Python fallback to revert to. v1.7.1 on PyPI remains as a tagged, known-working
snapshot for consumers who cannot migrate yet.

### CLI Restructure

The CLI currently lives in `src/entropic/cli.py` with a backend selection
pattern (try C engine, fall back to Python). After v1.9.15, the fallback
is removed:

```
BEFORE (v1.9.0–v1.9.14):
  cli.py → try import entropic_c → fall back to import entropic

AFTER (v1.9.15):
  cli.py → import entropic (the wrapper) → librentropic.so
```

Commands and their disposition:

| Command | Current Location | v1.9.15 Implementation |
|---|---|---|
| `ask` | `cli.py` → `AgentEngine` | `cli.py` → `EntropicEngine.run()` via wrapper |
| `status` | `cli.py` → `ConfigLoader` | `cli.py` → `EntropicEngine` state queries |
| `init` | `cli.py` → `ConfigLoader` | `cli.py` → config file generation (pure Python, data/ templates) |
| `download` | `cli_download.py` | `cli_download.py` — unchanged (pure Python HTTP) |
| `setup-cuda` | `cli_setup_cuda.py` | `cli_setup_cuda.py` — unchanged (pure Python system config) |
| `benchmark` | `benchmark/runner.py` | `cli.py` → `EntropicEngine` C benchmark (ported in v1.9.0) |
| `mcp-bridge` | `cli.py` → `ServerManager` | `cli.py` → `entropic_mcp_server_*` via wrapper |
| (headless) | `cli.py` → `AgentEngine` | `cli.py` → `EntropicEngine.run()` via wrapper |

### Bundled Data Migration

`src/entropic/data/` contains assets consumed by both the C engine and the
CLI. After deletion of `src/entropic/`, these assets must be discoverable:

| Asset | Consumer | Discovery |
|---|---|---|
| `bundled_models.yaml` | CLI `download`, C engine model resolution | `python/entropic/data/` (PyPI) or `DATA_DIR` (CMake) |
| `prompts/*.md` | C engine prompt assembly | `DATA_DIR` compile-time define or config override |
| `grammars/*.gbnf` | C engine grammar loading | Same |
| `tools/**/*.json` | C engine tool schema loading | Same |

The C engine already has compile-time `DATA_DIR` discovery (v1.8.1). The
PyPI wheel bundles assets in `python/entropic/data/` and the wrapper passes
the data directory path to the C engine at configure time:

```python
class EntropicEngine:
    def __init__(self, config_path=None, data_dir=None):
        if data_dir is None:
            data_dir = str(Path(__file__).parent / "data")
        # Pass to C engine so it finds bundled prompts/grammars/tools
        _lib.entropic_set_data_dir(self._handle, data_dir.encode())
```

### Phase 3: Examples Rewrite

Rewrite `examples/hello-world/` and `examples/pychess/` to use the C API
via the wrapper.

#### hello-world

Current imports:
```python
from entropic import (
    AgentEngine, ConfigLoader, EngineCallbacks,
    LoopConfig, ModelOrchestrator, setup_logging, setup_model_logger,
)
```

Rewritten to:
```python
from entropic import EntropicEngine
```

The current example does:
1. `ConfigLoader` → load config YAML
2. `setup_logging` / `setup_model_logger` → configure Python logging
3. `ModelOrchestrator` → load model into VRAM
4. `AgentEngine` → create engine with loop config
5. `EngineCallbacks` → wire streaming output
6. `engine.run(prompt)` → async generator

The rewritten example does:
1. `EntropicEngine(config_path=...)` → C engine handles config + model loading
2. `engine.run_streaming(prompt, on_token=...)` → synchronous with callback
3. No async — the C engine's agentic loop is synchronous from the caller

```python
"""Hello World — minimal entropic integration.

Demonstrates:
    - Single-tier config with bundled lead identity
    - Streaming output via callback
    - App context injection
"""
from pathlib import Path
from entropic import EntropicEngine

EXAMPLE_ROOT = Path(__file__).resolve().parent

def main() -> None:
    config_path = EXAMPLE_ROOT / "config.yaml"
    engine = EntropicEngine(config_path=str(config_path))

    print("entropic hello-world (C engine)")
    print("Type 'quit' to exit.\n")

    while True:
        try:
            prompt = input("You: ").strip()
        except EOFError:
            break
        if not prompt or prompt.lower() in ("quit", "exit", "q"):
            break

        engine.run_streaming(
            prompt,
            on_token=lambda tok: print(tok, end="", flush=True),
        )
        print("\n")

    del engine
    print("Bye!")

if __name__ == "__main__":
    try:
        main()
    except KeyboardInterrupt:
        print("\nInterrupted.")
```

Key differences:
- No `asyncio` — C engine is synchronous
- No separate `ConfigLoader`, `ModelOrchestrator`, `EngineCallbacks` — single
  `EntropicEngine` handles everything
- No `LoopConfig` Python object — loop config is in the YAML or passed as JSON
- Streaming via callback parameter, not async generator

#### pychess

The pychess example is more complex: it registers a custom MCP server
(`ChessServer`) and uses two-tier routing (thinker → executor). The
rewrite must demonstrate C engine plugin loading.

Current flow:
1. `load_config()` → custom config loader
2. `ModelOrchestrator` → model loading
3. `ChessServer()` → Python MCP server instance
4. `ServerManager.register_server(chess_server)` → register before init
5. `AgentEngine` → engine with custom server manager
6. `engine.run(prompt, system_prompt=board_context)` → async generator

Rewritten flow:
1. `EntropicEngine(config_path=...)` → C engine
2. Chess MCP server must be a C plugin `.so` OR use the external MCP
   protocol (stdio/SSE from v1.8.7) to register as an external server
3. `engine.run_streaming(prompt)` → synchronous with callback

**Decision point:** The `ChessServer` is currently a Python class inheriting
from `BaseMCPServer`. In v1.9.15, it cannot import from the deleted Python
engine. Two options:

**Option A: External MCP server (stdio)**
- `chess_server.py` becomes a standalone MCP server using the `mcp` Python
  package (stdio transport)
- Registered in `.mcp.json` alongside the pychess config
- Pro: no C compilation needed for the example
- Con: adds `mcp` as a dependency for the example (not the engine)

**Option B: C plugin `.so`**
- Rewrite `ChessServer` in C as a `libchess_server.so` plugin
- Loaded via `entropic_load_mcp_plugin()`
- Pro: demonstrates the plugin architecture
- Con: requires C compilation, heavier example

**Recommendation: Option A.** The pychess example's purpose is demonstrating
engine integration, not C plugin development. An external MCP server over
stdio is the path of least resistance and exercises the external MCP
support from v1.8.7. A separate C plugin example can be added later.

## Risk Analysis

### Risk 1: Undiscovered Python Engine Imports

**Threat:** A file outside `src/entropic/` imports from the Python engine
and breaks silently.

**Mitigation:** Phase 1 inventory includes a repo-wide grep for all
`entropic` imports. Phase 8 repeats the grep after deletion.

### Risk 2: CLI Command Behavior Regression

**Threat:** CLI commands behave differently through the wrapper than they
did through the Python engine.

**Mitigation:** The v1.9.0 migration already ported CLI commands to the C
engine. v1.9.15 removes the fallback path — the commands themselves don't
change behavior, only the import path changes. Model tests (ported in
v1.9.0) validate end-to-end behavior.

### Risk 3: Bundled Data Discovery Failure

**Threat:** After moving `data/` from `src/entropic/data/` to
`python/entropic/data/`, the C engine can't find prompts/grammars/tools.

**Mitigation:** The wrapper passes the data directory to the C engine
explicitly. CMake installs use `DATA_DIR` compile-time define. Both paths
are tested in existing integration tests.

### Risk 4: PyPI Wheel Size

**Threat:** Bundling `librentropic.so` (which statically links llama.cpp
with CUDA support) produces a very large wheel.

**Mitigation:** This is a known tradeoff documented in v2.0.0 release plan.
The CUDA wheel is large (~500MB+) but this is comparable to PyTorch wheels.
Source builds (`pip install --no-binary`) produce platform-native builds.
Non-CUDA (CPU-only) wheels are much smaller.

### Risk 5: pychess Example Complexity

**Threat:** Converting pychess from an in-process MCP server to an external
stdio server adds complexity that obscures the example's purpose.

**Mitigation:** The pychess port should be MINIMAL — import path changes
only, not a full rewrite. A full rewrite to demonstrate C plugin architecture
is a post-2.0.0 concern. For v1.9.15, the goal is: examples work against
the C engine via the Python wrapper. The chess server becomes a standalone
external MCP server (Option A) with minimal structural changes.
