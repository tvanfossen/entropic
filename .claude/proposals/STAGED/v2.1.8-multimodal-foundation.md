# v2.1.8 — Multimodal foundation

## Session prompt

This is the first patch of the 2.2.0 release plan. The goal is to make vision/multimodal a real, demonstrable feature end-to-end — not just an inference-layer capability that the facade discards.

Today: `entropic_inference_generate` parses `content_parts` correctly (v1.9.11 shipped this), but `entropic_run` accepts only a flat `const char*` and structurally drops images before they reach the backend. This patch closes that gap, adds the routing logic to pick a vision-capable tier when image parts arrive, ships a bundled vision-language model so the path is testable out of the box, and demonstrates it via the OpenAI server example. A small ride-along (`entropic_context_usage`) ships in the same patch since it's trivial.

After this patch:
- Consumers can call `entropic_run_messages(handle, messages_json, ...)` with text + image content parts and get a coherent response.
- The OpenAI server example accepts `data:` URL images via `/v1/chat/completions`.
- Multi-tier configs route image-bearing messages to a vision-capable tier automatically.
- A bundled VLM (Qwen2.5-VL-7B-Instruct, or a Qwen3.x-VL successor if released) provides the default demo path.

## Design decisions already made (do not re-litigate)

These were captured in the v2.2.0 planning session and must not be re-opened during implementation:

- **#37 — two parallel ABI paths.** `entropic_run` is preserved unchanged. `entropic_run_messages` is added alongside. No shim, no JSON ser/parse on the text-only fast path. Rationale: avoid perf regression on text-only consumers; keep ABI backwards-compatible (strictly additive — 2.1.8 is a patch, not a major).
- **#38 — image sources.** v1 supports `data:` URLs (base64), `file://`, and absolute paths. `http(s)://` URLs return HTTP 400 with a clear SSRF-related error. Adding HTTP fetch is easy in a future patch if needed.
- **#41 — tier-lock + image collision policy.** If a consumer locks to a text-only tier and then sends an image, the engine returns `ENTROPIC_ERROR_NO_VISION_TIER` cleanly. It does NOT auto-switch tiers (would violate the lock contract) and it does NOT silently drop image parts (would hide a bug). Consumers that want auto-routing don't lock.
- **#41 — capability defaults.** Missing `capabilities` field on a tier defaults to `[text]`. This preserves back-compat for all existing tier configs.
- **#42 — VLM only.** Image-generation / image-edit models (Qwen-Image, Qwen-Image-Edit) are NOT in scope; they ship a different ABI and are tracked in gh#43.
- **#39 — two-scalar API.** `entropic_context_usage(handle, size_t* used, size_t* capacity)`. Not a JSON struct. Matches the cost profile of `entropic_context_count`. Per-tier breakdown deferred to a future minor.

## Scope

### C ABI additions (strictly additive)

```c
// Facade
ENTROPIC_EXPORT entropic_error_t entropic_run_messages(
    entropic_handle_t handle,
    const char* messages_json,
    entropic_run_result_t* out_result);

ENTROPIC_EXPORT entropic_error_t entropic_run_messages_streaming(
    entropic_handle_t handle,
    const char* messages_json,
    entropic_token_callback_t cb,
    void* user_data,
    entropic_run_result_t* out_result);

// Context pressure
ENTROPIC_EXPORT entropic_error_t entropic_context_usage(
    entropic_handle_t handle,
    size_t* tokens_used,
    size_t* capacity);

// New error code
ENTROPIC_ERROR_NO_VISION_TIER
```

### Engine plumbing

- `AgentEngine::run_turn(std::vector<Message>)` overload preserving `content_parts` through `ContextManager`, hooks, and tool loop.
- Shared messages-JSON parser extracted from `src/inference/inference_c_api.cpp:125-167` into a utility usable by both the inference path and the facade.
- `ContextManager` serialization audited for string-only assumptions; either preserves `content_parts` through compaction, or documents that image parts are dropped during compaction (with rationale).

### Orchestrator routing

- Tier config schema adds `capabilities: [text, vision, ...]` field (defaults to `[text]`).
- Facade inspects incoming `content_parts`. If any `type: image`, requires a tier with `vision` capability.
- Routing logic in `src/inference/orchestrator.cpp` selects a vision-capable tier when image parts present, returns `ENTROPIC_ERROR_NO_VISION_TIER` if none configured.

### Bundled vision tier

- Add `qwen2_5_vl_7b` (or successor) entry to `data/bundled_models.yaml` with `path:`, `mmproj:`, `capabilities: [text, vision]`, and matching `adapter:` key (may need a dedicated `qwen2_5_vl` adapter — confirm at implementation time, do not assume Qwen35Adapter reuse).
- Fetch tooling downloads base GGUF + mmproj as a pair.
- License + linking-exception compatibility documented in registry comments.

### OpenAI server example

- Rewrite `examples/openai-server/src/main.cpp::flatten_messages` (L76) → new parser walks the OpenAI content-parts array and emits canonical messages-JSON.
- Base64 decoder for `data:image/...;base64,...` payloads; tempfile written under a per-request tempdir, deleted on success / error / streaming-completion.
- `file://` and absolute paths pass through as `path`.
- `http(s)://` URLs return HTTP 400.
- Vision-tier presence check before dispatch; 400 with actionable guidance if image arrives without a vision-capable tier active.

### Tests

- Unit: messages-JSON parser (text-only, text+image, malformed); routing decision (text → any tier, image → vision tier only).
- Integration: two-tier config (text + vision), end-to-end text+image turn through `entropic_run_messages` returns coherent caption.
- Model test: vision benchmark fixture under `tests/fixtures/images/` with known-content image; vision tier loads + first-token smoke + caption fidelity check.
- OpenAI server test: `data:` URL request → 200 with caption; absent-vision-tier → 400.

## Out of scope (deferred to other patches)

- Adapter expansion for Qwen3.5 siblings / Qwen3.6 / Gemma 4 / Nemotron 3 — that's v2.1.9.
- Mid-generation user message queue — v2.1.10.
- Speculative decoding + SecondaryModelLoader — v2.1.11.
- Per-tier breakdown of `entropic_context_usage` — future minor.
- Tool-result image attachments (assistant turns remain text-only).
- `http(s)://` URL fetch in the OpenAI server (SSRF surface — separate hardening).
- Image-generation / image-edit models — separately tracked in gh#43.
- mmproj hot-swap (multi-tier with mixed text/vision tiers must declare upfront).
- Per-request mmproj loading (load is one-shot at tier load).

2. `entropic_context_usage` returns matching values to what `core.context_manager` logs (`Context: N/M tokens`).
3. `ENTROPIC_ERROR_NO_VISION_TIER` returned cleanly when image arrives at a non-vision-capable tier; diagnostic message names the active tier and lists vision-capable tiers (if any).

5. End-to-end: bundled vision tier + text+image turn through `entropic_run_messages` produces a coherent caption response.
6. OpenAI server: `data:` URL request returns 200 with caption; `file://` + absolute path requests also succeed; `http(s)://` returns 400; image without vision tier returns 400 with actionable message.
7. Tempfile cleanup: confirmed on success, error, and streaming-completion paths (no leaks under repeated requests).
8. Pre-commit gates pass (knots, flake8, doxygen-guard, unit tests).
9. Doxygen on every new public symbol includes `@brief` and `@version 2.1.8`.

documents that Phases 5–7 (`LlamaCppBackend` mmproj loading,
multimodal generation, text-only fallback) and Phase 9 (compaction
content_parts awareness) were **deferred**. Confirmed by inspection
of the current code:

    them.

  Net effect today: a consumer can hand a multimodal `messages_json`
  to `entropic_inference_generate`, the parser builds correct
  `Message::content_parts`, and the backend silently drops the image
  parts in generation. v1.9.11's "graceful text-only fallback" path
  is not actually implemented either — image parts just don't reach
  any code that knows what to do with them.

**Impact on v2.1.8 validation criteria.** The facade-side work
(gh#37 ABI, gh#39 context_usage, gh#41 capabilities + routing,

completing v1.9.11 Phases 5–7 in `LlamaCppBackend`.

**libmtmd availability.** Confirmed present at
`extern/llama.cpp/tools/mtmd/{mtmd.h,mtmd-helper.h}`. Phases 5–7 are
mechanically achievable in-session — not blocked on upstream.

### 2026-05-13 — gh#42 simplification (user redirect)

**User correction.** Initial gh#42 plan assumed a separate VLM tier
(Qwen2.5-VL-7B-Instruct). User asked whether the existing primary
Qwen3.5-35B-A3B already does multimodal. Verified via the unsloth
HF page:

- `unsloth/Qwen3.5-35B-A3B-GGUF` is tagged
  **`image-text-to-text`** with a unified vision-language foundation
  (vision encoder built into the base model, not a separate VLM
  line).
- The same repo ships three mmproj options as siblings of the
  primary GGUF: `mmproj-BF16.gguf` (903 MB), `mmproj-F16.gguf`
  (899 MB), `mmproj-F32.gguf` (1.79 GB).

**Revised gh#42 scope.** No second tier. The existing `primary`
entry in `data/bundled_models.yaml` gains an `mmproj:` URL pointing
at `mmproj-F16.gguf` (smallest balanced quant) and gets
`capabilities: [text, vision]`. Fetch tooling already downloads
files by registry key — just needs to pull the mmproj alongside the
base when the field is present.

**Benefits:**
  - No additional VRAM footprint for a parallel VLM. The mmproj
    adds ~0.9 GB to the primary's ~13.1 GB footprint, matching the
    VRAM budget table v1.9.11 already documented.
  - Routing simplification (gh#41): primary tier alone carries
    `vision`, so routing-on-image-content lands on the primary
    automatically. Mixed text-only consumers still hit primary as
    today; image consumers get vision capability for free.
  - Family alignment preserved — same tokenizer, same adapter,
    same identity prompts.

**Adjusted validation criteria** (functional, not procedural):
criterion #5 (end-to-end caption) still requires v1.9.11 Phases
5–7 wired in `LlamaCppBackend` so the mmproj actually gets loaded
and exercised. That work is still in scope; only the model bundle
shape changes.
