# v2.1.9 — Model registry expansion + 3 new adapter classes

## Session prompt

This patch broadens the bundled model registry from 2 entries (Qwen3.5-A3B primary, Qwen3-0.6B router) to 11, and adds three new chat adapter classes to handle the new families. Each model gets its own registry key using a consistent `<family>_<size_or_variant>` naming convention; tier role assignment is a separate config concern, so the same key can be swapped into `primary`, `mid`, `draft`, `router`, etc.

After this patch:
- 9 new model registry entries are available for tier configs to reference.
- Three new adapter classes exist: `Qwen36Adapter`, `Gemma4Adapter`, `Nemotron3Adapter`.
- Each new adapter has unit tests verifying render + tool-call parse correctness, and at least one model test exercising real inference against a bundled GGUF in the family.
- Tier swapping is purely a config edit — no code changes needed to switch primary from Qwen3.5-A3B to Qwen3.6-A3B, or to use Gemma 4 26B-A4B as a primary alternate.

## Design decisions already made (do not re-litigate)

- **No version lumping.** Each model generation gets its own adapter class. `Qwen35Adapter` stays as-is for Qwen3.5; `Qwen36Adapter` is a new distinct class for Qwen3.6. If template logic happens to overlap, a `QwenBaseAdapter` intermediate can be extracted **as a result of implementation**, not as a precondition.
- **Naming convention.** `<family>_<size_or_variant>`, lowercase, underscores. Examples: `qwen3_5_9b`, `qwen3_6_a3b`, `gemma4_e2b`, `nemotron3_nano_4b`. Documented in a comment block at the top of `data/bundled_models.yaml`.
- **Adapter key matches family.** Config field `adapter: "qwen35"` / `"qwen36"` / `"gemma4"` / `"nemotron3"`. No cross-family aliases — keys do not silently accept other generations.
- **Nemotron has an architecture-verification gate.** Before any `Nemotron3Adapter` code is written, confirm: (a) is the model hybrid Mamba-Transformer? (b) what is the `llm_arch` tag from `gguf-dump`? (c) is the chat template Llama-derived or Nemotron-specific? (d) what is the tool-call format? (e) does the model emit reasoning traces? If hybrid Mamba and llama.cpp state handling is unstable, the adapter is deferred to STALLED — no workaround.
- **Single-GPU constraint, one at a time.** Dev machine VRAM target. Quants pick is implementer's call; document each pick in a per-entry comment in `bundled_models.yaml`.
- **License verification per family.** Apache-2.0 / Gemma Terms of Use / NVIDIA Open Model License must each be checked against engine license (LGPL-3.0-or-later) + BISSELL linking exception. Document in registry comments.

## Scope

### Registry expansion (`data/bundled_models.yaml`)

Add 9 new entries:

```yaml
qwen3_5_0_8b:    { source: unsloth/Qwen3.5-0.8B,            adapter: qwen35,   ... }
qwen3_5_2b:      { source: unsloth/Qwen3.5-2B,              adapter: qwen35,   ... }
qwen3_5_4b:      { source: unsloth/Qwen3.5-4B-GGUF,         adapter: qwen35,   ... }
qwen3_5_9b:      { source: unsloth/Qwen3.5-9B,              adapter: qwen35,   ... }
qwen3_6_a3b:     { source: unsloth/Qwen3.6-35B-A3B-GGUF,    adapter: qwen36,   ... }
gemma4_a4b:      { source: unsloth/gemma-4-26B-A4B-it-GGUF, adapter: gemma4,   ... }
gemma4_e4b:      { source: unsloth/gemma-4-E4B-it-GGUF,     adapter: gemma4,   ... }
gemma4_e2b:      { source: unsloth/gemma-4-E2B-it-GGUF,     adapter: gemma4,   ... }
nemotron3_nano_4b: { source: unsloth/NVIDIA-Nemotron-3-Nano-4B-GGUF, adapter: nemotron3, ... }
```

Each entry includes the canonical fields (`path`, `context_length`, `recommended_quant`, license, etc.). Test fixture `tests/data/bundled_models.yaml` mirrors at least one entry per family.

### Adapter implementations

For each new adapter (`Qwen36Adapter`, `Gemma4Adapter`, `Nemotron3Adapter`):

1. New `{name}_adapter.{h,cpp}` under `src/inference/adapters/`.
2. Subclass `ChatAdapter` (or a shared `Qwen…Base` / `Gemma…Base` only if overlap is real).
3. Implement: `render_messages`, `parse_tool_calls`, `format_tool_result`, `format_tools`, `clean_content`, `format_system_with_vision`, `format_content_parts`.
4. Register in `src/inference/adapters/adapter_registry.cpp` with the appropriate key.
5. Doxygen on every public function (`@brief`, `@version 2.1.9`).

`Nemotron3Adapter` only after the architecture-verification gate documented in gh#47 passes.

### Tests

- **Unit tests** per adapter under `tests/unit/inference/{name}_adapter_test.cpp`:
  - Round-trip render: system + user + assistant + tool-result multi-turn
  - Tool-call parse: well-formed, malformed, partial / interrupted streams
  - Reasoning-trace handling (Nemotron, if applicable)
  - `format_content_parts` for multimodal user messages (Qwen36 — Gemma4 only if vision variant is bundled later)
- **Model tests** per adapter under `tests/model/`:
  - At least one bundled GGUF per family loads
  - First-token smoke
  - One tool-call fixture per family produces a parseable call

5. Unit tests pass for each new adapter: render round-trip, tool-call parse (well-formed + malformed), multi-turn, reasoning-trace handling where applicable.

7. Each bundled GGUF's license documented in `bundled_models.yaml` comments.
8. Nemotron architecture-verification gate documented in the `Nemotron3Adapter` file header (arch tag, template format, tool-call format, reasoning-trace handling, hybrid-Mamba status).
9. Doxygen on every new public symbol includes `@brief` and `@version 2.1.9`.
10. Pre-commit (knots, flake8, doxygen-guard, unit tests) clean.
