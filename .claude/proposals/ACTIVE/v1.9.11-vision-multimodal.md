## Scope

Multimodal inference for the Entropic engine. After v1.9.11, the engine
can process messages containing images alongside text, load vision projection
models (mmproj) for vision-capable models, preprocess images for inference,
and fall back gracefully when the loaded model has no vision capability.

This builds on:
- **v1.8.2** (Inference Backend): `InferenceBackend` concrete base,
  `LlamaCppBackend`, `ChatAdapter` / `Qwen35Adapter`, `ModelOrchestrator`,
  VRAM lifecycle (COLD/WARM/ACTIVE)
- **v1.8.4** (Engine Loop): `AgentEngine`, `ContextManager`, message handling,
  `const char* messages_json` wire format

**What v1.9.11 adds:**
- Multimodal message content: `content` field supports both plain text strings
  and content arrays `[{"type":"text","text":"..."}, {"type":"image","path":"..."}]`
- mmproj (multimodal projection) loading as a companion model file alongside
  the primary GGUF model
- Image preprocessing pipeline: decode, resize, normalize to tensor format
  expected by the vision encoder
- Adapter support for vision content formatting (Qwen3.5 vision variants)
- Backward-compatible text-only fallback: text-only models silently skip
  image content parts with no error

**What v1.9.11 does NOT include:**
- Video input (frame extraction, temporal reasoning)
- Image generation or editing (vision-to-text only)
- Base64 encoding/decoding helpers (consumer responsibility)
- MCP tools for image capture or screenshot
- Streaming image chunks (images are input, not output)
- Per-tier vision enable/disable (all tiers sharing a vision model get vision)
- Vision-specific routing (router remains text-only classification)

**Consumer use cases:**
- Screenshot analysis: user sends a screenshot, model describes or debugs UI
- Document OCR: model reads text from images of documents
- Diagram understanding: model interprets architecture diagrams, flowcharts
- Visual debugging: model analyzes error screenshots, log images
- Homeschool: student sends a photo of a math problem, model explains

## Architecture

### Content Model: Multimodal Messages

Messages gain a polymorphic `content` field. At the C API boundary
(`const char* messages_json`), this is transparent — the JSON already
supports both string and array content per the OpenAI format. Internally,
the C++ `Message` struct needs a representation that handles both.

#### JSON Wire Format

```json
// Text-only (existing, unchanged)
{"role": "user", "content": "Explain this code"}

// Multimodal (new)
{"role": "user", "content": [
  {"type": "image", "path": "/tmp/screenshot.png"},
  {"type": "text", "text": "What's wrong with this UI?"}
]}

// Image with URL (data URI)
{"role": "user", "content": [
  {"type": "image", "url": "data:image/png;base64,iVBOR..."},
  {"type": "text", "text": "Describe this image"}
]}
```

#### C++ Content Representation

```cpp
/// @file include/entropic/types/content.h
/// @brief Multimodal content types for messages.
/// @version 1.9.11

namespace entropic {

/// @brief Content part type discriminant.
/// @version 1.9.11
enum class ContentPartType {
    TEXT,   ///< Plain text content
    IMAGE,  ///< Image content (path or data URI)
};

/// @brief A single content part within a multimodal message.
/// @version 1.9.11
///
/// Messages with simple string content are represented as a single
/// TEXT part. Multimodal messages have multiple parts. The engine
/// preserves part ordering — image position relative to text matters
/// for some models.
struct ContentPart {
    ContentPartType type;       ///< Part type discriminant
    std::string text;           ///< Text content (type == TEXT)
    std::string image_path;     ///< Local file path (type == IMAGE)
    std::string image_url;      ///< URL or data URI (type == IMAGE)
    int width = 0;              ///< Image width after preprocessing (0 = not yet processed)
    int height = 0;             ///< Image height after preprocessing (0 = not yet processed)
};

/// @brief Extract concatenated text from content parts.
/// @param parts Content parts array.
/// @return All text parts joined with spaces, images skipped.
/// @version 1.9.11
std::string extract_text(const std::vector<ContentPart>& parts);

/// @brief Check if content parts contain any image parts.
/// @param parts Content parts array.
/// @return true if at least one IMAGE part exists.
/// @version 1.9.11
bool has_images(const std::vector<ContentPart>& parts);

} // namespace entropic
```

#### Message Struct Extension

The existing `Message` struct (v1.8.0, `include/entropic/types/message.h`)
carries `std::string content`. v1.9.11 adds a `std::vector<ContentPart>
content_parts` field. The two fields coexist:

```cpp
/// @brief Message struct — v1.9.11 multimodal extension.
///
/// For text-only messages, `content` is populated and `content_parts`
/// is empty. For multimodal messages, `content_parts` is populated and
/// `content` holds the extracted text (for backward compatibility with
/// code that reads `content` directly).
///
/// The JSON parser sets both fields: multimodal JSON populates
/// `content_parts` and sets `content = extract_text(content_parts)`.
/// Text-only JSON sets `content` directly and leaves `content_parts`
/// empty.
struct Message {
    std::string role;                          ///< "system", "user", "assistant", "tool"
    std::string content;                       ///< Text content (always populated)
    std::vector<ContentPart> content_parts;    ///< Multimodal parts (empty for text-only)
    // ... existing fields (tool_calls, metadata) unchanged
};
```

**Backward compatibility:** Every existing callsite reads `content` as a
string. This continues to work because `content` is always populated with
the text portion. Only code that needs image data reads `content_parts`.

**JSON serialization:** When serializing a `Message` to JSON for the C API
boundary:
- If `content_parts` is empty: `"content": "text string"` (existing format)
- If `content_parts` is non-empty: `"content": [{"type":"text",...}, ...]`
  (array format)

This means the wire format is transparent to consumers — they see standard
OpenAI-format messages.

### mmproj Loading

The multimodal projection model (mmproj) is a separate GGUF file that
contains the vision encoder weights. It maps image pixel data to token
embeddings that the language model can process. For Qwen3.5, this is an
early-fusion architecture — vision was trained jointly with language.

#### llama.cpp Vision API

llama.cpp provides multimodal support through `libmtmd` (multimodal
media). The relevant API surface:

```c
// mtmd.h — multimodal context
struct mtmd_context;

// Create multimodal context from model + mmproj path
struct mtmd_context * mtmd_init(
    const struct llama_model * model,
    const char * mmproj_path,
    const struct mtmd_context_params * params);

void mtmd_free(struct mtmd_context * ctx);

// Tokenize text+image input into a sequence of chunks
// Each chunk is either text tokens or image embeddings
int mtmd_tokenize(
    struct mtmd_context * ctx,
    const struct mtmd_input * input,    // array of text/image parts
    int n_input,
    struct mtmd_input_chunks ** output);

// Get embeddings for an image chunk (runs the vision encoder)
int mtmd_get_chunk_embeddings(
    struct mtmd_context * ctx,
    const struct mtmd_input_chunk * chunk,
    float ** embeddings,
    int * n_embeddings);

// Helper: load image from file path
struct mtmd_bitmap * mtmd_helper_bitmap_init_from_file(const char * path);

// Helper: load image from memory buffer
struct mtmd_bitmap * mtmd_helper_bitmap_init_from_buf(
    const unsigned char * buf, size_t len);
```

The `mtmd_tokenize` function handles the interleaving of text tokens and
image embeddings. The engine calls `mtmd_get_chunk_embeddings` for each
image chunk, then feeds the mixed token/embedding sequence to `llama_decode`.

#### ModelConfig Extension

```cpp
/// @brief Model configuration — v1.9.11 additions.
struct ModelConfig {
    // ... existing fields from v1.8.1 ...

    std::filesystem::path mmproj_path;  ///< Vision projector GGUF path (empty = text-only)
    // ... rest unchanged
};
```

When `mmproj_path` is non-empty, the backend creates an `mtmd_context`
alongside the `llama_model`. When empty (default), vision is disabled
and behavior is identical to pre-v1.9.11.

#### YAML Config

```yaml
models:
  tiers:
    primary:
      path: primary  # resolves via bundled_models.yaml
      mmproj_path: ~/models/gguf/Qwen3.5-35B-A3B-mmproj-F16.gguf
      adapter: qwen35
      context_length: 8192
```

`mmproj_path` follows the same resolution rules as `path`: tilde expansion,
absolute/relative resolution. It does NOT go through `bundled_models.yaml`
(mmproj files are model-specific, not bundled registry entries).

### Image Preprocessing Pipeline

Image preprocessing converts raw image files into the format expected by
the vision encoder. The `mtmd_helper_bitmap_init_from_file` function in
llama.cpp handles basic image loading (JPEG, PNG, BMP, GIF via stb_image).
However, the engine needs a preprocessing layer above this for:

1. **Format validation** — reject unsupported formats early with clear errors
2. **Size constraints** — models have maximum image dimensions; oversized
   images must be resized
3. **Memory management** — large images consume significant RAM during
   preprocessing

```cpp
/// @file include/entropic/inference/image_preprocessor.h
/// @brief Image preprocessing for multimodal inference.
/// @version 1.9.11

namespace entropic {

/// @brief Image preprocessing configuration.
/// @version 1.9.11
struct ImagePreprocessConfig {
    int max_width = 1280;      ///< Maximum image width (resized if exceeded)
    int max_height = 1280;     ///< Maximum image height (resized if exceeded)
    size_t max_file_size = 20 * 1024 * 1024;  ///< Maximum file size in bytes (20MB)
    bool preserve_aspect = true;  ///< Preserve aspect ratio when resizing
};

/// @brief Preprocessed image ready for vision encoder.
/// @version 1.9.11
struct PreprocessedImage {
    std::vector<uint8_t> pixel_data;  ///< RGB pixel data
    int width;                         ///< Image width
    int height;                        ///< Image height
    int channels;                      ///< Color channels (3 = RGB)
    std::string source_path;           ///< Original file path (for logging)
};

/// @brief Image preprocessor — validates, resizes, and normalizes images.
/// @version 1.9.11
///
/// Concrete class (not a base class — no expected variants). Handles
/// the pipeline: load → validate → resize → normalize → output.
///
/// @par Thread safety:
/// Stateless after construction. Multiple threads can call
/// preprocess() concurrently on different images.
class ImagePreprocessor {
public:
    /// @brief Construct preprocessor with config.
    explicit ImagePreprocessor(const ImagePreprocessConfig& config);

    /// @brief Preprocess an image from file path.
    /// @param path File path to image.
    /// @return Preprocessed image ready for vision encoder.
    /// @throws std::runtime_error on invalid format, oversized file, or read error.
    /// @version 1.9.11
    PreprocessedImage preprocess_file(const std::filesystem::path& path);

    /// @brief Preprocess an image from memory buffer.
    /// @param data Raw image data (JPEG, PNG, etc.).
    /// @param len Data length in bytes.
    /// @param source_label Label for logging (e.g., "data_uri").
    /// @return Preprocessed image.
    /// @throws std::runtime_error on invalid format or decode error.
    /// @version 1.9.11
    PreprocessedImage preprocess_buffer(
        const uint8_t* data,
        size_t len,
        const std::string& source_label);

private:
    ImagePreprocessConfig config_;  ///< Preprocessing configuration

    /// @brief Resize image if it exceeds max dimensions.
    void resize_if_needed(PreprocessedImage& img);

    /// @brief Validate image dimensions and format.
    void validate(const PreprocessedImage& img);
};

} // namespace entropic
```

The preprocessor uses stb_image (already bundled with llama.cpp) for
decoding and stb_image_resize2 for resizing. These are header-only
libraries — no additional dependencies.

### Data URI Handling

Content parts with `image_url` containing `data:image/...;base64,...`
are decoded in-memory. The preprocessor receives the decoded bytes via
`preprocess_buffer()`. File paths go through `preprocess_file()`.

Resolution logic in `LlamaCppBackend`:

```
ContentPart.image_path non-empty → preprocess_file(image_path)
ContentPart.image_url starts with "data:" → base64_decode → preprocess_buffer()
ContentPart.image_url starts with "http" → ENTROPIC_ERROR_UNSUPPORTED_URL
    (engine does not fetch remote URLs — consumer must download first)
```

Remote URL fetching is explicitly out of scope. The engine processes local
files and data URIs. Consumers that need remote images must download them
before passing to the engine. This avoids network dependencies in the
inference library.

### LlamaCppBackend Extension

The backend gains vision capability gated on `mmproj_path` in config.

```cpp
/// @file src/inference/llama_cpp_backend.h
/// @brief v1.9.11 additions to LlamaCppBackend

class LlamaCppBackend : public InferenceBackend {
    // ... existing from v1.8.2 ...

protected:
    /// @brief Load model — extended for mmproj.
    /// @version 1.9.11 override of v1.8.2
    ///
    /// When config.mmproj_path is non-empty:
    /// 1. Load base model via llama_model_load (existing)
    /// 2. Initialize mtmd_context with mmproj path
    /// 3. Store mtmd_ctx_ for use during generation
    ///
    /// When mmproj_path is empty: identical to v1.8.2 behavior.
    bool do_load(const ModelConfig& config) override;

    /// @brief Unload model — extended for mtmd cleanup.
    void do_unload() override;

    /// @brief Generate response — multimodal-aware.
    /// @version 1.9.11 override extension
    ///
    /// If messages contain image content parts AND mtmd_ctx_ is loaded:
    /// 1. Preprocess images via ImagePreprocessor
    /// 2. Build mtmd_input array from content parts
    /// 3. Call mtmd_tokenize to get interleaved chunks
    /// 4. For each chunk: text → llama_decode, image → mtmd_get_chunk_embeddings → llama_decode
    /// 5. Sample and return generated text
    ///
    /// If messages contain image parts but mtmd_ctx_ is NULL (text-only model):
    /// - Strip image parts silently
    /// - Log warning: "Image content ignored — model has no vision capability"
    /// - Generate from text content only
    ///
    /// If no image parts: existing text-only code path (unchanged).
    GenerationResult do_generate(
        const std::vector<Message>& messages,
        const GenerationParams& params) override;

private:
    // ... existing members ...

    mtmd_context* mtmd_ctx_ = nullptr;  ///< Multimodal context (NULL = text-only)
    ImagePreprocessor image_preprocessor_;  ///< Image preprocessing pipeline
    bool has_vision_ = false;  ///< Quick check: is vision available?
};
```

### Multimodal Generation Flow

```
Consumer sends messages_json with image content
    |
    v
C API: entropic_inference_generate(backend, messages_json, params, &result)
    |
    v
Facade: parse messages_json → vector<Message>
    |  (content arrays parsed into content_parts)
    v
InferenceBackend::generate() — base class 80%:
    |  1. State check (ACTIVE)
    |  2. Lock generation mutex
    |  3. Pre-generate hook (v1.9.1)
    |  4. Log "generating, {} messages, {} with images"
    v
LlamaCppBackend::do_generate() — implementation 20%:
    |
    +-- has_images(messages)?
    |     |
    |     NO → existing text-only path (unchanged from v1.8.2)
    |     |
    |     YES → has_vision_?
    |             |
    |             NO → strip images, warn, text-only path
    |             |
    |             YES ↓
    |
    v
build_multimodal_input(messages):
    |  For each message:
    |    For each content_part:
    |      TEXT → add as text input
    |      IMAGE → preprocess → add as image input
    v
mtmd_tokenize(mtmd_ctx_, input, n_input, &chunks)
    |
    v
For each chunk in chunks:
    |  TEXT_CHUNK → llama_batch with token IDs → llama_decode
    |  IMAGE_CHUNK → mtmd_get_chunk_embeddings → llama_batch with embeddings → llama_decode
    v
Sample tokens until EOS or max_tokens
    |
    v
Return GenerationResult (text only — model output is always text)
```

### Streaming with Vision

Streaming generation (`do_generate_streaming`) follows the same
multimodal input path. The difference is in the sampling loop: tokens
are emitted via the `on_token` callback as they are generated. The
image preprocessing and encoding happen before the first token is
sampled — there is a latency cost for the vision encoder that the
consumer sees as a delay before the first token.

```
[image preprocess] → [vision encode] → [first token delay] → [streaming tokens...]
```

The consumer's `on_token` callback is not called during preprocessing
or vision encoding. It fires starting with the first sampled token.
This is the natural behavior — no special handling needed.

### Adapter: Vision Content Formatting

Model output is always text. Adapters parse assistant responses, which
never contain image content parts. This means **most adapter code is
unchanged**.

The one adapter touch point is system prompt assembly. When vision is
active, the adapter may need to include vision-specific instructions in
the system prompt (e.g., "You have vision capability and can analyze
images sent by the user.").

```cpp
/// @file include/entropic/inference/adapters/adapter_base.h
/// @brief v1.9.11 additions to ChatAdapter

class ChatAdapter {
public:
    // ... existing from v1.8.2 ...

    /// @brief Format system prompt with optional vision context.
    /// @param base_system Base system prompt text.
    /// @param has_vision Whether the model has vision capability.
    /// @return Formatted system prompt.
    /// @version 1.9.11
    ///
    /// Default implementation returns base_system unchanged.
    /// Vision-capable adapters override to append vision instructions.
    virtual std::string format_system_with_vision(
        const std::string& base_system,
        bool has_vision) const;

    /// @brief Convert multimodal content parts to adapter-specific format.
    /// @param parts Content parts from a message.
    /// @return JSON string in the format expected by the model's chat template.
    /// @version 1.9.11
    ///
    /// Default: OpenAI-format content array. Adapters override if the
    /// model expects a different image reference format.
    virtual std::string format_content_parts(
        const std::vector<ContentPart>& parts) const;
};
```

The `Qwen35Adapter` override:

```cpp
/// @file src/inference/adapters/qwen35_adapter.h
/// @brief v1.9.11 additions to Qwen35Adapter

class Qwen35Adapter : public ChatAdapter {
public:

```

### Text-Only Fallback

The fallback behavior when a text-only model receives image content:

1. `has_vision_` is false (no mmproj loaded)
2. `do_generate` detects image parts in messages
3. Image parts are stripped from the message content
4. A warning is logged: `"Image content ignored — model '{}' has no vision
   capability (no mmproj loaded)"`
5. Generation proceeds with text content only
6. No error returned — this is a graceful degradation, not a failure

This design means consumers can send multimodal messages to any model
without checking capabilities first. Vision models process images;
text-only models silently ignore them.

### ContextManager: Image-Aware Compaction

When context is compacted (v1.8.4), messages with image content parts
need special handling:

- Image content parts are **dropped during compaction**. They consume
  significant token budget (vision embeddings) and cannot be meaningfully
  summarized.
- The compaction summary notes that images were present: "User sent
  image at turn N" (metadata preservation, not image preservation).
- `content_parts` is cleared on compacted messages; `content` retains
  the text summary.

This is handled in `CompactionManager` — no changes to `ContextManager`
itself. The compaction logic already classifies messages by value density;
image parts are treated as low-density content that gets dropped
(similar to tool result stubs).

### C API: Transparent Content Arrays

The C API (`entropic.h`) requires no new functions for multimodal support.
The existing `entropic_inference_generate()` and `entropic_run()` accept
`const char* messages_json`. Content arrays in the JSON are parsed by the
facade's message deserializer and converted to `Message` structs with
populated `content_parts`.

The only C API addition is a capability query:

```c
/// @brief Check if a model has vision capability.
/// @param handle Engine handle.
/// @param model_id Model identifier (tier name or model key).
/// @return 1 if vision-capable (mmproj loaded), 0 if text-only.
/// @version 1.9.11
ENTROPIC_EXPORT int entropic_model_has_vision(
    entropic_handle_t handle,
    const char* model_id);
```

And new error codes:

```c
/// @brief New error codes for v1.9.11.
ENTROPIC_ERROR_IMAGE_LOAD_FAILED,     ///< Image file could not be read or decoded
ENTROPIC_ERROR_IMAGE_TOO_LARGE,       ///< Image exceeds max_file_size
ENTROPIC_ERROR_MMPROJ_LOAD_FAILED,    ///< mmproj GGUF failed to load
ENTROPIC_ERROR_UNSUPPORTED_URL,       ///< Remote URL not supported (must be local file or data URI)
```

### CMake Integration

Vision support is always compiled in — there is no `ENTROPIC_VISION` build
flag. The mmproj loading is opt-in at runtime via config (`mmproj_path`).
If no mmproj is configured, the vision code path is never exercised.

The `libmtmd` library from llama.cpp is linked into
`librentropic-inference-*.so` alongside `llama.cpp` itself. Since
llama.cpp is already a submodule dependency, mtmd comes for free.

stb_image and stb_image_resize2 are included via llama.cpp's vendored
copies — no separate dependency management needed.

## VRAM Budget

Vision has a VRAM cost. The mmproj adds to the base model's footprint:

| Component | Estimated VRAM |
|-----------|---------------|
| Qwen3.5-35B-A3B IQ3_XXS | ~13 GB |
| mmproj-F16 | ~0.5-1.5 GB |
| KV cache (8K context) | ~1-2 GB |
| **Total** | **~14.5-16.5 GB** |
| Available (16GB GPU) | 16 GB |

This is tight on 16GB. Mitigation strategies (all consumer-configured):

1. **Reduce context length** when vision is active (e.g., 4K instead of 8K)
2. **Use quantized mmproj** if available (mmproj-Q8_0 or similar)
3. **Partial GPU offload** — reduce `gpu_layers` to free VRAM
4. **Dynamic context** — engine could report available VRAM via metrics,
   consumer adjusts context length accordingly

The engine does NOT auto-reduce context length. VRAM management is the
consumer's responsibility via config. If the model + mmproj + KV cache
exceeds VRAM, llama.cpp will report an allocation failure that the
engine surfaces as `ENTROPIC_ERROR_LOAD_FAILED`.

## Validation Criteria

### V1: Text-only backward compatibility

Given messages with string content (no content arrays), after v1.9.11,
generation produces identical results to pre-v1.9.11. `content_parts`
is empty. No behavioral change for text-only workflows.

### V2: Content array parsing

Given a JSON message with `"content": [{"type":"text","text":"Hello"},
{"type":"image","path":"/tmp/test.png"}]`, the parser produces a `Message`
with two `content_parts` entries and `content == "Hello"`.

### V3: Text extraction from content parts

Given content parts `[IMAGE, TEXT("a"), IMAGE, TEXT("b")]`,
`extract_text()` returns `"a b"`. `has_images()` returns true.

### V4: mmproj loading

Given a `ModelConfig` with valid `mmproj_path`, `do_load()` creates
`mtmd_ctx_` and sets `has_vision_ = true`. Given empty `mmproj_path`,
`mtmd_ctx_` is NULL and `has_vision_ = false`.

### V5: mmproj load failure

Given an invalid `mmproj_path` (non-existent file), `do_load()` returns
`ENTROPIC_ERROR_MMPROJ_LOAD_FAILED`. Base model is not loaded (clean
failure, no partial state).

### V6: Image preprocessing — valid image

Given a valid PNG file under `max_file_size`, `preprocess_file()` returns
a `PreprocessedImage` with correct dimensions and RGB data.

### V7: Image preprocessing — resize

Given a 4000x3000 image with `max_width=1280, max_height=1280`,
the preprocessor resizes to fit within bounds while preserving aspect
ratio. Output dimensions are 1280x960.

### V8: Image preprocessing — oversized file

Given a file exceeding `max_file_size`, `preprocess_file()` throws
with a message mentioning the size limit.

### V9: Image preprocessing — invalid format

Given a non-image file (e.g., `.txt`), `preprocess_file()` throws
with a message mentioning unsupported format.

### V10: Multimodal generation — vision model

Given a model with mmproj loaded and a message with image + text
content, `do_generate()` processes the image through the vision encoder
and generates a text response that references the image content.

### V11: Text-only fallback — no vision model

Given a model WITHOUT mmproj and a message with image + text content,
`do_generate()` strips image parts, logs a warning, and generates from
text only. No error returned.

### V12: Adapter vision formatting

Given `has_vision == true`, `Qwen35Adapter::format_system_with_vision()`
appends vision instructions to the system prompt. Given `has_vision ==
false`, the system prompt is unchanged.

### V13: Compaction drops images

Given a conversation with multimodal messages that triggers compaction,
after compaction: `content_parts` is empty on compacted messages, `content`
contains text summary, and the summary notes "images were shared at turn N".

### V14: Capability query

`entropic_model_has_vision(handle, "primary")` returns 1 when primary has
mmproj loaded, 0 when it does not.

### V15: Data URI handling

Given a content part with `image_url: "data:image/png;base64,..."`,
the backend decodes the base64, preprocesses the image, and generates
a response referencing image content.

### V16: Remote URL rejection

Given a content part with `image_url: "https://example.com/img.png"`,
the backend returns `ENTROPIC_ERROR_UNSUPPORTED_URL`.

### V17: Streaming with vision

Given a vision-capable model and a multimodal message, streaming
generation emits tokens via `on_token` callback after vision encoding
completes. The first token is delayed by vision encoding time. All
subsequent tokens stream normally.

### V18: SLOC and complexity

All functions pass knots quality gates: cognitive complexity <= 15,
SLOC <= 50, returns <= 3.

### V19: Doxygen complete

Every public symbol in new headers has `@brief`, `@param`, `@return`,
`@version`, `@threadsafety` where applicable.

## Implementation Notes (from review)

1. **mmproj VRAM accounting.** mmproj files are 600MB+. The WARM→ACTIVE
   transition must account for mmproj VRAM alongside the base model. The
   `ModelConfig.mmproj_path` field (added in this version) should be used by
   the orchestrator's VRAM budget calculation. On 16GB (13GB model + 0.5-1.5GB
   mmproj + KV cache), context length may need automatic reduction. Document
   that the engine does NOT auto-reduce context — the consumer manages via config.
