## Scope

Post-TUI-separation cleanup. This is the last Python milestone before the C++ port
begins. The resulting public API surface IS the behavioral specification that
v1.8.x faithfully translates to C.

Three goals:

1. **Remove** all TUI-specific code from the engine repo
2. **Stabilize** `LibraryConfig` as the sole config type the engine accepts
3. **Document** public vs internal API boundaries with deprecation warnings

### What stays

- `AgentEngine`, `ModelOrchestrator`, `ServerManager`, `PromptManager`
- `LibraryConfig` and all its nested configs (models, routing, generation,
  permissions, mcp, compaction)
- `Presenter` ABC + `HeadlessPresenter` (engine-side contract for any UI)
- CLI subcommands: `ask`, `init`, `download`, `setup-cuda`, `benchmark`,
  `mcp-bridge`, `status`
- `Message`, `ToolCall`, `ToolResult`, `GenerationResult` core types

### What goes

- `EntropyConfig` (moves to `entropic-tui` in v1.7.2 — v1.7.3 removes the import)
- `Application` class (TUI orchestrator — entirely TUI concern)
- `Rich` / `Console` imports from engine code
- `SQLiteStorage` lifecycle management in `Application` (the storage
  module itself stays in engine — it's used by `DelegationManager`
  and `CompactionManager` for conversation persistence)
- `SessionManager`, `CommandRegistry` (session CRUD, slash commands = TUI)
- Voice config/callbacks wiring (voice is a consumer concern)
- `app_context_tui.md` bundling (TUI bundles its own)
- TUI-specific fields: `quality`, `ui`, `storage`, `lsp`, `voice`,
  `log_file`, `commands_dir`

## Implementation Plan

### Phase 1: Config Boundary

**Goal:** `AgentEngine` and all engine internals accept `LibraryConfig`, not `EntropyConfig`.

1. **Change `AgentEngine.__init__` signature**: `config: EntropyConfig | None` becomes
   `config: LibraryConfig | None`. Default changes from `EntropyConfig()` to
   `LibraryConfig()`.
2. **Audit all `self.config.` references in engine code** — engine only touches
   `compaction`, `models`, `routing`, `generation`, `permissions`, `mcp`,
   `log_level`, `constitution`, `app_context`, `inject_model_context`,
   `vram_reserve_mb`, `config_dir`. All are `LibraryConfig` fields already.
3. **Change `ServerManager` config type** to `LibraryConfig`. The manager uses
   `config.mcp`, `config.lsp` — LSP config must move to `LibraryConfig` OR
   the LSP integration must be injected rather than config-driven.
   - **Decision needed:** LSP diagnostics are engine-useful (proactive error
     detection on file edits). Move `LSPConfig` into `LibraryConfig` as an
     engine capability, not a TUI feature.
4. **`ResponseGenerator`, `ContextManager`, `ToolExecutor`** — audit config
   access, confirm all within `LibraryConfig` surface.

### Phase 2: Remove Application Class

**Goal:** No `app.py` in engine repo.

1. **Delete `src/entropic/app.py`** — 949 lines of TUI orchestration.
   All responsibilities move to `entropic-tui`:
   - Rich console / Status spinner usage
   - SQLite storage lifecycle
   - Session CRUD (`_new_session`, `_switch_session`, etc.)
   - Command registry wiring
   - Voice mode VRAM callbacks
   - Queue consumer for external MCP
   - `single_turn()` method (a thin wrapper — replace with a 10-line
     example in docstring or `examples/`)
2. **Update `cli.py`** — the default command (no subcommand) currently
   launches `Application(config).run()`. Post-cleanup:
   - Default command prints help text (engine is a library, not an app)
   - `ask` subcommand uses `AgentEngine` directly (no `Application` wrapper)
   - Remove `--headless` flag (engine is always headless)
3. **Delete `src/entropic/ui/tui_presenter.py`** — TUI presenter moves to
   `entropic-tui`. The `Presenter` ABC and `HeadlessPresenter` stay.
4. **Delete `src/entropic/ui/tui.py`** — Textual app, pure TUI.
5. **Delete TUI-only modules**: `core/commands.py` (slash commands),
   `core/session.py` (session manager), `core/queue.py` (MCP message queue
   for TUI integration), `core/tasks.py` (task manager for external MCP).

### Phase 3: Stabilize Public API

**Goal:** `__init__.py` exports exactly what library consumers need.

1. **Tier the exports** in `__init__.py` with comments:
   ```
   # === Public API (stable, documented) ===
   # === Extension Points (stable, for custom backends/tools) ===
   # === Internal (may change, deprecation warnings) ===
   ```
2. **Public API surface** (the C API behavioral spec):
   - Config: `LibraryConfig`, `ModelConfig`, `TierConfig`, `ModelsConfig`,
     `RoutingConfig`, `GenerationConfig`, `CompactionConfig`,
     `PermissionsConfig`, `MCPConfig`, `ConfigLoader`, `validate_config`,
     `save_permission`
   - Engine: `AgentEngine`, `AgentState`, `EngineCallbacks`, `LoopConfig`
   - Types: `Message`, `ToolCall`, `ToolResult`, `GenerationResult`,
     `ModelBackend`, `ModelTier`
   - Orchestrator: `ModelOrchestrator`, `RoutingResult`, `BackendFactory`
   - Presenter: `Presenter` (ABC), `HeadlessPresenter`, `StatusInfo`
   - Prompts: `PromptManager`, `TierIdentity`, `load_tier_identity`,
     `parse_prompt_file`, frontmatter types
3. **Extension points** (stable but secondary):
   - MCP: `BaseMCPServer`, `BaseTool`, `ServerResponse`, `ToolRegistry`,
     `ServerManager`, `InProcessProvider`, `load_tool_definition`
   - Adapters: `ChatAdapter`, `get_adapter`, `register_adapter`
4. **Deprecation wrapper for `EntropyConfig`**: If any external code imports
   `EntropyConfig` from `entropic`, emit `DeprecationWarning` pointing to
   `entropic-tui` or `LibraryConfig`. Keep the import working for one minor
   version (remove at v1.8.0).
5. **Remove from `__all__`**: `EntropyConfig` (deprecated), logging setup
   helpers (internal), `ToolValidationError` (internal), `ToolProvider`
   (internal base).
6. **Add `__all__` enforcement test**: test that importing `*` from
   `entropic` produces exactly the documented set.

### Phase 4: CLI Cleanup

**Goal:** CLI is a thin layer over `AgentEngine` for scripting.

1. **Default command** (`entropic` with no args): print help + version.
   No TUI launch.
2. **`ask` subcommand** rework:
   ```python
   engine = AgentEngine(orchestrator=orch, server_manager=mgr, config=cfg)
   async for msg in engine.run(message):
       if msg.role == "assistant":
           print(msg.content, end="", flush=True)
   ```
   No `Application` class. No Rich. Plain stdout streaming.
3. **`status` subcommand** — keep, but remove `config.quality.enabled`
   reference (quality is TUI config). Show only engine-relevant status:
   models, routing, log level.
4. **Keep `mcp-bridge` subcommand** — the stdio↔socket bridge is transport
   infrastructure used by any external MCP client (Claude Code etc.), not
   a TUI concern. Headless users are the primary audience.
5. **`init` subcommand** — keep, but remove `quality` section from
   generated config template (TUI concern). Generate `LibraryConfig`-shaped
   YAML only.

### Phase 5: Presenter Interface Refinement

**Goal:** `Presenter` ABC is the clean engine-to-UI contract.

1. **Remove voice callbacks** from `Presenter` ABC (`set_voice_callbacks`) —
   voice is a consumer concern, not an engine contract.
2. **Remove `set_queue_consumer`** from `Presenter` ABC — MCP queue
   integration is TUI-specific.
3. **Evaluate `prompt_injection`** — mid-generation injection is an engine
   capability. Keep on `Presenter` ABC.
4. **Evaluate `print_todo_panel`** — todos are an engine feature (directive
   system). Keep on `Presenter` ABC.
5. **Document each method** with whether it's required vs optional (default
   no-op). Current mix of abstract + concrete defaults is correct but
   undocumented.

## Validation Criteria

### Must pass

1. `entropic ask "Hello"` works — loads model, generates response, exits
2. `entropic` (no args) prints help text, does not launch TUI
3. `entropic --version` prints version
4. `entropic status` shows models, routing, log level (no quality/voice)
5. `entropic init` generates `LibraryConfig`-shaped YAML (no quality/ui/voice)
6. All existing unit tests pass (adapted for `LibraryConfig`)
7. `from entropic import LibraryConfig, AgentEngine, ModelOrchestrator` works
8. `from entropic import EntropyConfig` emits `DeprecationWarning`
9. No `import rich` or `import textual` anywhere in engine source
10. No `EntropyConfig` reference in engine source (except deprecation shim)
11. `__all__` enforcement test passes
12. Pre-commit passes (flake8, ruff, mypy, pytest, cognitive complexity)

### External validation

13. `entropic-tui` (separate repo) can import `entropic.LibraryConfig`,
    subclass `Presenter`, wire `AgentEngine`, and reproduce current TUI
    behavior. This is validated in v1.7.2, confirmed still working after
    v1.7.3 cleanup.

## Dependencies

- **v1.7.2 complete**: TUI must be extracted before engine-side cleanup.
  Deleting `app.py` before the TUI repo exists loses the reference impl.
- **No C++ dependency**: This is pure Python cleanup.
- **entropic-tui pinned to v1.7.1**: TUI continues working against the
  frozen v1.7.1 engine while v1.7.3 cleans up. TUI updates to v1.7.3
  API separately.
