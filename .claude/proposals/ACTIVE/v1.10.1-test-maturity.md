## Scope

Establish enforceable quality gates that block release if unmet. After
v1.10.1, the engine has:

- Per-library line coverage thresholds (enforced in CI)
- Automated regression tests ported from Python-era manual test findings (test1-test6)
- CI merge gates: PRs cannot merge with failing tests or coverage below threshold
- doxygen-guard `@req` enforcement promoted from WARNING to ERROR â€” every
  public API function has at least one `@req` tag validated by CI
- A clear, auditable answer to "is this engine ready to ship?"

This is the LAST milestone before v2.0.0 release.

**Key shift from earlier design:** `@test` annotations replaced by
doxygen-guard `@req` tags (v1.10.0 change). The traceability chain is:
`@req REQ-xxx` â†’ `docs/requirements.yaml` â†’ `acceptance_criteria` â†’
test suite mapping in `.doxygen-guard.yaml`. doxygen-guard's impact
analysis replaces the custom `parse_test_annotations.py` script.

## Implementation Plan

### Phase 1: Coverage Targets

Per-library line coverage thresholds enforced by `scripts/check_coverage.py`
against `scripts/coverage_thresholds.json`.

| Library | Target | Rationale |
|---|---|---|
| `librentropic-types` | 95% | Pure data types, fully testable |
| `librentropic-config` | 90% | Config loading, validation â€” all CPU-testable |
| `librentropic-core` | 85% | Engine loop, state machine â€” some paths need integration |
| `librentropic-mcp` | 85% | Server base, registry, executor â€” mock-friendly |
| `librentropic-storage` | 85% | SQLite operations â€” all CPU-testable |
| `librentropic-inference-*` | 70% | Hardware-dependent; model tests cover generation paths |
| `librentropic` (facade) | 80% | Thin delegation layer |
| MCP server plugins | 80% | Individual server logic, mock-friendly |

Thresholds live in a data file (`scripts/coverage_thresholds.json`), not
hardcoded. Updating a threshold is a visible, reviewable change.

Coverage exclusion patterns are explicit with justification:
- `*/inference/cuda_*` â€” GPU-specific, tested in model suite
- `*/platform/win32_*` â€” platform-specific fallbacks
- `*/entropic_config.h` â€” generated code

### Phase 3: CI Merge Gates

GitHub branch protection on `develop` and `main`:

| Rule | `develop` | `main` |
|---|---|---|
| Required status checks | unit, integration, regression, lint, static-analysis | ALL + coverage-check |
| Require review | 0 (Claude merges feature branches) | 1 (user merges develop) |

Coverage enforcement on `main` from day one. Promoted to `develop` once
all libraries meet thresholds.

### Phase 4: doxygen-guard @req Enforcement

v1.10.0 configured doxygen-guard with `@req` tag validation as WARNING
(unmapped functions logged but don't block). v1.10.1 promotes to ERROR.

**Enforcement rule:** Every public function in `entropic.h` and every
function in interface headers (`interfaces/i_*.h`) MUST have at least
one `@req REQ-xxx` tag. New API functions cannot be added without linking
to a requirement.

doxygen-guard validates:
1. `@req` values match pattern `^REQ-[A-Z]+-\d{3}$`
2. Referenced REQ IDs exist in `docs/requirements.yaml`
3. Every public function has at least one `@req` tag

doxygen-guard pre-commit hook runs in validate mode. CI runs with
`--strict` equivalent (all violations are errors).

**Impact analysis for test selection:** `doxygen-guard impact --staged`
reports which requirements are affected by staged changes. The test
mapping in `.doxygen-guard.yaml` maps requirements to ctest regex
patterns. `scripts/select_tests.py` consumes this output.

## Validation Criteria

1. Per-library coverage meets thresholds (`check_coverage.py` exit 0)
2. All 6 regression test files exist and pass
3. doxygen-guard validates all public headers without errors
4. Every function in `entropic.h` has at least one `@req` tag
5. CI coverage job blocks merge on failure
6. CI regression job runs alongside unit/integration
7. Branch protection configured on `develop` and `main`
8. Coverage report committed to `test-reports/coverage/`
9. Release readiness checklist: all 16 items verified
