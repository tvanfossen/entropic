## Scope

Establish enforceable quality gates that block release if unmet. After
v1.10.1, the engine has:

- Per-library line coverage thresholds (enforced in CI)
- Automated regression tests ported from Python-era manual test findings (test1-test6)
- CI merge gates: PRs cannot merge with failing tests or coverage below threshold
- A clear, auditable answer to "is this engine ready to ship?"

Remaining milestones before v2.0.0: v1.10.2 (C++ model tests — the
critical gap discovered post-v1.9.15) and v1.10.3 (doxygen traceability).

**Note:** doxygen-guard `@req` enforcement, documentation review, and
impact-based test selection are handled in v1.10.3. C++ model tests
(porting the 42 Python model tests deleted in v1.9.15) are handled in
v1.10.2.

## Implementation Plan

### Phase 1: Coverage Targets

Per-library line coverage thresholds enforced by `scripts/check_coverage.py`
against `scripts/coverage_thresholds.json`.

| Library | Target | Rationale |
|---|---|---|
| `librentropic-types` | 95% | Pure data types, fully testable |
| `librentropic-config` | 90% | Config loading, validation — all CPU-testable |
| `librentropic-core` | 85% | Engine loop, state machine — some paths need integration |
| `librentropic-mcp` | 85% | Server base, registry, executor — mock-friendly |
| `librentropic-storage` | 85% | SQLite operations — all CPU-testable |
| `librentropic-inference-*` | 70% | Hardware-dependent; model tests cover generation paths |
| `librentropic` (facade) | 80% | Thin delegation layer |
| MCP server plugins | 80% | Individual server logic, mock-friendly |

Thresholds live in a data file (`scripts/coverage_thresholds.json`), not
hardcoded. Updating a threshold is a visible, reviewable change.

Coverage exclusion patterns are explicit with justification:
- `*/inference/cuda_*` — GPU-specific, tested in model suite
- `*/platform/win32_*` — platform-specific fallbacks
- `*/entropic_config.h` — generated code

### Phase 2: Regression Suite from Manual Test Findings

Convert test1-test6 patterns (documented in `tests/manual/README.md` by
v1.10.0) into automated C++ tests in `tests/regression/`.

All regression tests use mock backends (CPU-only, CI-runnable):

| File | Source | What it validates |
|---|---|---|
| `test_basic_response.cpp` | test1 | Response cleanliness, no special tokens |
| `test_multi_turn_context.cpp` | test2 | Context accumulation, compaction trigger |
| `test_tool_calling_loop.cpp` | test3 | Generate→parse→execute→generate, circuit breaker |
| `test_delegation_pipeline.cpp` | test4 | Lead→child delegation, pipeline propagation |
| `test_router_classification.cpp` | test5 | Router uses complete() not generate(), consistency |
| `test_identity_behavior.cpp` | test6 | Tool filtering, constitution, lead has exactly 3 tools |

---

### Phase 3: CI Merge Gates

GitHub branch protection on `develop` and `main`:

| Rule | `develop` | `main` |
|---|---|---|
| Required status checks | unit, integration, regression, lint, static-analysis | ALL + coverage-check |
| Require review | 0 (Claude merges feature branches) | 1 (user merges develop) |

Coverage enforcement on `main` from day one. Promoted to `develop` once
all libraries meet thresholds.

## Validation Criteria

1. Per-library coverage meets thresholds (`check_coverage.py` exit 0)
2. All 6 regression test files exist and pass
3. CI coverage job blocks merge on failure
4. CI regression job runs alongside unit/integration
5. Branch protection configured on `develop` and `main`
6. Coverage report committed to `test-reports/coverage/`
7. Release readiness checklist: all items verified
