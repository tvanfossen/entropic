## Scope

Core inference library: `librentropic-inference-{cuda,vulkan,cpu}.so`. After
v1.8.2, the engine can load GGUF models, manage their VRAM lifecycle through
COLD/WARM/ACTIVE states, generate text (batch and streaming with cancel),
perform raw text completion for the router, and format/parse chat through
adapters.

This is a faithful port of:
- `src/entropic/core/base.py` -- `ModelBackend` ABC, `ModelState`, `GenerationResult`
- `src/entropic/inference/backend.py` -- `GenerationConfig`
- `src/entropic/inference/llama_cpp.py` -- `LlamaCppBackend` (three-state lifecycle, generate, stream, complete)
- `src/entropic/inference/adapters/base.py` -- `ChatAdapter` base (tool formatting, think blocks, XML/JSON parsing)
- `src/entropic/inference/adapters/qwen35.py` -- `Qwen35Adapter` (XML tool calls, `<tools>` tags, `<tool_response>`)
- `src/entropic/inference/orchestrator.py` -- `ModelOrchestrator` (tier routing, model pool dedup, VRAM lifecycle, handoff)

v1.8.1 provided config structs (`ModelConfig`, `GenerationParams`), prompt
manager, and bundled model registry. v1.8.2 consumes those structs to load
models and generate.

No engine loop, no MCP, no delegation. Pure inference + chat adapters +
orchestration.

### Phase 2: GenerationParams Struct

Already defined in `include/entropic/types/config.h` by v1.8.0 (per roadmap).
Validate the struct matches what inference needs:

```cpp
/// @brief Parameters for a single generation request.
/// @par Maps to Python GenerationConfig + per-request kwargs.
struct GenerationParams {
    int max_tokens = 4096;            ///< Max tokens to generate
    float temperature = 0.7f;         ///< Sampling temperature
    float top_p = 0.9f;               ///< Nucleus sampling threshold
    int top_k = 40;                   ///< Top-K sampling
    float repeat_penalty = 1.1f;      ///< Repetition penalty
    std::vector<std::string> stop;    ///< Stop sequences
    std::string grammar;              ///< Optional GBNF grammar string
    int logprobs = 0;                 ///< Top log-probs per token (0 = disabled)
    int reasoning_budget = -1;        ///< Think token budget (-1 = unlimited)
    bool enable_thinking = true;      ///< Enable <think> blocks
};
```

`reasoning_budget` maps to the Python `chat_template_kwargs["enable_thinking"]`
concept but extends it: -1 = unlimited thinking, 0 = disabled (equivalent to
`enable_thinking=false`), N > 0 = budget N tokens for `<think>` block. The
backend enforces this by monitoring think-block token count during generation
and forcing the `</think>` close tag when budget is exhausted.

`cache_type_k` and `cache_type_v` are model-level config in `ModelConfig`
(set once at load time), not per-request params. They map to
`llama_model_params.cache_type_k` / `cache_type_v`.

### Phase 3: InferenceBackend Concrete Base Class

The core of v1.8.2. Template method pattern: public methods own the state
machine and threading, protected virtuals for subclass-specific logic.

```cpp
/// @file include/entropic/inference/backend.h
/// @brief InferenceBackend concrete base class.
///
/// Owns: lifecycle state machine, transition mutex, metrics, logging.
/// Subclasses override: do_load, do_activate, do_deactivate, do_unload,
/// do_generate, do_generate_streaming, do_complete.
///
/// @par Thread safety
/// - state() is lock-free (std::atomic<ModelState>)
/// - load/activate/deactivate/unload acquire transition_mutex_
/// - generate/generate_streaming/complete require ACTIVE state,
///   do NOT acquire transition_mutex_ (generation is concurrent with
///   state queries, but not with state transitions)
///
/// @par State machine
/// @code
///   COLD ──warm()──→ WARM ──activate()──→ ACTIVE
///     ↑                ↑                    │
///     │                └──deactivate()──────┘
///     └────────────unload()─────────────────┘
/// @endcode

#pragma once

#include <entropic/types/config.h>
#include <entropic/types/generation_result.h>
#include <entropic/types/message.h>

#include <atomic>
#include <filesystem>
#include <functional>
#include <mutex>
#include <string>
#include <string_view>
#include <vector>

namespace entropic {

class InferenceBackend {
public:
    virtual ~InferenceBackend() = default;

    // ── Lifecycle (template methods — own the state machine) ──

    /// @brief Load model into CPU RAM (COLD → WARM).
    /// mmap + mlock, n_gpu_layers=0. No-op if already WARM/ACTIVE.
    /// @return true on success.
    bool load(const ModelConfig& config);

    /// @brief Promote to GPU (WARM → ACTIVE). If COLD, loads first.
    /// @return true on success.
    bool activate();

    /// @brief Release GPU layers (ACTIVE → WARM). No-op if not ACTIVE.
    void deactivate();

    /// @brief Full unload (→ COLD). Releases all RAM + VRAM.
    void unload();

    /// @brief Convenience: load() + activate().
    bool load_and_activate(const ModelConfig& config);

    // ── Generation (require ACTIVE state) ──

    /// @brief Generate a complete response.
    /// @param messages Conversation history.
    /// @param params Generation parameters.
    /// @return GenerationResult with content, token count, timing.
    GenerationResult generate(
        const std::vector<Message>& messages,
        const GenerationParams& params);

    /// @brief Generate with per-token streaming callback.
    /// @param messages Conversation history.
    /// @param params Generation parameters.
    /// @param on_token Called for each token. Receives string_view (valid
    ///        only for duration of callback).
    /// @param cancel Atomic flag. Set to true to abort generation.
    ///        Checked between tokens — latency is one token.
    /// @return GenerationResult with final content and timing.
    GenerationResult generate_streaming(
        const std::vector<Message>& messages,
        const GenerationParams& params,
        std::function<void(std::string_view token)> on_token,
        std::atomic<bool>& cancel);

    /// @brief Raw text completion without chat template.
    /// @param prompt Raw prompt string (passed directly to model).
    /// @param params Generation parameters.
    /// @return GenerationResult.
    ///
    /// Used by the router for digit-based classification. Distinct from
    /// generate() which applies chat templates via the adapter.
    GenerationResult complete(
        const std::string& prompt,
        const GenerationParams& params);

    // ── Queries (lock-free) ──

    /// @brief Current lifecycle state.
    /// @return ModelState (COLD, WARM, or ACTIVE). Lock-free read.
    ModelState state() const { return state_.load(std::memory_order_acquire); }

    /// @brief True when state is ACTIVE.
    bool is_active() const { return state() == ModelState::ACTIVE; }

    /// @brief Count tokens using model's tokenizer.
    /// @return Token count, or len/4 estimate if model not loaded.
    int count_tokens(const std::string& text) const;

    /// @brief Model's context window size.
    int context_length() const { return config_.context_length; }

    /// @brief Stored config.
    const ModelConfig& config() const { return config_; }

protected:
    // ── Subclass overrides (20%) ──

    /// @brief Load model into CPU RAM. Called under transition_mutex_.
    /// @param config Validated model config.
    /// @return true on success. Set last_error_ on failure.
    virtual bool do_load(const ModelConfig& config) = 0;

    /// @brief Promote loaded model to GPU. Called under transition_mutex_.
    virtual bool do_activate() = 0;

    /// @brief Release GPU, keep CPU. Called under transition_mutex_.
    virtual void do_deactivate() = 0;

    /// @brief Full unload. Called under transition_mutex_.
    virtual void do_unload() = 0;

    /// @brief Subclass generation. Called only when ACTIVE.
    virtual GenerationResult do_generate(
        const std::vector<Message>& messages,
        const GenerationParams& params) = 0;

    /// @brief Subclass streaming generation. Called only when ACTIVE.
    virtual GenerationResult do_generate_streaming(
        const std::vector<Message>& messages,
        const GenerationParams& params,
        std::function<void(std::string_view token)> on_token,
        std::atomic<bool>& cancel) = 0;

    /// @brief Subclass raw completion. Called only when ACTIVE.
    virtual GenerationResult do_complete(
        const std::string& prompt,
        const GenerationParams& params) = 0;

    /// @brief Subclass token counting. Called only when model loaded.
    virtual int do_count_tokens(const std::string& text) const = 0;

    // ── State management (base class uses these) ──

    std::string last_error_;  ///< Last error message for diagnostics

private:
    std::atomic<ModelState> state_{ModelState::COLD};
    ModelConfig config_;
    std::mutex transition_mutex_;  ///< Guards state TRANSITIONS only
};

} // namespace entropic
```

**Base class logic (the 80%):**

1. **`load()`**: Acquires `transition_mutex_`. If COLD, calls `do_load()`,
   sets state to WARM. If already WARM/ACTIVE, no-op. Logs timing.
2. **`activate()`**: Acquires `transition_mutex_`. If COLD, calls `load()`
   first. If WARM, calls `do_activate()`, sets state to ACTIVE. Logs timing.
3. **`deactivate()`**: Acquires `transition_mutex_`. If ACTIVE, calls
   `do_deactivate()`, sets state to WARM. Logs timing.
4. **`unload()`**: Acquires `transition_mutex_`. Calls `do_unload()` regardless
   of state (idempotent). Sets state to COLD.
5. **`generate()`**: Checks `is_active()`. If not, returns error result. Calls
   `do_generate()`. Measures wall-clock time. Does NOT acquire transition mutex
   — generation runs concurrently with state queries.
6. **`generate_streaming()`**: Same guard. Calls `do_generate_streaming()`.
   The `cancel` flag is checked between tokens by the subclass.
7. **`complete()`**: Same guard. Calls `do_complete()`.
8. **`count_tokens()`**: If model loaded (WARM or ACTIVE), delegates to
   `do_count_tokens()`. Otherwise returns `text.size() / 4`.

**Transition rules (enforced by base class):**

```
                    Transition            Precondition     Postcondition
                    ──────────            ────────────     ─────────────
                    load()                COLD             WARM
                    activate()            WARM (or COLD)   ACTIVE
                    deactivate()          ACTIVE           WARM
                    unload()              any              COLD
                    load_and_activate()   COLD             ACTIVE
```

Invalid transitions are no-ops with INFO log (not errors). This matches
the Python behavior where `warm()` on an already-warm model returns silently.

### Phase 4: LlamaCppBackend — Direct C API

The 20% subclass. Uses llama.cpp's C API directly — NOT llama-cpp-python.
This is the critical difference from the Python implementation.

```cpp
/// @file src/inference/llama_cpp_backend.h
/// @brief LlamaCppBackend — llama.cpp C API integration.
///
/// @par Key differences from Python LlamaCppBackend:
/// - Direct llama.cpp C API (llama_model_load, llama_decode, etc.)
///   instead of llama-cpp-python wrapper
/// - No Python GIL — generation runs natively
/// - No asyncio bridge — streaming is synchronous with callback
/// - VRAM lifecycle via llama_model_load (WARM) +
///   llama_new_context_with_model (ACTIVE) separation
///
/// @par VRAM lifecycle mapping:
/// - COLD: nothing allocated
/// - WARM: llama_model loaded (CPU mmap+mlock, n_gpu_layers=0),
///   no llama_context
/// - ACTIVE: llama_context created with n_gpu_layers from config,
///   KV cache allocated on GPU

#pragma once

#include <entropic/inference/backend.h>
#include <llama.h>

namespace entropic {

class LlamaCppBackend : public InferenceBackend {
protected:
    bool do_load(const ModelConfig& config) override;
    bool do_activate() override;
    void do_deactivate() override;
    void do_unload() override;

    GenerationResult do_generate(
        const std::vector<Message>& messages,
        const GenerationParams& params) override;

    GenerationResult do_generate_streaming(
        const std::vector<Message>& messages,
        const GenerationParams& params,
        std::function<void(std::string_view token)> on_token,
        std::atomic<bool>& cancel) override;

    GenerationResult do_complete(
        const std::string& prompt,
        const GenerationParams& params) override;

    int do_count_tokens(const std::string& text) const override;

private:
    // ── llama.cpp handles ──
    llama_model* model_ = nullptr;    ///< Loaded model (WARM+)
    llama_context* ctx_ = nullptr;    ///< Inference context (ACTIVE only)
    const llama_vocab* vocab_ = nullptr;  ///< Vocabulary (from model_)

    // ── Internal generation ──

    /// @brief Tokenize text using model vocabulary.
    std::vector<llama_token> tokenize(
        const std::string& text, bool add_special) const;

    /// @brief Detokenize a single token to string.
    std::string detokenize(llama_token token) const;

    /// @brief Apply chat template to messages, return formatted prompt.
    /// Uses the GGUF-embedded jinja template via llama_chat_apply_template.
    std::string apply_chat_template(
        const std::vector<Message>& messages,
        const GenerationParams& params) const;

    /// @brief Core decode loop — shared by generate and generate_streaming.
    /// @param tokens Input token sequence (already tokenized).
    /// @param params Generation parameters.
    /// @param on_token Per-token callback (nullptr for batch mode).
    /// @param cancel Cancel flag (nullptr for batch mode).
    /// @return GenerationResult.
    GenerationResult decode_loop(
        const std::vector<llama_token>& tokens,
        const GenerationParams& params,
        std::function<void(std::string_view)> on_token,
        std::atomic<bool>* cancel);

    /// @brief Sample next token using params (temperature, top_p, top_k, etc).
    llama_token sample(const GenerationParams& params) const;

    /// @brief Apply GBNF grammar constraint to sampler.
    void apply_grammar(
        const std::string& grammar_str,
        llama_sampler* smpl) const;

    /// @brief Create sampler chain from generation params.
    /// Owns the returned pointer (caller frees via llama_sampler_free).
    llama_sampler* create_sampler(const GenerationParams& params) const;
};

} // namespace entropic
```

**VRAM lifecycle — llama.cpp C API mapping:**

| State | llama.cpp resources | Transition call |
|---|---|---|
| COLD | none | — |
| WARM | `llama_model*` (mmap, mlock, n_gpu_layers=0) | `do_load()`: `llama_model_load_from_file()` |
| ACTIVE | `llama_model*` + `llama_context*` (KV cache on GPU) | `do_activate()`: `llama_context_new()` with `n_gpu_layers` from config |

**WARM → ACTIVE transition** reloads the model with GPU layers rather than
creating a context on the CPU-only model. This matches the Python behavior
where `_swap_model()` frees the old `Llama` instance and creates a new one
with different `n_gpu_layers`. The llama.cpp C API allows
`llama_model_load_from_file()` with different params — mlock ensures pages
stay resident, so the reload is fast (~1-3s PCIe transfer, no disk I/O).

Alternative considered: keep one `llama_model*` and create/destroy
`llama_context*` with different GPU layer counts. This would be cleaner
but llama.cpp ties GPU layer offloading to model load, not context creation.
The model-reload approach matches Python's proven behavior.

**`do_generate()` flow:**

1. `apply_chat_template()` — format messages using GGUF-embedded template
2. `tokenize()` — convert formatted prompt to tokens
3. `decode_loop()` — process prompt tokens, then auto-regressive generation
4. Build `GenerationResult` with content, token count, timing

**`do_generate_streaming()` flow:**

Same as `do_generate()` except `decode_loop()` calls `on_token` after each
generated token and checks `cancel` flag between iterations. No queue bridge
needed — C++ streaming is synchronous. The C boundary wraps this with a
raw function pointer callback.

**`do_complete()` flow:**

1. `tokenize()` — tokenize raw prompt directly (NO chat template)
2. `decode_loop()` — same loop, but prompt is raw text
3. Return `GenerationResult`

This is the critical distinction: `complete()` bypasses the chat template
entirely. The router uses this for digit classification where the prompt
ends with `"user message" ->` and the model continues with `"2"`.

**Per-request `enable_thinking` handling:**

The Python backend swaps chat handlers at runtime to inject
`enable_thinking=False` into the Jinja template. In C++, this is handled
by `apply_chat_template()` which passes `enable_thinking` as a template
variable to `llama_chat_apply_template()`. No handler swapping needed —
the template variable is per-call.

If `reasoning_budget == 0`, `enable_thinking` is set to `false`. If
`reasoning_budget > 0`, `enable_thinking` is `true` and the decode loop
monitors the think-block token count, forcibly emitting `</think>` when
the budget is exhausted.

### Phase 5: ChatAdapter Base Class

Concrete base class with 80% of parsing/formatting logic. Subclasses override
only model-specific behavior.

```cpp
/// @file include/entropic/inference/adapters/adapter_base.h
/// @brief ChatAdapter concrete base class.
///
/// @par Responsibilities (80% in base):
/// - System prompt assembly (identity + context + tools)
/// - Think-block extraction and stripping (<think>...</think>)
/// - Tagged tool call parsing (<tool_call>JSON</tool_call>)
/// - Bare JSON tool call parsing (fallback)
/// - Malformed JSON recovery
/// - Tool result formatting (default: plain text)
/// - Response completeness detection (think-aware)
/// - Logging
///
/// @par Subclass overrides (20%):
/// - parse_tool_calls() — model-specific parsing strategy
/// - _format_tools() — tool definition format
/// - format_tool_result() — if model needs special wrapping
/// - chat_format() — chat format identifier

#pragma once

#include <entropic/types/message.h>
#include <entropic/types/tool_call.h>

#include <string>
#include <string_view>
#include <unordered_set>
#include <utility>
#include <vector>

namespace entropic {

/// @brief Parsed tool call result: cleaned content + extracted calls.
struct ParseResult {
    std::string cleaned_content;
    std::vector<ToolCall> tool_calls;
};

class ChatAdapter {
public:
    /// @param tier_name Identity tier (e.g. "eng", "lead").
    /// @param identity_prompt Assembled identity prompt (from PromptManager).
    ChatAdapter(std::string tier_name, std::string identity_prompt);
    virtual ~ChatAdapter() = default;

    // ── System prompt assembly (base class, not overridden) ──

    /// @brief Assemble system prompt: identity + context + tools.
    /// @param base_prompt Application context (todo state, project info).
    /// @param tools MCP tool definitions as JSON objects.
    /// @return Assembled system prompt string.
    std::string format_system_prompt(
        const std::string& base_prompt,
        const std::vector<std::string>& tool_jsons) const;

    // ── Tool call parsing (subclass overrides) ──

    /// @brief Parse tool calls from model output.
    /// @param content Raw model output.
    /// @return ParseResult with cleaned content and extracted tool calls.
    virtual ParseResult parse_tool_calls(const std::string& content) const = 0;

    // ── Tool result formatting ──

    /// @brief Format a tool result as a user message.
    /// Default: "Tool `name` returned:\n\nresult\n\nContinue..."
    /// Override for model-specific wrapping (e.g. <tool_response> tags).
    virtual Message format_tool_result(
        const ToolCall& tool_call,
        const std::string& result) const;

    // ── Response completeness ──

    /// @brief Check if response represents task completion.
    /// Think-aware: think-only content without tool calls = still working.
    bool is_response_complete(
        const std::string& content,
        const std::vector<ToolCall>& tool_calls) const;

    /// @brief Chat format identifier (e.g. "chatml").
    /// Return empty string to use GGUF-embedded template.
    virtual std::string chat_format() const = 0;

protected:
    // ── Shared parsing primitives (available to subclasses) ──

    /// @brief Parse <tool_call>JSON</tool_call> tagged blocks.
    std::vector<ToolCall> parse_tagged_tool_calls(
        const std::string& content) const;

    /// @brief Parse bare JSON lines containing "name" key.
    std::vector<ToolCall> parse_bare_json_tool_calls(
        const std::string& content) const;

    /// @brief Extract and return <think>...</think> content.
    std::string extract_thinking(const std::string& content) const;

    /// @brief Strip all <think>...</think> blocks from content.
    std::string strip_think_blocks(const std::string& content) const;

    /// @brief Attempt JSON recovery on malformed tool call string.
    /// Tries: trailing comma removal, quote normalization, brace matching.
    std::optional<ToolCall> try_recover_json(const std::string& json_str) const;

    /// @brief Convert string value to typed JSON value.
    /// Handles: booleans, nulls, numbers, JSON arrays/objects.
    nlohmann::json convert_typed_value(const std::string& value) const;

    // ── Tool formatting (subclass can override) ──

    /// @brief Format tool definitions for system prompt.
    /// Default: markdown headings + JSON schema. Override for model-specific
    /// format (e.g. Qwen3.5 <tools> tags).
    virtual std::string format_tools(
        const std::vector<std::string>& tool_jsons) const;

    std::string tier_name_;
    std::string identity_prompt_;
    mutable std::unordered_set<std::string> tool_prefixes_;
};

} // namespace entropic
```

**Note on nlohmann/json in adapter:** The adapter is internal to the inference
`.so` — nlohmann/json does NOT appear in any interface header. The adapter
uses it internally for JSON tool parsing. At the `.so` boundary, tool calls
cross as `const char* json` strings.

### Phase 6: Qwen35Adapter

```cpp
/// @file src/inference/adapters/qwen35_adapter.h
/// @brief Qwen 3.5 chat adapter.
///
/// @par XML tool call format:
/// <tool_call>
/// <function=filesystem.read_file>
/// <parameter=path>/src/main.cpp</parameter>
/// </function>
/// </tool_call>
///
/// @par Tool definitions use OpenAI function format in <tools> tags.
/// @par Tool results use <tool_response>...</tool_response> wrapping.

#pragma once

#include <entropic/inference/adapters/adapter_base.h>

namespace entropic {

class Qwen35Adapter : public ChatAdapter {
public:
    using ChatAdapter::ChatAdapter;

    std::string chat_format() const override { return "chatml"; }

    /// @brief Parse XML function calls, fallback to tagged JSON.
    ParseResult parse_tool_calls(const std::string& content) const override;

    /// @brief Wrap tool result in <tool_response> tags.
    Message format_tool_result(
        const ToolCall& tool_call,
        const std::string& result) const override;

protected:
    /// @brief Format tools in <tools> tags with OpenAI function JSON.
    std::string format_tools(
        const std::vector<std::string>& tool_jsons) const override;

private:
    /// @brief Parse <function=name><parameter=key>value</parameter></function>.
    std::vector<ToolCall> parse_xml_function_calls(
        const std::string& content) const;

    /// @brief Extract <parameter=key>value</parameter> pairs from function body.
    std::unordered_map<std::string, nlohmann::json> extract_xml_parameters(
        const std::string& func_body) const;

    /// @brief Remove tool calls and think blocks from content.
    std::string clean_content(const std::string& content) const;
};

} // namespace entropic
```

**Parsing strategy mirrors Python exactly:**

1. Try `parse_xml_function_calls()` — regex `<function=([^>]+)>(.*?)</function>`
2. If no XML calls found, fallback to `parse_tagged_tool_calls()` (base class)
3. Clean content: strip `<tool_call>`, `<function=...>`, and `<think>` blocks

### Phase 8: ModelOrchestrator

Tier routing, model pool deduplication, VRAM lifecycle management. This is
internal to the inference `.so` — not exposed at the C boundary (the engine
loop in v1.8.4 uses it internally).

```cpp
/// @file include/entropic/inference/orchestrator.h
/// @brief ModelOrchestrator — multi-model lifecycle and routing.
///
/// @par Responsibilities:
/// - Model pool deduplication (one backend per unique .gguf path)
/// - Per-tier adapters (identity-specific, independent of shared backend)
/// - VRAM lifecycle: one ACTIVE main tier, router always ACTIVE
/// - Tier routing via router model (raw completion, digit classification)
/// - Handoff rule enforcement
/// - Swap logic: keep_warm → WARM, otherwise → COLD
///
/// @par Thread safety:
/// Uses std::mutex for tier swap operations. State queries on individual
/// backends are lock-free (atomic). Generation calls are not serialized
/// — multiple concurrent generations on the ACTIVE backend are allowed.

#pragma once

#include <entropic/inference/backend.h>
#include <entropic/inference/adapters/adapter_base.h>
#include <entropic/types/config.h>

#include <memory>
#include <mutex>
#include <string>
#include <unordered_map>
#include <vector>

namespace entropic {

/// @brief Result metadata from a routing decision.
struct RoutingResult {
    std::string tier_name;              ///< Selected tier
    std::string previous_tier;          ///< Previous tier (empty if first)
    std::string model_raw;              ///< Raw model output (e.g. "2")
    std::string swap_action = "none";   ///< "none", "reused", "loaded"
    double routing_ms = 0.0;            ///< Total routing time
};

class ModelOrchestrator {
public:
    /// @brief Initialize from config. Creates backends and adapters.
    /// @param config Full engine config (models, routing sections).
    /// @param prompt_mgr For identity prompt assembly.
    bool initialize(const EntropyConfig& config, /* PromptManager& */);

    /// @brief Shutdown — unload all models.
    void shutdown();

    // ── Generation ──

    /// @brief Generate using routed or explicit tier.
    /// @param messages Conversation history.
    /// @param params Generation parameters.
    /// @param tier_name Explicit tier, or empty for routing.
    /// @return GenerationResult with routing/swap timing.
    GenerationResult generate(
        const std::vector<Message>& messages,
        const GenerationParams& params,
        const std::string& tier_name = "");

    /// @brief Streaming generation.
    GenerationResult generate_streaming(
        const std::vector<Message>& messages,
        const GenerationParams& params,
        std::function<void(std::string_view)> on_token,
        std::atomic<bool>& cancel,
        const std::string& tier_name = "");

    // ── Routing ──

    /// @brief Route to appropriate tier using router model.
    /// @param messages Current conversation (extracts latest user message).
    /// @return Tier name.
    std::string route(const std::vector<Message>& messages);

    // ── Queries ──

    RoutingResult last_routing_result() const;
    std::string last_used_tier() const;
    std::vector<std::string> loaded_models() const;
    std::vector<std::string> available_models() const;
    bool can_handoff(const std::string& from, const std::string& to) const;
    ChatAdapter* get_adapter(const std::string& tier_name) const;

private:
    // ── Model pool (one backend per unique path) ──
    std::unordered_map<std::string, std::shared_ptr<InferenceBackend>> model_pool_;

    // ── Tier → backend mapping (many-to-one via shared_ptr) ──
    std::unordered_map<std::string, std::shared_ptr<InferenceBackend>> tiers_;

    // ── Per-tier adapters (one-to-one, identity-specific) ──
    std::unordered_map<std::string, std::unique_ptr<ChatAdapter>> adapters_;

    // ── Router (separate small model, always ACTIVE) ──
    std::shared_ptr<InferenceBackend> router_;

    // ── Routing state ──
    std::unordered_map<std::string, std::string> tier_map_;  ///< digit → tier name
    std::unordered_map<std::string, std::unordered_set<std::string>> handoff_rules_;
    std::string default_tier_;
    std::string loaded_main_tier_;
    RoutingResult last_routing_result_;
    std::vector<std::string> tier_history_;  ///< Recent tiers, max 5

    std::mutex swap_mutex_;  ///< Guards tier swap operations

    // ── Internal ──

    /// @brief Get model for tier, loading/swapping as needed.
    InferenceBackend* get_model(const std::string& tier_name);

    /// @brief Find already-loaded backend reusable for requested tier.
    /// With dedup, tiers sharing same .gguf share same backend instance.
    InferenceBackend* find_reusable(const std::string& tier_name);

    /// @brief Deactivate current main tier (WARM or COLD per keep_warm).
    void deactivate_current_if_needed(InferenceBackend* incoming);

    /// @brief Classify task using router model (raw completion).
    std::pair<std::string, std::string> classify_task(
        const std::vector<Message>& messages);

    /// @brief Extract latest user message from history.
    static std::string extract_latest_user_message(
        const std::vector<Message>& messages);
};

} // namespace entropic
```

**Model pool deduplication:**

Multiple tiers can reference the same `.gguf` file (e.g., `eng` and `qa`
both use `Qwen3.5-35B-A3B-UD-IQ3_XXS.gguf`). The orchestrator creates ONE
`LlamaCppBackend` instance per unique resolved path and stores it in
`model_pool_`. The `tiers_` map stores `shared_ptr` to the same backend.
This avoids duplicate VRAM compute buffer allocation (~1.1GB each).

Each tier still gets its own `ChatAdapter` instance — adapters are
identity-specific (different system prompts, different tool sets) even when
the underlying model is shared.

**Swap logic (mirrors Python `_deactivate_current_if_needed`):**

When switching from tier A to tier B and they use DIFFERENT backends:
- If A's config has `keep_warm=true`: A goes ACTIVE → WARM (fast reactivation)
- If A's config has `keep_warm=false`: A goes ACTIVE → COLD (full unload)

When switching between tiers sharing the SAME backend: no swap needed,
the backend is already ACTIVE. Log "reused" in RoutingResult.

**Router:**
- Separate `InferenceBackend` instance (small model, e.g. Qwen3-0.6B-Q8_0)
- Always ACTIVE — never swapped out
- Uses `complete()` (raw text, no chat template) for digit classification
- Classification prompt built from tier names/focus using `build_classification_prompt()`
  (ported from Python prompts module in v1.8.1)

### Phase 9: C Boundary Wrappers

Thin wrappers that bridge the C interface to the C++ implementation.
Exception catching at every boundary.

```cpp
/// @file src/inference/inference_c_api.cpp
/// @brief C API wrapper for InferenceBackend.

extern "C" {

ENTROPIC_EXPORT entropic_error_t entropic_inference_generate(
    entropic_inference_backend_t backend,
    const char* messages_json,
    const char* params_json,
    char** result_json)
{
    try {
        auto* impl = reinterpret_cast<entropic::InferenceBackend*>(backend);
        auto messages = entropic::parse_messages_json(messages_json);
        auto params = entropic::parse_params_json(params_json);
        auto result = impl->generate(messages, params);
        *result_json = entropic::serialize_result_json(result);
        return ENTROPIC_OK;
    } catch (const std::exception& e) {
        entropic::set_last_error(backend, e.what());
        return ENTROPIC_ERROR_GENERATE_FAILED;
    }
}

ENTROPIC_EXPORT entropic_error_t entropic_inference_generate_streaming(
    entropic_inference_backend_t backend,
    const char* messages_json,
    const char* params_json,
    void (*on_token)(const char* token, size_t len, void* user_data),
    void* user_data,
    int* cancel_flag)
{
    try {
        auto* impl = reinterpret_cast<entropic::InferenceBackend*>(backend);
        auto messages = entropic::parse_messages_json(messages_json);
        auto params = entropic::parse_params_json(params_json);

        // Bridge: C int* cancel_flag → C++ std::atomic<bool>&
        std::atomic<bool> cancel{false};

        auto cpp_callback = [on_token, user_data, cancel_flag, &cancel]
            (std::string_view token) {
                on_token(token.data(), token.size(), user_data);
                if (cancel_flag && *cancel_flag) {
                    cancel.store(true, std::memory_order_release);
                }
            };

        impl->generate_streaming(messages, params, cpp_callback, cancel);
        return ENTROPIC_OK;
    } catch (const std::exception& e) {
        entropic::set_last_error(backend, e.what());
        return ENTROPIC_ERROR_GENERATE_FAILED;
    }
}

} // extern "C"
```

**Streaming cancel bridge:** The C boundary uses `int* cancel_flag` (C-safe).
The wrapper creates a local `std::atomic<bool>` and checks the C flag in
the token callback, propagating cancellation to the C++ atomic. This means
cancellation latency is one token (checked after each token emission).

## Architecture-Agnostic Design Note

The `InferenceBackend` base class and `ChatAdapter` base class are designed
to be architecture-agnostic. While `LlamaCppBackend` is the only concrete
implementation in v1.8.2, the interfaces accommodate:

- **Transformer models**: standard token-by-token generation (current)
- **GDN/recurrent models** (Falcon-H1): same generate API, different
  internal state management (hidden state vs KV cache). The `do_generate`
  virtual allows recurrent backends to manage state differently.
- **NPU backends** (Axera AX630C): different model format (.axmodel vs
  .gguf), but same `load/activate/generate` lifecycle.

The `complete()` method is particularly important for architecture
agnosticism — it provides raw text completion without assuming chat template
structure, which may differ across architectures.

Backend abstraction is formally addressed in v1.9.13, but the interfaces
designed here must not preclude those future backends.

## Versioned Backend Pattern

`LlamaCppBackend` MUST be implemented as a versioned subclass from day one.
Name it against the pinned llama.cpp submodule commit (e.g.,
`LlamaCppBackend_b4979` or whatever the hash is). This establishes the
pattern for future llama.cpp upgrades — a new subclass overrides only the
changed methods, the old subclass compiles clean against the old pin.

```
InferenceBackend (80% — lifecycle, state, metrics, threading)
  └─ LlamaCppBackend (15% — common llama.cpp patterns, decode loop)
       └─ LlamaCppBackend_bXXXX (5% — pinned-commit-specific API calls)
```

CMake selects the subclass based on submodule version:

```cmake
# Set by llama.cpp submodule version detection
if(LLAMA_CPP_API_VERSION GREATER_EQUAL 500)
    set(LLAMA_BACKEND_SRC "src/inference/llama_cpp_backend_v500.cpp")
else()
    set(LLAMA_BACKEND_SRC "src/inference/llama_cpp_backend_v400.cpp")
endif()
```

The first versioned subclass IS the example. When llama.cpp upgrades later,
the diff between subclasses shows exactly what changed.

**Note on hierarchy depth:** This is four layers (C interface → InferenceBackend
→ LlamaCppBackend → LlamaCppBackend_bXXXX), which is deeper than the
architecture's three-layer rule (#4). This is acceptable because
`LlamaCppBackend` and its versioned subclass are both internal to the same
`.so` — they don't add a boundary. The C interface still sees one opaque
handle. This is implementation layering, not architectural layering.

## Implementation Notes (from review)

The implementing session MUST address these during development:

1. **Verify `llama_chat_apply_template()` exists** in the pinned llama.cpp
   commit. The exact API may differ from what's written here. Check the
   actual submodule revision.

2. **Sampler chain assembly order matters.** Grammar is applied first per
   llama.cpp PR #17937. The chain: grammar → penalties → temperature →
   top-k → top-p. The Python wrapper hid this — C++ must build it manually.

3. **`llama_batch` lifecycle for prompt processing.** Batched prefill
   (`llama_batch_init` → `llama_decode` → `llama_batch_free`) is needed
   for efficient prompt processing vs single-token generation. The Python
   wrapper hid this detail.

4. **Token count mode in C interface Doxygen.** `entropic_inference_count_tokens`
   returns exact count when model is loaded, `len/4` estimate otherwise.
   The C header Doxygen must document which mode was used so callers know
   the accuracy of the result.

5. **Memory ownership on C streaming callback.** The `on_token` callback
   receives `const char* token, size_t len` — caller MUST copy if they
   need to keep it beyond the callback. This ownership contract must be
   explicit in the C header Doxygen comment.

6. **Model failure mid-streaming.** If the backend errors during streaming
   generation (model crash, CUDA OOM, decode failure): partial content
   generated so far is appended to context as-is. `GenerationResult` gets
   an `error` field (error code + message) alongside `content` (partial).
   The engine's error counter increments. Iteration continues (the engine
   can retry or handle the error). The streaming callback receives no
   further tokens after the error. The cancel flag is NOT set — error and
   cancel are distinct conditions.
