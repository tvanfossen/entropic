## Scope

Engine introspection and self-diagnosis tools for the Entropic engine. After
v1.9.12, the model can read its own configuration, loaded identities, tool
definitions, recent action history, and engine state. A `/diagnose` command
triggers model self-reflection on previous actions, identifying reasoning
errors and flagging potential engine-related root causes.

This builds on:
- **v1.8.5** (Tool System): `EntropicServer` in `librentropic-mcp-entropic.so`,
  `MCPServerBase`, `ToolRegistry`, `ToolExecutor`, `ServerManager`
- **v1.8.6** (Delegation + Identity): `DelegationManager`, `IdentityFrontmatter`,
  `PhaseConfig`, identity resolution, delegation depth and pipeline awareness

**What v1.9.12 adds:**
- `entropic.diagnose` MCP tool: returns structured JSON snapshot of engine state
  including current config, loaded identities, available tools, recent tool call
  history, engine state, and metrics. The model uses this to self-reflect on
  failures and identify reasoning errors or engine-related root causes.
- `entropic.inspect` MCP tool: narrower queries for specific engine state
  (single identity, single tool definition, specific config section, specific
  metric). Avoids flooding context with a full snapshot when the model only
  needs one piece of information.
- Doxygen-generated architecture documentation bundled as a data file so the
  model can reference engine architecture when diagnosing issues.
- `/diagnose` command integration: consumer-facing command that injects a
  diagnostic prompt instructing the model to call `entropic.diagnose`, analyze
  its recent actions, and produce structured self-assessment.

**What v1.9.12 does NOT include:**
- State mutation (all tools are read-only introspection)
- Runtime config modification (read config, don't write it)
- Automated remediation (model reflects and suggests, human decides)
- External monitoring integration (Prometheus, Grafana export)
- Audit log analysis (v1.9.5 provides raw data; v1.9.12 reads engine state,
  not JSONL files)

**What v1.9.12 adds beyond the Python codebase:**
The Python engine has no introspection tools. Models cannot read their own
configuration or tool definitions. Diagnosis is entirely manual (developer
inspects logs). v1.9.12 gives the model the same observability that the
developer has, enabling self-diagnosis within the agentic loop.

## Architecture

### Tool Placement

Both tools live in `librentropic-mcp-entropic.so` alongside the existing
entropic server tools (todo, delegate, pipeline, complete, phase_change,
prune_context). They are registered as additional `ToolBase` subclasses
on `EntropicServer`.

```
librentropic-mcp-entropic.so
├── EntropicServer (MCPServerBase subclass)
│   ├── TodoTool
│   ├── DelegateTool
│   ├── PipelineTool
│   ├── CompleteTool
│   ├── PhaseChangeTool
│   ├── PruneContextTool
│   ├── DiagnoseTool      ← NEW (v1.9.12)
│   └── InspectTool        ← NEW (v1.9.12)
```

### Data Flow

```
Model generates:
  → tool_call: entropic.diagnose({})
  → EntropicServer::execute("diagnose", "{}")
    → DiagnoseTool::execute("{}")
      → reads engine state via EngineStateProvider
      → assembles DiagnoseSnapshot struct
      → serializes to JSON
    → ServerResponse { result: "<JSON snapshot>", directives: [] }
  → result appended to context as tool result message
  → model generates analysis based on snapshot
```

### EngineStateProvider Interface

The diagnose and inspect tools need read access to engine internals that
live across multiple `.so` boundaries (config in `librentropic-config.so`,
identity in `librentropic-core.so`, tools in `librentropic-mcp.so`, metrics
in `librentropic-core.so`). Rather than coupling the entropic MCP server
to every other library, the engine facade assembles a read-only state
snapshot and passes it to `EntropicServer` at construction time.

```
┌────────────────────────────┐
│      librentropic.so       │  Facade — assembles snapshot provider
│      (entropic.h)          │
└─────────┬──────────────────┘
          │ constructs
          ▼
┌────────────────────────────┐
│ EngineStateProvider        │  Read-only interface, owned by facade
│ (callback struct)          │
│                            │
│ get_config()    → JSON     │  Config snapshot from librentropic-config
│ get_identities() → JSON    │  Loaded identities from PromptManager
│ get_tools()     → JSON     │  Tool list from ServerManager
│ get_history()   → JSON     │  Recent tool calls from ToolExecutor
│ get_state()     → JSON     │  Engine state from AgentEngine
│ get_metrics()   → JSON     │  Metrics from AgentEngine
│ get_docs()      → string   │  Doxygen content from bundled data file
└────────────────────────────┘
          │ passed to
          ▼
┌────────────────────────────┐
│ EntropicServer             │  librentropic-mcp-entropic.so
│   DiagnoseTool             │
│   InspectTool              │
└────────────────────────────┘
```

The `EngineStateProvider` is a struct of C function pointers, keeping
the `.so` boundary pure C. No C++ types cross the boundary.

```c
/// @brief Read-only engine state provider for introspection tools.
/// @version 1.9.12
///
/// Callback struct passed to EntropicServer at construction time.
/// Each callback returns a JSON string (caller must free with
/// entropic_free()). The facade implements these callbacks by
/// querying the appropriate subsystem.
typedef struct {
    /// @brief Get current engine configuration as JSON.
    char* (*get_config)(void* user_data);

    /// @brief Get loaded identities as JSON array.
    char* (*get_identities)(void* user_data);

    /// @brief Get available tools as JSON array.
    char* (*get_tools)(void* user_data);

    /// @brief Get recent tool call history as JSON array.
    /// @param max_entries Maximum number of recent entries to return (0 = all).
    char* (*get_history)(int max_entries, void* user_data);

    /// @brief Get engine state as JSON (agent state, delegation depth, etc.).
    char* (*get_state)(void* user_data);

    /// @brief Get engine metrics as JSON.
    char* (*get_metrics)(void* user_data);

    /// @brief Get bundled Doxygen documentation as text.
    /// @param section Section name to retrieve (NULL = full doc).
    char* (*get_docs)(const char* section, void* user_data);

    /// @brief Opaque user data pointer passed to all callbacks.
    void* user_data;
} entropic_state_provider_t;
```

### No New `.so` Boundary

Diagnose and inspect tools add to the existing `librentropic-mcp-entropic.so`.
The `EngineStateProvider` crosses the `.so` boundary as a C callback struct
(passed at server construction via `entropic_mcp_server_configure()`). This
follows the existing pattern established in v1.8.5.

## /diagnose Command Integration

The `/diagnose` command is a consumer-side entry point (the TUI or CLI
issues it). It works by injecting a specially-crafted user message that
instructs the model to use `entropic.diagnose` and produce structured
self-assessment.

### Injected Prompt

```
[SYSTEM DIRECTIVE: SELF-DIAGNOSIS]

Analyze your recent actions and identify any issues. Follow these steps:

1. Call entropic.diagnose to get a full engine state snapshot.
2. Review the tool call history for:
   - Repeated failures (same tool, same error)
   - Duplicate tool calls (circuit breaker risk)
   - Tool calls that returned errors
   - Unexpected state (wrong phase, wrong tier)
3. Review your reasoning for:
   - Actions that didn't achieve the stated goal
   - Unnecessary tool calls
   - Missing context that led to errors
4. Produce a structured assessment:
   - FINDINGS: What went wrong (be specific)
   - ROOT CAUSE: Why it went wrong (reasoning error vs engine issue vs config issue)
   - RECOMMENDATION: What to do differently

Be honest and specific. The goal is accurate self-assessment, not self-defense.
```

### Consumer Integration

The `/diagnose` command is NOT part of `librentropic-mcp-entropic.so`.
It is a consumer-side feature. The engine provides the tools; the consumer
provides the command.

```
Consumer (TUI/CLI)
    │
    │ User types /diagnose
    │
    ▼
Consumer injects diagnostic prompt as user message
    │
    ▼
Engine processes normally:
    model sees diagnostic prompt
    → calls entropic.diagnose
    → receives engine state snapshot
    → generates self-assessment
    │
    ▼
Consumer receives assessment as normal assistant response
```

The engine provides a convenience function to generate the diagnostic
prompt text, so consumers don't hardcode it:

```c
/// @brief Get the diagnostic prompt text for /diagnose command.
/// @param handle Engine handle.
/// @param prompt_out Output: diagnostic prompt string. Caller must free.
/// @return ENTROPIC_OK on success.
/// @version 1.9.12
ENTROPIC_EXPORT entropic_error_t entropic_get_diagnostic_prompt(
    entropic_handle_t handle,
    char** prompt_out);
```

## Recent Tool Call History

`entropic.diagnose` and `entropic.inspect(target="history")` need access
to recent tool call history. This is NOT the v1.9.5 audit log (which
requires v1.8.8 and v1.9.1 as dependencies). Instead, v1.9.12 uses a
lightweight in-memory ring buffer maintained by `ToolExecutor`.

### ToolCallRecord

```cpp
/// @brief Lightweight record of a recent tool call for introspection.
/// @version 1.9.12
///
/// Stored in an in-memory ring buffer by ToolExecutor. This is NOT
/// the audit log (v1.9.5) — it is a fixed-size window of recent
/// calls for diagnostic introspection only. No persistence.
struct ToolCallRecord {
    size_t sequence;                    ///< Monotonic sequence number
    std::string tool_name;              ///< Fully-qualified tool name
    std::string params_summary;         ///< Condensed params (keys only, no large values)
    std::string status;                 ///< "success", "error", "timeout"
    std::string result_summary;         ///< First 200 chars of result (for context)
    double elapsed_ms;                  ///< Execution time
    std::string error_detail;           ///< Error message if status != "success"
    int iteration;                      ///< Loop iteration when called
};
```

### Ring Buffer in ToolExecutor

```cpp
/// @brief Fixed-size ring buffer of recent tool calls for introspection.
/// @version 1.9.12
///
/// Capacity set at construction (default 100). When full, oldest entries
/// are overwritten. Thread-safe reads via shared_mutex (multiple readers,
/// single writer).
class ToolCallHistory {
public:
    explicit ToolCallHistory(size_t capacity = 100);

    /// @brief Record a tool call (called by ToolExecutor after execution).
    void record(const ToolCallRecord& entry);

    /// @brief Get the N most recent entries (newest first).
    std::vector<ToolCallRecord> recent(size_t count) const;

    /// @brief Get all entries (up to capacity).
    std::vector<ToolCallRecord> all() const;

    /// @brief Current number of stored entries.
    size_t size() const;

private:
    std::vector<ToolCallRecord> buffer_;
    size_t head_ = 0;
    size_t count_ = 0;
    size_t capacity_;
    mutable std::shared_mutex mutex_;
};
```

This is distinct from duplicate detection (which uses a hash map of
tool_name+args for dedup). The history ring buffer stores sequential
records with timing and status for diagnostic purposes.

## Implementation Plan

### Phase 1: ToolCallHistory Ring Buffer

In-memory ring buffer for recent tool call tracking. Independent of
all other components — can be tested in isolation.

```cpp
/// @brief Fixed-size ring buffer of recent tool calls.
/// @version 1.9.12
class ToolCallHistory {
public:
    /// @brief Construct with buffer capacity.
    /// @param capacity Maximum entries to retain (default 100).
    /// @version 1.9.12
    explicit ToolCallHistory(size_t capacity = 100);

    /// @brief Record a completed tool call.
    /// @param entry Tool call record.
    /// @version 1.9.12
    void record(const ToolCallRecord& entry);

    /// @brief Get the N most recent entries (newest first).
    /// @param count Maximum entries to return.
    /// @return Vector of records, newest first.
    /// @version 1.9.12
    std::vector<ToolCallRecord> recent(size_t count) const;

    /// @brief Get all stored entries (oldest first).
    /// @return Vector of all records.
    /// @version 1.9.12
    std::vector<ToolCallRecord> all() const;

    /// @brief Serialize recent entries to JSON array string.
    /// @param count Maximum entries to include.
    /// @return JSON array of ToolCallRecord objects.
    /// @version 1.9.12
    std::string to_json(size_t count) const;

    /// @brief Current number of stored entries.
    /// @version 1.9.12
    size_t size() const;

private:
    std::vector<ToolCallRecord> buffer_;
    size_t head_ = 0;
    size_t count_ = 0;
    size_t capacity_;
    mutable std::shared_mutex mutex_;
};
```

**Files:**
- `include/entropic/mcp/tool_call_history.h` (new)

### Phase 2: ToolExecutor Integration

Wire `ToolCallHistory` into `ToolExecutor`. After each tool execution,
record a `ToolCallRecord`. The history is owned by ToolExecutor and
exposed via a const reference for reading.

Changes to `ToolExecutor`:
```cpp
// New member
ToolCallHistory history_;

// New accessor
const ToolCallHistory& history() const { return history_; }

// In execute_tool(), after execution:
history_.record({
    .sequence = next_sequence_++,
    .tool_name = call.name,
    .params_summary = summarize_params(call.args_json),
    .status = result_status,
    .result_summary = truncate(result_content, 200),
    .elapsed_ms = elapsed_ms,
    .error_detail = error_msg,
    .iteration = ctx.metrics.iterations
});
```

`summarize_params()` extracts only the top-level keys from the params
JSON, discarding large values. This keeps the history lightweight while
preserving enough context for diagnosis.

**Files:**
- `include/entropic/mcp/tool_executor.h` (modified — add history_ member and accessor)
- `src/mcp/tool_executor.cpp` (modified — record after execution)

### Phase 4: DiagnoseTool + InspectTool

Implement the two new tools as `ToolBase` subclasses registered on
`EntropicServer`.

`DiagnoseTool::execute(args_json)`:
```
1. Parse args: include_docs (bool), history_limit (int)
2. Call state_provider.get_state() → engine state JSON
3. Call state_provider.get_config() → config JSON
4. Call state_provider.get_identities() → identities JSON
5. Call state_provider.get_tools() → tools JSON
6. Call state_provider.get_history(history_limit) → history JSON
7. Call state_provider.get_metrics() → metrics JSON
8. IF include_docs: call state_provider.get_docs(NULL) → docs text
9. Assemble DiagnoseSnapshot JSON
10. Return as ServerResponse.result
```

`InspectTool::execute(args_json)`:
```
1. Parse args: target (string), key (optional string)
2. Switch on target:
   - "config": state_provider.get_config(), optionally filter by key
   - "identity": state_provider.get_identities(), filter by key (tier name)
   - "tool": state_provider.get_tools(), filter by key (tool name)
   - "state": state_provider.get_state()
   - "metrics": state_provider.get_metrics()
   - "history": state_provider.get_history(parse_int(key) or 10)
   - "docs": state_provider.get_docs(key)
3. Return filtered result as ServerResponse.result
```

Both tools return empty `directives` arrays (read-only, no side effects).

**Files:**
- `src/mcp/servers/entropic_server.cpp` (modified — add DiagnoseTool, InspectTool)
- `include/entropic/mcp/servers/entropic_server.h` (modified — declare new tools)

### Phase 6: Facade Wiring

Implement the `EngineStateProvider` callbacks in the facade. Each callback
queries the appropriate subsystem and returns a JSON string.

```cpp
// In facade initialization:
entropic_state_provider_t provider = {};
provider.user_data = engine_context;
provider.get_config = [](void* ud) -> char* {
    auto* ctx = static_cast<EngineContext*>(ud);
    return strdup(ctx->config_loader->to_json().c_str());
};
provider.get_identities = [](void* ud) -> char* {
    auto* ctx = static_cast<EngineContext*>(ud);
    return strdup(ctx->prompt_manager->identities_json().c_str());
};
provider.get_tools = [](void* ud) -> char* {
    auto* ctx = static_cast<EngineContext*>(ud);
    return strdup(ctx->server_manager->list_tools().c_str());
};
provider.get_history = [](int max_entries, void* ud) -> char* {
    auto* ctx = static_cast<EngineContext*>(ud);
    return strdup(ctx->tool_executor->history().to_json(max_entries).c_str());
};
provider.get_state = [](void* ud) -> char* {
    auto* ctx = static_cast<EngineContext*>(ud);
    return strdup(ctx->engine->state_json().c_str());
};
provider.get_metrics = [](void* ud) -> char* {
    auto* ctx = static_cast<EngineContext*>(ud);
    return strdup(ctx->engine->metrics_json().c_str());
};
provider.get_docs = [](const char* section, void* ud) -> char* {
    auto* ctx = static_cast<EngineContext*>(ud);
    return strdup(ctx->docs_loader->get_section(section).c_str());
};
```

The provider is passed to `EntropicServer` via
`entropic_mcp_server_configure()` as part of the config JSON
(serialized as an opaque pointer field, or via a dedicated
`set_state_provider()` call before initialization).

**Files:**
- `src/facade/entropic.cpp` (modified — populate EngineStateProvider callbacks)

```c
/// @brief Get the diagnostic prompt text for /diagnose command.

///        Caller must free with entropic_free().
/// @return ENTROPIC_OK on success.
/// @version 1.9.12
ENTROPIC_EXPORT entropic_error_t entropic_get_diagnostic_prompt(
    entropic_handle_t handle,
    char** prompt_out);
```

No other C API additions are needed. The `entropic.diagnose` and
`entropic.inspect` tools are invoked through the normal tool execution
path (`entropic_mcp_server_execute()`). Consumers trigger `/diagnose`
by injecting the diagnostic prompt as a user message and running the
engine loop normally.

## Validation Criteria

### V1: diagnose returns complete snapshot

Call `entropic.diagnose({})`. Verify response contains all sections:
engine, config, identities, tools, history, metrics. Verify `docs`
is null when `include_docs` is false.

### V2: diagnose with docs

Call `entropic.diagnose({"include_docs": true})`. Verify `docs` field
is non-null and contains subsystem sections.

### V3: diagnose history_limit

Record 50 tool calls. Call `entropic.diagnose({"history_limit": 5})`.
Verify history array has exactly 5 entries (most recent).

### V4: inspect config

Call `entropic.inspect({"target": "config"})`. Verify full config
returned. Call with `key: "model"`. Verify only model section returned.

### V5: inspect identity by name

Call `entropic.inspect({"target": "identity", "key": "eng"})`. Verify
eng identity returned. Call with non-existent key. Verify error message.

### V6: inspect tool by name

Call `entropic.inspect({"target": "tool", "key": "filesystem.read_file"})`.
Verify tool definition returned. Call with non-existent key. Verify
error message.

### V7: inspect state

Call `entropic.inspect({"target": "state"})`. Verify agent_state,
delegation_depth, active_phase, iteration present.

### V8: inspect metrics

Call `entropic.inspect({"target": "metrics"})`. Verify total_iterations,
total_tool_calls, total_tool_errors present.

### V9: inspect history

Call `entropic.inspect({"target": "history", "key": "3"})`. Verify
3 most recent entries returned.

### V10: inspect docs section

Call `entropic.inspect({"target": "docs", "key": "inference"})`.
Verify only inference section returned.

### V11: ToolCallHistory ring buffer

Record 150 entries into a history with capacity 100. Verify size() is
100. Verify oldest 50 entries were overwritten. Verify `recent(10)`
returns the 10 newest entries in newest-first order.

### V12: ToolCallHistory thread safety

Record entries from 4 concurrent threads. Read concurrently. Verify
no data races (run under ThreadSanitizer).

### V13: ToolCallHistory params_summary

Record a tool call with large params (10KB JSON). Verify
`params_summary` contains only top-level keys, not full values.
Verify `result_summary` is truncated to 200 chars.

### V14: EngineStateProvider callbacks

Verify each callback returns valid JSON parseable by
`nlohmann::json::parse()`. Verify `get_history(0)` returns all
entries. Verify `get_docs("inference")` returns only the inference
section.

### V15: Diagnostic prompt

Call `entropic_get_diagnostic_prompt()`. Verify non-null string
returned. Verify prompt contains "entropic.diagnose" instruction.

### V16: No state mutation

Call `entropic.diagnose` and `entropic.inspect` repeatedly. Verify
engine state is unchanged after each call (no side effects). Verify
empty directives array in ServerResponse.

### V17: diagnose during delegation

Execute diagnose from a child context at delegation_depth=1. Verify
snapshot shows correct delegation_depth, locked_tier, and child-scoped
tool history.

### V18: SLOC <= 50 per function

All functions pass knots quality gates.

### V19: Doxygen complete

Every public symbol in new/modified headers has `@brief`, `@param`,
`@return`, `@version`.
