# TUI Overhaul: Full-Featured Reference Implementation

## Problem Statement

The current TUI is a basic chat interface with streaming, tool approval,
and voice mode. It doesn't expose most of the engine's capabilities:

| Engine Capability | Current TUI Support |
|-------------------|-------------------|
| Identity system (13 identities) | None — shows tier name only |
| Grammar-constrained output | None — no grammar visualization |
| Multi-stage pipelines | None — no pipeline progress |
| VRAM warm/active states | None — no resource visibility |
| Model swap / identity swap | None — invisible to user |
| Runtime MCP servers | None — no management UI |
| GPU resource profiles | None — no tuning UI |
| Time-based generation caps | None — no time budget visibility |
| Pipeline inspector/debugging | None |
| Interstitial pruner | None — invisible infrastructure |

The TUI should be the **full-featured reference implementation** that
exercises every engine feature. It's the "here it is working" proof
that the library delivers, and the test bed for validating that all
features compose correctly.

## What Changes After Prerequisite Work

### After P1-022 (VRAM Orchestration)

The engine exposes `ResourceSnapshot`:
```python
@dataclass
class ResourceSnapshot:
    total_vram_mb: int
    used_vram_mb: int
    active_model: str | None
    active_gpu_layers: int
    warm_models: list[str]
    warm_ram_mb: int
    kv_cache_mb: int
```

TUI needs: VRAM dashboard widget showing GPU utilization, warm models,
active model, KV cache pressure.

### After P1-024 (Identity Library)

13 identities with pipeline composition. TUI needs:
- Active identity indicator (not just tier name)
- Pipeline progress bar (which stage, how many remaining)
- Grammar-constrained output formatting (structured vs free-form)
- Identity-specific styling (different colors/icons per cognitive mode)

### After P2-019 (Engine Decomposition)

Clean subsystem boundaries. TUI's Presenter interface maps cleanly to
subsystem events:
- ToolExecutor events → tool approval/completion widgets
- ContextManager events → context bar, compaction indicator
- ResponseGenerator events → streaming display, tier selection

### After P2-026 (Runtime MCP)

Runtime server management. TUI needs:
- Server list (name, status, tool count)
- Connect/disconnect commands
- Health status indicators

### After P2-020 + P2-021 (GPU Tuning)

Resource profiles and time caps. TUI needs:
- Active resource profile indicator
- Per-generation time budget / elapsed display
- tok/s readout

| Zone | Content | Widget |
|------|---------|--------|

### New Widgets

| Widget | Purpose | Source data |
|--------|---------|-------------|
| `PipelineStrip` | Show active pipeline: stages completed, current, remaining | Engine auto_chain events |
| `VRAMBar` | VRAM utilization bar with model breakdown | `ResourceSnapshot` from orchestrator |
| `ModelStateList` | List models with state indicators (●/○/·) | Orchestrator model states |
| `MCPServerList` | Runtime + built-in servers with health status | `ServerManager.list_servers()` |
| `IdentityBadge` | Per-message identity indicator with color coding | Message metadata |
| `TimeBudgetBar` | Generation time budget vs elapsed | ThroughputTracker data |
| `TokSecReadout` | Rolling tok/s display | ThroughputTracker data |

### Pipeline Visualization

When a multi-stage pipeline runs, the `PipelineStrip` shows:

```
[diagnoser] → [pruner] → [planner] → code_writer → validator
   done         done       done        ▶ active      pending
```

Each stage transition triggers a strip update. The strip persists until
the pipeline completes or the user sends a new message.

### Identity-Attributed Messages

Each assistant message shows which identity produced it:

```
┌─ Assistant (planner) ─────────────────────┐
│ Steps:                                     │
│ 1. Read current validation in input.py     │
│ 2. Add type checking for "name" field      │
│ 3. Add empty string rejection              │
└────────────────────────────────────────────┘
```

Grammar-constrained identities show structured output with syntax
highlighting. Free-form identities show markdown as today.

### Resource Panel (Collapsible)

The bottom panel showing models and MCP servers is collapsible (toggle
with a keybinding). Default: collapsed. Shows summary in header bar
(VRAM usage, server count). Expanded: full model list and server list.

## Presenter Interface Changes

The Presenter ABC needs new methods for features from prerequisite work:

```python
class Presenter(ABC):
    # Existing methods preserved...

    # New: Identity + Pipeline
    @abstractmethod
    def on_identity_change(self, identity: str, tier: str) -> None: ...
    @abstractmethod
    def on_pipeline_stage(self, stage: int, total: int, identity: str) -> None: ...
    @abstractmethod
    def on_pipeline_complete(self) -> None: ...

    # New: VRAM + Resources
    @abstractmethod
    def on_resource_update(self, snapshot: ResourceSnapshot) -> None: ...
    @abstractmethod
    def on_model_state_change(self, model: str, state: str) -> None: ...

    # New: MCP Servers
    @abstractmethod
    def on_server_connected(self, name: str, tools: list[str]) -> None: ...
    @abstractmethod
    def on_server_disconnected(self, name: str) -> None: ...
    @abstractmethod
    def on_server_health_change(self, name: str, status: str) -> None: ...

    # New: Generation metrics
    @abstractmethod
    def on_generation_metrics(self, tok_s: float, elapsed_ms: int, budget_ms: int | None) -> None: ...
```

HeadlessPresenter implements all new methods as no-ops (logging only).

## Risks & Considerations

- **Widget complexity**: 7 new widgets plus layout changes. Risk of
  visual clutter. Mitigate: collapsible panels, sensible defaults
  (resource panel collapsed, pipeline strip only visible during pipelines).
- **Presenter bloat**: 10+ new methods on Presenter ABC. If this grows
  further, consider event-based dispatch (single `on_event(EventType, data)`
  method) instead of method-per-event.
- **Textual limitations**: Some visualizations (VRAM bar, pipeline strip)
  may require custom Textual widgets. Textual's reactive system handles
  updates well, but complex layouts can be fragile.
- **HeadlessPresenter maintenance**: Every new Presenter method needs
  a no-op in HeadlessPresenter. Constructor injection (from P2-019)
  helps — consumers can inject a custom presenter that only implements
  methods they care about.
