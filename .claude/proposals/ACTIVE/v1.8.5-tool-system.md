## Scope

MCP tool system: `librentropic-mcp.so` (base + registry + executor + permissions)
plus six built-in server plugins (`librentropic-mcp-{filesystem,bash,git,diagnostics,web,entropic}.so`).
After v1.8.5, the engine can generate a response containing tool calls, parse them,
route to the correct server, execute with permission checking and duplicate detection,
and feed results back into the generation loop. The tool-calling loop is closed.

This is a faithful port of:
- `src/entropic/mcp/servers/base.py` -- `BaseMCPServer`, `ServerResponse`, `load_tool_definition`
- `src/entropic/mcp/tools.py` -- `BaseTool`, `ToolRegistry`
- `src/entropic/mcp/manager.py` -- `ServerManager`: server lifecycle, tool routing, permission checking, `.mcp.json` discovery
- `src/entropic/core/tool_executor.py` -- `ToolExecutor`: approval flow, duplicate detection, circuit breaker, tier allowlist enforcement, bash_commands allowlist
- `src/entropic/mcp/servers/filesystem.py` -- `FilesystemServer`: read/write/edit/glob/grep/list_directory, read-before-write, size gate, LSP diagnostics on edit
- `src/entropic/mcp/servers/bash.py` -- `BashServer`: shell execution
- `src/entropic/mcp/servers/git.py` -- `GitServer`: status/diff/add/commit/branch/checkout/reset/log
- `src/entropic/mcp/servers/diagnostics.py` -- `DiagnosticsServer`: LSP client, diagnostic diffs
- `src/entropic/mcp/servers/web.py` -- `WebServer`: web_fetch, web_search
- `src/entropic/mcp/servers/entropic.py` -- `EntropicServer`: todo, delegate, pipeline, complete, phase_change, prune_context
- Tool JSON definitions under `src/entropic/data/tools/`

v1.8.4 provided the engine loop, directive processing, context management, and
compaction. v1.8.5 fills in the tool execution branch that v1.8.4 left as dead code.

**What this version does NOT include:**
- Delegation execution (v1.8.6) -- `DelegationManager`, worktrees, pipeline execution
- External MCP servers (v1.8.7) -- stdio/SSE transport to third-party servers
- Session persistence (v1.8.8) -- SQLite storage, permission persistence to disk
- Per-identity MCP authorization (v1.9.4) -- per-caller key sets

**What this version DOES include beyond the Python port:**
- `ContextAnchor` support as base class infrastructure on `ServerResponse` -- tools
  optionally declare an anchor key strategy, the base class emits the directive
  automatically. `read_file` is the first consumer: `key="file:{path}"`.
- Investigation of llama.cpp autoparser for tool call parsing (outcome documented,
  not necessarily adopted)

## Architecture

### Library Decomposition

```
librentropic-mcp.so              MCPServerBase + ToolRegistry + ToolExecutor
                                  + ServerManager + PermissionManager
    │
    ├── librentropic-mcp-filesystem.so    read/write/edit/glob/grep/list_directory
    ├── librentropic-mcp-bash.so          shell execution
    ├── librentropic-mcp-git.so           git operations
    ├── librentropic-mcp-diagnostics.so   LSP client + diagnostics
    ├── librentropic-mcp-web.so           web_fetch + web_search
    └── librentropic-mcp-entropic.so      todo/delegate/pipeline/complete/phase_change/prune
```

Each plugin `.so` exports:
```c
extern "C" ENTROPIC_EXPORT int entropic_plugin_api_version();
extern "C" ENTROPIC_EXPORT entropic_mcp_server_t entropic_create_server();
```

For static builds (`ENTROPIC_STATIC=ON`), the CMake-generated
`entropic_static_plugins.cpp` registers enabled servers at compile time.

### ServerResponse Wire Format

All tool results cross the `.so` boundary as JSON:

```json
{
  "result": "Human-readable result text",
  "directives": [
    {"type": "context_anchor", "params": {"key": "file:src/main.cpp", "content": "..."}},
    {"type": "stop_processing"}
  ]
}
```

The engine's `DirectiveProcessor` (v1.8.4) already handles all directive types.
Tool results that have no directives return an empty `"directives": []` array.

### ContextAnchor on Tool Results

Base class infrastructure, not per-tool special case:

```
MCPServerBase::execute()
    1. Dispatch to registered ToolBase::execute()
    2. IF tool returns ServerResponse with directives: pass through
    3. IF tool declares anchor_key_strategy: auto-wrap result in ContextAnchor directive
    4. Serialize to JSON ServerResponse envelope
```

Tools opt in by overriding `anchor_key()`:

```cpp
/// @brief Generate anchor key for this tool result.
/// @param args_json Tool call arguments as JSON string.
/// @return Anchor key string, or empty string for no anchor.
/// @version 1.8.5
///
/// Override in tools whose results should REPLACE previous results
/// with the same key rather than accumulating in context.
/// Default returns "" (no anchoring -- results append normally).
///
/// Takes args as a JSON string (not nlohmann::json) to avoid exposing
/// nlohmann/json in the plugin authoring header. Third-party plugin
/// authors include server_base.h — keeping it free of third-party
/// types is consistent with the architecture's interface isolation rule.
virtual std::string anchor_key(
    const std::string& args_json) const { return ""; }
```

Filesystem `read_file` overrides: `return "file:" + arguments["path"].get<string>()`.
Subsequent reads of the same file replace the previous anchor. Other tools
(git.status, diagnostics) can adopt later by overriding `anchor_key()` with
no infrastructure changes.

### Permission Model

Default-deny. The `PermissionManager` evaluates tool calls against allow/deny
lists with glob pattern matching (fnmatch semantics).

```
Approval flow (ToolExecutor):
    1. auto_approve_tools config flag? → approve
    2. Explicitly in allow list (glob match)? → approve
    3. No approval callback registered (headless)? → approve
    4. Prompt user via callback → ToolApproval enum
    5. Always Allow / Always Deny → save to runtime permission list
       (persistence to disk deferred to v1.8.8)
```

Permission patterns support tool-level and argument-level granularity:
- `filesystem.*` -- all filesystem tools
- `bash.execute:python*` -- bash commands starting with "python"
- `git.commit` -- git commit specifically

Server subclasses override `get_permission_pattern()` for finer granularity.
The default returns `"server.tool_name"`.

### Duplicate Detection + Circuit Breaker

```
ToolExecutor::process_tool_calls():
    FOR each tool_call in batch:
        key = hash(tool_name + sorted_args_json)
        IF key in recent_tool_calls AND !server.skip_duplicate_check(tool_name):
            consecutive_duplicate_attempts++
            IF >= 3: inject "STOP: you are stuck" message, break
            ELSE: inject "already called, use previous result" message
            continue
        consecutive_duplicate_attempts = 0
        execute tool normally
        IF result is not error: record key → result in recent_tool_calls
```

Side-effect tools that must always execute override `skip_duplicate_check()`:
- `filesystem.read_file` -- updates FileAccessTracker state
- `entropic.delegate`, `entropic.pipeline` -- validation depends on runtime state

### Tier Allowlist Enforcement

Defense against hallucinated tool calls. Each tier's config specifies
`allowed_tools` (optional list). ToolExecutor rejects calls not in the list
before they reach the server. Additionally, `bash.execute` calls are checked
against the tier/phase `bash_commands` prefix allowlist.

## Implementation Plan

### Phase 1: MCPServerBase + ToolBase + ToolRegistry

Port `BaseMCPServer`, `BaseTool`, `ToolRegistry`, `ServerResponse`, `load_tool_definition`.

`MCPServerBase` is the concrete base class (80% logic):
- Owns a `ToolRegistry`
- `register_tool(ToolBase*)` registers tools
- `list_tools()` returns JSON array of tool definitions
- `execute(tool_name, args_json)` dispatches to registry, wraps result in
  ServerResponse JSON envelope, auto-injects ContextAnchor if tool declares one
- `get_permission_pattern()` -- default tool-level, virtual for override
- `skip_duplicate_check()` -- default false, virtual for override

`ToolBase` is the abstract base for individual tools:
- Loads definition from JSON file or inline definition
- `execute(args_json)` is pure virtual
- `anchor_key(args_json)` returns "" by default

`ToolRegistry`:
- `register(ToolBase*)` -- stores by name
- `has_tool(name)` -- lookup
- `get_tools()` -- returns all definitions as JSON array
- `dispatch(name, args_json)` -- routes to tool's execute()

`load_tool_definition(tool_name, server_prefix, data_dir)`:
- Reads JSON from `{data_dir}/{server_prefix}/{tool_name}.json`
- Validates required fields (name, inputSchema)
- Returns parsed JSON object

Tool JSON files are shared with the Python engine (same `data/tools/` directory).

### Phase 2: PermissionManager

Port the permission checking from `ServerManager._check_permission()`,
`_is_allowed()`, `_is_denied()`, `_pattern_matches()`.

```cpp
/// @brief Permission manager for MCP tool access control.
/// @version 1.8.5
///
/// Default-deny permission model. Tools must be explicitly allowed
/// or approved via callback. Deny list takes precedence over allow list.
class PermissionManager {
public:
    /// @brief Check if a tool call is explicitly denied.
    /// @param tool_name Fully-qualified tool name (e.g., "filesystem.read_file").
    /// @param pattern Tool pattern with args (e.g., "filesystem.read_file:/path").
    /// @return true if denied.
    /// @version 1.8.5
    bool is_denied(const std::string& tool_name,
                   const std::string& pattern) const;

    /// @brief Check if a tool call is explicitly allowed (skip prompting).
    /// @param tool_name Fully-qualified tool name.
    /// @param pattern Tool pattern with args.
    /// @return true if explicitly in allow list.
    /// @version 1.8.5
    bool is_allowed(const std::string& tool_name,
                    const std::string& pattern) const;

    /// @brief Add a permission pattern at runtime.
    /// @param pattern Permission pattern string.
    /// @param allow true for allow list, false for deny list.
    /// @version 1.8.5
    void add_permission(const std::string& pattern, bool allow);

private:
    std::vector<std::string> allow_list_;
    std::vector<std::string> deny_list_;

    /// @brief Check if a tool matches a permission pattern (fnmatch semantics).
    /// @version 1.8.5
    static bool pattern_matches(const std::string& tool_name,
                                const std::string& full_pattern,
                                const std::string& permission_pattern);
};
```

### Phase 5: Built-in Server — Filesystem

Port `filesystem.py`. The largest server. Six tools: `read_file`, `write_file`,
`edit_file`, `glob`, `grep`, `list_directory`.

Key behaviors:
- **Read-before-write**: `FileAccessTracker` records reads. Write/edit rejected
  if file not read or content changed externally since read.
- **Size gate**: `read_file` blocks reads exceeding `max_read_bytes` (computed
  from model context window or explicit config).
- **ContextAnchor**: `read_file` declares `anchor_key = "file:{path}"`.
  Subsequent reads replace the previous anchor.
- **LSP diagnostics on edit with rollback**: When `diagnostics_on_edit` is
  configured, `write_file`/`edit_file` follow a three-step protocol:
  1. Save original file content to memory before mutation
  2. Apply the write/edit
  3. Request LSP diagnostics within `diagnostics_timeout` seconds
  4. If new errors introduced AND `fail_on_errors` is true: restore original
     content and return error with diagnostic details
  This replicates the Python v1.7.0 behavior (`FilesystemConfig.fail_on_errors`,
  `diagnostics_timeout`). Reference: `src/entropic/mcp/servers/filesystem.py`
  `_check_diagnostics_after_edit()` method.
- **Path security**: Paths resolved relative to project root. Traversal outside
  root rejected unless `allow_outside_root` configured.
- **Brace expansion**: Glob patterns support `{a,b,c}` syntax.
- **Skip directories**: `.git`, `node_modules`, `__pycache__`, `.venv`, etc.

```cpp
/// @brief Filesystem MCP server with read-before-write enforcement.
/// @version 1.8.5
class FilesystemServer : public MCPServerBase {
public:
    FilesystemServer(const std::filesystem::path& root_dir,
                     const FilesystemConfig& config,
                     int model_context_bytes = 0);

    /// @brief read_file must always execute (updates FileAccessTracker).
    /// @version 1.8.5
    bool skip_duplicate_check(const std::string& tool_name) const override;
};
```

Individual tools are `ToolBase` subclasses registered in the constructor:
`ReadFileTool`, `WriteFileTool`, `EditFileTool`, `GlobTool`, `GrepTool`,
`ListDirectoryTool`. Each delegates to `FilesystemServer` methods.

`ReadFileTool::anchor_key()` returns `"file:" + args["path"]`.

### Phase 11: Engine Integration

Wire ToolExecutor into AgentEngine (v1.8.4). The engine loop's tool execution
branch becomes live code.

Changes to `AgentEngine::execute_iteration()`:
```
IF tool_calls:
    set_state(ctx, WAITING_TOOL)
    results = tool_executor.process_tool_calls(ctx, tool_calls)
    FOR each result:
        append to ctx.messages
        process directives if present
        IF stop_processing: break
    set_state(ctx, EXECUTING)  // back to next iteration
```

The facade (`librentropic.so`) wires ServerManager into ToolExecutor into
AgentEngine at engine creation time.

### i_mcp_server.h (established in architecture doc)

The plugin interface is already defined in `docs/architecture-cpp.md`. Each
plugin `.so` exposes:

```c
/// @brief Opaque handle to an MCP server instance.
/// @version 1.8.5
typedef struct entropic_mcp_server* entropic_mcp_server_t;

/// @brief Get the server name.
/// @version 1.8.5
const char* entropic_mcp_server_name(entropic_mcp_server_t server);

/// @brief List tools as JSON array string.
/// @version 1.8.5
char* entropic_mcp_server_list_tools(entropic_mcp_server_t server);

/// @brief Execute a tool. Returns ServerResponse JSON envelope.
/// @version 1.8.5
char* entropic_mcp_server_execute(
    entropic_mcp_server_t server,
    const char* tool_name,
    const char* args_json);

/// @brief Destroy a server instance.
/// @version 1.8.5
void entropic_mcp_server_destroy(entropic_mcp_server_t server);

/// @brief Free a string allocated by the server.
/// @version 1.8.5
void entropic_free(void* ptr);
```

Plugin exports:
```c
extern "C" ENTROPIC_EXPORT int entropic_plugin_api_version();
extern "C" ENTROPIC_EXPORT entropic_mcp_server_t entropic_create_server();
```

### Server-Specific Factory Parameters

Some servers need construction parameters (root_dir, config). The
`entropic_create_server()` signature is parameterless for uniformity.
Server-specific configuration is passed via a second init call:

```c
/// @brief Configure a server instance after creation.
/// @param server Server handle.
/// @param config_json JSON configuration string.
/// @return ENTROPIC_OK on success.
/// @version 1.8.5
entropic_error_t entropic_mcp_server_configure(
    entropic_mcp_server_t server,
    const char* config_json);
```

This keeps the plugin ABI stable while allowing per-server configuration.

1. **Tool-calling loop completes**: generate -> parse -> execute -> generate -> complete
2. **All 25 tool JSON definitions load and validate**: every file in `data/tools/` parses
3. **Permission deny blocks execution**: denied tool returns error, model receives feedback
4. **Permission allow skips prompting**: allowed tools auto-approve
5. **Duplicate detection works**: identical call returns cached result
6. **Circuit breaker fires at 3**: three consecutive duplicates inject "stuck" message
7. **Tier allowlist enforced**: hallucinated tool calls rejected with feedback
8. **Bash commands allowlist enforced**: disallowed prefixes rejected
9. **Filesystem read-before-write**: write without prior read returns error
10. **Filesystem size gate**: oversized files blocked with suggestion

12. **Edit rollback on diagnostic errors**: LSP errors trigger content revert

## Implementation Notes (from adversarial review)

1. **Invert headless auto-approve.** Step 3 in the approval flow: "No callback
   registered → DENY" (not approve). Consumers must explicitly set
   `auto_approve_tools=true` for unrestricted headless mode. This prevents
   accidental full-access in deployments that forget to wire the callback.

2. **Add `entropic_mcp_server_set_working_dir()` to `i_mcp_server.h`.** Base
   class default is no-op. Directory-aware servers (filesystem, bash, git)
   implement it. This enables v1.8.6's `ScopedWorktree` to swap directories
   across `.so` boundaries without breaking isolation. This is an additive
   change to the architecture doc's interface definition.

3. **ToolExecutor depends on `librentropic-types.so` only.** Not core internals.
   LoopContext fields that ToolExecutor reads (locked_tier, active_phase,
   effective_tool_calls, etc.) must be defined in types headers, not core
   headers. This preserves the dependency boundary.

4. **`bash_commands` is behavioral guidance, not a security sandbox.** Document
   explicitly: prefix matching reduces accidental model misuse but is
   fundamentally bypassable (`python` allows `python -c "os.system(...)"`).
   The PermissionManager (Layer 1 allow/deny) is the security boundary.

5. **Add `realpath()` symlink check to filesystem server.** After path
   resolution, call `realpath()` and verify the resolved path is still within
   `root_dir`. Reject if not. Prevents symlink-based path traversal (CWE-22).

6. **`entropic_plugin_api_version()` must bump when base class virtuals change.**
   A plugin built against an older vtable loaded into a newer engine will crash.
   Document that API version increments whenever `MCPServerBase` or `ToolBase`
   virtual method signatures change.

**After Phase 4** (ToolExecutor + PermissionManager). At this point, tool calls dispatch to the correct MCP server, the three-layer permission system (allow/deny patterns, tier allowlist, approval callback) enforces access control, and duplicate detection prevents repeated identical calls. The six built-in server implementations (filesystem, bash, git, diagnostics, web, entropic) can be added incrementally in later phases without changing the executor or permission infrastructure.
