## Scope

Final v1.8.x milestone. No new subsystems — this is a review, stabilization, and
verification pass on the public C API surface (`entropic.h`). After v1.8.9, a
consumer can write a program against `entropic.h`, compile and link it from an
external CMake project, and rely on documented thread safety guarantees, a stable
error handling contract, and semantic versioning on the API itself.

Every function accumulated across v1.8.0–v1.8.8 is audited for:

1. **Signature consistency** — naming conventions, parameter ordering, return types
2. **Doxygen completeness** — every function has `@brief`, `@param`, `@return`,
   `@version`, `@threadsafety`, and error documentation
3. **Thread safety classification** — each function labeled as thread-safe,
   requires serialization, or single-threaded-init-only
4. **Memory ownership clarity** — who allocates, who frees, lifetime guarantees
5. **Error contract completeness** — all error codes reachable, callback registration
   well-defined, per-handle error state documented

Additionally, two new function groups are introduced:

- **Identity management**: `entropic_load_identity()`, `entropic_get_identity()`
- **Hook registration**: `entropic_register_hook()` (stub — implementation in v1.9.1)

And the API versioning scheme is formalized:

- `entropic_api_version()` returns an integer incremented on any breaking change
- `entropic_version()` returns the full semver string

**What this version does NOT include:**
- New subsystem implementation (all done in v1.8.0–v1.8.8)
- Hook handler implementations (v1.9.1)
- Auto-generated Python wrapper (v1.9.0)
- Per-caller MCP authorization (v1.9.4)

## Implementation Plan

### Phase 1: Signature Audit

Review every function in `entropic.h` for consistency against these rules:

**Naming conventions:**
- All functions prefixed with `entropic_`
- Subsystem grouping: `entropic_{subsystem}_{action}` (e.g., `entropic_inference_load`)
- Handles are `entropic_{type}_t` typedefs to opaque struct pointers
- Enums are `entropic_{name}_t` with `ENTROPIC_` prefixed values

**Parameter ordering:**
- Handle always first
- Input parameters before output parameters
- Output parameters are pointer-to-pointer (`char**`) for allocated strings
- `user_data` always last in callback signatures

**Return types:**
- Functions that can fail return `entropic_error_t`
- Functions returning allocated strings return `char*` (caller frees via `entropic_free()`)
- Functions returning borrowed strings return `const char*` (owned by handle)
- Factory functions return the opaque handle (null on failure)

**Audit deliverable:** A diff correcting any inconsistencies found. If no
inconsistencies, the audit is a no-op (which is the expected outcome if prior
versions followed the architecture doc).

### Phase 3: Thread Safety Classification

Classify every function into one of three categories:

**Thread-safe (can be called from any thread at any time):**
- `entropic_version()`
- `entropic_api_version()`
- `entropic_interrupt(handle)` — designed for cross-thread cancellation
- `entropic_last_error(handle)` — per-handle, but read-only snapshot
- `entropic_detect_backend()`
- `entropic_free(ptr)` — stateless deallocation

**Serialized per-handle (one call at a time per handle, different handles are independent):**
- `entropic_run(handle, ...)`
- `entropic_run_streaming(handle, ...)`
- `entropic_configure(handle, ...)`
- `entropic_configure_from_file(handle, ...)`
- `entropic_register_mcp_server(handle, ...)`
- `entropic_load_identity(handle, ...)`
- `entropic_get_identity(handle, ...)`
- `entropic_register_hook(handle, ...)`
- `entropic_set_error_callback(handle, ...)`
- `entropic_storage_open(handle, ...)`
- `entropic_storage_close(handle, ...)`
- All `entropic_inference_*` functions (serialized per-backend)
- All `entropic_mcp_server_*` functions (serialized per-server)

**Single-threaded init/destroy (must not be called concurrently with anything):**
- `entropic_create()` — returns handle, no concurrency concern
- `entropic_destroy(handle)` — must not race with any other call on that handle
- `entropic_mcp_server_destroy(server)`

Each function's Doxygen block gets a `@threadsafety` tag documenting its category.

### Phase 4: Doxygen Completeness Pass

Every function in `entropic.h` must have:

```c
/**
 * @brief One-line description.
 *
 * Extended description if behavior is non-obvious.
 *
 * @param handle Engine handle (or other parameters).
 * @return ENTROPIC_OK on success. Possible errors:
 *         - ENTROPIC_ERROR_INVALID_HANDLE
 *         - ENTROPIC_ERROR_INVALID_CONFIG
 *
 * @threadsafety Serialized per-handle.
 * @version 1.8.0
 *
 * @par Memory ownership
 * Caller must free result_json with entropic_free().
 *
 * @par Example
 * @code
 * entropic_error_t err = entropic_configure(h, config_json);
 * if (err != ENTROPIC_OK) {
 *     fprintf(stderr, "Config error: %s\n", entropic_last_error(h));
 * }
 * @endcode
 */
```

Minimum required tags per function:
- `@brief` — always
- `@param` — for each parameter
- `@return` — with enumerated possible error codes
- `@version` — version when the function was introduced
- `@threadsafety` — from Phase 3 classification

Optional but encouraged:
- `@par Memory ownership` — for functions returning allocated memory
- `@par Example` — for non-obvious usage
- `@see` — cross-references to related functions

### Phase 7: API Versioning Scheme

Formalize the contract:

- `entropic_version()` returns the full semver string (e.g., `"1.8.9"`).
  Static lifetime — pointer valid for the lifetime of the process.
- `entropic_api_version()` returns an integer starting at `1`.
  Incremented when:
  - A function signature changes
  - A function is removed
  - An enum value is renumbered
  - Behavior of an existing function changes in a breaking way
- Adding new functions does NOT increment the API version (additive changes
  are non-breaking).
- The API version integer is defined in `entropic_config.h` as
  `ENTROPIC_API_VERSION` for compile-time checks.
- Plugin ABI version (`entropic_plugin_api_version()`) is SEPARATE from the
  C API version. Plugin ABI version tracks changes to the plugin interface
  contracts (`i_mcp_server.h`, `i_inference_backend.h`).
